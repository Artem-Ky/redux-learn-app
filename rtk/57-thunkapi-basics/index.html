Что тут происходит

Один thunk fetchProduct(id) использует всё сразу:

  • thunkAPI.getState() — если товар уже есть в cache и свежий, не идём в сеть;
  • thunkAPI.dispatch(...) — side-action analytics/productViewed шлёт в отдельный slice;
  • thunkAPI.extra — apiClient, подключённый через configureStore({middleware: gdm => gdm({thunk:{extraArgument: apiClient}})}).

getState() — cache check

const fetchProduct = createAsyncThunk( 'products/fetch', async (id, { getState, extra, dispatch }) => { const root = getState() as RootState const cached = root.products.byId[id] if (cached && Date.now() - cached.fetchedAt < 10_000) { return cached // ← не ходим в сеть } dispatch(analyticsEvent({ kind: 'product-view', id })) // side-action return await extra.apiClient.get(`/api/products/${id}`) }, )

Подводные камни

1. getState() возвращает АКТУАЛЬНОЕ состояние, не snapshot старта thunk'а. Между dispatch(pending) и resolve'ом промиса другой thunk/reducer мог всё изменить. Если нужен snapshot — сохраните в const before = getState() на входе.
2. extra типизируется ТОЛЬКО через ThunkApiConfig. Если не указать { extra: typeof apiClient } — получите extra: unknown. Есть ещё createAsyncThunk.withTypes<{ extra: ApiClient; state: RootState }>() для «curried»-формы.
3. dispatch(thunkB()) изнутри thunk'а — работает. Но: если внутри thunk'а await-ить другой thunk — получаете action-объект, а не payload. Для payload'а — await dispatch(thunkB()).unwrap().
4. extraArgument — один объект на весь store. Если нужны разные клиенты (dev/prod, mocked), заведите «container»: extraArgument: { api, history, logger }.
