Counter с состоянием в localStorage

Этот урок демонстрирует persistence через preloadedState. Counter сохраняется в localStorage. При перезагрузке страницы (F5) состояние восстанавливается через preloadedState. В DevTools первый @@INIT уже содержит non-zero value.

localStorage["rtk-lesson18-counter"]
(пусто)
value = 0
Попробуйте:
  1. Кликните +1 несколько раз → значение в localStorage обновится автоматически.
  2. Нажмите F5 → страница перезагрузится, value восстановится из localStorage.
  3. В DevTools (внизу) первый @@INIT покажет уже non-zero state.
  4. «Очистить localStorage» → перезагрузка → counter снова с нуля.
Теория — preloadedState

Из доки

«preloadedState: An optional initial state value to be passed to the Redux createStore function. This may be useful for resuming state from the server in universal apps, or to restore a previously serialized user session.»
— preloadedState option
«preloadedState: опциональное начальное значение state, передаваемое в Redux-функцию createStore. Может быть полезно для восстановления state с сервера в universal-приложениях или для восстановления ранее сериализованной пользовательской сессии.»

3 типичных сценария

  1. localStorage hydration — сохраняем state при изменениях, читаем при старте.
  2. SSR (Next.js, Remix) — сервер рендерит state, клиент гидрирует.
  3. Тесты — создаём store с уже задуманным state для конкретного теста.

Шаблон localStorage hydration

// 1) Загрузить state function loadState(): Partial<RootState> | undefined { try { const raw = localStorage.getItem('app-state') if (!raw) return undefined return JSON.parse(raw) } catch { return undefined } } // 2) Создать store const store = configureStore({ reducer, preloadedState: loadState() }) // 3) Подписаться на изменения и сохранять store.subscribe(() => { const state = store.getState() localStorage.setItem('app-state', JSON.stringify(state)) })

Подводный камень: только часть state

Часто вы хотите сохранять только часть state (например, auth + theme), но не cache RTK Query (он временный). Решение — селективно:

store.subscribe(throttle(() => { const { auth, theme } = store.getState() localStorage.setItem('app-state', JSON.stringify({ auth, theme })) }, 1000))

preloadedState и initialState slice

Если у вас и preloadedState: { counter: { value: 5 } }, и в slice initialState: { value: 0 } — RTK использует preloadedState. Это поведение Redux: при первом call'е reducer'а state === preloadedState[key], а не undefined, поэтому initialState проигнорируется.

Подводный камень: схема изменилась

Если вы выпустили новую версию приложения с новой структурой state, а у пользователя в localStorage старая схема — приложение упадёт. Решения:
  • Версионирование: { version: 2, state: {...} } + миграции;
  • redux-persist (поддерживает миграции);
  • Простая try/catch + localStorage.clear() при ошибке.

SSR-вариант (Next.js)

// _app.tsx (упрощённо) function App({ Component, pageProps }) { const store = useMemo(() => configureStore({ reducer, preloadedState: pageProps.initialReduxState // от getServerSideProps }), [pageProps.initialReduxState]) return <Provider store={store}><Component {...pageProps} /></Provider> }

redux-persist vs ручной hydration

Если нужны миграции схемы, blacklist/whitelist, debounce — берите redux-persist. Для простых кейсов (counter, theme, user-prefs) preloadedState + subscribe достаточно.
