Одна cache entry · pages[] · fetchNextPage / fetchPreviousPage

build.infiniteQuery — специальный endpoint с initialPageParam, getNextPageParam, getPreviousPageParam. Все страницы живут в одной cache entry как массив pages[].

Теория — 100. build.infiniteQuery

Цитата — Infinite Queries

«Infinite queries allow you to fetch data in pages and manage pagination state as part of the cached entry. You define initialPageParam, how to compute the next/previous page param from the latest response, and optionally a maximum number of pages to retain.»
— RTKQ · Infinite Queries

Полное определение endpoint'а

getFeed: build.infiniteQuery<Post[], void, number>({ // query принимает { queryArg, pageParam }: query: ({ pageParam }) => `feed?page=${pageParam}`, infiniteQueryOptions: { initialPageParam: 1, // computed на основе ПОСЛЕДНЕЙ загруженной страницы getNextPageParam: (lastPage, allPages, lastPageParam, allPageParams) => { // вернуть следующий pageParam, или undefined если страниц нет return lastPage.length > 0 ? lastPageParam + 1 : undefined }, // опционально — для "загрузить назад" (редко нужно) getPreviousPageParam: (firstPage, allPages, firstPageParam) => { return firstPageParam > 1 ? firstPageParam - 1 : undefined }, // опционально — сколько страниц держать в памяти maxPages: 5, }, }),

Что возвращает useInfiniteQuery

const { data, // { pages: Post[][], pageParams: number[] } currentData, // то же, но для текущего arg fetchNextPage, // () => void fetchPreviousPage, hasNextPage, // boolean hasPreviousPage, isFetchingNextPage, isFetchingPreviousPage, isLoading, // первая страница isFetching, // любая загрузка refetch, isUninitialized, } = useGetFeedInfiniteQuery()

Под капотом — как оно реализовано

build.infiniteQuery — это сахар над build.query с кастомными serializeQueryArgs (один ключ на все страницы), merge (добавляет страницу в массив), forceRefetch (по смене pageParam). До v2.x это паттерн собирали вручную (см. урок 83, пример 2).

Курсор-пагинация (вместо номеров страниц)

''}`, infiniteQueryOptions: { initialPageParam: null, getNextPageParam: (lastPage) => lastPage.nextCursor, }, }),

Подводные камни

1. data.pages — это массив массивов, не плоский.
Для рендера обычно data.pages.flat() или data.pages.flatMap(...). Не забывай key в React: {`page-${pageParams[i]}-item-${item.id}`} или проходи через allPosts.map с item.id.

2. getNextPageParam возвращает undefined → hasNextPage=false.
Это сигнал «конец данных». Не возвращай null (будет truthy для cursor-кейса, где null — валидный cursor) в неподходящем месте.

3. maxPages + fetchNext = циклическая память.
С maxPages: 5 старые страницы удаляются при подгрузке новых. Скроллишь назад — опять подгрузка. Без getPreviousPageParam это сломается.

4. refetch() на infiniteQuery — перезапрашивает ВСЕ страницы.
И именно через тот же pageParam массив, что был в cache. Это дорого — думай, нужен ли refetch или достаточно invalidate + fetchNextPage заново.

5. Инвалидация через теги — reset на page 1.
Когда mutation invalidates тег, entry пере-fetch'ится со стартовым pageParam, все загруженные страницы сбрасываются. UX: пользователь теряет скролл.

6. initialPageParam должен быть serializable.
Нельзя передать Symbol или Date — serializability middleware будет ругаться. Строки/числа/null — OK.
