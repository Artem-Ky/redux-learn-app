100 × dispatch — сколько уведомлений получит store.subscribe?

autoBatchEnhancer группирует «low-priority» экшены (с флагом meta[SHOULD_AUTOBATCH]: true) и уведомляет подписчиков один раз на пачку. В RTK 2.x этот enhancer включён по умолчанию — достаточно проставить флаг через prepareAutoBatched (или руками) и батчинг работает.

Обратите внимание на третью кнопку: после 50 «отложенных» экшенов мы диспатчим один обычный — enhancer синхронно flush'ит накопленное уведомление перед ним.

Теория — 71. autoBatchEnhancer: приоритизация уведомлений

Цитата из исходника

«Watches for any actions with a specific action.meta[SHOULD_AUTOBATCH]: true field… If some low-pri actions have been dispatched and a notification microtask is queued, then a normal priority action is dispatched, the enhancer will go ahead and notify all subscribers synchronously.»
— redux-toolkit.js.org/api/autoBatchEnhancer
«Следит за экшенами с meta[SHOULD_AUTOBATCH]: true. Если накоплена очередь low-priority экшенов и пришёл обычный экшен — enhancer синхронно уведомит подписчиков перед обработкой этого обычного экшена.»

В RTK 2.x enhancer включён по умолчанию

Достаточно просто configureStore({ reducer }) — getDefaultEnhancers() сам добавит autoBatchEnhancer({ type: 'raf' }). Руками его подключать не нужно; нужно только проставить флаг в тех экшенах, которые хочешь батчить.

Исходник — getDefaultEnhancers

// packages/toolkit/src/getDefaultEnhancers.ts export const buildGetDefaultEnhancers = <M extends Middlewares<any>>( middlewareEnhancer: StoreEnhancer, ): GetDefaultEnhancers<M> => function getDefaultEnhancers(options) { const { autoBatch = true } = options ?? {} // 👀 default: true let enhancerArray = new Tuple<StoreEnhancer[]>(middlewareEnhancer) if (autoBatch) { enhancerArray.push( autoBatchEnhancer( typeof autoBatch === 'object' ? autoBatch : undefined, // можно передать { type: 'tick' } через autoBatch: {...} ), ) } return enhancerArray as any }

Исходник — сердце autoBatchEnhancer.dispatch

// packages/toolkit/src/autoBatchEnhancer.ts — строки ~46-128 (сокращено) export const autoBatchEnhancer = (options: AutoBatchOptions = { type: 'raf' }): StoreEnhancer => (next) => (...args) => { const store = next(...args) let notifying = true // флаг: пропускаем оригинальные listener'ы? let shouldNotifyAtEndOfTick = false // накопилась ли очередь let notificationQueued = false // уже запланирована микро-задача const listeners = new Set<() => void>() const queueCallback = options.type === 'tick' ? queueMicrotask : options.type === 'raf' ? window.requestAnimationFrame : options.type === 'callback' ? options.queueNotification : createQueueWithTimer(options.timeout) const notifyListeners = () => { notificationQueued = false if (shouldNotifyAtEndOfTick) { shouldNotifyAtEndOfTick = false listeners.forEach((l) => l()) // ← ОДИН вызов на всю пачку } } return Object.assign({}, store, { subscribe(listener) { // обёртка: listener срабатывает ТОЛЬКО если notifying === true const wrappedListener = () => notifying && listener() const unsubscribe = store.subscribe(wrappedListener) listeners.add(listener) return () => { unsubscribe(); listeners.delete(listener) } }, dispatch(action) { try { // 1) нет флага SHOULD_AUTOBATCH — notifying=true → listener'ы сработают как обычно notifying = !action?.meta?.[SHOULD_AUTOBATCH] shouldNotifyAtEndOfTick = !notifying if (shouldNotifyAtEndOfTick) { if (!notificationQueued) { notificationQueued = true queueCallback(notifyListeners) // ставим задачу один раз на тик } } return store.dispatch(action) } finally { notifying = true // возвращаемся в нормальный режим } }, }) }

prepareAutoBatched — однострочник

// packages/toolkit/src/autoBatchEnhancer.ts, строки 5-10 export const prepareAutoBatched = <T>() => (payload: T): { payload: T; meta: unknown } => ({ payload, meta: { [SHOULD_AUTOBATCH]: true }, // SHOULD_AUTOBATCH === 'RTK_autoBatch' })

Использование: createSlice.reducers.tick = { reducer, prepare: prepareAutoBatched<void>() }. Или вручную: { type: 'feed/append', meta: { RTK_autoBatch: true }, payload }.

Как фокус с «синхронным flush» реализован

Когда приходит обычный экшен и в очереди уже ждёт notifyListeners:

  • внутри store.dispatch(action) оригинальный redux store пробегается по своим listener'ам — это обёртки;
  • перед этим мы уже выставили notifying = true — значит wrappers вызовут реальные listener'ы;
  • накопленное «low-priority» уведомление отдаётся синхронно, потому что мы всё равно уже в середине dispatch;
  • запланированная задача (raf/tick) когда-нибудь выполнит notifyListeners, но shouldNotifyAtEndOfTick уже false — будет no-op.

4 типа очереди — когда что выбирать

type Как реализован Когда выбрать
raf requestAnimationFrame UI-обновления (~60fps). Default для RTK. Синхронизирует с рендером браузера.
tick queueMicrotask Хочешь flush как можно скорее — в конце текущего цикла микро-задач (до paint). Нативный Promise.resolve().then() работает так же.
timer setTimeout(cb, timeout) Фон/логи/аналитика — можно отложить в macrotask. Задержка настраивается.
callback твоя функция Custom scheduler (React's unstable_scheduleCallback, idle callback, etc).
Подводные камни autoBatch:
  • Flush на первый «обычный» экшен может быть неожиданным. Если у тебя есть дорогой подписчик (ре-рендер большого дерева), и ты диспатчишь 1000 батчед + затем 1 обычный — этот обычный заставит ре-рендер синхронно со всеми 1000 изменениями. Это нормально (цель enhancer'а), но если ожидаешь «один setTimeout(0) → один render» — учти, что любой не-батчед action прерывает паузу.
  • raf не тикает в фоновых вкладках. Если вкладка неактивна — requestAnimationFrame останавливается. Batched-уведомления копятся, пока пользователь не вернётся. Для приложений, которые должны работать в фоне (например, data-feed), используй { type: 'timer', timeout: 16 } — setTimeout работает всегда.
  • queueMicrotask (tick) может отложиться, если стек глубокий. Микро-задачи выполняются в конце текущей JS-задачи. Если dispatch лежит внутри await → await → dispatch chain, микро-задача flush'нется между ними. Для визуальных группировок лучше raf.
  • Tuple порядка enhancer'ов важен. autoBatchEnhancer должен быть после middlewareEnhancer — именно поэтому в getDefaultEnhancers он push'ится в конец. Иначе middleware видел бы обёртки.

Когда НЕ использовать autoBatch

  • Один dispatch — одна анимация. Если у тебя по клику диспатчится один экшен и UI реагирует на него — батчить нечего.
  • Тесты с store.subscribe expectations. Если тест ждёт expect(listener).toHaveBeenCalledTimes(100), батчинг сломает это. Либо снимай флаг, либо configureStore({ enhancers: (g) => g({ autoBatch: false }) }).
  • react-redux уже имеет свой батчинг. useSelector с shallowEqual + React 18 automatic batching часто закрывают задачу. autoBatchEnhancer — для прямых подписчиков (через store.subscribe) и для очень частых экшенов (websocket feed, scroll events).
