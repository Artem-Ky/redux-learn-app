withLazyLoadedSlices<T> и WithSlice

Lazy injection (урок 35) даёт runtime-гибкость, но TS не знает о lazy slice'ах. .withLazyLoadedSlices<LazyState>() объявляет, какие ветки могут появиться. RTK делает их опциональными в RootState.

import { combineSlices, type WithSlice } from '@reduxjs/toolkit' // Способ 1: вручную через type type LazyLoadedSlices = { chart?: ChartState cart?: CartState search?: SearchState } const rootReducer = combineSlices(coreSlice, userSlice) .withLazyLoadedSlices<LazyLoadedSlices>() type RootState = ReturnType<typeof rootReducer> // => { core: CoreState; user: UserState; chart?: ChartState; cart?: CartState; search?: SearchState } // Способ 2: через WithSlice helper для каждого slice type LazyLoadedSlices = WithSlice<typeof chartSlice> & WithSlice<typeof cartSlice> // => { chart?: ChartState } & { cart?: CartState }

ReturnType<typeof rootReducer> в TS (демо)

Текущий runtime state

Теория — WithSlice и withLazyLoadedSlices

Из доки

«By default, the type returned from combineSlices assumes that all slices passed in are already required. To declare types for slices that may be lazy-loaded later, use the withLazyLoadedSlices<T>() method or the WithSlice helper type.»
— combineSlices — TypeScript
«По умолчанию тип, возвращённый из combineSlices, предполагает, что все переданные slice'ы уже обязательны. Чтобы объявить типы для slice'ов, которые могут быть lazy-загружены, используйте метод withLazyLoadedSlices<T>() или helper-тип WithSlice.»

WithSlice — устройство

// Сигнатура: type WithSlice<S extends Slice | { reducerPath: string, reducer: Reducer }> = { [K in S['reducerPath']]?: ReturnType<S['reducer']> } // Пример: type ChartLazy = WithSlice<typeof chartSlice> // = { chart?: ChartState } // Композиция нескольких slice'ов через intersection: type LazyState = WithSlice<typeof chartSlice> & WithSlice<typeof cartSlice> // = { chart?: ChartState; cart?: CartState }

Зачем именно опциональность

State injected slice'а появляется только после injectInto. До этого state.chart === undefined. TS должен это знать, иначе state.chart.points в селекторе сломается без проверки.

// Плохой селектор (если NOT lazy): const selectPoints = (state: RootState) => state.chart.points // Если chart не injected — runtime crash // Хороший селектор (с lazy типизацией): const selectPoints = (state: RootState) => state.chart?.points ?? [] // TS заставит написать оптимистичный fallback

useSelector + injected slice

// В компоненте: function ChartView() { useEffect(() => { chartSlice.injectInto(rootReducer) }, []) const points = useAppSelector( (state) => state.chart?.points ?? [] // fallback пока не injected ) return <Chart data={points} /> }

Альтернатива: обязательные slice'ы

Если уверены, что slice будет injected до первого рендера компонента (например, в useEffect родителя), можно не объявлять как опциональный — но это опасно. Лучше всегда оставлять опциональными, а в селекторе делать non-null assertion после монтирования.

// "уверены, что injected" — non-null assertion в одном месте const selectChart = (state: RootState) => state.chart! // ! // Лучше через guard: function selectChart(state: RootState) { if (!state.chart) throw new Error('chartSlice not injected') return state.chart }

Подводный камень: rootReducer.inject() и типизация

Если используете низкоуровневый rootReducer.inject({ reducerPath, reducer }) (без slice), TS не сможет вывести тип. Лучше всегда работать через slice.injectInto(rootReducer).

Пример с lazy + код-сплит

// store/index.ts export const rootReducer = combineSlices(coreSlice) .withLazyLoadedSlices<LazyState>() export type RootState = ReturnType<typeof rootReducer> // features/chart/index.tsx import { rootReducer } from '@/store' export const chartSlice = createSlice({ name: 'chart', ... }).injectInto(rootReducer) export default function ChartView() { ... } // App.tsx const ChartView = React.lazy(() => import('@/features/chart'))
