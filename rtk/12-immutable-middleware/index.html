Что middleware ловит, а что — пропускает

Это middleware делает shallow + deep сравнение state до и после dispatch. Если ссылка на nested объект осталась той же, но содержимое изменилось — это мутация. Throw. Реальные сценарии ниже:

1. Прямая мутация в reducer (без Immer)

const reducer = (state, action) => {
  state.value += 1   // ← мутация!
  return state
}

2. Push в массив без копии

const reducer = (state, action) => {
  state.list.push(action.payload)  // ← мутация массива
  return state
}

3. Мутация state ВНЕ reducer

// прямой доступ к состоянию
store.getState().value = 999  // ← никогда

4. Корректно: возврат нового объекта

const reducer = (state, action) => ({
  ...state,
  value: state.value + 1
})

Теория — как работает immutable check

Из доки

«Redux expects that all state updates are done immutably. The Immutable State Invariant Middleware deeply compares state values for mutations. It can detect mutations in reducers during a dispatch, and also mutations that occur between dispatches (such as in a component or a selector).»
— immutabilityMiddleware API
«Redux ожидает, что все обновления state делаются иммутабельно. Immutable State Invariant Middleware выполняет глубокое сравнение значений state на мутации. Он может обнаружить мутации в reducer'ах во время dispatch, а также мутации, происходящие между dispatch'ами (например, в компоненте или селекторе).»

Алгоритм (упрощённо)

  1. Перед каждым dispatch middleware делает trackedState = trackProperties(state) — рекурсивно строит дерево с метаинформацией.
  2. После выполнения reducer'а делает detectMutations(trackedState, newState).
  3. Если найдена ссылочно-неизменённая нода с другим содержимым — throw Invariant failed: A state mutation was detected at "path".
  4. Также делает второй проход между dispatch'ами — ловит мутации из компонентов и селекторов.

Опции

interface ImmutableStateInvariantMiddlewareOptions { isImmutable?: (value) => boolean // custom isImmutable check ignoredPaths?: (string | RegExp)[] // игнорировать эти пути warnAfter?: number // warn если check > N мс (default 32) } // Пример использования configureStore({ reducer, middleware: (gdm) => gdm({ immutableCheck: { ignoredPaths: ['someBigBlob', /^cache\./], warnAfter: 100, } }) })

Почему Immer не вызывает ошибок

createSlice оборачивает reducer в produce() (immer). Внутри produce вы пишете state.value++, но это не мутация исходного state — это мутация Proxy-draft. immer возвращает новый объект только для изменённых веток (структурный sharing). Поэтому immutable middleware не видит проблемы.

// Внутри createSlice происходит: const wrappedReducer = (state, action) => { return produce(state, draft => { yourReducer(draft, action) // здесь мутируйте draft сколько угодно }) }

Производительность

Immutable check рекурсивно обходит весь state. Для большого state (1000+ entities) это может быть медленно. Используйте ignoredPaths для тяжёлых веток (например, нормализованные данные RTK Query) или отключите middleware полностью с предупреждением.

Источник

immutableStateInvariantMiddleware.ts (~150 строк, реализация detectMutations).

Подводный камень: NODE_ENV

Этот middleware отключён в production-сборке. Если у вас баг с мутацией, который проявляется только в production — он мог скрываться, потому что middleware не предупредил. Решение: запускайте production-сборку локально с включённым immutable check (передайте immutableCheck: true явно, не полагаясь на дефолт).
