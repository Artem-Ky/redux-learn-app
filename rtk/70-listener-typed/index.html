startAppListening + addAppListener — один раз задали типы, больше не мучаемся с as RootState

В уроках 64–69 в каждом effect'е писали api.getState() as RootState. В «боевом» проекте такое неудобно. Решение: создать один typed startListening и использовать его везде. thunk'и тоже можно диспатчить изнутри listener'а — AppDispatch их принимает без каста.

app/store.ts
инстанс listenerMiddleware + store
export const listener = createListenerMiddleware() export const store = configureStore({ reducer: rootReducer, middleware: (g) => g() .prepend(listener.middleware), }) export type RootState = ReturnType< typeof store.getState> export type AppDispatch = typeof store.dispatch
app/listener.ts
typed helpers (отдельно, чтобы не было circular)
import { addListener } from '@reduxjs/toolkit' import type { RootState, AppDispatch } from './store' export const startAppListening = listener.startListening .withTypes<RootState, AppDispatch>() export const addAppListener = addListener .withTypes<RootState, AppDispatch>()
features/users/listeners.ts
конкретные listener'ы — используют startAppListening
import { startAppListening } from '../../app/listener' import { userLoggedIn, fetchProfile } from './usersSlice' startAppListening({ actionCreator: userLoggedIn, effect: async (action, api) => { // api.getState() → RootState // api.dispatch(thunk()) — без каста await api.dispatch( fetchProfile(action.payload.id)) }, })

Официальный рецепт

«For use with configureStore, we can export pre-typed versions of startListening and addListener to replace RootState and AppDispatch types in every listener's API.»

«Для configureStore экспортируйте "pre-typed" версии startListening и addListener — чтобы не писать в каждом listener'е RootState и AppDispatch руками.»

Исходник — TypedStartListening из types.ts

// packages/toolkit/src/listenerMiddleware/types.ts, строки 650–711 export type TypedStartListening< StateType, DispatchType extends Dispatch = ThunkDispatch<StateType, unknown, UnknownAction>, ExtraArgument = unknown, > = AddListenerOverloads< UnsubscribeListener, StateType, DispatchType, ExtraArgument > & { /** Pre-typed startListening с RootState/AppDispatch/Extra */ withTypes: < OverrideStateType extends StateType, OverrideDispatchType extends Dispatch = ThunkDispatch< OverrideStateType, unknown, UnknownAction >, OverrideExtraArgument = unknown, >() => TypedStartListening< OverrideStateType, OverrideDispatchType, OverrideExtraArgument > }

github.com/reduxjs/redux-toolkit/blob/master/packages/toolkit/src/listenerMiddleware/types.ts#L650-L711

Что даёт withTypes<S, D>()

// БЕЗ withTypes — каст в каждом effect'е listener.startListening({ actionCreator: loginSuccess, effect: (a, api) => { const s = api.getState() as RootState // 🙁 каст api.dispatch(fetchProfile(s.auth.user!.id) as any) // 🙁 thunk не типизирован }, }) // С withTypes — всё сразу правильно startAppListening({ actionCreator: loginSuccess, effect: (a, api) => { const s = api.getState() // 😊 RootState api.dispatch(fetchProfile(s.auth.user!.id)) // 😊 AppDispatch знает thunk'и }, })

Runtime-добавление listener'а через addAppListener

// Внутри хука/компонента — без прямого импорта listenerMiddleware function useAnalyticsListener() { const dispatch = useAppDispatch() useEffect(() => { const unsubscribe = dispatch(addAppListener({ matcher: isAnyOf(userLoggedIn, userLoggedOut), effect: (action, api) => { api.getState() // RootState track(action.type) }, })) return unsubscribe as () => void }, [dispatch]) }
Два частых подводных камня типизации:
  • Circular imports между store.ts и listener.ts. Если listener.ts импортирует RootState из store.ts, а сам инстанс listenerMiddleware живёт в store.ts — получаете цикл. Разбивайте на три файла: (1) создание listenerMiddleware в app/listener.ts, (2) store.ts импортирует из (1), (3) features/*/listeners.ts берут startAppListening из (1) + RootState из (2) только как type (import type).
  • addListener.withTypes тоже нужен. Экшен addListener используется для runtime-подписки через dispatch(addAppListener(...)). Без .withTypes() тип payload остаётся unknown — получаете те же касты, что и раньше, просто в другом месте. addListener.withTypes<RootState, AppDispatch>() — обязательный парный вызов.

ThunkDispatch внутри AppDispatch — thunk'и без каста

// AppDispatch автоматически содержит ThunkDispatch // (configureStore добавляет thunk в default middleware) type AppDispatch = typeof store.dispatch // ≈ ThunkDispatch<RootState, unknown, UnknownAction> // ⇒ thunk'и принимаются: api.dispatch(fetchProfile(42)) // Promise<fetchProfile.fulfilled | rejected> const result = await api.dispatch(fetchProfile(42)).unwrap()
