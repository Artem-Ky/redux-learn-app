Теория — 61. requestId и защита от race condition

Цитата

«A unique ID generated for this particular dispatch. This allows you to correlate the life-cycle actions that belong to the same thunk dispatch. […] You can use this in your reducer to ensure you handle only the latest request.»
— createAsyncThunk — requestId
«Уникальный ID, сгенерированный для данного конкретного dispatch'а. Позволяет связать life-cycle actions, принадлежащие одному запуску thunk'а. В reducer можно использовать для гарантии обработки только самого свежего запроса.»

Паттерн last-wins

interface SearchState { currentRequestId: string | null results: Post[] } extraReducers: (b) => { b.addCase(search.pending, (s, a) => { s.currentRequestId = a.meta.requestId // ← записали свежий }) b.addCase(search.fulfilled, (s, a) => { if (a.meta.requestId !== s.currentRequestId) { return // ← это УСТАРЕВШИЙ ответ } s.results = a.payload }) b.addCase(search.rejected, (s, a) => { if (a.meta.requestId === s.currentRequestId) { s.currentRequestId = null } }) }

Альтернатива: abort() предыдущего

let lastPromise: ReturnType<typeof search> | null = null function onInput(query: string) { lastPromise?.abort() // cancel предыдущий lastPromise = store.dispatch(search(query)) }

abort() экономит трафик (сеть тоже прерывается, если вы пробросили signal), requestId — дешевле, но сеть продолжает работать впустую.

Подводные камни

1. nanoid — 21 символ, НЕ стабилен между сессиями. Если вы пишете тесты на expect(action.meta.requestId).toBe('abc123') — получите flaky. Для тестов передавайте idGenerator: () => 'fixed-id' в опции.
2. Одинаковый arg → РАЗНЫЕ requestId. createAsyncThunk не дедуплицирует по аргументу. Два dispatch(fetchUser(1)) подряд создадут два независимых запроса. Если нужна дедупликация — condition.
3. requestId в nanoid — 21 символ (в некоторых статьях пишут 7 — это старые версии). Убедитесь, смотря в исходник выше: nanoid(size = 21).
4. requestId одинаков для pending / fulfilled / rejected одного dispatch'а. Но разный — для разных dispatch'ей, даже с одним arg. Это гарантия RTK.
5. Правильная реализация — обновлять currentRequestId в pending, проверять в fulfilled и rejected. Иногда забывают про rejected — и «зависший» error от старого запроса перебивает успех свежего.
