Теория — 86. Mutations

Цитата — Mutations docs

«Mutations are used to send data updates to the server and apply the changes to the local cache. Mutations can also invalidate cached data and force re-fetches. Unlike query hooks, mutation hooks return a tuple: the first item is the "trigger" function, and the second item contains an object with status flags, the requestId, and data from the latest invocation.»
— RTKQ · Mutations
«Mutations — это POST/PUT/DELETE (любой side-effect на сервере). Hook возвращает [trigger, state]. trigger(arg) стартует запрос, state — это {isLoading, isSuccess, isError, data, error, originalArgs, reset, requestId}.»

Исходник — buildMutationHooks.ts (концептуально)

// @reduxjs/toolkit/src/query/react/buildHooks.ts — упрощённо const useMutation: UseMutation<...> = ({ fixedCacheKey } = {}) => { const [promise, setPromise] = useState<MutationActionCreatorResult | undefined>() const triggerMutation = useCallback((arg: any) => { const promise = dispatch(initiate(arg, { fixedCacheKey })) setPromise(promise) return promise // 👀 Promise + .unwrap() + .reset() }, [dispatch, initiate, fixedCacheKey]) const reset = useCallback(() => { if (promise) promise.reset() setPromise(undefined) // освобождает subscription }, [promise]) const { endpointName, data, status, isLoading, isSuccess, isError, error } = useSelector(mutationStateSelector, shallowEqual) return [triggerMutation, { ...flags, requestId, originalArgs, reset }] }

trigger() возвращает Promise с .unwrap() и .reset()

const [addPost] = useAddPostMutation() async function onSubmit() { try { const result = await addPost({ title, body }).unwrap() // result — это типизированный ResultType (не весь action) showToast(`Created #${result.id}`) } catch (err) { // err — это action.error или action.payload (если rejectWithValue) showErrorToast(err) } } // Без unwrap: await addPost(...) — резолвится всегда, с { data? | error? }

fixedCacheKey — несколько компонентов разделяют статус

// ComponentA: const [trigger, state] = useAddPostMutation({ fixedCacheKey: 'shared-add-post' }) // ComponentB (где-то в другом месте дерева): const [, stateB] = useAddPostMutation({ fixedCacheKey: 'shared-add-post' }) // stateB увидит ТОТ ЖЕ isLoading / data что state в A, хотя trigger был там. // Без fixedCacheKey каждый вызов — отдельный requestId → локальный state.

Подводные камни

1. По умолчанию mutation state унмаунтится вместе с компонентом.
Когда компонент размонтируется, useMutation делает unsubscribe и state.api.mutations[requestId] очищается. Если хочешь, чтобы статус пережил unmount (например, оптимистичная мутация и сразу навигация) — передай fixedCacheKey.

2. trigger() возвращает НОВЫЙ promise каждый вызов.
Если нажать 3 раза подряд — будет 3 отдельных requestId, и только последний попадёт в hook state. Предыдущие промисы остаются валидными для .unwrap(), но hook state перезатирается. Чтобы отменять предыдущий → сам вызывай promise.abort().

3. .unwrap() бросает, .then/await без unwrap — нет.
await addPost(arg) всегда резолвится, с {data} | {error} формой. await addPost(arg).unwrap() — бросает при rejected. Выбор зависит от стиля обработки.

4. reset() очищает hook-level state, но НЕ server-side.
Это сбрасывает isLoading/isSuccess/data обратно в isUninitialized. Запрос на сервер УЖЕ прошёл (если он стартовал) — откат надо делать отдельным endpoint'ом или через optimistic updates (урок 94).

5. invalidatesTags — триггер refetch, НЕ авто-update cache.
После mutation RTKQ помечает соответствующие cache entries «stale» и вызывает refetch на активных подписках. Если на cacheKey никто не подписан — refetch не произойдёт, но при следующем mount он произойдёт автоматически. См. урок 87.

6. Mutation hooks — БЕЗ useFetchingRef.
У queries есть isFetching (перезапрос для того же cacheKey), у mutations — нет: каждое trigger создаёт отдельный requestId. Если нужно «идёт ли сейчас запрос» → смотри isLoading либо считай активные вручную.
