Probe: snapshot → мутация → snapshot — в ОДНОМ reducer'е

Один reducer runProbe делает три действия подряд в одной Immer-транзакции: (1) вызывает selectAvgPriceUnsafe / selectAvgPriceSafe и сохраняет результаты; (2) мутирует каждую цену +100 ₽; (3) снова вызывает оба селектора. Теория — 42. createDraftSafeSelector и Immer drafts

Цитата из доки RTK

«If an immer-drafted object is passed into the resulting selector's first argument, the selector will act on the current draft value, instead of returning a cached value that might be possibly outdated if the draft has been modified since.»
— RTK docs — createDraftSafeSelector
«Если в селектор передан объект-draft от Immer, селектор будет работать с текущим значением draft'а, а не возвращать закешированный результат, который мог устареть, если draft уже был изменён.»

Сценарий, в котором обычный createSelector ломается

Важная деталь: баг не в «повторных dispatch'ах». Между dispatch'ами Immer создаёт новый draft → ссылки другие → cache miss → пересчёт корректный. Баг возникает, когда селектор вызывается несколько раз внутри одного reducer'а, а между вызовами происходит мутация.

const selectAvgPrice = createSelector( [(s) => s.items], (items) => items.reduce((acc, b) => acc + b.price, 0) / items.length, ) reducers: { runProbe: (state) => { const before = selectAvgPrice(state) // miss → кладём в кеш [itemsProxy → X] state.items.forEach(b => { b.price += 100 }) // 👀 items Proxy — тот же const after = selectAvgPrice(state) // HIT → вернёт X (STALE) } }

Почему cache hit на том же Proxy

  1. Input selector (s) => s.items возвращает один и тот же Proxy для state.items в пределах одной Immer-транзакции. Мутация элементов (b.price += 100) не создаёт новый массив — только помечает proxy как «изменённый».
  2. lruMemoize(1) в обычном createSelector сравнивает аргументы по ===. Второй вызов → prevArgs[0] === curArgs[0] → HIT.
  3. Возвращается закешированный результат, вычисленный до мутации. То самое stale.

Визуально: во что превращается вход у двух селекторов

Почему дефолтный memoize здесь — weakMapMemoize, а не lruMemoize

Обратите внимание на последнюю строку исходника:

createDraftSafeSelectorCreator(weakMapMemoize) // vs createSelector = createSelectorCreator(lruMemoize) // дефолт у обычного

weakMapMemoize хранит дерево WeakMap по идентичности аргументов. Старые snapshot'ы живут, пока их кто-то держит (например, time-travel в DevTools или компонент, который ещё не размонтировался). Размер «кеша» не фиксирован 1 — удобно, когда селектор применяют к разным snapshot'ам параллельно.

Схема: как работает unwrap

Подводные камни

1. Селектор в reducer'е — это code smell. Reducer — чистая функция, делает минимум. Производные значения (avgPrice, sortedBy, activeCount) — обычно задача селектора компонента, а не reducer'а. Почему: кладя пересчитанное поле в state, вы дублируете источник истины и рискуете рассинхроном. Исключение: если результат нужен другому reducer'у или в extraReducers.
2. current() — O(structure), каждый вызов. current(draft) рекурсивно обходит весь subtree и делает plain copy. Для slice с 10k todo это заметно, особенно если селектор вызывается в цикле. Почему: Immer хранит только diffs (structural sharing), и для plain snapshot'а их надо «склеить» с оригиналом. Сузьте scope: current(state.items) вместо current(state).
3. isDraft проверяет только первый аргумент. Wrapper делает isDraft(value) ? current(value) : value только для args[0]. Если селектор принимает несколько state-ссылок (selectJoin(stateA, stateB)) — второй не распакуется. Почему: RTK предполагает, что в Redux первый аргумент — всегда root/slice state, а остальные — параметры (id, фильтр и т. п.).
4. Нет мемоизации «между» вызовами в reducer'е. Каждый вызов createDraftSafeSelector на draft'е делает новый current() → новая ссылка → cache miss → всегда пересчёт. Почему: плата за свежесть. Если селектор дорогой и вызывается часто в одном reducer'е — сохраните результат в локальную переменную вручную, мемоизация тут не помогает.
5. Один селектор — одна память кеша. Если снаружи (в компоненте) вы зовёте selectAvgPrice(plainState), а внутри reducer'а selectAvgPriceSafe(state) — это два разных селектора с разными кешами. Они не шарят результаты. Правильно: одна createDraftSafeSelector-инстанция на оба места.

Правильная стратегия

// selectors.ts import { createDraftSafeSelector } from '@reduxjs/toolkit' export const selectAvgPrice = createDraftSafeSelector( [(s: BooksState) => s.items], (items) => items.reduce((a, b) => a + b.price, 0) / (items.length || 1), ) // booksSlice.ts — безопасно вызывать и в reducer'е, и в компоненте reducers: { runProbe: (state) => { const before = selectAvgPrice(state) // ✅ current() распакует draft state.items.forEach(b => b.price += 100) const after = selectAvgPrice(state) // ✅ новый snapshot → пересчёт } }
