Порядок 5 шагов — почему именно так

  1. createAction — самый маленький, безопасный refactor: заменяет 1 константу + 1 функцию на 1 вызов. Reducer продолжает работать, потому что createAction('x')().type === 'x'.
  2. createReducer — меняет switch на builder. Immer начинает работать — можно писать state.value++.
  3. createSlice — объединяет actions + reducer. После этого action creators из шага 1 удаляются — slice их генерирует.
  4. configureStore — меняет createStore + applyMiddleware + composeWithDevTools на одну строку.
  5. createAsyncThunk — трогает каждый thunk отдельно; самый объёмный шаг, но самый ценный.

Что важно помнить при миграции большого проекта

  • Мигрируйте по одному slice за раз. Старый switch-reducer и новый slice.reducer прекрасно сосуществуют в combineReducers.
  • Типы action creators совместимы. createAction('x') возвращает функцию с .type === 'x' — все места, где раньше был import { INCREMENT }, меняются на increment.type.
  • extraReducers — builder callback обязателен для TS. Не используйте объектную форму, она deprecated в TS-проектах.
  • Удалите старые action types после createSlice. Легко забыть — и в проекте останутся мёртвые константы.

Что НЕ меняется

Концепция Redux (store, action, reducer, dispatch, subscribe) остаётся той же. React-биндинги (Provider, useSelector, useDispatch) — те же. Форма action'а { type, payload } — та же. Миграция — это замена инструмента, а не архитектуры.

Подводные камни

Не делайте миграцию в одном PR. 5 шагов выше — это 5 PR'ов. Каждый маленький, каждый безопасный. Одновременная замена action creators + reducer + store ломает code review и увеличивает вероятность регрессий.
extraReducers без builder callback — уже deprecated. extraReducers: { [type]: reducer } теряет types и убирает addMatcher. Пишите extraReducers: (builder) => builder.addCase(...).
Immer не всегда оптимален. В горячих путях (много raw-объектов, primitives) ручной spread иногда быстрее Immer-proxy. В 99% случаев — не важно, но знать надо (27 — immer pitfalls).
