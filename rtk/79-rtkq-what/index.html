Теория — 79. RTK Query: мотивация

Цитата — RTKQ Overview

«Web applications normally need to fetch data from a server in order to display it. Data fetching and caching is really a different set of concerns than state management. While you can use a state management library like Redux to cache data, the use cases are different enough that it's worth using tools that are purpose-built for the data fetching use case.»
— RTK Query Overview — Motivation
«Fetching и кэширование данных — другой класс задач, чем state management. Даже если ты можешь засунуть кэш в Redux, специализированный инструмент удобнее.»

RTKQ ≠ Redux. Разграничение зон

Что храним Redux / RTK slice RTK Query
UI-state: открытые модалки, фильтры, форма ✓ да нет
Auth-токен, role, текущий user ✓ да (client state) нет
Серверные данные: посты, юзеры, товары не надо ✓ да (server cache)
Loading / error / last-fetched флаги надоест писать руками ✓ авто
Dedup, TTL, refetch-on-focus, polling писать руками = не тривиально ✓ встроено

Подводные камни

1. RTK Query — НЕ замена Redux.
Если у тебя нет сервера — RTKQ не нужен. UI-state (модалки, фильтры, тема, auth-state) всё равно живёт в обычных slices. Правильная ментальная модель: RTKQ — это «клиентская обёртка над твоим REST/GraphQL API», а Redux — это «все остальные данные приложения».

2. Не пиши свой кэш поверх createAsyncThunk.
Dedup (два компонента подписались на одни данные → один запрос), TTL (через сколько выкидывать из кэша после отписки последнего подписчика), refetch-on-focus, refetch-on-reconnect, polling — всё это нетривиальные механики. В RTKQ они работают «из коробки» в 4 строках конфигурации. Если пишешь сам — готовься к багам race-condition'ов.

3. Обычно один createApi на весь проект.
Если у тебя два разных API (основной REST + отдельный сервис аналитики), их можно объединить в одном createApi через разные baseQuery внутри endpoint'ов. Два createApi — это две независимые tagTypes-вселенные: инвалидация из одного не увидит другое. См. урок 99 (injectEndpoints) для масштабирования.

4. RTKQ требует подключения api.middleware.
Без него hooks отрендерят isLoading=true навсегда — middleware отвечает за выполнение baseQuery. Типичный баг новичка; пример в уроке 81.

Bundle size — цитата из доки

«RTK Query adds a fixed one-time amount to your app's bundle size. Since RTK Query builds on top of Redux Toolkit and React-Redux, the added size varies depending on whether you are already using those. The estimated sizes are ~17 kB for RTK Query and ~19 kB for RTK Query + React.»
— RTK Query — Bundle Size
«~17 кБ ядро + ~19 кБ с React-биндингами. Фиксированный оверхед — дальше не растёт от количества endpoint'ов, потому что все они описываются через один общий builder.»
