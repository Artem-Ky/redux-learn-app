Discriminated union для error

import type { FetchBaseQueryError, SerializedError } from '@reduxjs/toolkit/query' function isFetchBaseQueryError( e: FetchBaseQueryError | SerializedError | undefined, ): e is FetchBaseQueryError { return e != null && 'status' in e } const { error } = api.useGetPostsQuery() if (isFetchBaseQueryError(error)) { error.status // ← ts знает: number | 'FETCH_ERROR' | ... error.data // ← unknown, нужно cast или zod-парсинг } else if (error) { error.message // SerializedError — string | undefined }

Кастомный baseQuery + typed error

type MyError = { code: string; message: string; status?: number } const myBaseQuery: BaseQueryFn<string, unknown, MyError> = async (url) => { try { const r = await fetch(url) if (!r.ok) return { error: { code: 'HTTP', message: r.statusText, status: r.status } } return { data: await r.json() } } catch (e) { return { error: { code: 'NETWORK', message: String(e) } } } } const api = createApi({ baseQuery: myBaseQuery, endpoints: ... }) // теперь error.code / .message — типизированы везде, никаких casts.

Подводные камни типизации

1. build.query<TResult, TArg> — TResult ПОСЛЕ transformResponse.
Если query возвращает {data: Post[]}, а transformResponse → Post[] — писать нужно build.query<Post[], void>, не {data: Post[]}.

2. TagTypes без as const = string.
Массив ['Post', 'User'] имеет тип string[]. Лучше ['Post', 'User'] as const — тогда TagTypes = 'Post' | 'User' и IDE подскажет в providesTags.

3. type в providesTag без as const = string.
{ type: 'Post', id: 5 } → TS выведет {type: string, id: number}. RTKQ скажет «tag 'Post' not in tagTypes» (потому что string ≠ 'Post'). Пиши { type: 'Post' as const, id } или используй satisfies.

4. BaseQueryFn generic порядок — Args, Result, Error, ExtraOptions, Meta.
Забудешь Error → будет unknown, везде cast. Пиши все 3 основных при custom baseQuery: BaseQueryFn<Args, Result, MyError>.

5. skipToken требует специальной типизации.
Если arg нельзя сделать optional (strict type) — useGetXQuery(id ?? skipToken) работает, и TS не ругается. Это правильнее чем useGetXQuery(id!, {skip: !id}).

6. Re-export хуков vs использование через api.X.
export const { useGetPostsQuery } = api теряет link на api в runtime — OK для TS, но в DevTools сложнее дебажить. Альтернатива: api.useGetPostsQuery() — немного длиннее, но явно где endpoint.
