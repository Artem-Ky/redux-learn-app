Теория — 41. createSelector, reselect и мемоизация

Цитата из официальной доки RTK

«The createSelector utility from the Reselect library, re-exported for ease of use.»
— redux-toolkit.js.org/api/createSelector
«Утилита createSelector из библиотеки Reselect, реэкспортированная для удобства использования.»

Почему реэкспорт?

Reselect — отдельная библиотека (~2 KB gzip), но Redux Toolkit включает её в свои зависимости и реэкспортирует те же функции. Это даёт три выгоды:

  • Один импорт — import { createSelector } from '@reduxjs/toolkit' вместо двух пакетов.
  • Версия synced с RTK — RTK сам следит за совместимостью с immer и reselect.
  • Tree-shaking не страдает — reselect экспортирует ESM, бандлер выбрасывает лишнее.

Исходник: что именно реэкспортирует RTK

// @reduxjs/toolkit/src/reselectImports.ts // https://github.com/reduxjs/redux-toolkit/blob/master/packages/toolkit/src/reselectImports.ts export { createSelectorCreator, weakMapMemoize } from 'reselect'

Это всё, что пакет импортирует напрямую из reselect внутри src. А вот что видно через index.ts (то есть пользователю):

// @reduxjs/toolkit/src/index.ts (фрагмент) export { createSelector, createSelectorCreator, lruMemoize, weakMapMemoize, unstable_autotrackMemoize, createStructuredSelector, } from 'reselect' // 👀 именно здесь RTK пробрасывает API reselect наружу

Анатомия createSelector

Что значит «identity check», а не deep equality

// reselect сравнивает так (lruMemoize): function referenceEqualityCheck(a, b) { return a === b } // Поэтому: state1.todos === state2.todos // если Immer/reducer не трогал todos — true → cache hit [...items] === items // false → cache miss, combiner пересчитает { a: 1 } === { a: 1 } // false! два разных объекта → miss

LRU по умолчанию, размер 1

// reselect/src/lruMemoize.ts — фрагмент // https://github.com/reduxjs/reselect/blob/master/src/lruMemoize.ts function createSingletonCache(equals) { let entry return { get(key) { if (entry && equals(entry.key, key)) { return entry.value // 👀 один слот — если ключ совпал, возвращаем value } return NOT_FOUND // иначе — пересчитать }, put(key, value) { entry = { key, value } // перетираем, размер всегда 1 }, clear() { entry = undefined }, } }

Это делает мемоизацию лёгкой, но одноразовой: стоит вызвать селектор с другим набором входов — прошлый кеш затирается. Про кеш на несколько записей — урок 43.

Что такое «expensive computation»?

В плане Redux-а это любая операция, которая создаёт новую ссылку:

  • arr.filter, arr.map, arr.slice, [...arr] — O(n), возвращают новый массив.
  • Object.keys / Object.values / Object.entries — новый массив каждый раз.
  • { ...obj }, Object.fromEntries — новый объект.
  • .toSorted(), .reduce((a, x) => ...) — O(n log n) или O(n).

Сама операция может быть быстрой. Но если компонент делает её внутри селектора, то каждый dispatch вызывает useSelector (до RTK react-redux это было true для каждого подписчика), === между старым и новым результатом даёт false, React-Redux считает «state изменился» → компонент перерисовывается.

Подводные камни

1. Reselect не делает глубокое сравнение. Если твой input-selector возвращает state.todos.filter(t => t.active) — это новый массив на каждый вызов, и === всегда даст false. Почему: reselect использует referenceEqualityCheck, потому что глубокое сравнение O(n×m) убивает смысл мемоизации (быстрее просто пересчитать). Правило: input-selector должен тянуть сырую ссылку из state, а фильтрация/трансформация — только в resultFunc.
2. Default cache size = 1. selectTodoById(state, 1), затем selectTodoById(state, 2) — второй вызов вытеснит первый результат. В списке из 50 todos каждый render вытесняет соседей → 0% попаданий в кеш. Почему так сделано: минимум памяти, максимум предсказуемости. Решение: weakMapMemoize (урок 43) или createSelector.withCache.
3. Инлайновый селектор внутри useSelector. useSelector(state => state.todos.filter(...)) — каждый вызов компонента создаёт новую стрелку, у которой нет персистентного замыкания. Даже если бы внутри был createSelector — его бы пересоздали. Почему: кеш живёт в замыкании селектора. Выносите createSelector в модуль-скоуп.
4. Не импортируйте createSelector из вложенных entry points. Не надо import { createSelector } from '@reduxjs/toolkit/dist/reselectImports'. Это не публичный API — сломается при любом минорном апдейте. Почему: RTK гарантирует совместимость только через корневой @reduxjs/toolkit и @reduxjs/toolkit/query.
5. Мемоизация — per instance. Два компонента, каждый useSelector(selectActive) — один кеш (хорошо). Но useSelector(makeSelectActive()) — у каждого свой селектор и свой кеш (плохо, кешлайф == render lifecycle компонента). Для параметризованных селекторов используйте useMemo(() => makeSelectActive(), []).
