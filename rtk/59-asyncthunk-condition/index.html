Что тут происходит

Кнопка «Fetch posts» — spam-кликабельная. Внутри thunk есть condition: (arg, api) => api.getState().posts.loading !== 'pending': если запрос уже в полёте, thunk не запускается (ни pending, ни fulfilled — ничего). Счётчик «реальных запросов» показывает настоящий рейт. Тоггл dispatchConditionRejection включает побочную фичу — rejected-action с meta.condition === true.

Типичные сценарии

1. Дедупликация — не запускать второй параллельный запрос

Классика: пользователь нажимает кнопку несколько раз подряд. Пока первый запрос в полёте, condition возвращает false — дубли не улетают в сеть.

const fetchPosts = createAsyncThunk<Post[], void, { state: RootState }>( 'posts/fetch', async (_, { extra }) => (extra as Api).getPosts(), { condition: (_arg, { getState }) => { const { loading } = getState().posts return loading !== 'pending' // уже грузится ⇒ пропускаем }, }, )

2. Кэш по id — skip, если объект уже в store

Entity-lookup: не идём в сеть за тем, что уже загружено. Аргумент id доступен первым параметром.

const fetchProduct = createAsyncThunk<Product, number, { state: RootState }>( 'products/fetchById', async (id, { extra }) => (extra as Api).getProduct(id), { condition: (id, { getState }) => { const cached = getState().products.byId[id] return !cached // есть — skip }, }, )

3. Кэш с TTL — skip, пока данные свежие

Тот же кэш, но с временем жизни. Повторный fetch не уйдёт, пока запись моложе 10 секунд.

const CACHE_MS = 10_000 const fetchProfile = createAsyncThunk<Profile, string, { state: RootState }>( 'profile/fetch', async (userId, { extra }) => (extra as Api).getProfile(userId), { condition: (userId, { getState }) => { const rec = getState().profiles.byId[userId] if (!rec) return true // нет — грузим return Date.now() - rec.fetchedAt > CACHE_MS }, }, )

4. Фичефлаг — не дёргать API, если фича выключена

Код call-site'ов остаётся чистым: dispatch(loadAnalytics()). Включение/выключение фичи живёт рядом с thunk'ом, а не в каждом компоненте.

const loadAnalytics = createAsyncThunk<Stats, void, { extra: Api }>( 'analytics/load', async (_, { extra }) => extra.getAnalytics(), { condition: () => featureFlags.isEnabled('analytics-v2'), }, )

5. Async condition — ждать готовности внешней системы

Condition может быть async. Полезно когда нужен auth-токен, WebSocket, или фичефлаги с сервера. Возврат true ⇒ pending диспатчится; false ⇒ skip.

const fetchMe = createAsyncThunk<Me, void, { extra: Container }>( 'me/fetch', async (_, { extra }) => extra.api.getMe(), { condition: async (_, { extra }) => { const ok = await extra.auth.ready() // Promise<boolean> return ok }, }, )

6. Валидация аргумента — skip для некорректных id

Если аргумент заведомо невалиден — не гоняем сеть. Альтернатива rejectWithValue из урока 58: там вы хотите ошибку в state, тут — молча пропустить.

const fetchUser = createAsyncThunk<User, number>( 'users/fetch', async (id) => api.getUser(id), { condition: (id) => Number.isInteger(id) && id > 0, }, )

7. Polling с back-pressure — не накапливать хвост

Таймер раз в секунду дёргает thunk, но если сеть тормозит — condition не даст запросам наложиться друг на друга. Комбо «interval + condition» ≈ дешёвый throttle.

const pollOrders = createAsyncThunk<Order[], void, { state: RootState }>( 'orders/poll', async (_, { extra }) => (extra as Api).getOrders(), { condition: (_, { getState }) => getState().orders.inflight === false, }, ) setInterval(() => store.dispatch(pollOrders()), 1000)

8. Обработка skip на caller'е — .unwrap() + ConditionError

Если нужно реагировать на «пропустили» в компоненте — матчим rejected по meta.condition или ловим ConditionError от .unwrap().

// Вариант А: без throw — анализируем результат action-объекта const res = await store.dispatch(fetchPosts()) if (fetchPosts.rejected.match(res) && res.meta.condition) { toast('Запрос уже в полёте — подождите') } // Вариант Б: с .unwrap() — throw + catch try { const posts = await store.dispatch(fetchPosts()).unwrap() } catch (err) { const e = err as { name?: string; message?: string } if (e.name === 'ConditionError') /* skip */ else /* настоящая ошибка сети */ }

Что вы получаете от thunk'а, пропущенного condition

const res = await store.dispatch(fetchPosts()) // Когда condition вернул false: res.type === 'posts/fetch/rejected' // ← rejected-action ВЕРНУТ (даже если не задиспатчен) res.meta.condition === true // ← наш флаг res.meta.aborted === false res.payload === undefined res.error.name === 'ConditionError' res.error.message === 'Aborted due to condition callback returning false.'

dispatchConditionRejection

По умолчанию (если dispatchConditionRejection: false) RTK не диспатчит rejected-action при condition-отказе. Это хорошо: DevTools не замусоривается. Если включить — получите rejected-action в логе с meta.condition === true. Удобно, если reducer хочет сбросить флаг или залогировать попытку.

Почему condition ≠ if до dispatch'а

Внешне «if (!state.x.loading) dispatch(thunk())» выглядит так же. Разница:
1. condition живёт вместе с thunk'ом. Все компоненты, дёргающие dispatch(thunk()), автоматически защищены.
2. condition async. Можно await-ить внутри (fetch'ить фичефлаг, ждать ready-сигнал).
3. Caller всё равно получает promise. С .unwrap() — корректный ConditionError, можно красиво обработать.

Подводные камни

1. condition может быть async. Если вернёте Promise<boolean>, RTK await-ит его. Значит — между dispatch(thunk()) и pending может быть «пустой» промежуток, в течение которого второй dispatch(thunk()) тоже пройдёт condition. Это настоящий race condition.
2. dispatchConditionRejection — false по умолчанию. Если ваш reducer завязан на обязательный rejected — включайте явно.
3. options.condition получает только getState и extra (не dispatch, не signal). Нельзя диспатчить из condition — это pre-condition, а не часть thunk'а.
4. Resolved promise от condition'а ≠ fulfilled. Если condition вернул false — результат thunk'а всё равно rejected (с meta.condition=true). .unwrap() бросит ConditionError. Обрабатывайте отдельно.
