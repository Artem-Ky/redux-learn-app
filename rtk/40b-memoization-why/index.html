Теория — 40b. Мемоизация как паттерн и почему Redux её любит

Что значит «мемоизация»

Мемоизация — это кеширование результата чистой функции по её аргументам. Если функцию вызывают повторно с теми же аргументами — возвращается сохранённый результат, без повторного вычисления. В Redux это обычно означает: «пока вход селектора не менялся по ссылке — не пересчитывай производные данные (filter/map/sort/reduce)».

Цитата: Redux docs — Deriving Data with Selectors

«Selectors are typically used as memoized functions. A memoized function caches the last input arguments and result value. If called again with the same inputs, it returns the cached result instead of recalculating. This is important for performance when selectors do expensive operations like filtering or transforming arrays.»
— redux.js.org/usage/deriving-data-selectors
«Селекторы обычно реализуют как мемоизированные функции. Такая функция помнит последние входные аргументы и результат. При повторном вызове с теми же аргументами она возвращает кеш вместо пересчёта. Это важно для производительности, когда селектор делает тяжёлые операции — фильтрацию или трансформацию массивов.»

Наивная мемоизация одного аргумента — 10 строк

function memoize1<A, R>(fn: (a: A) => R): (a: A) => R { let lastArg: A | undefined let lastResult: R let hasCache = false return (arg) => { if (hasCache && arg === lastArg) { // 👀 === по ссылке — как reselect return lastResult // cache hit: пересчёта нет } lastArg = arg lastResult = fn(arg) // cache miss: вызываем настоящую функцию hasCache = true return lastResult } } // Использование: const selectPublished = memoize1((items: Post[]) => items.filter((p) => p.published), )

Это и есть «кеш размером 1» (LRU cache size = 1). Reselect делает то же самое, только для нескольких входов + оборачивает всё в createSelector.

Как reselect обобщает это на несколько входов

// Упрощённый createSelector function createSelector(inputSelectors, resultFunc) { let lastInputs: unknown[] | undefined let lastResult: unknown return (state) => { const inputs = inputSelectors.map((sel) => sel(state)) const hit = lastInputs !== undefined && inputs.every((v, i) => v === lastInputs![i]) // 👀 ссылочное сравнение каждого входа if (hit) return lastResult lastInputs = inputs lastResult = resultFunc(...inputs) return lastResult } }

Настоящая reselect ещё сложнее: кастомные equalityFn, argsMemoize, resultEqualityCheck, LRU/WeakMap стратегии, devmode warnings. Но ядро — именно это.

stabilityCheck в dev-моде: react-redux сам вызывает твой селектор дважды и сравнивает результаты. Если даже на одинаковом state селектор возвращает новый объект — warning в консоли с ссылкой на официальный гайд по мемоизации. Это прямое подтверждение, что мемоизация = must-have для селекторов, возвращающих производные данные.

Что мемоизация ≠

Это мемоЭто НЕ мемо
Кеш по ссылке аргумента (===) HTTP-кеш по URL / TTL
Живёт в замыкании функции, lifecycle = lifecycle замыкания Долговременный кеш (IndexedDB, Service Worker)
Работает на чистых функциях (pure) Мемоизация side-effect функций (fetch) — это другой паттерн (каширование запроса)
Обычно cache size = 1 в reselect LRU/LFU с несколькими слотами (weakMapMemoize, lruMemoize(N))

Когда мемо не даёт выигрыша

Если твой селектор — простое поле state => state.user.name, то:

  • Input — та же ссылка, что в Redux-store (ничего не создаём).
  • === работает из коробки (благодаря structural sharing из урока 40a).
  • createSelector здесь — лишний: та же скорость, но с overhead на проверку.

Мемо нужна, только когда в селекторе создаётся новый объект/массив. Красивый ориентир:

Подводные камни

1. Cache size = 1: параметризованный селектор убивает кеш. selectPostById(state, postId): вызови его с id=1, затем с id=2 — второй вызов вытеснит кеш первого. В Array.map(post => selectById(state, post.id)) реальный hit-rate = 0%. Почему так: сделано для предсказуемости и минимума памяти. Решение: weakMapMemoize (урок 43) или createSelector.withCache.
2. Мемо живёт в замыкании — создание селектора внутри render = каждый render новый кеш. const sel = useSelector(createSelector(...)) — каждый render компонента создаёт новый createSelector, у которого новый пустой кеш. Hit-rate = 0%. Почему: кеш = let lastArg в замыкании, которое пересоздаётся. Решение: вынеси createSelector в модуль-scope или в useMemo(() => createSelector(...), []).
3. Input-селектор возвращает новый объект → мемо бесполезна. createSelector([s => s.posts.items.filter(...)], ...) — input-селектор сам делает .filter(), каждый раз новый массив. === между старым и новым массивом всегда false → combiner пересчитывает каждый раз. Почему: reselect сравнивает вход к combiner по ===, и новые массивы всегда разные. Правило: input-selector = cheap pluck ссылки из state, трансформации — только в combiner.
4. Мемо не заменяет иммутабельность — она её требует. Если reducer мутирует state напрямую (state.posts.items.push(x) без Immer), новый items — это та же ссылка, что и старый. Селектор увидит «items не менялись» → cache hit → UI не обновится. Почему: мемо доверяет ===. Решение: всегда пиши reducer через createSlice или Immer; никогда не мутируй состояние вне draft.
5. «Мемоизация» в нашем смысле не равна React.memo. React.memo — обёртка компонента, которая сравнивает props shallow'ом. Мемоизация селекторов — кеш функции. Оба полезны, но для разного: createSelector — чтобы селектор вернул ту же ссылку (чтобы React.memo сработал). Без мемо-селектора React.memo часто бесполезен: props — новые объекты каждый раз.
