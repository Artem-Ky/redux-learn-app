Теория — Архитектура и зависимости

Дословно из доки

«The Redux Toolkit package is intended to be the standard way to write Redux logic. It includes utilities to help simplify many common use cases, including store setup, creating reducers and writing immutable update logic, and even creating entire "slices" of state at once.»
— Getting Started
«Пакет Redux Toolkit задуман как стандартный способ писать Redux-логику. Он включает утилиты, упрощающие многие типичные сценарии, в том числе настройку store, создание reducer'ов и написание иммутабельных обновлений, и даже создание целых «slice'ов» state сразу.»

Зачем зонтичный пакет

Раньше для production-приложения нужно было собрать минимум 5 пакетов:

  • redux — core store
  • react-redux — биндинги для React
  • redux-thunk — async actions
  • reselect — мемоизация селекторов
  • immer — иммутабельные обновления

Каждая со своими версиями, breaking changes, типами. RTK решает это: одна установка, версии под контролем команды Redux, единая документация, согласованные API.

Почему именно эти зависимости

Это исторически сложившийся стек best-practice от Дэна Абрамова (создателя Redux) и Марка Эриксона (мейнтейнера). Все 4 библиотеки тоже мейнтейнятся командой Redux (immer и reselect — нет, но они стандарт де-факто для Redux-сообщества). RTK — это «упаковка best practices» в один пакет.

Совместимость версий

Если у вас в проекте уже есть immer или reselect отдельно — npm/pnpm могут поставить вторую копию в node_modules. Это опасно: разные инстансы immer не понимают друг друга (isDraft(x) вернёт false для draft из другой копии). Проверьте: npm ls immer.

Что НЕ входит в RTK

  • react-redux — это peer dep (не реэкспортируется напрямую кроме useDispatch/useSelector в /query/react)
  • redux-saga — saga middleware, alternative thunk; ставится отдельно
  • redux-observable — RxJS-based middleware; отдельно
  • redux-persist — для localStorage; отдельно
  • redux-form, connected-react-router — отдельно (и устаревшие)
