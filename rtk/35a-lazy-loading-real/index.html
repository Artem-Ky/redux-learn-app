Зачем lazy loading в Redux-приложении

Урок 35 показал механику slice.injectInto(). В этом уроке — реальный code-splitting: feature-модули лежат в отдельных .ts-файлах, попадают в отдельные chunks, грузятся через await import('./features/...') и видны отдельной строкой в DevTools → Network.

«Splitting up reducer code becomes more of a concern with larger apps. We may want to keep some reducers out of the initial bundle.»
— Redux — Code Splitting
  • First paint / TTI. Юзер увидит header и loading раньше, если тяжёлая feature не в main.
  • Pay-as-you-go. Admin-панель грузит только админ (~1% трафика). Зачем платить остальным 99%?
  • Изоляция риска. Сломалась feature-chunk — остальное приложение работает.
  • Параллельные релизы. Команда может деплоить feature-chunk отдельно от core.
Статически vs лениво — один и тот же slice, две разные судьбы

main.tsx — ВСЁ статически в main.js

import { chartSlice } from './chart-feature' import { cartSlice } from './cart-feature' import { analyticsSlice } from './analytics-feature' const store = configureStore({ reducer: combineReducers({ chart: chartSlice.reducer, cart: cartSlice.reducer, analytics: analyticsSlice.reducer, }), }) // build output: // main.js ~ 85 KB — всё внутри

✗ Analytics грузится первым даже если юзер туда никогда не зайдёт

main.tsx — лениво main + 3 chunks

const rootReducer = combineSlices(coreSlice) .withLazyLoadedSlices<LazyState>() // по клику — настоящий dynamic import() async function loadAnalytics() { const mod = await import('./features/analytics-feature') mod.register(rootReducer, store, hostEl, con) } // build output: // main.js ~ 25 KB // chart-feature.*.js ~ 1 KB ← грузится при клике // cart-feature.*.js ~ 2 KB ← // analytics-feature.*.js ~ 4 KB ←

✓ main меньше, тяжёлая analytics лежит в отдельном файле

Критически важно: чтобы Vite/Rollup вынесли модуль в отдельный chunk — import должен быть динамическим (await import('...')) и статически НЕ использоваться. Если где-то сверху файла есть import x from './features/analytics-feature' — модуль попадёт в main chunk, и dynamic import в том же файле станет бесполезным.
Демо · нажмите и увидите в Network новый chunk

Откройте DevTools → Network → фильтр JS. Нажимайте «Загрузить фичу» — браузер запросит новый *-feature-*.js файл. До клика он недоступен. Подсказка: hover на кнопку предзагрузит chunk — в Network появится строчка ещё до клика.

Самый лёгкий feature: slice с массивом чисел. Всё RTK уже в main — только свой reducer.
cart not loaded
Средний: createEntityAdapter внутри slice'а. Adapter попадёт в этот chunk, а не в main.

Демонстрация в миниатюре: наши фичи маленькие (это курс, а не Figma). В реальном приложении analytics-chunk с реальным графическим движком будет ~50–200 KB. Именно такие модули окупают lazy-подход.

React.lazy + Suspense + injectInto

React.lazy() — это обёртка вокруг того же dynamic import(), но с интеграцией в React render-цикл. Когда компонент загружается, React показывает <Suspense fallback>. После загрузки — перерисовывает.

import { lazy, Suspense } from 'react' const AnalyticsView = lazy(() => import('./AnalyticsView')) function App() { const [open, setOpen] = useState(false) return ( <> <button onClick={() => setOpen(true)}>Open analytics</button> {open && ( <Suspense fallback={<div>Loading analytics…</div>}> <AnalyticsView /> </Suspense> )} </> ) }

Живое демо — React.lazy + Suspense + ErrorBoundary

Жизненный цикл ленивой загрузки

click → setOpen(true) │ ▼ React рендерит <AnalyticsView /> │ ▼ lazy() внутренне throws Promise ─────► <Suspense> catches, рендерит fallback │ ▼ браузер: GET /assets/AnalyticsView-*.js │ ▼ выполняется top-level модуля: - import React - (если нужно) slice.injectInto(...) - export default AnalyticsView │ ▼ React re-renders │ ▼ <AnalyticsView /> реально монтируется dispatch(...) работает, slice уже в store

Комбо: React.lazy + slice.injectInto в одном модуле

// features/analytics/AnalyticsPage.tsx — сам будет lazy chunk import { createSlice } from '@reduxjs/toolkit' import { rootReducer } from '@/app/store' const analyticsSlice = createSlice({ ... }) // ↓ выполнится на top-level при первой загрузке chunk'а // reducer уже в rootReducer к моменту render'а const injected = analyticsSlice.injectInto(rootReducer) export default function AnalyticsPage() { const data = useAppSelector(injected.selectors.selectData) return <Chart data={data} /> }
Почему top-level, а не в useEffect? Если делать inject в useEffect, то первый render произойдёт с state.analytics === undefined, и ваши селекторы упадут. На top-level модуля inject срабатывает до того, как React начнёт рендерить компонент.
Decision matrix · lazy или не lazy

Не всё надо лениво грузить. Оверхед — лишний HTTP-запрос, время подождать Suspense fallback, сложность debug'а. Правило: lazy окупается, если вес feature > 5 KB и используется < 50% юзеров.

Сценарий Lazy? Причина
Header, Nav, AuthGate ❌ нет Нужны на первом пикселе. Lazy = визуальный сбой.
Core Redux slices (auth, user, theme) ❌ нет Нужны везде. Разбрасывать по chunks — пересбор одного и того же.
/dashboard после логина ✅ да Не-залогиненный юзер никогда не увидит.
Editor с Monaco / tiptap (~1 MB) ✅ обязательно Иначе main bundle неприемлем.
Feature под A/B-тестом (50% off) ✅ да Платит только variant B. Вариант A вообще не грузит.
Маленькая форма в диалоге ⚠ вряд ли < 3 KB — оверхед import'а не окупается.
Admin /users/:id/edit ✅ да 1% трафика. Экономия для 99% юзеров.
i18n словари (fr, de, ja) ✅ по локали Один язык — один chunk. Не тяните все 20.
Mobile-only фича isMobile && load(...) ✅ условно Desktop-юзер вообще не грузит.
Settings / Profile / Account ✅ да Посещают редко, но каждая — большая.
Notification bell (иконка в header) ❌ нет Видна сразу, маленькая — lazy нечего грузить.
Предзагрузка · modulepreload, prefetch-on-hover

Lazy — это не обязательно «пусть юзер подождёт». Chunk можно начать тянуть заранее, пока юзер ещё не кликнул. Когда кликнет — chunk уже в кеше.

Prefetch on hover (используется в этом уроке)

// как только курсор на кнопке — начинаем грузить (без await, без UI) btn.addEventListener('mouseenter', () => { import('./features/chart-feature') // Promise игнорируется }, { once: true }) // при клике: Promise уже resolved (или почти), import() мгновенный btn.addEventListener('click', loadChart)

modulepreload — hint браузеру

// в idle-момент (requestIdleCallback или setTimeout): const link = document.createElement('link') link.rel = 'modulepreload' link.href = '/assets/analytics-feature-abc.js' document.head.appendChild(link) // Отличие от обычного fetch: браузер распарсит и скомпилирует JS // заранее, но не выполнит. Выполнение — при реальном import().
Vite сам ставит <link rel="modulepreload"> для статических импортов. Для динамических — вручную или пишете плагин. Проверьте build.modulePreload.
Подводные камни

1. State появляется только на следующем dispatch'е

После injectInto state[path] === undefined, пока не прилетит action. Решение: сразу после inject сделать store.dispatch({ type: '@@INIT_INJECTED/<name>' }). Это noop-action — все reducers получат текущий state, injected reducer получит undefined и вернёт initialState.

const injected = slice.injectInto(rootReducer) store.dispatch({ type: '@@INIT_INJECTED/analytics' }) // теперь state.analytics === initialState

2. Двойной inject в React.StrictMode

StrictMode в dev вызывает эффекты дважды. Если injectInto на top-level модуля — не страшно, модуль выполнится один раз. Если в useEffect — вызовется дважды. injectInto идемпотентен: второй вызов вернёт тот же slice, reducer не заменится.

3. Selectors падают до inject'а

Если state.analytics === undefined, а селектор пишет state.analytics.events — runtime TypeError. Решение: state.analytics?.events ?? [] ИЛИ не рендерьте компонент, пока feature не injected.

// ✗ падает если state.analytics undefined const selectEvents = (state: RootState) => state.analytics.events // ✓ безопасно const selectEvents = (state: RootState) => state.analytics?.events ?? []

4. Race condition · одновременные загрузки

Если юзер быстро кликает «Open» дважды — браузер всё равно загрузит chunk один раз (module dedup по URL). Но ваш handler может запустить inject + mount дважды. Решение: Map<feature, Promise> с in-flight промисами (см. main.tsx).

5. Chunk не загрузился

Сеть упала, сервер вернул 404 (старый deploy), браузер заблокировал — import() реджектится. Обязательный retry с экспоненциальным backoff:

async function loadWithRetry<T>( factory: () => Promise<T>, attempts = 3, ): Promise<T> { for (let i = 0; i < attempts; i++) { try { return await factory() } catch (e) { if (i === attempts - 1) throw e await new Promise(r => setTimeout(r, 2 ** i * 300)) } } throw new Error('unreachable') } // usage: const mod = await loadWithRetry(() => import('./features/analytics-feature'))

6. Stale chunks после deploy

Юзер открыл страницу в понедельник (main-abc.js). Во вторник вы задеплоили новую версию (main-def.js, analytics-XYZ.js). Старый main у юзера всё ещё в памяти — он пытается import('./features/analytics-OLD-hash.js'), а такого файла на сервере уже нет → 404.

Решения: (а) CDN хранит старые chunks несколько дней, (б) при import()-ошибке делаете window.location.reload(), (в) service worker с proper caching strategy.

7. SSR + lazy

В SSR (Next.js, Remix) React.lazy традиционно не работал — Suspense нельзя было «подождать» на сервере. В React 19 это решили streaming SSR. Детали — в React docs — Suspense.

8. Hot Module Replacement

В dev-режиме при редактировании features/chart-feature.ts Vite перезагружает только этот chunk. Reducer в store остаётся — injectInto идемпотентен. Но state не сбросится: если вы поменяли shape initialState, старый state может оказаться несовместим. Решение — ручной reload страницы после изменения shape'а.

Теория — lazy loading в RTK + React

Цитата из RTK docs

«combineSlices returns an enhanced reducer with an inject method that allows you to inject a new slice/reducer at runtime. This is most useful for large applications where some slices are loaded only when needed (code splitting).»
— combineSlices — Injecting reducers
«combineSlices возвращает расширенный reducer с методом inject, который позволяет внедрять новый slice в runtime. Наиболее полезно в больших приложениях, где некоторые slice'ы подгружаются только при необходимости (code splitting).»

Цитата из React docs

«lazy lets you defer loading a component's code until it is rendered for the first time. … The loading code is typically included in a separate bundle from the rest of your app.»
— React — lazy
«lazy позволяет отложить загрузку кода компонента до его первого рендера. Код обычно выносится в отдельный bundle.»

Как Vite создаёт chunks

Rollup (его использует Vite) при build-е ищет все import()-выражения в коде. Для каждого уникального target'а создаёт отдельный output chunk. Если target используется из нескольких chunks — создаётся общий shared chunk. Модули, импортируемые только статически — остаются в том chunk, откуда импортируются.

Зачем writeable rootReducer?

Обычный combineReducers возвращает замороженную функцию — добавить reducer нельзя. combineSlices возвращает функцию-с-методом inject: под капотом это Map<reducerPath, reducer>, в которую можно добавлять записи в runtime. На каждом dispatch root reducer обходит Map и вызывает все.

injectInto vs rootReducer.inject

// rootReducer.inject — универсальный API, для любых reducer'ов rootReducer.inject({ reducerPath: 'feature', reducer: featureReducer, }) // slice.injectInto — для slice'ов, рекомендуемый const injected = slice.injectInto(rootReducer) // Возвращает "клон" slice'а — все actions/selectors работают // Идемпотентен: повторный вызов вернёт существующий

Connecting to routing — React Router 6.4+

// Современный React Router — встроенный lazy для роутов import { createBrowserRouter } from 'react-router-dom' const router = createBrowserRouter([ { path: '/analytics', lazy: async () => { const mod = await import('./features/analytics') return { Component: mod.default } }, }, ])
