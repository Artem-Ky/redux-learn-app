Теория — 81. api.reducer и api.middleware

Цитата — Quick Start

«The middleware that is generated by RTK Query manages cache lifetimes and expiration, and is required for caching and invalidation of queries and mutations to work correctly. Adding api.middleware is required.»
— Configure the Store
«Без api.middleware нет запросов — reducer будет в state, но он никогда не получит pending/fulfilled, т.к. никто их не диспатчит.»

Правильная сборка

import { configureStore } from '@reduxjs/toolkit' import { setupListeners } from '@reduxjs/toolkit/query' import { pokeApi } from './pokeApi' export const store = configureStore({ reducer: { [pokeApi.reducerPath]: pokeApi.reducer, // ← 1. reducer под ключ 'pokeApi' // прочие slices }, middleware: (getDefaultMiddleware) => getDefaultMiddleware().concat(pokeApi.middleware), // ← 2. concat — иначе запросы не пойдут }) setupListeners(store.dispatch) // ← 3. опц: refetchOnFocus / refetchOnReconnect

Что делает api.reducer

Это обычный slice-reducer, который:

  • Обрабатывает internal-actions api/config/..., api/executeQuery/pending|fulfilled|rejected, api/executeMutation/..., api/subscriptions/....
  • Хранит 5 объектов: config, queries, mutations, provided, subscriptions (подробно — урок 84).
  • Не делает ни одного fetch — он чисто state-хранилище.

Что делает api.middleware

Это три-в-одном middleware:

  • queryMiddleware — ловит initiate(), вызывает baseQuery, диспатчит pending → fulfilled/rejected.
  • mutationMiddleware — то же для мутаций + инвалидация тэгов.
  • cacheLifecycleMiddleware — keepUnusedDataFor-таймеры, отписка последнего subscribers'а → TTL → removeQueryResult.
  • invalidationMiddleware — слушает invalidatesTags, диспатчит refetch.
  • pollingMiddleware — таймеры poll intervals.

Исходник — подсобранный middleware

// @reduxjs/toolkit/src/query/core/buildMiddleware/index.ts (упрощённо) export function buildMiddleware(input) { const middlewares = [ buildQueryLifecycleHandler(input), // onQueryStarted buildDevMiddleware(input), // dev warnings buildCacheCollectionHandler(input), // 👀 keepUnusedDataFor buildInvalidationByTagsHandler(input), // 👀 invalidatesTags → refetch buildPollingHandler(input), // 👀 pollingInterval buildCacheLifecycleHandler(input), // onCacheEntryAdded (ws) buildBatchNotificationsHandler(input), // SHOULD_AUTOBATCH ] return (api) => (next) => (action) => { let result = next(action) middlewares.forEach((mw) => mw(...)) return result } }

setupListeners — зачем

import { setupListeners } from '@reduxjs/toolkit/query' setupListeners(store.dispatch) // ↓ внутри: window.addEventListener('focus', ...) + 'online', 'offline' // Диспатчит внутренние actions api/focus-changed, api/online-changed. // Middleware ловит их и триггерит refetch для опций refetchOnFocus / refetchOnReconnect.

Без setupListeners(store.dispatch) опции refetchOnFocus / refetchOnReconnect не сработают — см. урок 91.

Подводные камни

1. .concat(api.middleware), а НЕ .prepend().
RTKQ-middleware должно видеть все предыдущие слои (сериализацию, thunk, listener). prepend поставит его в начало и отломает, например, проверку сериализации payload'ов фейковых actions.
Исключение: listenerMiddleware ставится через prepend, а RTKQ — через concat. Это норма.

2. Забыл reducer — ошибка на старте.
Без [api.reducerPath]: api.reducer в reducer-map RTK бросит в dev: «Unable to determine the reducer path for this api slice». initiate() не сработает — state.pokeApi undefined.

3. Забыл middleware — тишина.
Самый коварный баг. Reducer есть, selector возвращает { status: 'uninitialized' }, hooks показывают isLoading: true навсегда. Никаких ошибок в консоли. Проверь: store.getState().pokeApi.config.middlewareRegistered — это флаг, который выставляет само middleware при первом action. Если false — middleware не подключён.

4. Два createApi с одним reducerPath.
Если сделать createApi({ reducerPath: 'api', ... }) дважды — при .concat(apiA.middleware, apiB.middleware) внутри one path ключи конфликтуют. Правило: каждый api = свой reducerPath.

5. setupListeners нужен раз за жизнь store, не на каждый mount.
Его вызов возвращает unsubscribe. В SPA — один раз при бутстрапе. В Next.js — per-request (по документации). НЕ вызывать в useEffect компонента.
