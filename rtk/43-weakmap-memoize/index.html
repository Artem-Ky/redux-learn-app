Теория — 43. weakMapMemoize, cache size и WeakMap-дерево

Цитата из reselect docs

«Creates a tree of WeakMap-based cache nodes based on the identity of the arguments it's been called with. This allows weakMapMemoize to have an effectively infinite cache size.»
— reselect.js.org/api/weakmapmemoize
«Создаёт дерево узлов кэша на основе WeakMap по идентичности аргументов, с которыми был вызван. Это даёт weakMapMemoize фактически неограниченный размер кэша.»

Проблема, которую решает weakMapMemoize: cache size = 1

До reselect v5 createSelector использовал lruMemoize c maxSize: 1. Для селектора без аргументов (selectAllTodos) этого хватало. Но для параметризованного — беда:

// Симуляция старого дефолта — явно lruMemoize(1) const createSelectorLru1 = createSelectorCreator(lruMemoize, { maxSize: 1 }) const selectTodoById = createSelectorLru1( [(s) => s.items, (_s, id) => id], (items, id) => items.find((t) => t.id === id), ) selectTodoById(state, 'a') // compute #1, кэш: {a → result} selectTodoById(state, 'b') // compute #2, кэш: {b → result} — 'a' вытеснен selectTodoById(state, 'a') // compute #3 — снова! hit-rate = 0%
Начиная с reselect v5 default поменяли. Теперь createSelector по умолчанию использует weakMapMemoize — именно потому, что cache-1 слишком часто кусал параметризованные селекторы. Источник: createSelectorCreator.ts: export const createSelector = createSelectorCreator(weakMapMemoize). Поэтому в демо выше первая кнопка — явный lruMemoize(1): без него «проблему» не увидеть, обычный createSelector ведёт себя как weakmap-режим.

Решение: weakMapMemoize — дерево WeakMap'ов по аргументам

Исходник: как строится дерево

// reselect/src/weakMapMemoize.ts — фрагмент // https://github.com/reduxjs/reselect/blob/master/src/weakMapMemoize.ts interface CacheNode<T> { s: 0 | 1 // status: unterminated / terminated v: T | void // value o: WeakMap<object, CacheNode<T>> | null // object cache p: Map<primitive, CacheNode<T>> | null // primitive cache } function memoized() { let cacheNode = fnNode for (let i = 0; i < arguments.length; i++) { const arg = arguments[i] if (typeof arg === 'object' && arg !== null) { let objectCache = cacheNode.o if (objectCache === null) cacheNode.o = objectCache = new WeakMap() const objectNode = objectCache.get(arg) if (objectNode === undefined) { // новая ветка cacheNode = createCacheNode() objectCache.set(arg, cacheNode) } else { cacheNode = objectNode // 👀 спускаемся вглубь дерева } } else { let primitiveCache = cacheNode.p if (primitiveCache === null) cacheNode.p = primitiveCache = new Map() const primitiveNode = primitiveCache.get(arg) if (primitiveNode === undefined) { cacheNode = createCacheNode() primitiveCache.set(arg, cacheNode) } else { cacheNode = primitiveNode } } } if (cacheNode.s === TERMINATED) { return cacheNode.v // 🎯 cache hit — возвращаем запомненное } const result = func.apply(null, arguments) // miss — считаем cacheNode.s = TERMINATED cacheNode.v = result return result }

Про два типа узлов: WeakMap для объектов, Map для примитивов

Если у вас selectTodoById(state, id), то state — это объект (уходит в WeakMap), а id — строка/число (уходит в обычный Map). Почему нельзя оба в WeakMap? Потому что WeakMap принимает ключи только типа object — это встроенное ограничение JS. Ссылка на примитив не может быть слабой — сборщик мусора не видит «ссылок» на строку, она всегда дублируется.

Почему WeakMap — GC-friendly

// Обычный Map — держит ключи в памяти вечно: const cache = new Map() cache.set(oldState, result) oldState = null // но cache ещё держит ссылку → утечка // WeakMap — ссылка слабая: const cache = new WeakMap() cache.set(oldState, result) oldState = null // если других ссылок нет — GC очистит запись

В контексте Redux это означает: история state'ов в time-travel-дебагере не растёт бесконечно при работе с weakMapMemoize. Как только snapshot state'а удалён из DevTools — его кеш-записи очищаются.

weakMapMemoize — default в createDraftSafeSelector

Напомним код из урока 42:

export const createDraftSafeSelector = createDraftSafeSelectorCreator(weakMapMemoize) // ☝ специально, потому что в reducer'е селектор получает разные state'ы (before/after). // lruMemoize(1) вытеснял бы каждый раз. weakMapMemoize держит все пока state жив.

Как включить weakMapMemoize в обычном createSelector

import { createSelector, weakMapMemoize } from '@reduxjs/toolkit' const selectTodoById = createSelector( [(s: RootState) => s.todos.items, (_s: RootState, id: string) => id], (items, id) => items.find((t) => t.id === id), { memoize: weakMapMemoize, argsMemoize: weakMapMemoize, }, )

Два memoize поля: memoize управляет кэшем результата combiner'а, argsMemoize — кэшем массива аргументов самой обёртки (снижает повторные вычисления === prev). Для полной «бесконечной» мемоизации ставьте оба.

Альтернатива weakMap: фабрика селекторов (селектор на id)

До weakMapMemoize классический приём против cache-1 был такой: не один «умный» селектор с большим кешем, а много маленьких — по одному на каждый id — каждый со своим lruMemoize(1). Все вместе они держатся в Map<id, selector>.

const factoryCache = new Map<string, TodoSelector>() function makeSelectorFor(id: string) { return createSelectorLru1( // ⚠ ключевое: input — САМ объект todo, а не items. [(s: RootState) => s.todos.items.find((t) => t.id === id)], (item) => item, ) } const selectTodoById = (state: RootState, id: string) => { let sel = factoryCache.get(id) if (!sel) { sel = makeSelectorFor(id); factoryCache.set(id, sel) } return sel(state) }

Почему это работает там, где weakmap не работает? Immer делает structural sharing: при toggle одного todo новый массив items содержит 49 ТЕХ ЖЕ ссылок на объекты + 1 новую. Если input-селектор достаёт именно объект (items.find(id)), то для 49 селекторов input === prev → hit. А weakmap смотрит на items как на ключ WeakMap — новая ссылка массива = осиротевшая ветка дерева, все 50 id пересчитаются.

То же работает и с weakmap, если переписать input так же: extracting не весь массив, а конкретный объект. Это третий режим в демо — «weakmap + input=items.find»:

const selectTodoById = createSelector( [ // input достаёт сам объект, а не массив (s: RootState, id: string) => s.todos.items.find((t) => t.id === id), (_s: RootState, id: string) => id, ], (item, _id): Todo | undefined => item, { memoize: weakMapMemoize, argsMemoize: weakMapMemoize }, )

Поведение при toggle: 49 hit, 1 miss — как у factory. Суть не в механизме хранения кеша (Map vs WeakMap), а в форме input-селектора. Фабрика просто подталкивает думать в правильных терминах: «что конкретно нужно этому селектору?».

Практика: если всё равно нужно сравнивать через items целиком — weakMapMemoize проще. Если селектор зависит от одного элемента и хочется пережить точечные мутации — нужен либо «умный» input (items.find(id)), либо фабрика, либо entityAdapter с selectById (урок 52).

Подводные камни

1. Примитивы не GC-ятся. Узлы с ключами-примитивами (id, category) лежат в обычном Map и живут, пока живёт сам селектор. Если у вас 10 000 уникальных id — кеш растёт. Почему: WeakMap не поддерживает примитивы как ключи. Смягчить: если id — это большой шардированный uuid, иногда лучше хранить кеш на уровне компонента с TTL, а не в селекторе.
2. weakMapMemoize теперь default, но только в reselect v5+. Если в проекте reselect v4 / старый RTK (< 2.0) — default всё ещё lruMemoize(1), и параметризованные селекторы миссят. Как проверить: cat node_modules/reselect/package.json. Почему важно: код «я просто createSelector вызвал» ведёт себя по-разному на разных мажорах reselect.
3. Инлайновый селектор разрушает всё. useSelector(s => selectTodoById(s, id)) где selectTodoById создан с weakMapMemoize — всё ок. Но useSelector(s => createSelector(...)(s)) внутри тела компонента — каждый render создаёт новый селектор с пустым кешем. Выносите в модуль-скоуп.
4. resultEqualityCheck — как спасти от ненужных ререндеров. Допустим, combiner возвращает items.map(t => t.id). Даже если items не менялся, ссылка на результат новая → подписчики реагируют. Опция resultEqualityCheck: shallowEqual (или свой deep) восстановит старую ссылку, если новое значение «такое же». Почему: reselect пер-результатно кеширует последний output через WeakRef и сравнивает твоим чекером.
5. ≈lruMemoize по скорости, но чуть медленнее. В доке reselect прямо написано: «roughly the same speed as lruMemoize, although likely a fraction slower». Разница в WeakMap.get vs линейный поиск в массиве при maxSize=1. Для 99% случаев незаметно.
