Теория — 50. updateOne / updateMany / takeNewKey

Как именно работает updateOne

updateOne({ id, changes }) ≡ Object.assign(entities[id], changes)

Это shallow assign — только верхний уровень. Любое nested-поле (объект или массив в changes) заменяет прежнее, не мёрджится.

Исходник — updateManyMutably + takeNewKey

function takeNewKey(keys, update, state): boolean { const original = state.entities[update.id] if (original === undefined) return false const updated = Object.assign({}, original, update.changes) // ← shallow! const newKey = selectIdValue(updated, selectId) const hasNewKey = newKey !== update.id if (hasNewKey) { keys[update.id] = newKey delete state.entities[update.id] // старый ключ выкидываем } state.entities[newKey] = updated // записываем под (возможно) новым return hasNewKey } function updateManyMutably(updates, state) { const updatesPerEntity = {} // 1) мержим несколько update для одного id (last wins per key) updates.forEach((update) => { if (update.id in state.entities) { updatesPerEntity[update.id] = { id: update.id, changes: { ...updatesPerEntity[update.id]?.changes, ...update.changes }, } } }) updates = Object.values(updatesPerEntity) const didMutateEntities = updates.length > 0 if (didMutateEntities) { // 2) для тех, у кого изменился id — перестраиваем ids const didMutateIds = updates.filter((u) => takeNewKey(newKeys, u, state)).length > 0 if (didMutateIds) { state.ids = Object.values(state.entities).map((e) => selectIdValue(e, selectId)) } } }

Смена id через updateOne

Если changes включает поле, от которого зависит selectId, — RTK автоматически перекладывает запись под новый ключ: удаляет старый ключ из entities, вставляет под новым, и пересчитывает ids. Это и делает takeNewKey.

Цитата — RTK docs

«updateOne and updateMany accept an Update<T> object that contains an entity ID and an object containing one or more new field values to update inside a changes field. The provided changes will be shallow-merged with the existing entity.»
«updateOne/updateMany принимают объект Update<T> с id и одним или несколькими изменёнными полями в changes. Изменения shallow-merge-ятся с существующим объектом.»

Подводные камни

1. Nested-поля затираются целиком.
Пример (см. демо): user.profile = { name:'Alice', age:30 }, updateOne({ id, changes: { profile: { name: 'Bob' } } }) → profile = { name: 'Bob' }. Поле age исчезло.

Решение A. Собрать полный новый nested-объект в компоненте: { changes: { profile: { ...oldProfile, name: 'Bob' } } }. Минус — нужен selectById перед dispatch.

Решение B. Написать свой reducer с Immer-мутацией:
reducers: { userNameChanged: (state, action) => { const u = state.entities[action.payload.id] if (u) u.profile.name = action.payload.name // ← mutate через Immer } }
Immer применит patch, age останется на месте.

2. updateMany: несколько update на один id → last wins per key.
Если в массиве два { id: 'a', changes: { price: 10 } } и { id: 'a', changes: { price: 20 } } — RTK делает { ...first.changes, ...second.changes }. Второй перезапишет первый. Но если поля разные — объединятся. Это не применение по очереди, а single merge.

3. updateOne игнорирует отсутствующий id.
Если state.entities[id] === undefined, takeNewKey возвращает false и ничего не мутирует. Никакого error/warning. Хочешь гарантию, что запись будет создана → upsertOne.

4. Смена id — Full Record rebuild. Если takeNewKey вернул true хоть один раз — state.ids пересобирается через Object.values(state.entities).map(selectId). Для больших коллекций это O(n). Не меняй id часто.
