Теория — 94. onCacheEntryAdded + streaming

Цитата — Streaming Updates

«onCacheEntryAdded is called when a new cache entry is added to the cache. It can be used to set up a long-lived connection (WebSocket, Server-Sent Events) and feed updates to the cache via updateCachedData, cleanly tied to the entry's lifecycle.»
— RTKQ · Streaming Updates
«Когда первый подписчик появился → onCacheEntryAdded запустится. Закрывай WS в cacheEntryRemoved — это момент когда последний подписчик отпал и keepUnusedDataFor догорел. Никакой ручной жизни соединения.»

Ждём первый fulfilled (initial history загружена в cache) await cacheDataLoaded // 2. Подписываемся на сообщения const listener = (event: MessageEvent) => { const msg: Message = JSON.parse(event.data) updateCachedData((draft) => { draft.push(msg) }) } ws.addEventListener('message', listener) } catch { /* cacheDataLoaded rejected — игнорим */ } // 3. Ждём пока entry «умрёт» (последний подписчик + keepUnusedDataFor догорел) await cacheEntryRemoved ws.close() }, }),

Альтернатива — SSE

async onCacheEntryAdded(arg, { updateCachedData, cacheDataLoaded, cacheEntryRemoved }) { const es = new EventSource('/stream/prices') try { await cacheDataLoaded es.addEventListener('price', (e) => { const tick = JSON.parse(e.data) updateCachedData((draft) => { draft[tick.symbol] = tick.price }) }) } catch {} await cacheEntryRemoved es.close() }

Полный цикл от mount до cleanup

// 1. Первый useGetMessagesQuery → subscription created → cache entry added // → RTKQ вызывает onCacheEntryAdded(arg, api) // 2. Initial fetch запускается параллельно → fulfilled // → cacheDataLoaded.resolve(data) // 3. Твой код: new WebSocket() → подписываешься на WS.onmessage // → каждое сообщение → updateCachedData((draft) => draft.push(msg)) // 4. Все подписчики отписались → refCount=0 → стартует keepUnusedDataFor // таймер → removeQueryResult → cacheEntryRemoved.resolve() // 5. Твой код после await: ws.close() → GC

Подводные камни

1. onCacheEntryAdded вызывается ОДИН раз на создание entry.
Не на каждый subscribe, не на refetch. Если entry уже есть — не вызовется. Если ты в ней читаешь getState — снимок только на момент создания.

2. Обязательно закрывай ресурс в cacheEntryRemoved.
Иначе WebSocket будет жить, даже когда компонент давно unmount'ился. Classic leak.

3. cacheDataLoaded может REJECT.
Если initial fetch упал — await cacheDataLoaded бросит. Оберни в try/catch иначе код после await не выполнится.

4. updateCachedData — Immer-рецепт, не replace data.
Внутри мутируй draft. Если хочешь полностью заменить — делай draft.splice(0); draft.push(...newArr) или возвращай новый массив (как в createReducer).

5. Не забудь про CORS/origin для WS в моках.
Mock-WS через in-memory EventTarget (как в этом демо) — OK для dev. В проде — ws:// vs wss:// + heartbeat + reconnect.

6. Streaming != polling.
WS/SSE — серверный push, минимум latency, бесконечно открытое соединение. Polling — клиент сам тыкает сервер с интервалом. Если сервер не умеет WS — polling через pollingInterval (урок 91).
