Два сценария дедупликации, которые RTKQ закроет одной строкой

Есть два разных типа дубликатов в запросах:
① Идемпотентные дубли — один и тот же запрос улетает два раза параллельно (кнопка дважды). Правильный ответ — второй не запускать. Это condition.
② Последовательные разные — пользователь печатает в search. Прошлый запрос больше не нужен. Правильный ответ — отменить старый, запустить новый. Это takeLatest.

① condition: спам-клики Fetch

Нажмите кнопку быстро несколько раз. condition проверяет getState().loading === 'pending' — пока запрос в полёте, новые отсекаются.

Вариант 1 — condition (идемпотентные дубли)

const fetchPosts = createAsyncThunk<Post[], void, { state: RootState }>( 'posts/fetch', async () => api.getPosts(), { condition: (_, { getState }) => getState().posts.loading !== 'pending', // кэш по статусу }, ) // Пока первый pending — все последующие dispatch(fetchPosts()) возвращают // rejected action с meta.condition=true и БЕЗ pending-диспатча.

Вариант 2 — takeLatest через listener

import { createListenerMiddleware } from '@reduxjs/toolkit' const listener = createListenerMiddleware() listener.startListening({ actionCreator: searchQueryChanged, effect: async (action, api) => { // ① Отменить все ПРЕДЫДУЩИЕ активные эффекты этого listener'а api.cancelActiveListeners() // ② Debounce — ждём 200ms тишины await api.delay(200) // ③ Запускаем fetch. signal привязан к этому эффекту — // если придёт следующий keystroke → cancelActiveListeners убьёт этот await const result = await api.dispatch(fetchSearch(action.payload)).unwrap() // (опционально) обновить UI через dispatch }, })

Per-id inflight flag (когда condition не хватает)

// Если у вас fetchProduct(id) — condition по loading:'pending' ломает параллельность: // fetch(1) в полёте, fetch(2) тоже отсечётся. Исправление — inflight per id. interface ProductsState { byId: Record<number, Product> inflight: Record<number, boolean> // ← новый флаг } const fetchProduct = createAsyncThunk<Product, number, { state: RootState }>( 'products/fetch', async (id) => api.getProduct(id), { condition: (id, { getState }) => !getState().products.inflight[id], }, ) // В slice reducer'ах: pending → inflight[id] = true, fulfilled/rejected → delete inflight[id]

В RTKQ

RTKQ делает ОБА варианта автоматически:
• Dedup по cacheKey: если useGetPostsQuery() вызывается из двух компонентов одновременно — сеть бьётся один раз. Кэш-ключ = endpoint + serializedArg.
• AbortController: unmount компонента или change аргумента — отменяет fetch.
Это то, что мы собираем руками через condition + listener.
