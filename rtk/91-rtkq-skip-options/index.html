Цитата — Query Hook Options

«Query hooks provide a lot of options to control their behavior. They can be passed as the second argument to each query hook, or as defaults on the endpoint definition, or globally via createApi.»
— RTKQ · Query Hook Options

Полный список

useGetXQuery(arg, { // ─── подписка ─── skip: false, // true → запроса не будет, data=undefined // ─── рефетч при mount / смене arg ─── refetchOnMountOrArgChange: false, // false | true | number (секунды) // ─── рефетч на window events ─── refetchOnFocus: false, // при window focus refetchOnReconnect: false, // при online event // ─── polling ─── pollingInterval: 0, // ms, 0 = выкл skipPollingIfUnfocused: false, // v2.x — не поллить вне фокуса // ─── оптимизация ─── selectFromResult: (r) => r, // см. урок 90 })

1. skip — не подписываться

// вариант A — boolean: const { data } = useGetUserQuery(userId, { skip: !userId }) // вариант B — skipToken (TS-дружелюбно, если arg не может быть undefined): import { skipToken } from '@reduxjs/toolkit/query' const { data } = useGetUserQuery(userId ?? skipToken) // Эффект: // - cacheKey НЕ генерируется, запроса нет // - подписчик НЕ регистрируется в state.api.subscriptions // - data = undefined, isUninitialized = true, isLoading = false // - существующий entry (от прошлого arg) остаётся до keepUnusedDataFor //

2. refetchOnMountOrArgChange — три формы

// false (default): mount → если entry fulfilled, берём cache, запроса НЕТ { refetchOnMountOrArgChange: false } // true: ВСЕГДА рефетчим на mount (игнорируем кеш) { refetchOnMountOrArgChange: true } // number — условная «свежесть»: { refetchOnMountOrArgChange: 30 } // рефетчим, если fulfilledTimeStamp старее 30 секунд // Смена arg — всегда работает одинаково: новый cacheKey → новый fetch (если нет HIT)

3. refetchOnFocus / refetchOnReconnect

// store.ts — БЕЗ ЭТОГО вызова НЕ работают: import { setupListeners } from '@reduxjs/toolkit/query' setupListeners(store.dispatch) // глобально в createApi: createApi({ refetchOnFocus: true, refetchOnReconnect: true, endpoints: ... }) // или на хуке: useGetXQuery(arg, { refetchOnFocus: true })

Реализация (setupListeners.ts): слушает window.focus и online → dispatch'ит actions api/focus и api/online → middleware для каждой подписки с включённым флагом делает forceRefetch.

4. pollingInterval — авто-рефетч по таймеру

{ pollingInterval: 5000 } // каждые 5 секунд, пока компонент подписан // если несколько подписчиков с разными интервалами — побеждает МИНИМАЛЬНЫЙ { pollingInterval: 5000, skipPollingIfUnfocused: true } // не долбить API, пока вкладка в фоне (экономим квоту/батарею)

Источник — buildMiddleware/polling.ts

// pollingInterval внутри — setInterval на подписке const handler = () => { if (skipPollingIfUnfocused && !document.hasFocus()) return api.dispatch(refetchQuery(cacheKey, { force: true })) } const timer = setInterval(handler, pollingInterval) // на unsubscribe → clearInterval

Подводные камни

1. Забыл setupListeners — refetchOnFocus молчит.
Это самая частая ошибка. Листенеры не навешиваются автоматически — нужен один раз setupListeners(store.dispatch) в store.ts. В SSR — передаёшь custom eventTargetForFocus (см. доку).

2. skipToken ≠ undefined.
useGetUserQuery(undefined) — создаёт cacheKey getUser(undefined) и делает fetch. useGetUserQuery(skipToken) — ничего не делает. Разница критична, когда сам arg может быть undefined.

3. pollingInterval: 100 — убиваешь сервер.
Polling — это брутфорс. Для реального «свежие данные как только появились» — используй onCacheEntryAdded + WebSocket (урок 94), или server-sent events.

4. refetchOnMountOrArgChange: true — больно, если компонент переоткрывается часто.
Каждое открытие = fetch. Если пользователь листает табы туда-сюда — куча запросов. Варианты: true + keepUnusedDataFor: 0 для критичных данных; или number для компромисса.

5. Polling + скрытая вкладка = батарея/квота.
С v2.x есть skipPollingIfUnfocused. Включай, если polling не критичен в фоне. Без этой опции setInterval работает всегда.

6. Опции наследуются по цепочке.
global (createApi) → endpoint (build.query) → hook (useGetX(arg, opts)). Последний уровень побеждает. Это полезно: включил refetchOnFocus: true глобально, на конкретном endpoint'е выключил через refetchOnFocus: false.
