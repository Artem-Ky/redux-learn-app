fetchedAt + condition = самый простой «expiring cache»

Идея: для каждой записи храним fetchedAt: timestamp. В condition проверяем Date.now() - fetchedAt < TTL. Если свежо — skip (кэш-хит). Если протухло — новый fetch. invalidate — просто reducer, который чистит cache или помечает записи как просроченные.

Паттерн «fetchedAt в entity»

interface Product { id: number title: string price: number fetchedAt: number // ← ключевое поле } const TTL_MS = 10_000 const fetchProduct = createAsyncThunk<Product, number, { state: RootState }>( 'products/fetch', async (id) => { const data = await api.getProduct(id) return { ...data, fetchedAt: Date.now() } }, { condition: (id, { getState }) => { const cached = getState().products.byId[id] if (!cached) return true // cache miss → fetch return Date.now() - cached.fetchedAt > TTL_MS // stale → fetch }, }, )

Три уровня свежести

// Разделение на fresh / stale / expired — полезно для UX stale-while-revalidate const FRESH_MS = 5_000 const EXPIRE_MS = 30_000 function freshness(fetchedAt: number): 'fresh' | 'stale' | 'expired' { const age = Date.now() - fetchedAt if (age < FRESH_MS) return 'fresh' // вообще не fetch if (age < EXPIRE_MS) return 'stale' // показать старое + fetch в фоне return 'expired' // ждать fetch, старое не показывать }

Per-id vs global invalidation

Per-id — invalidate one. Типовой случай: пользователь отредактировал товар, нужно перезапросить ИМЕННО этот товар. updateOne({ id, changes: { fetchedAt: 0 } }).
Global — invalidate all. Типовой случай: logout, смена языка, смена тенанта. removeAll() или пометить все как stale.
Tag-group — invalidate tag. Типовой случай: создали новый пост — инвалидируем список posts.list, но не трогаем posts.byId. Имитируется через отдельное поле lastListFetchedAt или tags: string[].

Параллельный запрос одного id (in-flight dedup)

// Проблема: 10 компонентов одновременно монтируются → 10 параллельных fetch того же товара. // condition по fetchedAt не спасёт — все 10 увидят entity=undefined и все пойдут в сеть. interface ProductsState extends EntityState<Product, number> { inflight: Record<number, boolean> } condition: (id, { getState }) => { const s = getState().products if (s.inflight[id]) return false // уже летит → не дублируем const cached = s.entities[id] if (!cached) return true return Date.now() - cached.fetchedAt > TTL_MS } // В extraReducers: pending → inflight[id]=true, fulfilled/rejected → delete inflight[id]

Что нельзя класть в store

❌ setTimeout / интервалы. Timer id — не сериализуемо, reducer perish. Таймеры держите ВНЕ store: в listener middleware или в usePoll hook.
❌ Promise / AbortController. Они не сериализуются, DevTools сломается. Храните только скаляры: fetchedAt, status, error.
❌ Date object. Immer не любит неплоские типы с методами. Date.now() возвращает number — это и кладите.

RTKQ analog

В RTKQ TTL называется keepUnusedDataFor (по умолчанию 60s) — это НЕ TTL свежести, а сколько держать данные после того, как последний подписчик отписался. Свежесть в RTKQ — это refetchOnMountOrArgChange: number | true + refetchOnFocus + refetchOnReconnect. Invalidation — через tagTypes + providesTags / invalidatesTags. Всё то, что мы сейчас собираем руками через fetchedAt и invalidateAll, RTKQ делает декларативно.
