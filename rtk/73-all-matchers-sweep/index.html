Исходник — isAnyOf и isAllOf

// packages/toolkit/src/matchers.ts, строки 22-62 const matches = (matcher: Matcher<any>, action: any) => { if (hasMatchFunction(matcher)) { return matcher.match(action) // action creator имеет .match } else { return matcher(action) // иначе — просто функция-предикат } } export function isAnyOf<Matchers extends Matcher<any>[]>( ...matchers: Matchers ) { return (action: any): action is ActionMatchingAnyOf<Matchers> => { return matchers.some((matcher) => matches(matcher, action)) } } export function isAllOf<Matchers extends Matcher<any>[]>( ...matchers: Matchers ) { return (action: any): action is ActionMatchingAllOf<Matchers> => { return matchers.every((matcher) => matches(matcher, action)) } }

Исходник — isPending

// packages/toolkit/src/matchers.ts, строки 128-140 export function isPending<AsyncThunks>(...asyncThunks: AsyncThunks | [any]) { if (asyncThunks.length === 0) { // ни одного thunk'а — смотрим по meta.requestStatus === 'pending' return (action: any) => hasExpectedRequestMetadata(action, ['pending']) } if (!isAsyncThunkArray(asyncThunks)) { // 1 аргумент и это не thunk — значит тебе передали прямо action return isPending()(asyncThunks[0]) } // Здесь asyncThunks — это [thunk1, thunk2, …]. Берём у каждого .pending и объединяем в isAnyOf. return isAnyOf(...asyncThunks.map((asyncThunk) => asyncThunk.pending)) }

То есть isPending(fetchA, fetchB) ≡ isAnyOf(fetchA.pending, fetchB.pending). isFulfilled, isRejected устроены точно так же — у них та же «три ветки» логика.

Исходник — hasExpectedRequestMetadata

// packages/toolkit/src/matchers.ts, строки 70-81 export function hasExpectedRequestMetadata( action: any, validStatus: readonly string[], ) { if (!action || !action.meta) return false const hasValidRequestId = typeof action.meta.requestId === 'string' const hasValidRequestStatus = validStatus.indexOf(action.meta.requestStatus) > -1 return hasValidRequestId && hasValidRequestStatus }

Именно поэтому «thunk action» опознаётся даже без ссылки на сам thunk — любой action с meta.requestId: string и meta.requestStatus: 'pending'|'fulfilled'|'rejected' считается thunk-action'ом. Если ты вручную соберёшь такой action, matchers его поймают — feature.

isRejectedWithValue — тонкость

// packages/toolkit/src/matchers.ts, строки 237-253 export function isRejectedWithValue(...asyncThunks) { const hasFlag = (action: any) => { return action && action.meta && action.meta.rejectedWithValue // 🔑 специальный флаг } if (asyncThunks.length === 0) { return isAllOf(isRejected(...asyncThunks), hasFlag) } if (!isAsyncThunkArray(asyncThunks)) { return isRejectedWithValue()(asyncThunks[0]) } return isAllOf(isRejected(...asyncThunks), hasFlag) }

rejectWithValue(...) проставляет meta.rejectedWithValue = true. Это позволяет отличить «thunk решил вернуть ошибку с payload» от «throw Error в payload-creator'е». Последний не помечается флагом — значит isRejectedWithValue его не поймает.

Подводные камни matchers:
  • isAllOf для одного action срабатывает редко. Он требует, чтобы все предикаты приняли один и тот же action. isAllOf(fetchA.pending, fetchB.pending) никогда не вернёт true — action может быть либо A.pending, либо B.pending, но не обоими сразу. isAllOf оправдан только для ортогональных предикатов: isAllOf(isFulfilled(), (a) => a.payload.userId === 42).
  • isPending() без аргументов ловит любой pending от любого thunk'а. Удобно для глобального loading'а — но также ловит thunk'и, которые ты не хотел тут обрабатывать. Если в проекте 30 thunk'ов и ты хочешь «мой локальный loader» только на 2 — пиши isPending(thunk1, thunk2).
  • isRejected ловит и throw, и rejectWithValue. Если тебе нужен именно «контролируемый reject» (с payload, типизированный) — isRejectedWithValue. Обычный isRejected ловит оба случая, но payload/error-поля у них разные: throw → action.error.message, rejectWithValue → action.payload.
  • Порядок addMatcher важен. Reducer проходит matchers по порядку. Если isAsyncThunkAction() поставить первым, он перехватит все thunk-экшены — следующие isFulfilled/isPending тоже сработают (не эксклюзивно!), но если там мутации state'а перекрывают друг друга — получишь неожиданный результат. Ставь «широкие» matchers последними.
  • Matchers через .match vs функция-предикат. RTK распознаёт matcher по наличию .match (это свойство action creator'а) через hasMatchFunction. Поэтому в isAnyOf можно смешивать: isAnyOf(fetchA.pending, myCustomPredicate, inc). Но самописный предикат теряет типизацию payload'а — результирующий тип будет any.
