Теория — 83. cacheKey и serializeQueryArgs

Цитата — Customizing Queries

«Each endpoint has a concept of a cacheKey, which is by default derived from a combination of the endpoint name and a serialized form of the query arguments. RTK Query uses this cacheKey to determine whether to perform a fresh request or return cached data.»
— RTKQ — serializeQueryArgs
«Ключ кэша — это endpointName + сериализованные args. По этому ключу RTKQ решает: вернуть из кэша или идти в сеть.»

Исходник — defaultSerializeQueryArgs

// @reduxjs/toolkit/src/query/defaultSerializeQueryArgs.ts export const defaultSerializeQueryArgs: SerializeQueryArgs<any> = ({ endpointName, queryArgs, }) => { let serialized = '' const cached = cache?.get(queryArgs) if (typeof cached === 'string') { serialized = cached } else { const stringified = JSON.stringify(queryArgs, (_, value) => isPlainObject(value) ? Object.keys(value) .sort() // 👀 сортировка ключей объекта .reduce<any>((acc, key) => { acc[key] = (value as any)[key] return acc }, {}) : value, ) if (isPlainObject(queryArgs)) cache?.set(queryArgs, stringified) serialized = stringified } return `${endpointName}(${serialized})` // 👀 финальный формат }

GitHub: defaultSerializeQueryArgs.ts

Примеры cacheKey

// Примитив: useGetPokemonByNameQuery('pikachu') // → cacheKey = getPokemonByName("pikachu") // Объект — ключи сортируются: useSearchUsersQuery({ page: 1, query: 'ann', sortBy: 'name' }) useSearchUsersQuery({ sortBy: 'name', page: 1, query: 'ann' }) // → ОДИН cacheKey: searchUsers({"page":1,"query":"ann","sortBy":"name"}) // Разные значения — разные ключи: useSearchUsersQuery({ page: 1, query: 'ann' }) useSearchUsersQuery({ page: 2, query: 'ann' }) // → ДВА cacheKey, два запроса, два cache-entry // void / undefined: useGetAllPokemonsQuery() // → cacheKey = getAllPokemons(undefined) — один для всех "безаргументных"

Кастомный serializeQueryArgs — когда менять

// Пример 1 — игнорим "transient" фильтры (UI-only): getPosts: build.query<Post[], GetPostsArgs>({ query: (args) => ({ url: 'posts', params: args }), serializeQueryArgs: ({ endpointName, queryArgs }) => { const { _localSort, ...keep } = queryArgs return `${endpointName}(${JSON.stringify(keep)})` }, }) // Пример 2 — один общий cache (пагинация в памяти): getInfiniteFeed: build.query<Feed, { page: number }>({ query: ({ page }) => `feed?page=${page}`, serializeQueryArgs: ({ endpointName }) => endpointName, // 👀 единственный ключ merge: (current, incoming) => { current.items.push(...incoming.items) }, forceRefetch: ({ currentArg, previousArg }) => currentArg?.page !== previousArg?.page, })

Пример 2 — это, по сути, основа build.infiniteQuery (урок 100).

Подводные камни

1. Новый объект каждый рендер → бесполезно, но НЕ ломает cache.
useGetX({ foo: 'bar' }) в компоненте каждый рендер создаёт новый объект-аргумент. Но RTKQ сначала сериализует (строка одинаковая) и только потом сверяет с кэшем — поэтому реально cache reuse работает. Но: useQuery hooks внутри делают useMemo по референсу args → при reference-change происходит лишний job по re-subscribe. Паттерн: оборачивать args в useMemo или держать на верхнем уровне.

2. Непримитивные / не-plain-object args.
Date, Map, class instance — не-plain. Их JSON.stringify не сортирует (функция sort-reviver работает только для plain-object). Один и тот же new Date('2024-01-01') в двух рендерах → одинаковая строка, но НЕ передавай экзотики: serializability middleware тоже ругнётся.

3. Порядок массивов ВАЖЕН.
searchUsers({ ids: [1, 2] }) и searchUsers({ ids: [2, 1] }) — два разных cacheKey. Если логически один и тот же запрос — передавай отсортированный массив или пиши свой serializeQueryArgs.

4. skipToken ≠ args.
Когда arg === skipToken, cache НЕ создаётся (запроса не было). cacheKey не генерируется. data остаётся от последнего реального arg.

5. Вложенные объекты — сортируются рекурсивно?
Да, reviver применяется к каждому plain-object рекурсивно (через JSON.stringify). Но если у тебя { filter: new URLSearchParams(...) } — внутрь reviver не зайдёт (не plain).
