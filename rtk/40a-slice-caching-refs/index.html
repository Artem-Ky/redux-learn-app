Теория — 40a. Что такое «кеш» в RTK slice на уровне ссылок

Почему это вообще работает

Redux-store хранит state как один объект. При каждом dispatch корневой reducer (combineSlices/combineReducers) вызывает slice-reducers поочерёдно. Если slice ничего не сделал — он возвращает ту же ссылку. Redux тогда знает, что state-дерево в этой ветке не поменялось. А благодаря Immer.produce это бесплатно даже для сложных вложенных объектов.

Цитата: документация Immer

«Immer produces immutable state by creating the next immutable state tree. The next tree is structurally shared with the previous tree, so no unnecessary copying occurs: only the paths that have changed are written to; the rest are reused.»
— immerjs.github.io/immer
«Immer создаёт следующее неизменяемое дерево состояния. Новое дерево разделяет структуру с предыдущим: ничего лишнего не копируется — перезаписываются только изменённые пути, остальное переиспользуется.»

Цитата: документация Redux

«Reducers must always return a new state value if the state has changed. If no changes are needed, return the existing state object as-is. Returning the same reference is how we tell Redux ‟nothing changed”.»
— redux.js.org/style-guide
«Reducer должен вернуть новое state-значение, только если state действительно изменился. Если изменений нет — верни существующий объект как есть. Возврат той же ссылки — это сигнал Redux'у «ничего не поменялось».»

Исходник RTK: createReducer → createNextState (Immer)

// @reduxjs/toolkit/src/createReducer.ts (строки 180–216) // https://github.com/reduxjs/redux-toolkit/blob/master/packages/toolkit/src/createReducer.ts return caseReducers.reduce((previousState, caseReducer): S => { if (caseReducer) { if (isDraft(previousState)) { // already a draft — просто мутируем const draft = previousState as Draft<S> const result = caseReducer(draft, action) if (result === undefined) return previousState return result as S } else if (!isDraftable(previousState)) { // примитив (0, "", false) — обходим Immer const result = caseReducer(previousState, action) if (result === undefined) return previousState return result as S } else { // 👀 главный случай: оборачиваем caseReducer в Immer.produce return createNextState(previousState, (draft: Draft<S>) => { return caseReducer(draft, action) }) } } return previousState }, state)

createNextState — это псевдоним для immer.produce. Ключевое свойство: если внутри caseReducer draft не модифицировался — produce возвращает ту же ссылку, что была на входе. Именно поэтому «noop reducer» → same ref.

Схема structural sharing

Что это даёт на практике

УровеньМеханизмНа чём держится
reducer Immer сохраняет ссылки неизменённых веток produce(prev, draft => ...)
store subscribers дёргаются всегда, но они могут сами сравнить prev === next нет авто-оптимизации на уровне Redux core
useSelector skip re-render, если selector(state) равен === refEquality по умолчанию
createSelector skip пересчёт combiner, если все входы равны === lruMemoize(1) + ссылки из Immer
entityAdapter selectAll закешировал массив, пока не добавили/удалили сущность createDraftSafeSelector над ids и entities

Что происходит при dispatch: 3 маршрута

// 1. reducer не match'ится ни одним case → slice возвращает prev as-is case 'UNKNOWN_ACTION': return state // same ref // 2. case match, но Immer draft не изменён → produce вернёт same ref createNextState(prev, (draft) => { if (!action.payload) return // noop path draft.name = action.payload // но если payload === prev.name, // Immer заметит «writes equal reads» // и вернёт prev (same ref) }) // 3. draft был реально изменён → новый объект на пути от корня createNextState(prev, (draft) => { draft.name = 'Alice' }) // → new ref

Подводные камни

1. Ручной return без Immer ломает structural sharing. Если ты пишешь return { ...state, user: { ...state.user, name: 'x' } } в reducer без Immer и меняешь только user, ты всё равно получишь новый state (это ожидаемо), и новый state.user (тоже ок). Но если случайно напишешь return { ...state, posts: [...state.posts] } — создашь новый массив posts, хотя не менял его.
Почему это плохо: все селекторы на state.posts получат cache miss и пересчитают. Решение: не трогай то, что не меняешь. С createSlice/Immer это бесплатно.
2. Присваивание того же значения — Immer может заметить, а может и нет. state.name = action.payload где payload === state.name — Immer обнаружит «write equals read» и вернёт ту же ссылку только для примитивов. Для объектов (state.user = { name: 'Alice' }) Immer видит новый объект и создаёт новый draft.
Почему так: Immer не делает deep equality — это было бы медленно. Решение: проверяй сам перед присваиванием: if (state.user.name !== payload) state.user.name = payload.
3. Subscribe дёргается на КАЖДЫЙ dispatch, даже NOOP. Redux core не фильтрует подписчиков: store.subscribe(fn) вызовет fn после любого action, даже если state — та же ссылка. Фильтрация — ответственность подписчика.
Почему: Redux не знает, чем интересуется подписчик, — может, он следит не за state, а за числом dispatch'ей. Решение: useSelector делает === на результат селектора, а не на state; autoBatchEnhancer батчит много dispatch'ей в один тик.
4. Идентичный selector output, но по ссылке — разный. state => state.user — хорошо (ссылка та же, пока user не меняется). state => ({ name: state.user.name }) — плохо: каждый вызов даёт новый объект, react-redux видит !==, перерисовывает компонент.
Почему: ссылочное сравнение не смотрит содержимое. Решение: shallowEqual вторым аргументом useSelector, либо createSelector с combiner'ом.
5. Extra reducers на slice не видят Immer-draft родителя. Если ты делаешь extraReducers(builder => builder.addCase(other.action, (state, action) => ...)), твой state — это draft только этого slice. Менять state — OK (Immer знает). Но не пытайся вернуть из reducer'а ссылку из другого slice — она в другом draft-tree.
Почему: combineSlices вызывает produce per-slice, не глобально.
