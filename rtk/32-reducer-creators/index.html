3 типа reducers через create-callback

В RTK 2.x появился альтернативный синтаксис: reducers: (create) => ({...}). Он даёт три helper'а: create.reducer, create.preparedReducer, create.asyncThunk. Главное преимущество — async thunk прямо в slice'е, без отдельного createAsyncThunk и extraReducers.

Чтобы использовать create.asyncThunk, нужно создать свой createAppSlice через buildCreateSlice({ creators: { asyncThunk: asyncThunkCreator } }). Это сделано для того, чтобы не пулить createAsyncThunk в bundle, если он вам не нужен.
import { buildCreateSlice, asyncThunkCreator, nanoid } from '@reduxjs/toolkit' export const createAppSlice = buildCreateSlice({ creators: { asyncThunk: asyncThunkCreator }, }) const todosSlice = createAppSlice({ name: 'todos', initialState: { items: [], loading: false, error: null }, reducers: (create) => ({ // 1. create.reducer — простой case reducer deleteTodo: create.reducer<string>((state, action) => { state.items = state.items.filter(t => t.id !== action.payload) }), // 2. create.preparedReducer — с prepare callback addTodo: create.preparedReducer( (text: string) => ({ payload: { id: nanoid(), text, done: false } }), (state, action) => { state.items.push(action.payload) } ), // 3. create.asyncThunk — full thunk inside slice! fetchTodos: create.asyncThunk( async (_, thunkApi) => { const r = await fetch('/api/todos') return r.json() }, { pending: (state) => { state.loading = true }, fulfilled: (state, action) => { state.items = action.payload }, rejected: (state, action) => { state.error = action.error.message }, settled: (state) => { state.loading = false }, } ), }) })

Теория — reducer creators (RTK 2.x)

Из доки

«Alternatively, the reducers field can be a callback which receives a "create" object. The main benefit of this is that you can create async thunks as part of your reducers.»
— Reducers creator callback
«Альтернативно, поле reducers может быть callback'ом, который получает объект "create". Главное преимущество — можно создавать async thunk'и как часть reducers.»

3 helper'а в create

Helper Эквивалент в старом API
create.reducer<P>((s, a) => …) обычная функция в reducers
create.preparedReducer(prepare, reducer) { reducer, prepare } object form
create.asyncThunk(payloadCreator, handlers) createAsyncThunk + extraReducers

buildCreateSlice — почему отдельная фабрика

import { createSlice } from '@reduxjs/toolkit' // Этот createSlice не знает про asyncThunk — bundle меньше createSlice({ name: 'x', reducers: (create) => ({ fetchUser: create.asyncThunk(...) // ← Error! asyncThunk creator not configured }) }) // Нужно собрать свой: import { buildCreateSlice, asyncThunkCreator } from '@reduxjs/toolkit' export const createAppSlice = buildCreateSlice({ creators: { asyncThunk: asyncThunkCreator } }) // Теперь работает createAppSlice({ name: 'x', reducers: (create) => ({ fetchUser: create.asyncThunk(...) }) })

Это паттерн "конфигурируй и используй". В большом приложении делаете один createAppSlice на проект и используете везде вместо createSlice.

create.asyncThunk — handlers объект

create.asyncThunk( payloadCreator, // тот же что в createAsyncThunk { pending: (state, action) => { ... }, fulfilled: (state, action) => { ... }, rejected: (state, action) => { ... }, settled: (state, action) => { ... }, // вызывается на fulfilled И rejected options: { idGenerator: () => customId(), condition: (arg, api) => { ... }, dispatchConditionRejection: true, } } )

settled — удобный helper, эквивалент addMatcher для обоих fulfilled+rejected. Часто используется чтобы выключить loading-flag.

Доступ к thunk action creator

// Slice автоматически экспортирует action creator вместе с pending/fulfilled/rejected todosSlice.actions.fetchTodos // сам thunk todosSlice.actions.fetchTodos.pending // pending action creator todosSlice.actions.fetchTodos.fulfilled // fulfilled action creator todosSlice.actions.fetchTodos.rejected // rejected action creator // В компоненте: dispatch(todosSlice.actions.fetchTodos())

Подводный камень: типизация state в asyncThunk

Внутри payloadCreator thunkApi.getState() возвращает unknown, потому что slice ещё не построен (циклическая типизация). Нужно делать ручной cast: const state = thunkApi.getState() as RootState.

Когда использовать новый callback API

  • Новые проекты RTK 2.x — да, рекомендуется.
  • Перевод slice'а с asyncThunk внутри — да, удобнее (всё в одном месте).
  • Простые slice без thunk — старый object-form тоже хорош. Дело вкуса.
  • Тесты — новый API создаёт ту же структуру; тесты не меняются.
