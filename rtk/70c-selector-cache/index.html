Когда === сломан — спасает createSelector

В 70b мы видели, что Immer сохраняет ссылки → === работает. Но как только селектор трансформирует данные (filter / map / sort / merge), результат — новый объект каждый раз. Даже если исходные данные не менялись. Компоненты ререндерят впустую.

createSelector из reselect добавляет кэш по ссылкам на inputs: если все input-селекторы вернули === те же значения — не пересчитываем, возвращаем закэшированный output.

Naive vs memoized — один и тот же вычисляемый output

selectAdminsNaive(s) = s.users.filter(...).map(...)
0
— last
selectAdminsMemo — createSelector
0
— last

«runs» = сколько раз функция вычислила результат. «last» = сколько было cache hits vs real runs в последнем пакете чтений.

Теория — 70c. Что именно кэширует reselect

Принцип

const selectAdminsMemo = createSelector( [(s: RootState) => s.users], // ← input-селекторы (users) => users // ← combiner — ВЫЧИСЛИТ результат .filter(u => u.role === 'admin') .map(u => u.name), ) // reselect хранит: last inputs[], last output. // На каждом вызове: // 1. Запустить input-селекторы → получить inputs'[] // 2. Сравнить inputs' с last inputs по === (default equality). // 3. Если все равны — вернуть last output (cache hit). // 4. Иначе — вызвать combiner, сохранить inputs' и output.

Когда создавать createSelector

Селектор Нужен createSelector?
s => s.users Нет — уже === стабильный.
s => s.user.name Нет — примитив.
s => s.users.map(u => u.name) ДА — новый массив каждый раз.
s => ({ a: s.x.a, b: s.y.b }) ДА — inline object.
s => s.todos.filter(t => !t.done).length Число — не обязательно, но вычисление тяжёлое → да.

Параметризованные селекторы — ловушка

createSelector по умолчанию — lruMemoize size 1. Это значит один слот кэша.
Если у вас компонент <BookById id={id} /> × 100 штук, каждый дёргает selectBookById(s, id) с разным id — кэш промахивается на каждом вызове. Решение: createSelector с memoize: weakMapMemoize (см. урок 43). Это дерево кэшей по WeakMap'ам — каждый id свой слот.

Размер кэша — аргумент memoizeOptions

import { createSelector, lruMemoize, weakMapMemoize } from '@reduxjs/toolkit' // ① lru size N — фиксированный размер, вытесняет старые const selectTopN = createSelector( [(s: RootState) => s.posts, (_: RootState, n: number) => n], (posts, n) => posts.slice(0, n), { memoize: lruMemoize, memoizeOptions: { maxSize: 10 } }, ) // ② weakMap — авто-GC, подходит когда ключи — объекты const selectByEntity = createSelector( [(s: RootState) => s.data, (_: RootState, entity: Entity) => entity], (data, entity) => data.filter(d => d.owner === entity), { memoize: weakMapMemoize }, )

Ключевое отличие от Layer 1

Layer 1 (structural sharing): «вернуть ту же ссылку если state не трогали» — пассивно, встроено.
Layer 2 (memoized selector): «вернуть тот же вычисленный результат если inputs === те же» — активно, настраивается.
Без Layer 1 Layer 2 бесполезен: если state.users меняет ссылку на каждый dispatch (даже когда содержимое то же) — memoize всегда будет промахиваться.

В RTKQ

RTKQ использует Layer 2 внутри — selectFromResult как раз строит createSelector под капотом. Внешние memoized селекторы над state.api.queries пишутся точно так же, как над обычным state.
