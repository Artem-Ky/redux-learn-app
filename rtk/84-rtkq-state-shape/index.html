Теория — 84. Анатомия state.api

Цитата — Usage with TypeScript

«RTK Query uses a single "API slice" per base API. Under the hood, this slice is actually composed of five separate sub-slices, combined via combineReducers: queries, mutations, provided, subscriptions, and config.»
— RTK source, buildSlice.ts
«Один api-slice внутри — это combineReducers из 5 мини-slice'ов. Ты никогда не пишешь их руками, но понимать, какое поле за что отвечает — полезно для отладки, для селекторов и для extraReducers, которые слушают api-actions.»

Исходник — combineReducers в buildSlice.ts

// @reduxjs/toolkit/src/query/core/buildSlice.ts — строки ~712 const combinedReducer = combineReducers({ queries: querySlice.reducer, // { [cacheKey]: QuerySubState } mutations: mutationSlice.reducer, // { [requestId]: MutationSubState } provided: invalidationSlice.reducer, // tagTypes → cacheKeys subscriptions: internalSubscriptionsSlice.reducer, config: configSlice.reducer, // online, focused, keepUnusedDataFor, ... }) const reducer: typeof combinedReducer = (state, action) => combinedReducer(resetApiState.match(action) ? undefined : state, action) // 👀 api.util.resetApiState() = state=undefined → combineReducers пересоздаёт всё initialState

Исходник — типы под-state'ов (apiState.ts)

// @reduxjs/toolkit/src/query/core/apiState.ts export type CombinedState<D, E, ReducerPath> = { queries: QueryState<D> // { [cacheKey]: QuerySubState | undefined } mutations: MutationState<D> // { [requestId]: MutationSubState | undefined } provided: InvalidationState<E> subscriptions: SubscriptionState // { [cacheKey]: Subscribers | undefined } config: ConfigState<ReducerPath> } export type InvalidationState<TagTypes> = { tags: { [_ in TagTypes]: { [id: string | number]: QueryCacheKey[] } } keys: Record<QueryCacheKey, FullTagDescription<any>[]> } export type ConfigState<ReducerPath> = { reducerPath: ReducerPath online: boolean focused: boolean middlewareRegistered: boolean | 'conflict' keepUnusedDataFor: number // default 60 (секунд) invalidationBehavior: 'delayed' | 'immediately' refetchOnMountOrArgChange: boolean | number refetchOnReconnect: boolean refetchOnFocus: boolean }

Где поля вообще — и когда на них смотреть

под-state индекс когда смотреть
queries cacheKey видеть, что закэшировано; проверять status pending/fulfilled/rejected
mutations requestId (или fixedCacheKey) отслеживать прогресс mutation; fixedCacheKey = чтобы несколько компонентов делили одну mutation
provided { tags, keys } дебаг invalidations; обратный индекс «тег → cacheKeys»
subscriptions cacheKey → requestId → options refCount живых подписчиков; после последнего unsubscribe запускается keepUnusedDataFor таймер
config — online/focused от setupListeners; middlewareRegistered — отладка «забыл concat»

Подводные камни

1. subscriptions НЕ входит в DevTools state по умолчанию.
Начиная с RTK 2.x, internalSubscriptionsSlice не попадает в Redux DevTools state-снимки — это сделано, чтобы time-travel не пытался восстановить «живые подписки». Если хочешь увидеть реальные subscribers — бери store.getState().api.subscriptions напрямую (как в этом демо) или используй api.internalActions.subscriptionsUpdated.

2. mutations индексируются по requestId, не по arg.
Повторный addPost({title}) с тем же telom создаёт новую запись с новым requestId. Если хочешь «один общий статус mutation» для нескольких компонентов — передавай fixedCacheKey: 'addPost' в .initiate(arg, { fixedCacheKey }).

3. provided.keys и provided.tags — денормализованные обе стороны.
RTKQ хранит ОБА индекса: tag → cacheKeys (для invalidate) и cacheKey → tags (для cleanup при удалении cache entry). Если напрямую мутировать только одну сторону через extraReducers — сломаешь invalidation.

4. config.middlewareRegistered = false → silent bug.
Если забыл concat(api.middleware), в dev-режиме появится warning, но в prod — просто тишина. Запросы будут висеть в pending вечно. Проверяй это поле в integration-тестах (см. урок 81).

5. resetApiState() — reducer=undefined, НЕ clear каждого slice по отдельности.
В обработке action {type: 'api/resetApiState'} RTKQ передаёт state=undefined в combineReducers → каждый sub-slice возвращает свой initialState. Это значит: текущие запросы в queries исчезают, но РЕАЛЬНЫЕ HTTP-запросы (AbortController) отдельно не abortятся. Используй initiate().unsubscribe() перед reset.
