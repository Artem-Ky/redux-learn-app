Структура ошибки

// fetchBaseQuery возвращает FetchBaseQueryError: type FetchBaseQueryError = | { status: number; data: unknown } // HTTP 4xx/5xx | { status: 'FETCH_ERROR'; error: string } // network fail | { status: 'PARSING_ERROR'; data: string; ... } // invalid JSON | { status: 'TIMEOUT_ERROR'; error: string } // timeout опция | { status: 'CUSTOM_ERROR'; error: string; data } // validateStatus вернул false // Кастомный baseQuery может возвращать SerializedError — стандартный Error Redux.

retry — встроенный wrapper

import { retry, fetchBaseQuery } from '@reduxjs/toolkit/query/react' const baseQueryWithRetry = retry( fetchBaseQuery({ baseUrl: '/api' }), { maxRetries: 3 }, ) const api = createApi({ baseQuery: baseQueryWithRetry, endpoints: (build) => ({ getPosts: build.query({ query: () => 'posts', // переопределить на уровне endpoint'а: extraOptions: { maxRetries: 5 }, }), }), })

Отказаться от retry для конкретного ответа

// Если 401 — нет смысла ретраить, токен не подменится сам const baseQueryWithRetry = retry(async (args, api, extraOptions) => { const result = await rawBase(args, api, extraOptions) if (result.error?.status === 401) { retry.fail(result.error) // 👀 bail out немедленно } return result }, { maxRetries: 3 })

Глобальный error logger через middleware

import { isRejectedWithValue } from '@reduxjs/toolkit' import type { Middleware } from '@reduxjs/toolkit' const errorLogger: Middleware = () => (next) => (action) => { if (isRejectedWithValue(action)) { // action.payload — это error от transformErrorResponse (или raw) toast.error(`API error: ${action.payload.status}`) } return next(action) } configureStore({ ..., middleware: (gdm) => gdm().concat(api.middleware, errorLogger), })

Подводные камни

1. retry — exponential backoff с джиттером.
Дефолтный backoff — Math.random() * 2^attempt * 300ms (примерно). Это ок для throttling-ловли, но если тебе нужен предсказуемый — передавай свой backoff.

2. retry ретраит и 4xx по умолчанию.
Это редко то что ты хочешь (404 ретраить = 3 лишних запроса за один 404). Используй retry.fail(error) для 4xx.

3. status может быть строкой.
Не делай if (error.status >= 500) — сломается на 'FETCH_ERROR'. Сначала typeof error.status === 'number'.

4. transformErrorResponse не видит retries.
Она применяется к финальной ошибке после всех retry. Если хочешь логгировать промежуточные попытки — это в кастомном baseQuery.

5. Failed query хранится в cache с error, data может быть от прошлого успеха.
Это «stale-while-error». UI решает: показывать старые данные с баннером «обновить не удалось» или полностью прятать.

6. Refetch на error — делай refetch(), не меняй arg.
q.refetch() — семантически правильно. Менять arg (setTick(t+1)) — хак, создаст новый cacheKey и размножит entries.
