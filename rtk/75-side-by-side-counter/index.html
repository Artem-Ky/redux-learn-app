Три независимых store работают параллельно

Один и тот же counter реализован тремя способами. Classic — createStore + ручной reducer. RTK — configureStore + createSlice. RTK Query — createApi с mutation к mock-API (setTimeout 300 мс вместо сети). У каждого — свой store; в DevTools внизу показан именно RTK store (ключевой для секции), но все три работают в памяти.

Почему counter в RTK Query — это «перебор»

Counter — это классический пример client state: он принадлежит UI, не существует на сервере. RTK Query создан для server state: GET/POST/PUT/DELETE, cache по параметрам, invalidation по тегам. Использовать mutation для +1 — это как забивать гвозди микроскопом. Здесь это сделано только для демонстрации формы API — на реальном проекте counter — это createSlice, а RTK Query держите для /api/todos, /api/users/:id и подобного.

Что видно из трёх фрагментов

  • Classic — 3 action types руками, switch в reducer, spread для immutability, явный createStore.
  • RTK — одна createSlice, Immer внутри, configureStore автоматически подключает DevTools и thunk.
  • RTK Query — нет явных action types; createApi сам генерирует reducer, middleware, actions (pending/fulfilled/rejected) и хуки. Cache живёт в state.counterApi.

Manual subscription в RTK Query (без React hooks)

Хуки useXxxMutation — это фасад над store.dispatch(api.endpoints.X.initiate(arg)). В этом уроке мы используем именно dispatch — чтобы показать, что RTK Query — это обычный Redux store, а React — просто одна из оболочек. Подписка на результат — через store.subscribe и api.endpoints.X.select(arg)(state).

Подводные камни

Classic Redux → много boilerplate. Action types, action creators, switch в reducer, spread для immutability — всё вручную. Для любой фичи нужно править 3–5 файлов. Именно поэтому RTK появился.
RTK Query — не замена createSlice. Для чисто клиентского state (модалки, темы, формы, счётчики) используйте createSlice. RTK Query — специально для server state, где нужны cache, dedup, invalidation, polling. Применение RTKQ к локальному counter (как здесь) — демо, а не рекомендация.
Три разных store — редкий случай. В реальном приложении обычно один configureStore, в котором соседствуют обычные slice'ы и api.reducer. Разделение в уроке — только для наглядности.
