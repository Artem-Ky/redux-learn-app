Теория — 68. TaskAbortError, runTask и takeLatest

TaskAbortError как единый объект отмены

«Listeners may fork long-running tasks. These tasks may be cancelled by the user or by the parent listener. TaskAbortError is thrown to signal the cancellation. Any awaited delay, pause, or take will throw this error.»
— RTK docs — cancellation
«Listener'ы могут форкать долгие задачи. Они отменяются пользователем или самим родителем. TaskAbortError сигнализирует об отмене — любой await на delay/pause/take получит именно этот объект.»

Рецепт debounce+takeLatest для поиска

listener.startListening({ actionCreator: searchTermChanged, effect: async (action, api) => { api.cancelActiveListeners() // 1. убираем предыдущих await api.delay(300) // 2. debounce: ждём 300мс // если за это время пришёл новый ввод — cancelActive // нас уже уронил через TaskAbortError try { const res = await fetch(`/api?q=${action.payload}`, { signal: api.signal, // 3. abort'им сам fetch }) const json = await res.json() api.dispatch(searchResults(json)) } finally { // 4. finally, не catch! TaskAbortError должен пропагироваться. // catch-нутый error сломает поток — тихо проглотит отмену. } }, })
Два коварных подводных камня cancellation:
  • Fetch сам по себе не отменяется. api.delay(300) бросит TaskAbortError, но если вы уже дошли до await fetch(...) — запрос уйдёт. Передавайте signal: api.signal в options: fetch(url, { signal: api.signal }). AbortController сам прервёт network и бросит DOMException AbortError.
  • try/catch вместо try/finally проглатывает TaskAbortError. Если после catch вы делаете dispatch(searchResults(...)), то отменённый запрос всё равно обновит state — race condition. Используйте finally, а результат ловите отдельно; если нужен catch — явно пробрасывайте: if (e instanceof TaskAbortError) throw e.
