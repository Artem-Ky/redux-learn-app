name = action prefix, reducerPath = ключ в state

В обычных slice'ах эти два поля совпадают (по умолчанию reducerPath = name). Различие важно для: 1) RTK Query (всегда reducerPath='api'), 2) сложных приложений с переименованием slice'ов без изменения action.type, 3) lazy-injection.

const usersSlice = createSlice({ name: 'users', // для action.type = 'users/add', 'users/remove' reducerPath: 'data/users', // (опц.) ключ в state — state['data/users'] initialState: { items: [] }, reducers: { ... } }) const root = combineSlices(usersSlice) // state['data/users'].items // Action.type всё ещё 'users/add' — независимо от reducerPath
Поле Что задаёт Когда меняется отдельно
name Префикс action.type: ${'`'}${'$'}{name}/${'$'}{reducerKey}${'`'} Никогда (это идентификатор slice'а)
reducerPath Ключ в root state: state[reducerPath] RTK Query, переименование, sharing slice между несколькими store

Теория — name vs reducerPath

Из доки

«reducerPath defaults to name. Use it when you want the slice's state to be stored under a different key from its action prefix. combineSlices uses reducerPath to determine the state key.»
— createSlice — reducerPath
«reducerPath по умолчанию равен name. Используйте, если хотите хранить state slice'а под другим ключом, отличным от префикса actions. combineSlices использует reducerPath для определения ключа state.»

В RTK Query — обязательное использование

const apiSlice = createApi({ reducerPath: 'api', // state.api baseQuery: fetchBaseQuery({ baseUrl: '/' }), endpoints: ... }) const store = configureStore({ reducer: { [apiSlice.reducerPath]: apiSlice.reducer, // state.api = apiSlice.reducer }, middleware: (gdm) => gdm().concat(apiSlice.middleware) }) // Все RTK Query actions имеют префикс 'api/...': // 'api/executeQuery/pending' // 'api/executeMutation/fulfilled' // и т.д.

Когда полезно разделять name и reducerPath

  1. Миграция: переименовали key в state, но не хотите менять action.type (старые middleware на них завязаны).
  2. Несколько копий slice: один и тот же createSlice используется в разных ветках state (но с разным reducerPath).
  3. RTK Query: name внутри createApi не настраивается, а reducerPath — да, и это разные вещи.

Деривация типа RootState

type RootState = ReturnType<typeof store.getState> // => { users: ..., counter: ..., 'data/users': ... } // ↑ ключи берутся из reducerPath // А action.type всё ещё: usersSlice.actions.add.type // 'users/add' (от name)

Подводный камень: ключ со слэшем

reducerPath: 'data/users' создаст ключ state['data/users'] со слэшем в имени. Это валидный ключ JS, но затрудняет работу — приходится использовать state['data/users'] вместо state.dataUsers. Лучше избегать спецсимволов.

Подводный камень: переопределение в configureStore

Если вы передаёте slice через объект-map в configureStore с другим ключом: configureStore({ reducer: { myKey: usersSlice.reducer } }) — это переопределит reducerPath для целей state ключа, но slice.selectors и slice.selectSlice всё равно будут смотреть в state[reducerPath] и сломаются. Используйте combineSlices или slice.getSelectors(rootSelector).
