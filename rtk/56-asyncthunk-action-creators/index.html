Три формы в одном builder

extraReducers: (b) => { // 1. Точечно на конкретный thunk b.addCase(fetchUsers.pending, (s) => { s.users.loading = true }) b.addCase(fetchUsers.fulfilled, (s, a) => { s.users.list = a.payload }) b.addCase(fetchUsers.rejected, (s, a) => { s.users.error = a.error.message }) // 2. Matcher для НЕСКОЛЬКИХ thunk'ов — общий loader b.addMatcher(isPending(fetchUsers, fetchPosts), (s) => { s.anyLoading = true }) // 3. thunk.settled — cleanup (fulfilled ИЛИ rejected) b.addMatcher(fetchPosts.settled, (s) => { s.posts.loading = false }) // 4. isPending() без аргументов — на ЛЮБОЙ pending b.addMatcher(isPending, (_, a) => console.log('any pending', a.type)) }

Подводные камни

1. thunk.pending — это action creator, не строка. addCase(thunk.pending.type, ...) — работает, но теряется типизация action. Всегда передавайте сам creator.
2. isPending(thunkA, thunkB) ≠ isPending. С аргументами — matcher ловит только эти thunk'и. Без аргументов — любой pending-action (тип payload теряется). В RTK это часто путают: для «общего загрузчика» всегда передавайте список.
3. Порядок в extraReducers — как в createReducer. Сначала addCase, затем addMatcher, наконец addDefaultCase. На один action матчится один addCase + все подходящие addMatcher по очереди.
4. Не путайте thunk.settled и isAnyOf(thunk.fulfilled, thunk.rejected). Это одно и то же (см. исходник выше), но settled — читабельнее и типизированно.
