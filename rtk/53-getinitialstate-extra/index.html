Теория — 53. getInitialState(additionalState, entities)

Сигнатура (RTK 2.x)

// Две перегрузки getInitialState( state?: undefined, entities?: Record<Id, T> | readonly T[], ): EntityState<T, Id> getInitialState<S extends object>( additionalState: S, entities?: Record<Id, T> | readonly T[], ): EntityState<T, Id> & S

Исходник — entity_state.ts

export function createInitialStateFactory<T, Id>( stateAdapter: EntityStateAdapter<T, Id>, ): EntityStateFactory<T, Id> { function getInitialState( additionalState: any = {}, entities?: readonly T[] | Record<Id, T>, ): any { const state = Object.assign(getInitialEntityState(), additionalState) return entities ? stateAdapter.setAll(state, entities) : state } return { getInitialState } }

GitHub: entities/entity_state.ts

Ключевое. Сначала Object.assign({ ids:[], entities:{} }, additionalState) — получаем плоский объект. Затем, если передан второй аргумент, вызывается adapter.setAll(state, entities), который мутирует ids/entities через createStateOperator. Если state plain (мы вне reducer'а) — оператор оборачивает в createNextState → безопасно.

Цитата — RTK docs

«The getInitialState function now has a second parameter that accepts initial entities. This allows you to seed the entity state with data at slice creation time, with no extra dispatch.»
— RTK 2.0 release notes
«В getInitialState теперь есть второй параметр — начальные entity. Можно засеивать данные при создании slice без лишнего dispatch.»

Типичный паттерн «async flags + entity»

type LoadingStatus = 'idle' | 'pending' | 'fulfilled' | 'rejected' const postsSlice = createSlice({ name: 'posts', initialState: postsAdapter.getInitialState({ loading: 'idle' as LoadingStatus, error: null as string | null, }), reducers: { fetchPending: (s) => { s.loading = 'pending'; s.error = null }, fetchFulfilled: (s, a) => { s.loading = 'fulfilled'; postsAdapter.setAll(s, a.payload) }, fetchRejected: (s, a) => { s.loading = 'rejected'; s.error = a.payload }, }, })

Плюсы плоской структуры

  • Одна slice — одна фича. UI читает state.posts.loading и state.posts.entities из одного места.
  • React-компоненты подписываются через useSelector на удобные селекторы: selectLoading, selectAll — сам подстейт вложен, нам не важно.
  • Когда мигрируешь на createAsyncThunk (секция G) — схема уже готова.

Подводные камни

1. Не используй ключи ids или entities в additionalState.
Object.assign(initial, additionalState) в исходнике — если additionalState содержит эти поля, они перезапишут пустые. Потом setAll снова их пересоздаст (при наличии entities), но если entities не передан — останутся твои плохие данные. Используй любые другие имена: status, meta, filter.

2. setAll мутирует. Когда RTK вызывает stateAdapter.setAll(state, entities) в getInitialState — это работает, потому что state только что создан (ещё не заморожен Immer'ом). Но не делай adapter.setAll(plainExistingStore, …) руками — createStateOperator обернёт в createNextState (immutably) и вернёт новый объект; старый не поменяется. Всегда используй внутри reducer'а / сохраняй результат.

3. Селекторы должны быть готовы к loading/error. selectAll вернёт [] в состоянии pending. В UI всегда проверяй loading === 'pending' до того, как показать «пусто» — иначе покажешь empty-state при живом запросе.

4. Второй параметр entities — появился только в RTK 2.x. В 1.x надо было делать adapter.setAll(adapter.getInitialState(extra), initial) вручную. Если видишь старый код — просто перепиши на 2-аргументный.
