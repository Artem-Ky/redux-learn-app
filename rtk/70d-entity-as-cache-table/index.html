Bench: array.find (O(n)) vs entities[id] (O(1))

createEntityAdapter — это не просто «удобный CRUD». Это in-memory cache-таблица: шейп { ids, entities } превращает state в хеш-карту, где selectById(s, id) = entities[id] — это cache-hit за константное время.

Ментальная модель

Обычный массив items: Todo[] — это «лента». Чтобы найти запись по id, нужно обойти всю ленту. EntityAdapter добавляет индекс по id:

// ❌ Массив — cache бесполезен: Array.find = O(n) interface TodosState { items: Todo[] } const found = state.items.find(t => t.id === targetId) // ✅ EntityState — встроенный O(1) cache interface TodosState { ids: string[] entities: Record<string, Todo> } const found = state.entities[targetId] // O(1) — сразу из хеш-таблицы const exists = targetId in state.entities const count = state.ids.length

Сцепка с другими слоями

Layer 3 (adapter) + Layer 2 (memo): параметризованный селектор selectById у adapter'а — это lruMemoize size 1. В списке из 100 компонентов с разными id — промах в 99 из 100. Берите weakMapMemoize (Layer 2+3 вместе).
Layer 3 + Layer 5 (TTL): храните fetchedAt прямо в entity. Тогда condition проверяет свежесть по id за O(1).
Layer 3 + Layer 4 (dedup): loading: Record<id, 'pending' | 'idle'> — inflight-flag per id. Дедуп параметризованный, без коллизий.

Upsert vs set — mental model кэша

// upsertOne — "обновить запись в кэше, если она есть; добавить, если нет" // Это именно cache-update семантика. После ответа от сервера → upsertOne. upsertOne(state, { id: 'p1', price: 200 }) // → entities.p1 = { ...old, price: 200 } (shallow merge) // setOne — "заменить запись целиком" // Полезно когда сервер вернул ПОЛНЫЙ объект и вы уверены в схеме. setOne(state, { id: 'p1', price: 200, title: 'New' }) // → entities.p1 = { id:'p1', price: 200, title: 'New' } // addOne — "добавить только если нет" (кэш-miss) // silent no-op если уже есть. Полезно когда не хочешь перетереть свежие данные.

В RTKQ

RTKQ хранит свой cache не через createEntityAdapter, а через state.api.queries[cacheKey] — тоже plain object. Но философия та же: id-индекс для O(1) доступа. Многие кодбейсы используют createEntityAdapter ВМЕСТЕ с RTKQ: API возвращает список → transformResponse прогоняет через setAll. Это даёт плоский state и плюсы нормализации.
