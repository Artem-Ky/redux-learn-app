Теория — 44. slice.selectors, getSelectors и мемоизация

Цитата из доки RTK

«slice.selectors is a pre-built object containing the defined selectors, each already bound to the slice's state path… Each selector automatically receives the slice state as its first argument.»
— RTK docs — createSlice selectors
«slice.selectors — преднастроенный объект, где каждый селектор уже привязан к пути slice внутри root state. Каждый селектор автоматически получает state слайса первым аргументом.»

Вариант 1: автопривязка через selectSlice

Если вы передали reducer: { users: usersSlice.reducer } в configureStore, то slice.selectors.xxx сам возьмёт state из rootState[slice.reducerPath]. reducerPath по умолчанию = name.

const usersSlice = createSlice({ name: 'users', initialState: [] as User[], reducers: { /* ... */ }, selectors: { selectCount: (state) => state.length, // НЕ мемоизирован selectActive: createSelector( // мемоизирован [(state: User[]) => state], (users) => users.filter((u) => u.active), ), }, }) // В компоненте: const active = usersSlice.selectors.selectActive(rootState) // ^^ rootState, НЕ state.users — магия

Как это работает внутри — фрагмент исходника

// @reduxjs/toolkit/src/createSlice.ts — фрагмент (~30 строк) // https://github.com/reduxjs/redux-toolkit/blob/master/packages/toolkit/src/createSlice.ts function makeSelectorProps(reducerPath, injected = false) { function selectSlice(state) { let sliceState = state[reducerPath] if (typeof sliceState === 'undefined') { if (injected) { sliceState = getOrInsertComputed( // инъекция → initial state injectedStateCache, selectSlice, getInitialState, ) } else if (process.env.NODE_ENV !== 'production') { throw new Error( 'selectSlice returned undefined for an uninjected slice reducer', ) // 👀 кейс 3: вот оно } } return sliceState } function getSelectors(selectState = selectSelf) { return getOrInsertComputed(selectorCache, selectState, () => { const map = {} for (const [name, selector] of Object.entries(options.selectors ?? {})) { map[name] = wrapSelector(selector, selectState, /*...*/) // 👀 каждый selector оборачивается, чтобы получать slice state } return map }) } return { reducerPath, getSelectors, get selectors() { return getSelectors(selectSlice) }, // 👀 geter → ленивая привязка selectSlice, } }

Вариант 2: getSelectors с кастомным путём

Если state положен глубже (например, state.admin.users), а не по имени slice:

const store = configureStore({ reducer: { admin: combineReducers({ users: usersSlice.reducer }), }, }) // slice.selectors.selectActive(state) — упадёт: state.users === undefined // Делаем так: const adminUsersSelectors = usersSlice.getSelectors( (s: RootState) => s.admin.users, ) adminUsersSelectors.selectActive(rootState) // ✅ работает

Вариант 3: неверный shape — ошибка в dev, undefined в prod

// В development: throw new Error('selectState returned undefined for an uninjected slice reducer') // В production (NODE_ENV === 'production'): // ошибки НЕТ, sliceState === undefined передаётся дальше → селектор упадёт // при первой попытке прочитать .filter / .length, но сообщение будет уже неинформативным.

Ключевой нюанс: инлайновая функция в selectors: не мемоизирована

selectors: { selectActive: (state) => state.filter((u) => u.active), // ☝ НЕ мемоизирован! Вызовется при каждом dispatch, вернёт новый массив. selectActiveMemo: createSelector( [(state) => state], (state) => state.filter((u) => u.active), ), // ✅ мемоизирован — combiner пересчитается только если state изменился. }

Когда использовать getSelectors (а не slice.selectors)

  • state лежит не по slice.reducerPath. Например, вложен в combineReducers.
  • Тесты. В юнит-тесте state обычно mock-нутый без корневого shape-а.
  • Несколько инстансов одной slice. createModalSlice с разными ключами.
  • Server components / SSR. Когда state приходит из getServerSideProps.

combineSlices и .inject() — важный кейс

import { combineSlices } from '@reduxjs/toolkit' const rootReducer = combineSlices(usersSlice).withLazyLoadedSlices<{ posts?: PostsState }>() const store = configureStore({ reducer: rootReducer }) // Позже — инжектим posts: const injectedSlice = postsSlice.injectInto(rootReducer) // После инжекта: injectedSlice.selectors — новые обёртки, с fallback на getInitialState() // Если state ещё не инициализирован (reducer не зарегистрирован до первого селекта): injectedSlice.selectors.selectAll(rootState) // вернёт initialState, НЕ undefined

Подводные камни

1. Инлайн-функции в selectors: НЕ мемоизированы. Это просто присвоение: map[name] = wrapSelector(selector, selectState, ...). wrapSelector не оборачивает в createSelector. Почему: RTK следует принципу «explicit is better». Если хочешь memo — оберни явно в createSelector.
2. selectSlice() бросает в dev, молчит в prod. Это сделано для bundle size. Почему: process.env.NODE_ENV !== 'production' выпиливается минификатором. В prod вы получите silent failure при неверном shape. Решение: покрывать тестами и использовать reducerPath явно.
3. slice.selectors — getter, но кешируется. В исходнике: get selectors() { return getSelectors(selectSlice) }. getter, но getOrInsertComputed кеширует результат по ключу selectState. Значит повторное чтение slice.selectors не создаёт новые обёртки. Почему это важно: если бы не кешировалось, мемоизация сломалась бы — у каждого доступа свой замыкающий кеш.
4. После combineSlices.inject() state может быть undefined. До того, как хоть один reducer отработал, rootState.posts === undefined. injected-селекторы используют getInitialState() как fallback — но только если slice действительно был инжектирован через .injectInto(). Селекторы, полученные из обычной slice (не инжектированной), fallback'а не имеют и упадут.
5. Стабильность ссылок между инстансами slice.selectors. slice.getSelectors(fnA) и slice.getSelectors(fnB) — разные объекты с разными мемо-кешами. Если вы случайно используете оба в разных компонентах — cache не переиспользуется. Обычно достаточно одного.
