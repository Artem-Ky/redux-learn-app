Теги — это именованные cache-зависимости. Query предоставляет теги (providesTags: ['Post']), mutation инвалидирует теги (invalidatesTags: ['Post']). RTKQ строит обратный индекс state.api.provided['Post'] и после mutation автоматически перезапускает все query, которые висят на этих тегах.

Теория — 87. Tags · providesTags · invalidatesTags

Цитата — Automated Re-fetching

«RTK Query uses an abstraction called tags. A tag is just a name that you can give to a specific collection of data to control the caching and re-fetching behavior for queries dependent on that data. When a mutation is triggered that "invalidates" a tag, any query that "provides" that tag will be re-fetched, if subscribed to the cache.»
— RTKQ · Automated Re-fetching
«Тег — это имя-маркер. Query его предоставляет, mutation — инвалидирует. RTKQ автоматически ре-фетчит только те запросы, которые подписаны на тот же тег. Это замена ручному "dispatch refresh после создания".»

Три шага настройки

// 1. объявить tagTypes в createApi const api = createApi({ reducerPath: 'api', baseQuery: fetchBaseQuery({ baseUrl: '/' }), tagTypes: ['Post'], // 👀 белый список тегов endpoints: (build) => ({ // 2. query — providesTags getPosts: build.query<Post[], void>({ query: () => 'posts', providesTags: ['Post'], // 👀 «я предоставляю Post» }), // 3. mutation — invalidatesTags addPost: build.mutation<Post, NewPost>({ query: (body) => ({ url: 'posts', method: 'POST', body }), invalidatesTags: ['Post'], // 👀 «я ломаю все Post-кэши» }), }), })

Что хранится в state.api.provided

// apiState.ts — тип InvalidationState export type InvalidationState<TagTypes> = { tags: { [_ in TagTypes]: { [id: string | number]: QueryCacheKey[] } } keys: Record<QueryCacheKey, FullTagDescription<any>[]> } // После getPosts.fulfilled с providesTags:['Post']: state.api.provided = { tags: { Post: { '__internal_without_id__': ['getPosts(undefined)'] // string-tag → bucket '__internal' } }, keys: { 'getPosts(undefined)': [{ type: 'Post' }] } }

Алгоритм invalidation

Ключевой файл: invalidationByTags.ts. Там реализована логика «найти cacheKeys для tags → refetch / remove».

providesTags — функция result/error/arg

// Динамические теги — по каждому элементу отдельно: getPosts: build.query<Post[], void>({ query: () => 'posts', providesTags: (result, error, arg) => result ? [ ...result.map((p) => ({ type: 'Post' as const, id: p.id })), { type: 'Post', id: 'LIST' }, // 👀 спецтег для «списка» ] : [{ type: 'Post', id: 'LIST' }], }) // Гранулярные invalidates: updatePost: build.mutation<..., { id: number; ... }>({ query: ..., invalidatesTags: (result, error, arg) => [{ type: 'Post', id: arg.id }], }), // → ре-фетчится ТОЛЬКО getPost({id}) с тем же id, список НЕ ломается.

Детально про {type, id} и 'LIST' — урок 88.

Подводные камни

1. tagTypes — белый список. Используешь тег не из списка → TS ошибка + runtime warning.
RTKQ не позволяет случайно «изобрести» тег. Забыл добавить в tagTypes — TS подскажет сразу. Это удобно для рефакторинга.

2. invalidation срабатывает только для SUBSCRIBED queries.
Если на getPosts(undefined) сейчас нет активной подписки (никто не useQuery), после mutation RTKQ просто delete queries['getPosts(undefined)']. При следующем mount hook увидит, что cache пустой → сделает запрос. Это экономит сеть, но иногда сбивает: «почему UI не обновляется» → проверь, что компонент с useQuery реально в дереве.

3. invalidatesTags может быть функцией, которая получает result/error/arg.
Если мутация не удалась — теги могут быть другие (или пустой массив — чтобы не инвалидировать). Типичный паттерн: invalidatesTags: (result, error) => error ? [] : ['Post'].

4. Вызов invalidatesTags = один action «api/invalidateTags».
Даже если tags[] содержит 10 элементов — это ОДНА запись в DevTools. Но под капотом middleware может задиспатчить несколько getX/pending + remove. Отладка: смотри на action api/invalidateTags и его payload.

5. api.util.invalidateTags(['Post']) — ручной способ.
Можно инвалидировать вне мутации: dispatch(api.util.invalidateTags([{type:'Post', id:5}])). Полезно для WebSocket-push обновлений или «pull-to-refresh».
