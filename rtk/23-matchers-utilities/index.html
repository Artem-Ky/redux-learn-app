nanoid(), isAnyOf, isAllOf — утилиты в живом коде

Три утилиты, которые часто используются вместе с createAction и createReducer.

nanoid() — безопасный id-генератор

Встроен в RTK. Использует crypto.getRandomValues. 21 символ по умолчанию, URL-safe алфавит.

— здесь появятся сгенерированные id —

isAnyOf(...actionCreators)

Matcher, который возвращает true если хотя бы один из actionCreators совпал.

const userAction = isAnyOf(loggedIn, loggedOut, userUpdated) if (userAction(action)) { // action — один из трёх }
—

isAllOf(...actionCreators)

Matcher для случаев, когда нужно проверить, что action удовлетворяет нескольким условиям. Используется с type-predicates: isAllOf(isRejectedWithValue, isUserAction).

const userError = isAllOf(isRejectedWithValue, isUserAction) if (userError(action)) { // action — и rejected, и user-related }
—

isAsyncThunkAction, isRejectedWithValue, …

Специальные matcher'ы для createAsyncThunk actions. Подробно — в секции G.

import { isPending, isFulfilled, isRejected, isRejectedWithValue, isAsyncThunkAction, } from '@reduxjs/toolkit' isPending(myThunk)(action) // любой pending от myThunk isPending(myThunk, otherThunk)(action) isPending() // pending от любого thunk

Эти matcher'ы возвращают функцию, которую нужно вызвать с action. Идеально для builder.addMatcher(isPending(thunk), handler).

Теория — matchers и utilities

nanoid — из доки

«nanoid returns a non-secure random string ID. For most use cases this is sufficient. If you need cryptographically strong IDs, use a dedicated library.»
— nanoid in RTK
«nanoid возвращает не-секьюрную случайную строку-id. Для большинства случаев этого достаточно. Если нужны криптографически стойкие id — используйте специальную библиотеку.»

Почему RTK переэкспортирует nanoid

nanoid используется внутри createAsyncThunk для генерации requestId и в createEntityAdapter. RTK переэкспортирует его, чтобы вы могли использовать без доп. установки пакета nanoid.

isAnyOf — из доки

«isAnyOf takes one or more action creators or matchers as arguments. It returns a matcher that returns true if the action matches any of the provided conditions.»
— isAnyOf API
«isAnyOf принимает один или несколько action creator'ов или matcher'ов. Возвращает matcher, который возвращает true, если action удовлетворяет хотя бы одному из условий.»

isAllOf — из доки

«isAllOf ...returns true only if the action matches all of the provided conditions.»
— isAllOf API
«isAllOf …возвращает true, только если action удовлетворяет всем условиям сразу.»

Матчеры — что это вообще

Matcher — функция сигнатуры (action) => boolean (в TS — type-predicate). Именно так построен RTK: .match на actionCreator'е — это matcher. Комбинаторы isAnyOf и isAllOf объединяют несколько matcher'ов в один.

Полный список utility-matchers

  • isAction(action) — базовая проверка, что это action
  • isPlainObject(value) — plain object helper
  • isAsyncThunkAction(thunk1, thunk2, ...) — любой action этих thunk'ов
  • isPending(thunk?) — pending action
  • isFulfilled(thunk?) — fulfilled action
  • isRejected(thunk?) — rejected action
  • isRejectedWithValue(thunk?) — rejected c rejectWithValue
  • isAllOf(...)
  • isAnyOf(...)

Где использовать

// 1) builder.addMatcher builder.addMatcher(isRejected(fetchUser, fetchPosts), (state, action) => { state.error = action.error.message }) // 2) listenerMiddleware listener.startListening({ matcher: isAnyOf(loggedIn, userUpdated), effect: (action, api) => { /* ... */ } }) // 3) Условно в thunk'е if (isAction(maybeAction)) { ... }

Подводный камень: isPending(thunk) — фабрика

isPending(thunk) — это НЕ matcher, это фабрика. Она возвращает matcher. Нужно: builder.addMatcher(isPending(fetchUser), handler). Если пишете builder.addMatcher(isPending, handler) — это сработает как matcher для любого pending, что может быть нежелательно.

Подводный камень: isRejectedWithValue требует requestId

Если вручную создаёте «rejected»-action через createAction(..., prepare), то isRejectedWithValue(action) вернёт false, пока в meta нет и requestStatus: 'rejected', и requestId: string, и rejectedWithValue: true. Внутри RTK это hasExpectedRequestMetadata(action, ['rejected']) && meta.rejectedWithValue. createAsyncThunk расставляет всё это сам — вам думать не надо. Но в учебном коде / тестах нужно вручную:
const userFail = createAction('user/fetch/rejected', (msg: string) => ({ payload: msg, error: { message: msg, name: 'RejectedError' }, meta: { arg: undefined, requestId: nanoid(), // ← обязательно requestStatus: 'rejected' as const, // ← обязательно aborted: false, condition: false, rejectedWithValue: true, // ← обязательно }, }))
