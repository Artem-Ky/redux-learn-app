Что такое matcher — за 30 секунд

Matcher = функция (action) => boolean. Всё. Никакой магии RTK. Просто функция, которая отвечает «подходит ли этот action под мой критерий». В TypeScript мы дополнительно делаем её type-predicate (action is MyAction), чтобы внутри handler'а TS знал точный тип action.

// 1. Самая базовая форма — просто функция const isUserAction = (action: unknown) => isAction(action) && action.type.startsWith('user/') // 2. Type-predicate — даёт TS-typing внутри handler'а const isNotifyAction = (action: unknown): action is NotifyAction => { if (!action || typeof action !== 'object') return false return !!(action as any).meta?.notify } // 3. Использование в reducer builder.addMatcher(isNotifyAction, (state, action) => { // TS знает: action.meta.notify существует state.toasts.push(action.meta.notify) })

А isAnyOf, isAllOf, actionCreator.match, isPending, isRejectedWithValue — это просто фабрики, которые возвращают такую функцию. Можно (и нужно!) писать свои собственные.

5 форм matcher'ов на одном примере
// === Action === const userLoggedIn = createAction('user/loggedIn', (name: string) => ({ payload: { name }, meta: { notify: { kind: 'success', message: `Привет, ${name}!` } }, })) // === Все 5 форм matcher'а — все возвращают true для userLoggedIn('Алиса') === // (a) Произвольная функция — самая гибкая const m1 = (action: unknown) => isAction(action) && action.type === 'user/loggedIn' // (b) Type-predicate — добавляет typing const m2 = (action: unknown): action is { meta: { notify: any } } => !!(action as any)?.meta?.notify // (c) Встроенный .match на actionCreator — это уже type-predicate const m3 = userLoggedIn.match // под капотом: (action) => action.type === 'user/loggedIn' // (d) isAnyOf — комбинатор «ИЛИ» const m4 = isAnyOf(userLoggedIn, userLoggedOut, fileSaved) // (e) isAllOf — комбинатор «И»; принимает другие matcher'ы const m5 = isAllOf(isUserNamespace, isNotifyAction)

Когда какую форму использовать:

  • (a) функция — когда логика сложная (проверка shape, namespace, meta-полей).
  • (b) type-predicate — то же что (a), но всегда в TS-проектах. Без неё в handler'е action будет any или unknown.
  • (c) actionCreator.match — когда нужен один конкретный action. Эквивалент addCase(actionCreator, ...), но удобен в listenerMiddleware.
  • (d) isAnyOf — когда нужно перечислить N конкретных action'ов. Минус: легко забыть новый.
  • (e) isAllOf — когда нужно «И» между matcher'ами. Главное применение — комбинировать готовые (типа isRejectedWithValue) со своим namespace-фильтром.
Реальный кейс: toast-уведомления + analytics через matcher'ы

В приложении 4 разных action'а должны показывать toast (login / logout / file-saved / file-failed). Вместо того чтобы в каждом slice'е дублировать state.toasts.push(...), мы делаем один matcher по shape'у action.meta.notify и один addMatcher в reducer'е. То же — для analytics и lastActivity.

// Один matcher → автоматически работает для ВСЕХ action'ов с meta.notify, // даже тех, которые добавят через год. Никаких изменений в reducer. const isNotifyAction = (a: unknown): a is NotifyAction => !!(a as any)?.meta?.notify const reducer = createReducer(initial, (b) => { b .addCase('notifications/dismiss', (s, a) => { ... }) // (1) toast pipeline .addMatcher(isNotifyAction, (s, a) => { s.toasts.push(a.meta.notify) }) // (2) analytics pipeline .addMatcher(isAnalyticsAction, (s, a) => { s.events.push({ type: a.type, ... }) }) // (3) activity pipeline .addMatcher(isUserNamespace, (s) => { s.lastActivity = Date.now() }) })

Теория — matchers под лупой

Матчер — это ровно одно

«A matcher utility is a type guard function that returns true if the passed in argument matches the expected type.»
— Matching Utilities API
«Утилита-матчер — это type-guard функция, возвращающая true, если переданный аргумент удовлетворяет ожидаемому типу.»

Сигнатура: (value: unknown) => boolean. В TypeScript почти всегда — (value: unknown) => value is X. Это позволяет внутри if (matcher(action)) { ... } и в addMatcher(matcher, handler) получить точный тип action без as.

Зачем нужны (3 главных юзкейса)

  1. Сквозная логика в reducer'е — toast'ы, analytics, audit-log, lastActivity, optimistic flush. Один addMatcher вместо N addCase.
  2. listenerMiddleware — startListening({ matcher: isAnyOf(a, b) }). Эффект сработает на любой action, который проходит matcher.
  3. Type narrowing в обычном коде — if (myAction.match(action)) { ... } в условном thunk'е, тесте, devtool'е.

Реализация isAnyOf / isAllOf — внутри они тривиальны

// Из исходника RTK (matchers.ts): function isAnyOf(...matchers) { return (action) => matchers.some((m) => matches(m, action)) } function isAllOf(...matchers) { return (action) => matchers.every((m) => matches(m, action)) } const matches = (matcher, action) => hasMatchFunction(matcher) ? matcher.match(action) // если это actionCreator — берём .match : matcher(action) // иначе — вызываем как функцию

Поэтому в isAnyOf можно смешивать actionCreator'ы и matcher-функции — обёртка matches() их различает.

actionCreator.match — встроенный matcher

const userLoggedIn = createAction<{ name: string }>('user/loggedIn') // .match — это (action) => action.type === 'user/loggedIn' // Но как type-predicate: action is PayloadAction<{name: string}> userLoggedIn.match({ type: 'user/loggedIn', payload: { name: 'A' } }) // true userLoggedIn.match({ type: 'other' }) // false if (userLoggedIn.match(action)) { action.payload.name // ← TS знает что это string }

Когда писать СВОЙ matcher (а не использовать isAnyOf)

  • Когда action'ов много / они растут динамически (analytics/*, user/*).
  • Когда условие — не «один из», а «по shape'у» (action.meta.notify существует).
  • Когда нужно комбинировать с runtime-логикой (только error: true).
  • Когда isAnyOf(a, b, c) легко забыть обновить, а свой matcher по namespace — само-расширяющийся.

Полный type-safe matcher на shape — шаблон

import { type Action } from '@reduxjs/toolkit' interface NotifyMeta { notify: { kind: 'success' | 'error' | 'info'; message: string } } type NotifyAction = Action & { meta: NotifyMeta } export const isNotifyAction = (action: unknown): action is NotifyAction => { if (!action || typeof action !== 'object') return false const a = action as { meta?: { notify?: unknown } } return ( !!a.meta && typeof a.meta.notify === 'object' && a.meta.notify !== null && 'kind' in a.meta.notify && 'message' in a.meta.notify ) }

Принимаем unknown — потому что matcher должен переваривать и невалидные actions без runtime-ошибки. Внутри сужаем тип через серию проверок.

Подводные камни

1. Фабрика vs matcher. isPending, isRejected, isFulfilled, isRejectedWithValue — это и то и другое:
• isPending(myThunk) — фабрика, возвращает matcher, который ловит pending только этого thunk'а.
• isPending (без вызова) — сам matcher, ловит ЛЮБОЙ pending от любого thunk'а.
Будьте внимательны: addMatcher(isPending, h) ≠ addMatcher(isPending(thunk), h).
2. matcher вызывается на КАЖДЫЙ action. Reducer пробегает по всем matcher'ам при каждом dispatch. Не делайте в matcher'е тяжёлых вычислений (regex по большим строкам, JSON-сериализацию). Лучшие matcher'ы — O(1) проверка action.type или !!action.meta?.notify.
3. addMatcher выполняется ДОПОЛНИТЕЛЬНО к addCase. Если у вас addCase(userLoggedIn, h1) и addMatcher(isUserNamespace, h2) — при dispatch(userLoggedIn(...)) выполнятся оба handler'а. Это фича (см. урок 25), но если h1 и h2 пишут в один и тот же путь state — будет затирание.
4. Проверка action.type требует typeof check. Action технически может прийти null/undefined/без type (особенно в тестах и middleware). Поэтому всегда: typeof (action as any)?.type === 'string', а не сразу action.type.startsWith(...) — иначе runtime crash.

Matcher в listenerMiddleware

import { createListenerMiddleware } from '@reduxjs/toolkit' const listener = createListenerMiddleware() // Тот же matcher работает и в listener'е listener.startListening({ matcher: isNotifyAction, effect: (action, api) => { // в action — тип NotifyAction (благодаря type-predicate) analytics('toast.shown', action.meta.notify) }, }) // Или предикат listener.startListening({ predicate: (action, currentState, prevState) => (currentState as any).user.id !== (prevState as any).user.id, effect: (action, api) => { /* user changed */ }, })

Чек-лист «как написать свой matcher»

  1. Объяви TS-тип action'а, который ловишь: type MyAction = Action & { ... }.
  2. Принимай unknown, возвращай type-predicate action is MyAction.
  3. Сначала отбей null/undefined/non-object.
  4. Проверь только то, что точно нужно — никаких лишних расчётов.
  5. Экспортируй и переиспользуй: один matcher → addMatcher + listener + if-проверка в thunk'е.
