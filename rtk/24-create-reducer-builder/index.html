Todos reducer — старый стиль vs builder

createReducer принимает builder-callback, в котором вы цепочкой вызываете .addCase(action, handler). Внутри — Immer: мутируйте state без опасений. Старый object-map (ключи — action types) deprecated в RTK 2.x.

⚠ Deprecated: object-map (RTK 1.x)

const todos = createReducer(initial, { [addTodo.type]: (state, action) => { state.push(action.payload) }, [toggleTodo.type]: (state, { payload }) => { const t = state.find(x => x.id === payload) if (t) t.done = !t.done } })

⚠ Удалено в RTK 2.x. Используйте builder-callback.

✓ Рекомендовано: builder-callback

const todos = createReducer(initial, (builder) => { builder .addCase(addTodo, (state, action) => { state.push(action.payload) }) .addCase(toggleTodo, (state, { payload }) => { const t = state.find(x => x.id === payload) if (t) t.done = !t.done }) })

✓ Типобезопасный, поддерживает addMatcher, addDefaultCase (урок 25).

Теория — createReducer + builder

Из доки

«createReducer lets you supply a lookup table of action types to case reducer functions, rather than writing switch statements. It uses immer internally, so you can write code that "mutates" state inside the case reducers.»
— createReducer API
«createReducer позволяет задать соответствие между action type и case-reducer'ом — вместо того чтобы писать switch. Внутри используется immer, поэтому вы можете писать код, «мутирующий» state внутри case-reducer'ов.»

Сигнатура

function createReducer<S>( initialState: S | (() => S), builderCallback: (builder: ActionReducerMapBuilder<S>) => void ): Reducer<S> interface ActionReducerMapBuilder<S> { addCase(actionCreator, reducer): this addCase(actionType: string, reducer): this addMatcher(matcher, reducer): this // урок 25 addDefaultCase(reducer): this // урок 25 }

initialState может быть функцией

// Функция вызывается один раз при инициализации const reducer = createReducer( () => ({ value: 0, startedAt: Date.now() }), // ← lazy init (builder) => { ... } )

Полезно, если initialState зависит от side-effects (например, чтение localStorage). Функция вызывается один раз при первом @@INIT.

Правило чистоты reducer'а

Reducer ДОЛЖЕН быть чистой функцией: для одинакового входа — одинаковый выход.
  • Никаких Date.now(), Math.random(), nanoid() внутри.
  • Никаких API-вызовов (fetch).
  • Только чтение из action.payload и запись в state.
Все side-effects — в prepare-callback или в thunk.

Почему object-map deprecated

  1. Порядок matcher'ов потерян — при object-map вы не могли гарантировать порядок вызова handlers. Builder делает это явно: addCase вызывается в том порядке, в котором вы его написали.
  2. Нет addMatcher / addDefaultCase — для них нужен builder-паттерн, который сохраняет цепочку действий.
  3. TS-вывод хуже — builder лучше выводит типы через цепочку методов.

Можно комбинировать с строковым type

builder .addCase(addTodo, handler) // через actionCreator .addCase('legacy/action', handler) // через строку (редко нужно)

Подводный камень: обязательно addCase перед addMatcher

Порядок в builder важен: addCase → addMatcher → addDefaultCase. Если попытаетесь вызвать addCase после addMatcher — получите runtime error «addCase cannot be called after addMatcher». Подробнее в уроке 25.
