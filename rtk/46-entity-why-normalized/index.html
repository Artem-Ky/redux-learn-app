Теория — 46. Нормализация в Redux state

Проблема: «база данных» из массивов

Типичный начинающий хранит коллекции как items: Todo[]. Пока их 5 — всё быстро. Когда их 1000+ и мы каждый рендер делаем items.find(t => t.id === action.id) в reducer'е, селекторе и в компоненте — получаем O(n) × рендеры × компоненты. Обновление одного объекта требует нового массива (items.map) — ещё O(n) с аллокацией.

Цитата — Redux docs

«Treat a portion of your Redux store as if it were a database. Each type of data gets its own "table". Each "data table" should store the individual items in an object, with the IDs of the items as keys and the items themselves as the values. Any references to individual items should be done by storing the item's ID.»
— Structuring Reducers · Normalizing State Shape
«Относитесь к части вашего Redux-стора как к базе данных. У каждого типа данных — своя «таблица». Внутри таблицы элементы лежат в объекте { [id]: item }, а любые ссылки на отдельный элемент — через его id.»

Форма EntityState<T>

// Это та самая форма, которую использует createEntityAdapter (урок 47-48) interface EntityState<T, Id> { ids: Id[] // сохраняет порядок entities: Record<Id, T> // быстрый lookup O(1) }

Схема: как массив превращается в нормализованный state

Зачем 2 поля, а не одно?

  • entities — fast lookup и дедупликация (не хранить юзера дважды).
  • ids — сохраняет порядок (у объекта ключи формально упорядочены, но полагаться на это — плохо), даёт «витрину»: ids.map(id => entities[id]).

Дублирование — второй враг

В классическом дереве часто лежит selectedTodo: Todo и одновременно todos: Todo[] с тем же объектом. Обновляем один — забыли обновить другой — данные «разъезжаются». В нормализованной форме хранится только id: selectedTodoId: 'a3'. Единственный источник истины.

Когда НЕ нормализовать

Не нормализуйте просто ради того, что «так принято».
  • UI-state из 3 полей (isModalOpen, theme, filter) — overkill.
  • Коллекция < 20 элементов, которую никогда не ищут по id — overkill.
  • Дерево (tree of categories) — лучше оставить древовидным или хранить два поля: плоскую entities + отдельный rootIds.

Подводные камни нормализации

1. Дублирование id. Сам id лежит и в ids[i], и в entities[id].id. При смене id (редко) — надо поддерживать оба места (RTK это делает автоматически в updateOne, см. урок 50).

2. Потеря порядка. Если строите UI из Object.values(entities) — порядок не гарантирован. Используйте ids.map(id => entities[id]) (это и есть selectAll adapter'а). Сортировку контролирует sortComparer (см. урок 51).

3. Nested data. Если Todo ссылается на User — нельзя хранить весь user внутри каждого todo; лучше userId + отдельный usersAdapter. Это и есть normalizr-подход из документации.
