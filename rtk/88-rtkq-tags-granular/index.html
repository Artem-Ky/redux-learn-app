providesTags per-item · invalidatesTags точечно · 'LIST' как whole-collection

Тег 'Post' в виде строки = всё про Post — грубо. Тег { type: 'Post', id: 5 } = конкретный пост #5 — точечно. Тег { type: 'Post', id: 'LIST' } — отдельный «bucket» для списка; query списка его provides, мутация add его invalidates — и только список перезапрашивается, 5 кэшей деталей остаются нетронутыми.

Теория — 88. Granular tags · {type,id} · 'LIST'

Цитата — Advanced Invalidation with abstract tag IDs

«Providing a tag with an id enables more granular control over which cached entries will be invalidated. An ID can be any string or number. Tags can be further optimized by using 'LIST' as a special ID so you can invalidate the whole collection when adding a new entity without re-fetching every single detail query.»
— RTKQ · Advanced Invalidation with abstract tag IDs
«Гранулярность — это когда тег содержит id, чтобы инвалидировать конкретную запись, а не всю коллекцию. А спецтег 'LIST' — это «маркер списка»: query списка его provides, create-мутация его invalidates → ре-фетчится только список, детали не трогаются.»

Три уровня гранулярности

// ─── уровень 1: строковый тег — грубо ─── getPosts: { providesTags: ['Post'] } updatePost:{ invalidatesTags: ['Post'] } // ↑ updatePost({id:5}) перезапрашивает ВСЕ кэши с тегом 'Post': // getPosts + getPost(1) + getPost(2) + ... + getPost(10) — 11 запросов // ─── уровень 2: { type, id } — точечно ─── getPost(id): { providesTags: (r, e, id) => [{ type: 'Post', id }] } updatePost: { invalidatesTags: (r, e, arg) => [{ type: 'Post', id: arg.id }] } // ↑ updatePost({id:5}) перезапрашивает ТОЛЬКО getPost(5) — 1 запрос // Но! getPosts теперь не инвалидируется → список показывает устаревший title // ─── уровень 3: { type, id } + 'LIST' — смешанный ─── getPosts: { providesTags: (result) => result ? [ ...result.map((p) => ({ type: 'Post' as const, id: p.id })), { type: 'Post', id: 'LIST' }, // 👀 маркер «это список» ] : [{ type: 'Post', id: 'LIST' }], } getPost(id): { providesTags: (r, e, id) => [{ type: 'Post', id }] } updatePost: { invalidatesTags: (r, e, arg) => [{ type: 'Post', id: arg.id }] } // update → getPost(id) + getPosts (потому что список тоже provides {Post,id}!) — 2 запроса addPost: { invalidatesTags: [{ type: 'Post', id: 'LIST' }] } // add → только getPosts (который один provides 'LIST') — 1 запрос, детали нетронуты

Что на самом деле хранится в state.api.provided.tags

// После того как загрузились getPosts + getPost(3) + getPost(5): state.api.provided.tags = { Post: { '1': ['getPosts(undefined)'], // из списка — по id=1 '2': ['getPosts(undefined)'], '3': ['getPosts(undefined)', 'getPost(3)'], // 👀 ДВА поставщика! '4': ['getPosts(undefined)'], '5': ['getPosts(undefined)', 'getPost(5)'], // 👀 'LIST': ['getPosts(undefined)'], // спецтег } } // dispatch(api.util.invalidateTags([{type:'Post', id: 3}])): // ищем tags['Post']['3'] → ['getPosts(undefined)', 'getPost(3)'] // → перезапуск обеих (если subscribed) или delete (если нет). // dispatch(api.util.invalidateTags([{type:'Post', id: 'LIST'}])): // ищем tags['Post']['LIST'] → ['getPosts(undefined)'] // → перезапуск ТОЛЬКО списка, детали нетронуты.

Источник — invalidationByTags.ts (упрощённо)

// buildMiddleware/invalidationByTags.ts — handler на mutation.fulfilled/rejected function invalidateTags(tags: FullTagDescription<string>[], api) { const state = api.getState()[reducerPath] const toInvalidate = new Set<QueryCacheKey>() for (const tag of tags) { const provided = state.provided.tags[tag.type] if (!provided) continue const idBucket = tag.id !== undefined ? String(tag.id) : '__internal_without_id__' provided[idBucket]?.forEach((cacheKey) => toInvalidate.add(cacheKey)) // 👀 Set — дедуп } // тэг БЕЗ id — «все buckets» для этого типа (edge-case) for (const tag of tags.filter((t) => t.id === undefined)) { const provided = state.provided.tags[tag.type] if (provided) Object.values(provided).flat().forEach((k) => toInvalidate.add(k)) } for (const cacheKey of toInvalidate) { const q = state.queries[cacheKey] if (!q) continue if (hasSubscription(cacheKey)) api.dispatch(forceRefetch(cacheKey)) // refetch else api.dispatch(removeQueryResult({ queryCacheKey: cacheKey })) } }

Источник полностью: invalidationByTags.ts . Ключевая идея — Set дедуплицирует cacheKey, если один кэш провайдит несколько тегов.

Схема — три разных invalidatesTags

Паттерн «list + detail» — канонический

type PostTag = { type: 'Post'; id: number | 'LIST' } endpoints: (build) => ({ getPosts: build.query<Post[], void>({ query: () => 'posts', providesTags: (result) => result ? [ ...result.map(({ id }) => ({ type: 'Post' as const, id })), { type: 'Post' as const, id: 'LIST' }, // всегда есть ] : [{ type: 'Post' as const, id: 'LIST' }], // даже при ошибке }), getPost: build.query<Post, number>({ query: (id) => `posts/${id}`, providesTags: (_r, _e, id) => [{ type: 'Post', id }], }), addPost: build.mutation<Post, NewPost>({ query: (body) => ({ url: 'posts', method: 'POST', body }), invalidatesTags: [{ type: 'Post', id: 'LIST' }], // 👀 только список }), updatePost: build.mutation<Post, { id: number; patch: Partial<Post> }>({ query: ({ id, patch }) => ({ url: `posts/${id}`, method: 'PATCH', body: patch }), invalidatesTags: (_r, _e, arg) => [{ type: 'Post', id: arg.id }], // 👀 точечно }), deletePost: build.mutation<void, number>({ query: (id) => ({ url: `posts/${id}`, method: 'DELETE' }), invalidatesTags: (_r, _e, id) => [ { type: 'Post', id }, // getPost(id) → delete { type: 'Post', id: 'LIST' }, // список → refetch без него ], }), })

Подводные камни

1. tag.id = undefined ≠ tag.id = 'LIST'.
Если передать {type:'Post'} без id — RTKQ кладёт его в bucket '__internal_without_id__'. {type:'Post', id:'LIST'} — в bucket 'LIST'. Это два разных bucket'а. При invalidate {type:'Post'} без id — инвалидируется вся коллекция для этого типа (все buckets через Object.values().flat()), а не только '__internal_without_id__' — будь осторожен.

2. List-query provides теги и для LIST, и для каждого элемента.
Если просто сделать providesTags: [{type:'Post',id:'LIST'}] на getPosts, тогда updatePost({id:5}) НЕ ре-фетчит список, и ты увидишь устаревший title в списке (хотя detail обновился). Решение — всегда provides массив item-тегов + LIST-тег (как в паттерне выше).

3. Не используй as const — потеряешь типизацию.
{ type: 'Post' as const, id } — без as const TS выведет type: string, и RTKQ выдаст ошибку «tag 'Post' not in tagTypes». Пиши всегда as const или используй satisfies TagDescription[].

4. 'LIST' — это просто соглашение, не ключевое слово.
RTKQ не проверяет строку — можно писать 'ALL', 'FEED', '__list__'. Но весь мир знает про 'LIST' — держись стандарта.

5. Delete-мутация должна инвалидировать и item, и LIST.
Item-тег нужен, чтобы закешированный detail-query (если пользователь его ещё смотрит) пошёл за новыми данными → получит 404 → отобразит error. LIST-тег нужен, чтобы список обновился без удалённого элемента. Пропустить LIST = список покажет «призрак» до следующего mount.

6. invalidatesTags может быть функцией — используй, когда зависит от arg.
invalidatesTags: (r, e, arg) => error ? [] : [{type:'Post', id: arg.id}] — при ошибке не инвалидировать ничего (deterministic behavior). Возврат пустого массива — valid, invalidation просто не происходит.
