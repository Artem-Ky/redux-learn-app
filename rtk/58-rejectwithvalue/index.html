Один thunk login(creds) возвращает две разные ошибки:

  • Кнопка «throw Error» → обычное исключение. В reducer'е action.payload = undefined, action.error = SerializedError;
  • Кнопка «rejectWithValue» → return rejectWithValue({ code: 401, message: 'Bad password' }). В reducer'е action.payload = { code, message } (типизирован), action.error = { message: 'Rejected' }.

Теория — 58. rejectWithValue vs throw

Цитата

«If you need to customize the contents of the rejected action, you should catch any errors yourself, and then return a new value using the thunkAPI.rejectWithValue utility. Doing return rejectWithValue(errorPayload) will cause the rejected action to use that value as action.payload.»
— createAsyncThunk — Handling Thunk Errors
«Если нужно кастомизировать содержимое rejected action'а — ловите ошибки сами и return-ьте новое значение через thunkAPI.rejectWithValue. return rejectWithValue(errorPayload) заставит rejected использовать этот объект как action.payload.»

Два класса под капотом

// createAsyncThunk.ts, строки 64–86 class RejectWithValue<Payload, RejectedMeta> { private readonly _type!: 'RejectWithValue' constructor( public readonly payload: Payload, public readonly meta: RejectedMeta, ) {} } class FulfillWithMeta<Payload, FulfilledMeta> { private readonly _type!: 'FulfillWithMeta' constructor( public readonly payload: Payload, public readonly meta: FulfilledMeta, ) {} }

Как RTK их различает

// createAsyncThunk.ts, строки 688–702 .then((result) => { if (result instanceof RejectWithValue) { throw result // ← rejectWithValue → throw (но не настоящая ошибка) } if (result instanceof FulfillWithMeta) { return fulfilled(result.payload, requestId, arg, result.meta) } return fulfilled(result as any, requestId, arg) }) // ... } catch (err) { finalAction = err instanceof RejectWithValue ? rejected(null, requestId, arg, err.payload, err.meta) // ← payload/meta выставлены : rejected(err as any, requestId, arg) // ← payload undefined, error = serialized(err) }

Подводные камни

1. rejectWithValue — это return, не throw! throw rejectWithValue(...) тоже работает (в catch сработает instanceof проверка), но канонично — return rejectWithValue(...).
2. throw new Error('X') → action.payload === undefined. Если reducer пишет s.error = action.payload.message — получите crash на обычных ошибках. Всегда проверяйте: a.payload?.message ?? a.error.message.
3. .unwrap() различает их. При rejectedWithValue: true — throw action.payload (ваш объект). Иначе — throw action.error (SerializedError). Смотрите unwrapResult в исходнике.
4. Error в payload-creator'е → мимо try/catch. Если payloadCreator кидает неожиданное исключение (TypeError, ReferenceError…) — RTK ловит его try/catch'ем и делает rejected(err, …). То есть это не попадёт в catch, окружающий await dispatch(thunk()). Только .unwrap() их перебрасывает.
