Raw API response ↔ то, что попадает в cache

Бэкенд отдаёт «обёрнутый» ответ { data: [...], meta: {...} }. Мы через transformResponse разворачиваем и сразу нормализуем в { ids, entities } через createEntityAdapter. В кеш попадает уже готовая структура.

Теория — 89. transformResponse / transformErrorResponse

Цитата — Customizing Queries

«transformResponse is given the full response, and should return a value to be cached.
Where successful queries are stored in the cache with this value, failed queries pass their result through transformErrorResponse.»
— RTKQ · Customizing Query Responses
«transformResponse — последний шанс поменять данные до того, как они упадут в state.api.queries[cacheKey].data. Выполняется один раз на успешный ответ, результат мемоизируется в кеше.»

Сигнатура

getPosts: build.query<EntityState<Post>, void>({ query: () => 'posts', // response — сырой ответ baseQuery, meta — headers/status, arg — то, что передали в query transformResponse: (response: ApiEnvelope<Post[]>, meta, arg) => { return postsAdapter.setAll(postsAdapter.getInitialState(), response.data) }, transformErrorResponse: (error: FetchBaseQueryError, meta, arg) => { return { code: error.status, message: (error.data as any)?.message ?? 'unknown' } }, }),

Типичные применения

// ─── 1. Разворачивание envelope ─── transformResponse: (r: { data: Post[]; meta: Meta }) => r.data, // ─── 2. Нормализация через entity adapter ─── const adapter = createEntityAdapter<Post>() transformResponse: (r: Post[]) => adapter.setAll(adapter.getInitialState(), r), // cache.data = { ids: [1,2,3], entities: { 1:..., 2:... } } — O(1) lookup // ─── 3. Фильтрация/сортировка на клиенте ─── transformResponse: (r: Post[]) => r.filter(p => !p.draft).sort((a, b) => b.createdAt - a.createdAt), // ─── 4. Обогащение производными полями ─── transformResponse: (r: Post[]) => r.map(p => ({ ...p, summary: p.body.slice(0, 100) })), // ─── 5. Объединение данных с мета ─── transformResponse: (r: Post[], meta) => ({ items: r, total: Number(meta?.response?.headers.get('x-total-count') ?? r.length), }),

Почему transformResponse — правильное место для нормализации

Альтернативы — нормализовать в компоненте (useMemo) или в selectFromResult. Оба варианта перезапускают преобразование на каждом рендере. А transformResponse выполняется один раз на fulfilled-action, результат сохраняется в store. При всех дальнейших чтениях — та же ссылка на нормализованную структуру (structural sharing работает).

Источник — buildThunks.ts

// упрощённо из buildThunks.ts — как вызывается transformResponse const transformResponse = endpointDefinition.transformResponse ?? defaultTransformResponse const transformed = await transformResponse(result.data, result.meta, arg.originalArgs) return { ...result, data: transformed } // ↑ результат летит в queryThunk.fulfilled → buildSlice записывает transformed в state.api.queries[key].data

GitHub: buildThunks.ts. defaultTransformResponse — identity, просто возвращает data как есть.

transformErrorResponse — аналогично, но для ошибок

transformErrorResponse: (error: FetchBaseQueryError, meta, arg) => { // нормализация ошибки в единый формат: if (error.status === 'FETCH_ERROR') return { code: 'network', message: 'Нет сети' } if (error.status === 401) return { code: 'auth', message: 'Авторизуйтесь' } return { code: error.status, message: (error.data as any)?.message ?? 'unknown' } } // ↑ теперь action.payload.error имеет форму { code, message } — единая структура на весь app

Подводные камни

1. Async в transformResponse — можно, но осторожно.
Функция async поддерживается. RTKQ await-ит результат. Но помни: время трансформации складывается со временем fetch → UI ждёт дольше. Для тяжёлых трансформаций лучше делать синхронно, а тяжёлые вычисления — мемоизировать селекторами.

2. НЕ использовать transformResponse для derived data, зависящих от arg.
Трансформация выполняется один раз на fetch — если зависит от аргумента, который меняется без рефетча (например, UI-фильтр) — используй селектор, не trans.

3. Возврат mutating nested structure ломает structural sharing.
Если каждый вызов создаёт новые ссылки на items, даже если данные те же — RTKQ не увидит идентичности и перерендерит подписчиков. Внутри трансформации используй адаптер или immer.

4. meta может быть undefined.
Кастомный baseQuery может не передавать meta. Не полагайся на meta!.response — используй optional chaining.

5. Типизация: возвращаемый тип = тип cache.
build.query<TResult, TArg> — TResult это после трансформации. Если пишешь build.query<Post[], void> но transformResponse возвращает EntityState<Post> — ошибка типов. Всегда согласовывай.

6. В компоненте data — результат трансформации, не raw.
const { data } = useGetPostsQuery() — это уже EntityState, а не envelope. Доступ к raw-ответу из компонента не предусмотрен (кешируется только transformed).
