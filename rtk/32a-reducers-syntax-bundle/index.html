Два равноправных способа, разная цена в бандле

В RTK 2.x createSlice принимает reducers в двух формах. Они дают один и тот же рантайм, но по-разному влияют на размер production-бандла и работу tree-shaking. Урок отвечает на три вопроса: в чём разница по коду, почему asyncThunk вынесен в отдельный creator через buildCreateSlice, и что выбирать в новом проекте.

«Using the new callback syntax is entirely optional (the object syntax is still standard).»
— RTK 2.0 migration guide
Side-by-side: один и тот же slice

Один и тот же слайс, без асинхронщины. По runtime-поведению абсолютно эквивалентны: те же actions, те же type-строки 'todos/addTodo', тот же reducer. Отличается только синтаксис описания.

A. Object-syntax стандарт

import { createSlice, nanoid } from '@reduxjs/toolkit' const todosSlice = createSlice({ name: 'todos', initialState: { items: [] }, reducers: { // простой reducer deleteTodo: (state, action) => { state.items = state.items.filter( t => t.id !== action.payload ) }, // prepared reducer — объектная форма addTodo: { prepare: (text) => ({ payload: { id: nanoid(), text, done: false } }), reducer: (state, action) => { state.items.push(action.payload) }, }, }, })

B. Callback-syntax RTK 2.x

import { createSlice, nanoid } from '@reduxjs/toolkit' const todosSlice = createSlice({ name: 'todos', initialState: { items: [] }, reducers: (create) => ({ // обёрнут в create.reducer deleteTodo: create.reducer((state, action) => { state.items = state.items.filter( t => t.id !== action.payload ) }), // prepared — через create.preparedReducer addTodo: create.preparedReducer( (text) => ({ payload: { id: nanoid(), text, done: false } }), (state, action) => { state.items.push(action.payload) }, ), }), })
Рантайм идентичен. todosSlice.actions.addTodo.type === 'todos/addTodo' в обеих версиях. Callback-версия чуть дороже по синтаксису (обёртки create.reducer), но ничего не платит в бандле, пока вы не используете create.asyncThunk. Это ключевой момент — разбираем дальше.
Dependency graph — что попадает в production bundle

Выбери один из трёх способов использования createSlice. Узлы зависимостей подсветятся accent-цветом если они попадают в итоговый бандл, и потускнеют если tree-shaker их выкинул. Справа — ориентировочный прирост веса в min+gzip.

Теория — почему callback-API потребовал buildCreateSlice

Корневая проблема: ESM и условные импорты

«You can't do synchronous conditional imports in ES modules, and there's no good way to make the createAsyncThunk import optional. Either createSlice always depends on it (and adds that to the bundle size), or it can't use createAsyncThunk at all.»
— RTK 2.0 migration guide

Если бы createSlice напрямую умел create.asyncThunk, ему пришлось бы статически импортировать createAsyncThunk. Bundler (Vite/webpack/rollup) тогда обязан положить его в бандл у всех, включая тех, кому async thunk не нужен вообще. Tree-shaking тут не помогает: импорт безусловный.

── Если бы сделали «в лоб» ── createSlice.ts ──import──▶ createAsyncThunk.ts │ └──▶ всегда в bundle (даже если вы не звали asyncThunk) ── Как сделано в RTK 2.x ── createSlice.ts (ядро) asyncThunkCreator.ts │ │ │ <── НЕТ прямого импорта ──> │ │ │ └── buildCreateSlice({ creators: { asyncThunk } }) ──┐ │ вы САМИ собираете createAppSlice ───────┘ только тогда asyncThunk попадает в bundle

Три сценария — три разных bundle

Сценарий Попадает в bundle Ориентир. прирост (min+gzip)
A. createSlice + object-syntax createSlice, createReducer, createAction, immer, redux-thunk (как дефолтный middleware configureStore) ~9 KB (ядро RTK)
B. createSlice + callback, только sync reducers то же, что A. Ничего дополнительного. ~9 KB (как A, разница в байтах)
C. createAppSlice + asyncThunkCreator A + createAsyncThunk + asyncThunkCreator ~10–11 KB (+1–2 KB)

Цифры ориентировочные, на момент @reduxjs/toolkit 2.5+. Актуальные значения — на bundlephobia, а измерить в своём проекте проще всего через source-map-explorer или rollup-plugin-visualizer.

Почему всё-таки createAsyncThunk сам по себе tree-shakable

RTK публикует ESM-пакет с sideEffects: false в package.json. Когда вы import { createAsyncThunk } from '@reduxjs/toolkit', но не используете его — bundler его уберёт. Ключевое отличие: в сценарии C вы вызываете buildCreateSlice({ creators: { asyncThunk: asyncThunkCreator } }) — это использование, после которого бандлер больше не имеет права удалять asyncThunkCreator, а тот тащит createAsyncThunk.

Вывод. «Цена» callback-синтаксиса — не в самом синтаксисе, а в том, сколько creators вы подключили при buildCreateSlice. Callback-syntax + только create.reducer/create.preparedReducer — не стоит ничего лишнего в бандле.

Попробовать create.asyncThunk в обычном createSlice

import { createSlice } from '@reduxjs/toolkit' createSlice({ name: 'x', initialState: {}, reducers: (create) => ({ fetchUser: create.asyncThunk(...) // ⚠ throw: cannot use `create.asyncThunk` in this `createSlice` }) })

Цитата из доки: «The version of createSlice exported from RTK will throw an error if create.asyncThunk is called.» Проверь в демо кнопкой «попробовать create.asyncThunk в обычном createSlice» — ошибка уйдёт в ConsolePanel.

Финал: createAsyncThunk + extraReducers vs create.asyncThunk

До RTK 2.x был один канонический путь подключить async thunk к slice'у: вынести createAsyncThunk в отдельный файл и слушать его life-cycle через extraReducers. В RTK 2.x появился «новый» путь — create.asyncThunk прямо внутри reducers. Разберём оба по весу бандла, tree-shaking, переиспользованию и DX — чтобы было понятно, что выбирать.

Старый путь · thunk в отдельном файле RTK 1.x+

// features/todos/fetchTodos.ts import { createAsyncThunk } from '@reduxjs/toolkit' export const fetchTodos = createAsyncThunk( 'todos/fetchTodos', async () => { const r = await fetch('/api/todos') return r.json() }, ) // features/todos/todosSlice.ts import { createSlice } from '@reduxjs/toolkit' import { fetchTodos } from './fetchTodos' const todosSlice = createSlice({ name: 'todos', initialState: { items: [], loading: false }, reducers: { deleteTodo: (state, action) => { state.items = state.items.filter( t => t.id !== action.payload ) }, }, extraReducers: (builder) => { builder .addCase(fetchTodos.pending, (s) => { s.loading = true }) .addCase(fetchTodos.fulfilled, (s, a) => { s.items = a.payload; s.loading = false }) .addCase(fetchTodos.rejected, (s) => { s.loading = false }) }, })

Новый путь · thunk в create.asyncThunk RTK 2.x

// app/createAppSlice.ts — ОДИН раз на проект import { buildCreateSlice, asyncThunkCreator } from '@reduxjs/toolkit' export const createAppSlice = buildCreateSlice({ creators: { asyncThunk: asyncThunkCreator }, }) // features/todos/todosSlice.ts import { createAppSlice } from '../../app/createAppSlice' const todosSlice = createAppSlice({ name: 'todos', initialState: { items: [], loading: false }, reducers: (create) => ({ deleteTodo: create.reducer((state, action) => { state.items = state.items.filter( t => t.id !== action.payload ) }), fetchTodos: create.asyncThunk( async () => { const r = await fetch('/api/todos') return r.json() }, { pending: (s) => { s.loading = true }, fulfilled: (s, a) => { s.items = a.payload; s.loading = false }, rejected: (s) => { s.loading = false }, }, ), }), }) // thunk доступен как todosSlice.actions.fetchTodos

Сравнение по весу бандла

── Старый путь ── configureStore (~3 KB) + createSlice (~3 KB) + createReducer (<1 KB)  ╮ + createAction (<1 KB)  │ тянется даже без asyncThunk — + immer (~3 KB)  │ вы всё равно используете createSlice + redux-thunk (<1 KB, default mw) ╯ + createAsyncThunk (~1–2 KB)   ← импорт в файле thunk'а ────────────────────── ≈ 10–11 KB min+gzip ── Новый путь (createAppSlice + create.asyncThunk) ── configureStore (~3 KB) + createSlice (~3 KB) + createReducer (<1 KB) + createAction (<1 KB) + immer (~3 KB) + redux-thunk (<1 KB) + buildCreateSlice (<0.2 KB)   ← фабрика, крохотная + asyncThunkCreator (<0.3 KB)   ← адаптер + createAsyncThunk (~1–2 KB)   ← тащится через asyncThunkCreator ────────────────────── ≈ 10–11 KB min+gzip (+~0.3–0.5 KB поверх старого)

Ключевой факт: оба подхода тянут один и тот же createAsyncThunk. Основная «цена» thunk'ов — это сам createAsyncThunk (~1–2 KB gzip), и он присутствует в обоих вариантах. Новый путь добавляет поверх тонкий адаптер asyncThunkCreator + фабрику buildCreateSlice — это буквально ~300–500 байт min+gzip. В рамках typical-прода (сотни KB JS) разница не заметна.

Таблица сравнения

Критерий Старый путь (отдельный createAsyncThunk + extraReducers) Новый путь (create.asyncThunk)
Размер bundle (min+gzip) ~10–11 KB ~10–11 KB (+~0.3–0.5 KB на адаптер)
Tree-shaking асинхронного кода Работает: если thunk не импортирован — он не в bundle Работает так же: если slice не импортирован — adapter тоже не тянется
Code-splitting по фичам ⭐ Лучше: thunk и slice — разные модули, удобно React.lazy и injectEndpoints-паттерны Slice + thunk в одном chunk'е. Менее гранулярно, но проще
Переиспользование thunk'а несколькими slice'ами ⭐ Легко: один createAsyncThunk слушают extraReducers у N slice'ов Thunk принадлежит slice'у. Для шеринга придётся вынести в отдельный createAsyncThunk (т.е. вернуться к старому пути)
Тестируемость thunk-логики ⭐ Импорт fetchTodos из своего файла — чистый unit-тест Thunk достаётся через todosSlice.actions.fetchTodos — тесты выглядят чуть тяжелее
DX / читаемость Больше файлов, extraReducers(builder) boilerplate, .pending/.fulfilled/.rejected в addCase ⭐ Всё в одном файле, handlers прямо рядом с payload creator'ом, короче на ~30–40% строк
Типизация Явная: PayloadAction<T> в handlers, thunk типизирован генериками Выводится автоматически. Подводный камень: thunkApi.getState() возвращает unknown (см. урок 32)

Что выгоднее использовать — финальный вердикт

По bundle size — ничья. Разница ~300–500 байт min+gzip. В прод-приложении, где JS обычно 100+ KB, это не тот критерий, по которому стоит выбирать.

Выбирайте по архитектурным соображениям, а не по «экономии байтов»:

  • Берите новый путь (create.asyncThunk), если:
    • Thunk принадлежит одному slice'у — где описан, там и используется
    • Вы хотите минимум файлов и boilerplate
    • Проект новый, на RTK 2.x — это идиома последней версии
    • Все handlers (pending/fulfilled/rejected/settled) логически группируются с thunk'ом
  • Берите старый путь (отдельный createAsyncThunk + extraReducers), если:
    • Thunk слушают несколько slice'ов — переиспользование важнее компактности
    • Вы делаете агрессивный code-splitting по фичам (React.lazy, lazy slices через injectReducer)
    • Проект ещё на RTK 1.x или вы пишете npm-пакет со slice'ами (не навязывайте чужому бандлу asyncThunkCreator)
    • Thunk тянет чужие slice'ы — меньше риск циклов
  • Смешанный подход в крупном приложении — норма. Один createAppSlice на проект, «приватные» slice-specific thunks через create.asyncThunk, «общие» thunks — отдельными файлами и слушаете через extraReducers.

Официальная документация RTK не даёт однозначной рекомендации в пользу нового пути — в migration-guide явно сказано: «using the new callback syntax is entirely optional (the object syntax is still standard)». То есть старый путь остаётся каноничным; новый — удобная опция для кейсов, где thunk неотделим от slice'а.
