Теория — 93. Optimistic updates

Цитата — Optimistic Updates

«Optimistic updates... update the cache immediately when a request begins (within onQueryStarted), and roll back the cache update if the request fails. The updateQueryData thunk action creator returns a patchResult with an .undo() method.»
— RTKQ · Optimistic Updates
«Идея: верим, что сервер ответит «ок», и меняем UI сразу. Если сервер вернёт ошибку — откатываем. Пользователь видит мгновенный отклик вместо спиннера на 300ms.»

Канонический паттерн

updateTodo: build.mutation<Todo, { id: number; patch: Partial<Todo> }>({ query: ({ id, patch }) => ({ url: `todos/${id}`, method: 'PATCH', body: patch }), // 🟢 онлайн-апдейт — вызывается КОГДА mutation только стартовала async onQueryStarted({ id, patch }, { dispatch, queryFulfilled }) { // 1. Патчим cache — меняем draft через Immer const patchResult = dispatch( api.util.updateQueryData('getTodos', undefined, (draft) => { const todo = draft.find((t) => t.id === id) if (todo) Object.assign(todo, patch) }), ) // 2. Ждём результат сервера try { await queryFulfilled // сервер вернул — cache уже в нужном состоянии, ничего не делаем } catch { // 3. Rollback — inverse patches, cache откатился к тому что было patchResult.undo() } }, }),

onQueryStarted — жизненный цикл

// вызывается для queries И mutations onQueryStarted(arg, api: MutationLifecycleApi) { const { dispatch, getState, extra, requestId, queryFulfilled, // Promise — resolves при fulfilled, rejects при error getCacheEntry, // снимок cache entry } = api // любая async-логика: // - оптимистические патчи // - pessimistic (ждать → patch после fulfilled) // - side-effects (toast, navigate, analytics) }

Pessimistic update (тоже валидный паттерн)

// иногда лучше ДОЖДАТЬСЯ сервера (если нужен id из ответа) async onQueryStarted(arg, { dispatch, queryFulfilled }) { try { const { data: created } = await queryFulfilled dispatch( api.util.updateQueryData('getTodos', undefined, (draft) => { draft.push(created) // push с уже настоящим id }), ) } catch { /* тост «не удалось создать» */ } }

updateQueryData — сигнатура

api.util.updateQueryData( endpointName: string, args: ArgType, updateRecipe: (draft: Draft<T>) => void | T, ) // returns: { undo: () => void, // откат через inverse patches patches: Patch[], // RFC 6902 inversePatches: Patch[], } // Внутри — Immer produce с патчами. Если entry не существует — NOOP.

Подводные камни

1. updateQueryData не работает, если entry нет в cache.
Патч применяется только к существующей entry. Если никто ещё не подписан на getTodos — updateQueryData ничего не сделает. Используй upsertQueryData или сначала делай prefetch.

2. Несколько последовательных optimistic patches — каждый нужно undo отдельно.
Если пользователь жмёт кнопку дважды подряд, и первый запрос упал — надо undo'ить именно его patchResult. Храни их в Map по requestId.

3. patchResult.undo() применяет inversePatches как новый patch.
То есть между optimistic patch и undo могли быть ДРУГИЕ изменения (инвалидация успела перезагрузить данные). Undo откатит только ТВОЙ патч поверх нового состояния. Обычно это OK, но если сервер успел ответить invalidate → данные актуальные → undo даст странный результат. Рассмотри getCacheEntry для логики.

4. Если mutation invalidates тег — refetch перетрёт оптимистичный патч.
Порядок: onQueryStarted patch → mutation fulfilled → invalidatesTags → refetch → data сервера перезапишет. Для optimistic UI часто отключают invalidatesTags на этой mutation и полагаются только на optimistic patch.

5. queryFulfilled бросает rejection со спец структурой.
await queryFulfilled при ошибке бросает { error, meta, isUnhandledError } — это НЕ стандартный Error. Не полагайся на err.message.

6. onQueryStarted для queries — редко.
Это хук для mutations в 99% случаев. Для queries полезен только для логов/ analytics. Если в query — не забудь, что он вызывается на КАЖДЫЙ dispatch initiate, включая refetch.
