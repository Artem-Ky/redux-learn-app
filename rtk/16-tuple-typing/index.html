Почему обычный Array ломает типы

В RTK 2.x была введена обёртка Tuple вокруг массива middleware/enhancers. Это нужно потому, что TypeScript при concat на обычном массиве теряет литеральный тип и сводит элементы к общему типу Middleware — а это разрушает inference для thunk-actions.

Проблема: обычный массив

const store = configureStore({ reducer, middleware: () => [thunk, myLogger] // ← TS error в RTK 2.x }) // Argument of type '() => [...]' is not assignable to type // '(getDefaultMiddleware: ...) => Tuple<Middlewares>' // Type 'Middleware[]' is missing the following properties // from type 'Tuple<Middlewares>': prepend, concat

Решение 1: getDefaultMiddleware().concat(...)

const store = configureStore({ reducer, middleware: (gdm) => gdm().concat(myLogger) // ← Tuple сохраняет типы }) // store.dispatch остаётся типизированным: // AppDispatch = ThunkDispatch + Dispatch + ...

Решение 2: импортировать Tuple вручную

import { Tuple } from '@reduxjs/toolkit' const store = configureStore({ reducer, middleware: () => new Tuple(thunk, myLogger) // ← OK })

.prepend() — поставить ДО default

const listenerMw = createListenerMiddleware() const store = configureStore({ reducer, middleware: (gdm) => gdm().prepend(listenerMw.middleware) }) // listenerMiddleware ВСЕГДА должен быть первым // — иначе он не увидит actions, которые отвергнуты другими middleware

В DevTools ниже показан реально работающий store, собранный с использованием gdm().concat(loggerMiddleware). Кликайте кнопку — logger перехватывает action, DevTools тоже его записывает.

Теория — что такое Tuple

Из доки

«Tuple is an extension of Array, which preserves the types of its elements when calling .concat() and .prepend(). This is necessary because TypeScript's built-in Array.prototype.concat has a return type that loses tuple information.»
— middleware option docs
«Tuple — это расширение Array, которое сохраняет типы своих элементов при вызове .concat() и .prepend(). Это необходимо потому, что встроенный TypeScript-метод Array.prototype.concat имеет возвращаемый тип, теряющий tuple-информацию.»

Реализация (исходник)

// utils.ts (упрощённо) export class Tuple<Items extends ReadonlyArray<unknown>> extends Array<Items[number]> { constructor(...items: Items) { super(...items) Object.setPrototypeOf(this, Tuple.prototype) } concat<AdditionalItems extends ReadonlyArray<unknown>>( items: Tuple<AdditionalItems> | AdditionalItems ): Tuple<[...Items, ...AdditionalItems]> { return new Tuple(...this, ...items) } prepend<AdditionalItems extends ReadonlyArray<unknown>>( items: Tuple<AdditionalItems> | AdditionalItems ): Tuple<[...AdditionalItems, ...Items]> { return new Tuple(...items, ...this) } }

Ключ — typescript-сигнатура: concat<AdditionalItems extends ReadonlyArray<unknown>> возвращает [...Items, ...AdditionalItems]. Это сохраняет точный кортежный тип. У стандартного Array.concat возврат — T[], т.е. сужается до общего типа.

Когда замечаете разницу

Без Tuple store.dispatch теряет thunk-overload. Вы делаете store.dispatch(myThunk()) — и TS говорит «Argument of type 'ThunkAction<...>' is not assignable to parameter of type 'AnyAction'». Лекарство: всегда возвращайте Tuple из middleware callback.

То же для enhancers

Опция enhancers работает по тому же принципу: enhancers: (gde) => gde().concat(myEnhancer). Метод getDefaultEnhancers() возвращает Tuple, и concat сохраняет тип.

Почему не использовать as const

Можно было бы написать middleware: () => [thunk, myMw] as const — и TypeScript бы вывел кортежный тип. Но as const делает массив readonly, а internal-код RTK ожидает concat/prepend/push/.... Поэтому был выбран Tuple-класс.
