Что тут происходит

Две реализации одного и того же fetch users бок-о-бок. Левая — «ручной» thunk с тремя action creators и switch-reducer'ом. Правая — createAsyncThunk + extraReducers. Нажмите кнопку — оба store отрабатывают одинаково, но в коде разница разительная.

Слева — ручной thunk (≈40 строк)

—
// 3 action creators, switch, try/catch const fetchPending = () => ({ type: 'USERS/PENDING' }) const fetchFulfilled = (u) => ({ type: 'USERS/OK', payload: u }) const fetchRejected = (e) => ({ type: 'USERS/FAIL', error: e }) const fetchUsers = () => async (dispatch) => { dispatch(fetchPending()) try { dispatch(fetchFulfilled(await api())) } catch (e) { dispatch(fetchRejected((e as Error).message)) } } function reducer(state = init, action) { switch (action.type) { case 'USERS/PENDING': return {...state, loading:true, error:null} case 'USERS/OK': return {...state, loading:false, list:action.payload} case 'USERS/FAIL': return {...state, loading:false, error:action.error} default: return state } }

Справа — createAsyncThunk (≈12 строк)

—
// один createAsyncThunk, extraReducers на life-cycle const fetchUsers = createAsyncThunk( 'users/fetch', () => api(), ) const slice = createSlice({ name: 'users', initialState: { loading:false, list:[], error:null }, reducers: {}, extraReducers: (b) => { b.addCase(fetchUsers.pending, (s) => { s.loading=true; s.error=null }) b.addCase(fetchUsers.fulfilled, (s,a) => { s.loading=false; s.list=a.payload }) b.addCase(fetchUsers.rejected, (s,a) => { s.loading=false; s.error=a.error.message }) }, })

Цитата из документации

«createAsyncThunk accepts three parameters: a string action type value, a payloadCreator callback, and an options object. It returns a standard Redux thunk action creator that will automatically dispatch pending, fulfilled, or rejected actions based on the returned promise.»
— redux-toolkit.js.org/api/createAsyncThunk
«createAsyncThunk принимает три параметра: строку type, callback payloadCreator и объект опций. Возвращает обычный Redux thunk action creator, который автоматически диспатчит pending, fulfilled или rejected действия в зависимости от того, чем закончится promise.»

State machine

Любой async-запрос живёт в одном из пяти состояний. createAsyncThunk инкапсулирует переходы:

pending — запрос стартовал; fulfilled — payload вернулся; rejected — throw/rejectWithValue/abort/condition.

Что даёт createAsyncThunk «из коробки»

  • 3 автоматических action-creator'а: thunk.pending, thunk.fulfilled, thunk.rejected;
  • meta с arg, requestId, requestStatus, aborted, condition — для любого action'а;
  • AbortController и signal — прозрачная отмена;
  • condition — пропуск запуска до pending;
  • rejectWithValue — ошибка с типизированным payload;
  • promise с .unwrap() — для try/catch в UI-слое;
  • правильная сериализация ошибки через miniSerializeError (не теряем stack).

Фрагмент исходника — весь pending/fulfilled/rejected

// createAsyncThunk.ts, строки 539–599 const fulfilled = createAction( typePrefix + '/fulfilled', (payload, requestId, arg, meta?) => ({ payload, meta: { ...(meta || {}), arg, requestId, requestStatus: 'fulfilled' }, }), ) const pending = createAction( typePrefix + '/pending', (requestId, arg, meta?) => ({ payload: undefined, meta: { ...(meta || {}), arg, requestId, requestStatus: 'pending' }, }), ) const rejected = createAction( typePrefix + '/rejected', (error, requestId, arg, payload?, meta?) => ({ payload, error: (options?.serializeError || miniSerializeError)(error || 'Rejected'), meta: { ...(meta || {}), arg, requestId, rejectedWithValue: !!payload, requestStatus: 'rejected', aborted: error?.name === 'AbortError', condition: error?.name === 'ConditionError', }, }), )

Источник: createAsyncThunk.ts

Почему ручной thunk — проблема

1. Не стандартизировано. Каждая фича изобретает своё именование (USERS/PENDING, fetchTodos/start, posts/LOAD_BEGIN). DevTools — зоопарк.
2. Меты нет. requestId, arg, requestStatus либо отсутствуют, либо каждый раз пишутся руками — и с ошибками.
3. Нет отмены. AbortController и signal нужно прокидывать вручную; race condition guards тоже.
4. Ошибка теряется. Обычно пишут dispatch({type:'FAIL', error: e.message}) — и stack пропадает. RTK кладёт весь SerializedError.
