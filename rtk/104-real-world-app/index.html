createSlice (UI) + RTK Query (CRUD) + listenerMiddleware (analytics)

Мини-приложение «Книги». Всё вместе: RTK Query для CRUD, createSlice для локального UI-state (выделение, фильтр), listenerMiddleware для analytics events (в консоль).

Теория — 104. Real-world architecture

Разделение ответственности

── Client state (createSlice) ── - UI state: выбранная книга id, поиск, сортировка, фильтры - Непосредственно в памяти, без синхронизации с сервером - Reset на logout/navigation ── Server state (RTK Query) ── - Книги CRUD: getBooks, getBook, addBook, updateBook, deleteBook - Кеш + теги + optimistic updates - Истина живёт на сервере ── Side effects (listenerMiddleware) ── - Analytics: каждый addBook → ga.event('book_added') - Log audit на mutation fulfilled - Не требует нового useEffect в каждом компоненте

Паттерн store.ts

import { configureStore, createListenerMiddleware } from '@reduxjs/toolkit' import { booksApi } from './booksApi' import { uiSlice } from './uiSlice' const listener = createListenerMiddleware() // Подписка на all mutations → analytics listener.startListening({ matcher: isAnyOf( booksApi.endpoints.addBook.matchFulfilled, booksApi.endpoints.updateBook.matchFulfilled, booksApi.endpoints.deleteBook.matchFulfilled, ), effect: (action, { dispatch }) => { analytics.track(action.type, action.payload) }, }) export const store = configureStore({ reducer: { [booksApi.reducerPath]: booksApi.reducer, ui: uiSlice.reducer, }, middleware: (gdm) => gdm() .prepend(listener.middleware) // 👀 listener ПЕРЕД api .concat(booksApi.middleware), }) setupListeners(store.dispatch) // для refetchOnFocus

Что даёт такая композиция

  • RTKQ — тегами инвалидирует при любом add/update/delete. Оптимистические updates — через onQueryStarted + updateQueryData.
  • createSlice (UI) — хранит selectedBookId, searchQuery. Не зависит от сервера, в reducer — чистые функции.
  • listenerMiddleware — side effects (analytics, toast, navigate) на события. Не надо пихать useEffect в каждый компонент.
  • createEntityAdapter (опционально) — для normalized client cache поверх RTKQ, если нужны O(1) lookup или sorted views.

Подводные камни real-world

1. Не смешивай server state в createSlice.
Если список книг — в RTKQ, не дублируй в UI-slice. Единый источник истины. Для производных (отфильтрованных, отсортированных) — селекторы с createSelector.

2. Auth token — в UI-slice, но не в cache.
Token живёт в auth slice. RTKQ читает его через prepareHeaders в baseQuery. На logout — resetApiState + clear auth slice.

3. listenerMiddleware для cross-slice action.
Пример: при auth/logout → resetApiState. Единое место для reaction-логики.

4. Optimistic updates + tags = refetch перетрёт.
Если mutation invalidates тег, и у тебя optimistic patch — refetch после fulfilled перезапишет patch. Для optimistic UI убирай invalidate, полагайся на patch + server response.

5. Навигация — храни route-state в UI slice или URL.
React Router state vs Redux: обычно URL = source of truth для роута, Redux для UI детали. Не клади полный URL в Redux.

6. Error boundary — обяательно для RTKQ.
Ошибки в компоненте при рендере (data shape не тот) НЕ ловятся RTKQ. Error Boundary на уровне feature.
