15 реальных ошибок из продакшена — что ломает, как чинить

Список собран из issues redux-toolkit репо, моих код-ревью и опыта. Порядок — по частоте.

1. dispatch(actionCreator) вместо dispatch(actionCreator())

Забыл вызов — передаёшь САМУ функцию. actionCreatorInvariantMiddleware в dev это ловит и варнит, но в prod — молчит. State не обновляется, бага час ищется.

Fix: dispatch(increment()). Всегда вызывай с скобками.

2. Мутация в reducer возвращающем значение

(state) => { state.count++; return state.count; } — Immer ругается: «you can either mutate the draft OR return a new value, not both».

Fix: или только мутируй (state.count++), или только возвращай (return { ...state, count: state.count + 1 }). Нельзя совмещать.

3. non-serializable в state (Date, Map, class instance)

serializableStateInvariantMiddleware ловит: «non-serializable value was detected». DevTools time-travel ломается.

Fix: храни строку ISO (new Date().toISOString()) или number (Date.now()). Для Map/Set — конвертируй в объект/массив, или добавь в ignoredPaths.

4. extraReducers в объект-форме (deprecated)

Старый синтаксис extraReducers: { [action.type]: reducer } — удалён в RTK 2.x для TypeScript. Теперь только builder callback.

Fix: extraReducers: (builder) => builder.addCase(action, reducer).

5. Shallow update в entityAdapter.updateOne с nested

updateOne({ id, changes: { profile: { name: 'X' } } }) — age в profile потеряется (shallow merge).

Fix: делай полный profile или пиши свой reducer: state.entities[id].profile.name = 'X' (Immer draft).

6. Несколько createApi в одном проекте

Каждый createApi — отдельный reducer, отдельный middleware, отдельные теги. Tags одного api НЕ инвалидируют queries другого. Создаётся путаница.

Fix: один тонкий root createApi + injectEndpoints в feature-модулях. См. урок 99.

7. keepUnusedDataFor: 0 — ловушка для tabs

Переключаешь вкладку → unmount → entry умирает мгновенно → возврат → fetch. За 1 минуту активного переключения — 30 запросов.

Fix: 5-10 секунд достаточно. Или держи «высшую» подписку на уровне layout компонента (он не анмаунтится).

8. Забытый setupListeners

refetchOnFocus / refetchOnReconnect не работают молча — нет ошибки, просто не триггерится.

Fix: в store.ts один раз: setupListeners(store.dispatch) после configureStore.

9. serializeQueryArgs с нестабильными объектами в args

useGetPosts({ filter: new Date() }) — каждый рендер новый Date → новый cacheKey → бесконечный fetch. В DevTools видны 100+ entries.

Fix: нормализуй args в примитив (date.toISOString()) или оберни в useMemo. Или кастомный serializeQueryArgs.

10. Забытый as const в tag / payload

{ type: 'Post', id: 5 } — TS выведет type: string, RTKQ скажет «tag 'Post' not in tagTypes».

Fix: { type: 'Post' as const, id: 5 } или tagTypes: ['Post'] as const.

11. Забытый prepare для Date.now() / nanoid

Генерация id / timestamp в reducer — нарушает чистоту. Два dispatch с теми же args дадут разный state. Time-travel ломается.

Fix: используй prepare callback: addTodo: { reducer, prepare: (text) => ({ payload: { id: nanoid(), text, createdAt: Date.now() } }) }.

12. Двойной импорт createApi из разных entry points

import { createApi } from '@reduxjs/toolkit/query' (без хуков) vs from '@reduxjs/toolkit/query/react' (с хуками). Разные versions API в одном проекте → middleware conflict.

Fix: в React-проекте всегда импортируй из @reduxjs/toolkit/query/react. Одинаково везде.

13. useSelector на огромный объект без shallowEqual

useSelector(state => state.api.queries) — ререндерится на любое изменение любой cache entry. Тормозит всё приложение.

Fix: useSelector(selectSpecific, shallowEqual) или selectFromResult в хуке (см. урок 90).

14. Возврат draft из reducer

(state) => { state.foo = 1; return state } — Immer считает что ты вернул draft = это плохой паттерн, warn в dev.

Fix: ничего не возвращай если мутируешь: (state) => { state.foo = 1 }. Либо return { ...state, foo: 1 } без мутаций.

15. console.log(state) в reducer вместо current()

Лог покажет Immer Proxy со странной нотацией — не понятно что происходит.

Fix: import { current } from '@reduxjs/toolkit'; console.log(current(state)). Для состояния до мутаций — original(state).

Специфичные для RTK Query:

16. updateQueryData — noop если нет entry

Optimistic update не применяется — компонент с useGetX не смонтирован.

Fix: проверь через api.endpoints.X.select(arg)(state).status === 'fulfilled' или используй upsertQueryData.

17. Tags invalidation без подписчиков — ленивая

Mutation инвалидировала, но компонента со списком нет → refetch НЕ произойдёт. Следующий mount увидит «старые + invalid» → рефетч при открытии.

Fix: это осознанный выбор RTKQ. Если нужна мгновенная реакция — api.util.updateQueryData руками.

18. isLoading vs isFetching путаница

Показываешь if (isLoading) <Spinner /> — при refetch'е спиннер не появляется (это isFetching).

Fix: isLoading — только первая загрузка. isFetching — любая. Для спиннера на refetch используй isFetching. См. урок 96.
