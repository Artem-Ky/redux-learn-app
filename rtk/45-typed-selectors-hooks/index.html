Приложение в действии (React + react-redux 9.2)

Теория — 45. withTypes, identity-function и pre-typed hooks

Цитата из RTK TypeScript guide

«Starting in React-Redux v9.1, hooks have a built-in .withTypes() method for defining "pre-typed" versions of the useDispatch, useSelector, and useStore hooks.»
— RTK docs — withTypes
«Начиная с React-Redux v9.1, хуки имеют встроенный метод .withTypes() для определения «предтипизированных» версий useDispatch, useSelector и useStore.»

Что такое withTypes — «identity function с магией»

// react-redux/src/hooks/useSelector.ts — фрагмент // https://github.com/reduxjs/react-redux/blob/master/src/hooks/useSelector.ts export interface UseSelector<StateType = unknown> { // ... перегрузки вызова useSelector(selector, equalityFn?) ... withTypes: < OverrideStateType extends StateType, >() => UseSelector<OverrideStateType> // 👀 возвращает себя же, но с узким типом } // Реализация — буквально: Object.assign(useSelector, { withTypes: () => useSelector, // 👀 runtime — identity })

В runtime useSelector.withTypes<RootState>() возвращает тот же самый useSelector. Вся магия в TypeScript-сигнатуре: возвращаемый тип параметризован RootState, и TS теперь знает, что первый аргумент — селектор вида (state: RootState) => T.

createSelector.withTypes — тот же приём

// reselect/src/createSelectorCreator.ts — фрагмент // https://github.com/reduxjs/reselect/blob/master/src/createSelectorCreator.ts Object.assign(createSelector, { withTypes: () => createSelector // 👀 runtime identity }) // В TS — возвращает CreateSelectorFunction<..., OverrideStateType> // Использование: export const createAppSelector = createSelector.withTypes<RootState>() const selectTodoIds = createAppSelector( [state => state.todos], // state автоматически: RootState todos => todos.map((t) => t.id), )

Полный сет hooks.ts — стандарт индустрии

// hooks.ts — paste-and-go import { useDispatch, useSelector, useStore } from 'react-redux' import type { RootState, AppDispatch, AppStore } from './store' export const useAppDispatch = useDispatch.withTypes<AppDispatch>() export const useAppSelector = useSelector.withTypes<RootState>() export const useAppStore = useStore.withTypes<AppStore>() // Теперь в компонентах: const dispatch = useAppDispatch() // знает про thunks const users = useAppSelector((s) => s.users) // s: RootState

Как сужается тип — схема

Почему не импортировать RootState каждый раз вручную

// ❌ Старый способ — работает, но надо везде писать дженерик: import { useSelector } from 'react-redux' import type { RootState } from './store' const users = useSelector((s: RootState) => s.users) // ^^^^^^^^^^^^^ шум в каждом компоненте // ✅ Современный способ — один раз в hooks.ts: const users = useAppSelector((s) => s.users) // s уже типизирован

AppDispatch — не просто Dispatch

// store.ts export type AppDispatch = typeof store.dispatch // AppDispatch = ThunkDispatch<RootState, undefined, UnknownAction> & Dispatch<UnknownAction> // Благодаря этому: dispatch(fetchUsers()) // ✅ fetchUsers() возвращает thunk — TS не ругается // Без typed dispatch: dispatch(fetchUsers()) // ✖ Argument of type 'ThunkAction' is not assignable to...

Рекомендуемая структура проекта

src/ ├── app/ │ ├── store.ts // configureStore, RootState, AppDispatch │ ├── hooks.ts // useAppDispatch, useAppSelector, useAppStore │ └── selectors.ts // createAppSelector = createSelector.withTypes<RootState>() ├── features/ │ ├── users/ │ │ ├── usersSlice.ts │ │ ├── usersSelectors.ts // используют createAppSelector │ │ └── UsersList.tsx // используют useAppSelector │ └── ... └── main.tsx

Подводные камни

1. withTypes не валидирует runtime. Это identity function. Если вы через as привели state к неправильному типу — TS поверит, а runtime получит undefined.filter. Почему: стоимость runtime проверки типа = сериализация + сверка каждой грани. Для redux это стало бы тормозом. Решение: тесты + schema validation (zod) на границе API, если state приходит извне.
2. Не реэкспортируйте useSelector как useAppSelector. export { useSelector as useAppSelector } from 'react-redux' — на вид то же, но без .withTypes. У такого «хука» дефолтный StateType = unknown, и TS не знает shape. Почему: withTypes возвращает хук с конкретно этим StateType через дженерик-ограничение. Без вызова withTypes TS не получает RootState.
3. RootState от store.getState(), а не от { …InitialStates }. type RootState = ReturnType<typeof store.getState> — единственный правильный путь. Если собирать RootState вручную — пропустите autoInjected slices, RTK Query state, и подобное. Почему: configureStore добавляет свой собственный ключ для RTKQ, enhancers и т.д.
4. withTypes — ≥ react-redux 9.1. До 9.1 нужно было вручную создавать TypedUseSelectorHook: export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector. Если в projеct'е стоит старая версия — обновите, либо используйте старый паттерн.
5. Циркулярный импорт store ↔ hooks ↔ slice. Типичная ошибка: hooks.ts импортирует RootState из store.ts, который в свою очередь собирается из slice'ов, которые используют useAppSelector из hooks.ts. Почему: TS ещё способен это разрешить (типы «ленивы»), но bundler'ы могут словить ReferenceError в runtime. Решение: держите selector-функции в отдельном файле (usersSelectors.ts), а хуки используйте только в компонентах.
