Теория — 92. keepUnusedDataFor

Цитата — createApi · keepUnusedDataFor

«Defaults to 60 (seconds). The amount of time (in seconds) that the data should remain in the cache after the last subscriber has unsubscribed.»
— RTKQ · keepUnusedDataFor
«Это не «срок годности данных». Это «сколько entry доживёт после того, как все подписчики отвалятся». Пока хоть один useQuery на экране — данные живут бесконечно, никто их не протухает.»

Механика — 3 сцены

// 📍 Сцена 1: первый mount useGetPostsQuery() // в App // → state.api.queries['getPosts(undefined)'] = { status: 'pending' → 'fulfilled', data: [...] } // → subscriptions['getPosts(undefined)'] = { 'req-1': {...} } ← refCount = 1 // 📍 Сцена 2: второй mount (другой компонент, тот же query) useGetPostsQuery() // в List // → subscriptions[...] = { 'req-1': {...}, 'req-2': {...} } ← refCount = 2 // → data берём ИЗ КЭША, fetch НЕ идёт // 📍 Сцена 3: все unmount // Unmount App → refCount = 1, таймер НЕ стартует // Unmount List → refCount = 0 // → setTimeout(removeQueryResult, keepUnusedDataFor * 1000) // Если НОВЫЙ mount до истечения — clearTimeout, entry жива дальше // Если таймер догорел — delete state.api.queries[cacheKey]

Где задать

// глобально: createApi({ keepUnusedDataFor: 60, // default, секунды endpoints: ... }) // per-endpoint (приоритет): getPokemon: build.query({ query: (name) => `pokemon/${name}`, keepUnusedDataFor: 300, // 5 минут для покемонов }) // крайние значения: keepUnusedDataFor: 0 // удалять сразу в следующем тике (≈ 4ms) keepUnusedDataFor: Infinity // держать до resetApiState()

Зачем дефолт именно 60

Типичный UX: пользователь листает таб → возвращается через минуту → хочет мгновенный ответ. 60 секунд — баланс «не захламлять память» vs «не дёргать сеть попусту». Для критичных данных (курс валют) — уменьшай до 5-10. Для редко меняющегося (список стран) — ставь Infinity.

Подводные камни

1. keepUnusedDataFor ≠ «данные свежие N секунд».
Пока компонент подписан, RTKQ НЕ протухает данные сам. Держишь страницу час — час смотришь те же данные (если не было invalidate / refetch / polling).

2. keepUnusedDataFor: 0 — ловушка для последовательных рендеров.
Unmount компонента → микротик → entry удалена → новый mount → fetch. Если у тебя навигация туда-сюда (tabs), каждый клик = запрос. Лучше 1-5 секунд.

3. Infinity не освобождает память при logout.
Нужен dispatch(api.util.resetApiState()) на logout явно — иначе данные прошлого юзера останутся в store.

4. Не работает для invalidateTags без подписчиков.
Mutation инвалидировала тег, но подписчиков нет → entry помечена «невалид», но не удаляется. Умрёт только по keepUnusedDataFor таймеру. Следующий mount увидит «невалид» → пойдёт fetch заново.

5. Per-endpoint keepUnusedDataFor перекрывает глобальный.
Global=300, endpoint=5 → для этого endpoint'а 5, не 305. Иерархия: endpoint > global. Нет «наследования» — только override.

6. Таймер НЕ переживает resetApiState.
Reset очищает всё, включая pending timeouts. Новый fetch начнёт всё сначала.
