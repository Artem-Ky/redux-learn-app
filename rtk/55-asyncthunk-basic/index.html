Что такое thunk.pending / fulfilled / rejected

Это обычные createAction-креаторы с prepare callback. Каждый получает requestId, arg и кладёт их в action.meta.

// createAsyncThunk.ts, строки 539–599 — как создаются 3 креатора const fulfilled = createAction( typePrefix + '/fulfilled', (payload, requestId, arg, meta?) => ({ payload, meta: { ...(meta || {}), arg, requestId, requestStatus: 'fulfilled' }, }), ) const pending = createAction( typePrefix + '/pending', (requestId, arg, meta?) => ({ payload: undefined, meta: { ...(meta || {}), arg, requestId, requestStatus: 'pending' }, }), ) const rejected = createAction( typePrefix + '/rejected', (error, requestId, arg, payload?, meta?) => ({ payload, error: (options?.serializeError || miniSerializeError)(error || 'Rejected'), meta: { ...(meta || {}), arg, requestId, rejectedWithValue: !!payload, requestStatus: 'rejected', aborted: error?.name === 'AbortError', condition: error?.name === 'ConditionError', }, }), )

createAsyncThunk.ts#L539

Что лежит в action.meta

  • arg — ровно тот аргумент, который передали в thunk(arg). В reducer удобно использовать для diff ID;
  • requestId — nanoid, сгенерированный в момент dispatch. Одинаков для pending / fulfilled / rejected одного запроса;
  • requestStatus — дискриминант: 'pending' | 'fulfilled' | 'rejected';
  • aborted — true, если ошибка была AbortError (только в rejected);
  • condition — true, если сработал options.condition → false (только в rejected);
  • rejectedWithValue — true, если payload пришёл через rejectWithValue.

thunkAPI — второй параметр payload-creator'а

async (arg, thunkAPI) => { thunkAPI.dispatch // полноценный dispatch (можно слать другие actions/thunks) thunkAPI.getState() // RootState — актуальный, не snapshot старта thunkAPI.extra // extraArgument из middleware (apiClient, history…) thunkAPI.requestId // тот же, что в meta thunkAPI.signal // AbortSignal — пробрасываем в fetch thunkAPI.abort(reason?) // отменить this запрос thunkAPI.rejectWithValue // вернуть типизированную ошибку (см. урок 58) thunkAPI.fulfillWithValue // добавить meta к fulfilled (см. урок 62) }

Подводные камни

1. Promise от thunk никогда не отклоняется. Даже если payload-creator бросил исключение — возвращённый promise всё равно fulfilled. Он содержит rejected action, а не reject. То есть await store.dispatch(thunk()) никогда не уйдёт в catch. Для try/catch используйте .unwrap():
try { const user = await store.dispatch(fetchUserById(1)).unwrap() console.log('payload:', user) } catch (e) { console.error('реальная ошибка:', e) }
2. action.payload (rejectWithValue) vs action.error. В rejected есть и payload, и error, но они значат разное:
  • throw new Error('...') → action.payload === undefined, action.error = SerializedError;
  • return rejectWithValue({code:404}) → action.payload = {code:404}, action.error = { message: 'Rejected' }.
