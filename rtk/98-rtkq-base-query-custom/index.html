Custom baseQuery с авто-reauth на 401

Свой baseQuery — это функция (args, api, extraOpts) → { data } | { error }. Здесь: обёртка над fetchBaseQuery с refresh-token логикой: на 401 → POST /refresh → повторяет оригинальный запрос с новым токеном.

Теория — 98. Custom baseQuery

Цитата — Customizing Queries

«A baseQuery is just a function with the signature (args, api, extraOptions) => Promise<{ data } | { error }>. You can wrap it, compose with other base queries, or write your own from scratch for axios / graphql / custom transports.»
— RTKQ · Implementing a Custom baseQuery

Сигнатура baseQuery

type BaseQueryFn< Args = any, Result = unknown, Error = unknown, DefinitionExtraOptions = {}, Meta = {}, > = ( args: Args, api: BaseQueryApi, extraOptions: DefinitionExtraOptions, ) => Promise< | { data: Result; error?: undefined; meta?: Meta } | { error: Error; data?: undefined; meta?: Meta } >

Пример 1 — axios baseQuery

import axios, { type AxiosRequestConfig, type AxiosError } from 'axios' import type { BaseQueryFn } from '@reduxjs/toolkit/query' const axiosBaseQuery = ( { baseUrl }: { baseUrl: string } = { baseUrl: '' }, ): BaseQueryFn< { url: string; method?: AxiosRequestConfig['method']; data?: unknown; params?: unknown }, unknown, { status?: number; data?: unknown; message?: string } > => async ({ url, method, data, params }) => { try { const r = await axios({ url: baseUrl + url, method, data, params }) return { data: r.data } } catch (e) { const err = e as AxiosError return { error: { status: err.response?.status, data: err.response?.data, message: err.message } } } } createApi({ baseQuery: axiosBaseQuery({ baseUrl: '/api' }), ... })

Пример 2 — graphql-request baseQuery

import { GraphQLClient, ClientError } from 'graphql-request' const client = new GraphQLClient('/graphql') const graphqlBaseQuery: BaseQueryFn< { document: string; variables?: object } > = async ({ document, variables }) => { try { const data = await client.request(document, variables) return { data } } catch (e) { if (e instanceof ClientError) { return { error: { status: e.response.status, data: e.response.errors } } } return { error: { status: 'FETCH_ERROR', data: String(e) } } } }

Пример 3 — reauth обёртка (на 401)

import { fetchBaseQuery } from '@reduxjs/toolkit/query/react' import type { BaseQueryFn, FetchArgs, FetchBaseQueryError } from '@reduxjs/toolkit/query' import { Mutex } from 'async-mutex' const baseQuery = fetchBaseQuery({ baseUrl: '/api', prepareHeaders: (headers, { getState }) => { const token = (getState() as RootState).auth.token if (token) headers.set('Authorization', `Bearer ${token}`) return headers }, }) const mutex = new Mutex() // защита от параллельных refresh'ей const baseQueryWithReauth: BaseQueryFn< string | FetchArgs, unknown, FetchBaseQueryError > = async (args, api, extraOptions) => { await mutex.waitForUnlock() let result = await baseQuery(args, api, extraOptions) if (result.error?.status === 401) { if (!mutex.isLocked()) { const release = await mutex.acquire() try { const refresh = await baseQuery({ url: '/auth/refresh', method: 'POST' }, api, extraOptions) if (refresh.data) { api.dispatch(setToken((refresh.data as any).token)) result = await baseQuery(args, api, extraOptions) // повторяем оригинальный } else { api.dispatch(logout()) } } finally { release() } } else { await mutex.waitForUnlock() result = await baseQuery(args, api, extraOptions) } } return result }

BaseQueryApi — что доступно внутри

interface BaseQueryApi { signal: AbortSignal // отмена через initiate.abort() abort: (reason?: string) => void dispatch: ThunkDispatch<...> getState: () => RootState extra: unknown // твой thunk extra endpoint: string // имя endpoint'а type: 'query' | 'mutation' forced?: boolean // true если forceRefetch queryCacheKey?: string // для queries }

Подводные камни

1. Возвращаемая структура должна быть ровно {data} или {error}.
Не {data, error} оба, не голый data. Если оба — поведение undefined. Если бросил Error — RTKQ его оборачивает сам в SerializedError, но лучше явно.

2. Custom baseQuery → type error становится твой тип, не FetchBaseQueryError.
В endpoint определении, в transformErrorResponse, в component через q.error — всё будет твой shape. Не забудь проставить generic в BaseQueryFn<Args, Result, Error>.

3. Reauth без mutex = N параллельных refresh'ей.
Если 5 запросов одновременно вернули 401, без mutex все пятеро начнут refresh. Решение — async-mutex или свой флаг isRefreshing.

4. signal нужно прокидывать в fetch/axios.
Если не прокидываешь api.signal в transport — отмена через promise.abort() не сработает.

5. Composing baseQueries — можно через replay.
retry(wrapper(base), ...) — wrappers можно цепочкой: reauth внутри, retry снаружи. Порядок важен: retry не должен ретраить 401, иначе лупинг.

6. В custom baseQuery ты САМ ответственен за JSON parsing.
fetchBaseQuery делает это за тебя. Кастом — ты берёшь raw Response и решаешь как парсить. Не забудь про ошибки JSON.parse.
