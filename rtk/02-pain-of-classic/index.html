Боли классики vs лекарства RTK
Боль: чтобы поднять production-ready store, нужно вручную собрать combineReducers, applyMiddleware, composeEnhancers, подключить DevTools-extension и thunk. 25+ строк только на инфраструктуру.
Classic Redux ~28 строк
import { createStore, combineReducers,
         applyMiddleware, compose } from 'redux'
import thunk from 'redux-thunk'
import logger from 'redux-logger'
import counterReducer from './counter'
import todosReducer from './todos'

const rootReducer = combineReducers({
  counter: counterReducer,
  todos: todosReducer,
})

const composeEnhancers =
  (window as any).__REDUX_DEVTOOLS_EXTENSION_COMPOSE__
  || compose

const store = createStore(
  rootReducer,
  /* preloadedState */ undefined,
  composeEnhancers(
    applyMiddleware(thunk, logger)
  )
)

// Без проверок на immutability!
// Без проверок на serializability!
// devtools-extension не подключится без window-проверки.

Боль: для каждого action — константа + creator + кейс в switch + spread-копирование. Минимальный todos-slice занимает 50+ строк. RTK генерирует всё это из объекта.
todos.ts (классика) ~52 строки
const ADD = 'todos/added'
const TOGGLE = 'todos/toggled'
const REMOVE = 'todos/removed'

let nextId = 1
export const addTodo = (text) =>
  ({ type: ADD, payload: { id: nextId++, text, done: false } })
export const toggleTodo = (id) =>
  ({ type: TOGGLE, payload: id })
export const removeTodo = (id) =>
  ({ type: REMOVE, payload: id })

const initial = []

export default function todos(state = initial, action) {
  switch (action.type) {
    case ADD:
      return [...state, action.payload]
    case TOGGLE:
      return state.map(t =>
        t.id === action.payload
          ? { ...t, done: !t.done }
          : t
      )
    case REMOVE:
      return state.filter(t => t.id !== action.payload)
    default:
      return state
  }
}
// + забываем, что nextId вне reducer ломает HMR/SSR.
// + забываем про immutability check.
todosSlice.ts (RTK) ~22 строки
import { createSlice, nanoid, PayloadAction } from '@reduxjs/toolkit'

interface Todo { id: string; text: string; done: boolean }

const todosSlice = createSlice({
  name: 'todos',
  initialState: [] as Todo[],
  reducers: {
    addTodo: {
      reducer(state, action: PayloadAction<Todo>) {
        state.push(action.payload) // mutating через Immer!
      },
      prepare(text: string) {
        return { payload: { id: nanoid(), text, done: false } }
      },
    },
    toggleTodo(state, { payload: id }: PayloadAction<string>) {
      const t = state.find(t => t.id === id)
      if (t) t.done = !t.done
    },
    removeTodo(state, { payload: id }: PayloadAction<string>) {
      return state.filter(t => t.id !== id)
    },
  },
})

export const { addTodo, toggleTodo, removeTodo } = todosSlice.actions
export default todosSlice.reducer

Теория — Почему RTK существует

Дословно из документации

«It was originally created to help address three common concerns about Redux:
  • "Configuring a Redux store is too complicated"
  • "I have to add a lot of packages to get Redux to do anything useful"
  • "Redux requires too much boilerplate code"
We can't solve every use case, but in the spirit of create-react-app, we can try to provide some tools that abstract over the setup process and handle the most common use cases.»
— Getting Started — Purpose
«Изначально RTK был создан, чтобы решить три типичные жалобы на Redux:
  • «Настройка Redux store слишком сложная»
  • «Чтобы получить от Redux хоть какую-то пользу, приходится ставить много пакетов»
  • «Redux требует слишком много шаблонного кода»
Мы не можем решить все задачи сразу, но в духе create-react-app можем дать инструменты, которые абстрагируют процесс настройки и закрывают самые частые сценарии.»

Боль 1 — Setup

Классический createStore требует ручной композиции: combineReducers для нескольких slice'ов, applyMiddleware для middleware, compose для соединения нескольких enhancer'ов, и обязательная проверка window.__REDUX_DEVTOOLS_EXTENSION_COMPOSE__.

В RTK configureStore делает всё это за вас плюс добавляет 4 default middleware:

  • redux-thunk — для асинхронной логики (по умолчанию)
  • immutableStateInvariantMiddleware — ловит мутации state в reducer'ах (только dev)
  • serializableStateInvariantMiddleware — ловит несериализуемые значения в state/action (только dev)
  • actionCreatorInvariantMiddleware — ловит вызов dispatch(actionCreator) вместо dispatch(actionCreator()) (только dev)

Плюс автоматически подключает Redux DevTools и autoBatchEnhancer. Подробно об этом — уроки 11-15.

Боль 2 — Зоопарк пакетов

Чтобы получить полный набор инструментов в классике, нужно установить и согласовать версии: redux, redux-thunk, reselect, immer, redux-devtools-extension (или composeWithDevTools). В RTK все эти зависимости либо bundled, либо peer, либо re-exported из @reduxjs/toolkit. См. packages/toolkit/package.json.

Боль 3 — Boilerplate

В классическом Redux одно "событие" требует:

  1. Константу для action.type (защита от опечаток)
  2. Action creator (фабрика action-объекта)
  3. Case в reducer'е
  4. Spread-копирование для иммутабельного обновления

В RTK один ключ в объекте reducers: — это и action type (генерируется как ${name}/${key}), и action creator (доступен через slice.actions.key), и case-reducer. Плюс Immer позволяет писать мутирующий код, который RTK конвертирует в immutable update.

«createSlice accepts an object of reducer functions, a slice name, and an initial state value, and automatically generates a slice reducer with corresponding action creators and action types.»
— What's Included
«createSlice принимает объект reducer-функций, имя slice и начальное значение state, и автоматически генерирует slice-reducer вместе с соответствующими action creators и action types.»
