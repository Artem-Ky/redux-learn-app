100 строк · лайк одной — сколько перерендерилось?

Слева — строки подписаны на весь data (наивно). Справа — через selectFromResult, выбирая только свой пост. Жмёшь лайк — в левой колонке перерендериваются ВСЕ, в правой — только изменённая.

Теория — 90. selectFromResult

Цитата — Selecting data from a query result

«Sometimes you may have a parent component that subscribes to a query, and then multiple children that need to read individual items from that cached data. selectFromResult lets you select specific pieces of data, and ensures that the component only re-renders when that selected subset changes.»
— RTKQ · Queries — selectFromResult
«Каждый useQuery — это подписка. Без selectFromResult компонент ререндерится при любом изменении cache entry. С selectFromResult — только когда твой кусок сменил reference (проверка через shallow equality).»

Сигнатура

const { post, isLoading } = useGetPostsQuery(undefined, { selectFromResult: ({ data, isLoading }) => ({ post: data?.find((p) => p.id === id), isLoading, }), }) // ↑ возвращается твой объект; дефолтные поля (error, isFetching, ...) НЕ попадают — только те, что вернул сам.

Как работает «только нужное изменилось»

Под капотом — createSelector из reselect + shallowEqual на возвращаемый объект:

// упрощённая логика из buildHooks.ts const selectorFactory = () => createSelector( [selectQueryResult, (_, arg) => arg], (result) => selectFromResult!(result), ) // useSelector с shallowEqual const subset = useSelector(selectorFactory(), shallowEqual) // ↑ только если shallow-форма subset поменялась → ререндер

Вместе со structural sharing RTKQ (урок 84) это даёт: refetch массива той же формы → reference у items стабильны → find возвращает ту же ссылку → shallow-eq true → ререндера нет.

Три типичных паттерна

// ─── 1. Один item из списка ─── function PostRow({ id }) { const { post } = useGetPostsQuery(undefined, { selectFromResult: ({ data }) => ({ post: data?.find((p) => p.id === id) }), }) return <div>{post?.title}</div> } // ─── 2. Вычисленное значение ─── const { unreadCount } = useGetNotificationsQuery(undefined, { selectFromResult: ({ data }) => ({ unreadCount: data?.filter((n) => !n.read).length ?? 0, }), }) // ─── 3. Normalized lookup (с entity adapter) ─── const { post } = useGetPostsQuery(undefined, { selectFromResult: ({ data }) => ({ post: data?.entities[id] }), // O(1) })

Альтернатива — api.endpoints.X.select(arg)

Подводные камни

1. Новый объект каждый selectFromResult — shallowEqual всё равно спасёт.
selectFromResult: ({ data }) => ({ x: data?.foo }) возвращает новый объект при каждом вызове. Но shallow-eq сравнивает его поля. Если x имеет ту же ссылку — ререндера нет. Но если x это результат .filter()/.map()/.find() без стабильных входов — новая ссылка, ререндер. Используй структурно разделённые данные или memoized селекторы.

2. selectFromResult не уменьшает сетевой трафик.
Это только оптимизация рендеров. Подписка создаётся всё равно → fetch происходит как обычно. Если хочешь избежать запроса — это skip.

3. Если хук используется в 100 компонентах — 100 селекторов.
Каждый useQuery создаёт свой инстанс селектора. Это нормально — reselect делает хеш по ссылке входа, перформанс O(1) на каждый. Но не забывай: 100 подписок в state.api.subscriptions.

4. Не вызывай dispatch внутри selectFromResult.
Селектор вызывается в reactor'е React — side effects ломают concurrent rendering. Если нужно среагировать на данные — useEffect по вычисленному куску.

5. selectFromResult vs getSelectors (entity adapter).
Если в data лежит EntityState — getSelectors возвращают мемоизированные селекторы с weakMapMemoize. Комбинация: один общий entity, детали берут selectById через selectFromResult.

6. Вернул isFetching — потерял типизацию дефолта.
selectFromResult переопределяет форму возврата. Если вернул только {post} — isLoading и прочие поля недоступны. Нужны — включай в возврат.
