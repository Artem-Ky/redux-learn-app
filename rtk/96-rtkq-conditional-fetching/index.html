Пагинация — смотри разницу между data/currentData в момент смены страницы

Листай страницы. data = последний успешный ответ ЛЮБОГО arg. currentData = последний успешный для ЭТОГО arg. При смене страницы разница видна мгновенно.

Теория — 96. data/currentData · isLoading/isFetching

Цитата — Frequently Used Query Hook Return Values

«data — the latest returned result regardless of hook arg, if present.
currentData — the latest returned result for the current hook arg, if present.
isLoading — true only on the very first load (no previous data for this hook).
isFetching — true whenever a request is in flight (including background refetch).»
— RTKQ · Query hook return values

Разница data vs currentData на примере

// Текущее состояние: // - useGetPosts({page:1}) отработал успешно, cache HIT → data=[...page1 posts] // - меняем arg на {page:2} → новый cacheKey, идёт fetch useGetPostsQuery({ page: 2 }) // СРАЗУ после смены arg, пока fetch в процессе: // data = [...page1 posts] ← «последний успех любого arg» // currentData = undefined ← «для ЭТОГО arg ещё нет» // isFetching = true // isLoading = false ← это не «первая» загрузка, data уже был // Когда fulfilled: // data = [...page2 posts] // currentData = [...page2 posts]

UX-паттерны — что показывать

// ─── A. Показывать «призрак» старых данных (смазанная пагинация) ─── const { data, isFetching } = useGetPostsQuery({ page }) return ( <div style={{ opacity: isFetching ? 0.5 : 1 }}> {data?.map(...)} </div> ) // ─── B. Чистый UX — пусто пока нет данных для ТЕКУЩЕГО arg ─── const { currentData, isFetching } = useGetPostsQuery({ page }) return currentData ? <PostList posts={currentData} /> : <Skeleton /> // ─── C. First load vs background refetch ─── const { data, isLoading, isFetching } = useGetPostsQuery() if (isLoading) return <FullPageSpinner /> // впервые if (isFetching) return <TopBarLoader /> // обновление в фоне return <PostList posts={data!} />

Источник — buildHooks.ts (упрощённо)

const subSelector = createSelector( [selectQueryResult, (_, arg) => arg], (baseResult, queryArgs) => { return { ...baseResult, // currentData = data ТОЛЬКО если currentArg совпадает с sourceArg entry'и: currentData: queryArgs === baseResult.originalArgs ? baseResult.data : undefined, isFetching: baseResult.status === 'pending', isLoading: baseResult.status === 'pending' && baseResult.data === undefined, isSuccess: baseResult.status === 'fulfilled', isError: baseResult.status === 'rejected', } }, )

Подводные камни

1. isLoading — только на САМУЮ первую загрузку этого hook-инстанса.
После первого успеха при refetch isLoading=false, только isFetching=true. Это тонко: хук в другом компоненте, впервые смотрящий этот же query — увидит isLoading=false сразу (HIT из чужого кеша).

2. data «застревает» от прошлого arg.
Смена page=1 → 2 — data на 400ms остаётся страницей 1. Если не хочешь мигания — рисуй UI с currentData + skeleton.

3. isError с data !== undefined — возможно.
Если ранее был успех, а потом refetch упал — isError=true и одновременно data показывает старые данные. Решай: показывать stale data с toast, или прятать.

4. currentData=undefined при FIRST SUCCESS тоже.
Не только «пока грузим». Проверка: if (currentData) ... — безопаснее чем if (!isLoading).

5. isUninitialized ≠ isFetching=false + data=undefined.
Это когда skip=true или entry полностью удалена. isSuccess/isError=false, status='uninitialized'. Важно чтобы не показать «ошибку» пользователю когда просто skip.

6. status vs флаги.
Поле status — строка ('uninitialized'|'pending'|'fulfilled'|'rejected'). Флаги — производные от неё. Используй status если нужна switch-логика.
