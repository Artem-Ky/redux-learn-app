Четыре режима оркестрации — сравнение таймеров и поведения

dispatch(thunk()) возвращает обычный Promise. Значит мы можем комбинировать thunk'и любым нативным Promise-методом. Четыре главных паттерна — ниже. Нажми кнопку в любой карточке, смотри таймер и список задач.

1Sequential · ожидание по цепочке

await dispatch(fetchUser(1)).unwrap() await dispatch(fetchPosts(uid)).unwrap()
—

Общее время ≈ сумма. Второй ждёт первого.

2Parallel · Promise.all

await Promise.all([ dispatch(fetchA()), dispatch(fetchB()), dispatch(fetchC()), ])
—

Общее время ≈ max. Отваливается на первом reject.

3Race · Promise.race

await Promise.race([ dispatch(slowFetch()), dispatch(fastFetch()), ])
—

Кто быстрее — тот выигрывает. Медленный продолжает идти в фоне (zombie).

4AllSettled · довести всё до конца

await Promise.allSettled([ dispatch(fetchA()), dispatch(fetchB()), dispatch(failingFetch()), ])
—

Упавший не прерывает остальных. Получаем массив {status, value|reason}.

Теория — 74. Четыре паттерна композиции async thunk'ов

Цитата из доки

«The dispatched action returned by dispatch(someAsyncThunk()) is always a Promise that will be fulfilled with a pending/fulfilled/rejected action, never rejected itself. Use .unwrap() to get a regular rejected-promise if you want to use .then/.catch.»
— redux-toolkit.js.org/api/createAsyncThunk
«dispatch(thunk()) всегда даёт fulfilled Promise — даже если внутри thunk бросил Error. Чтобы получить «нормальный» отклоняющийся Promise и использовать try/catch — вызови .unwrap().»

Почему .unwrap() нужен

// БЕЗ unwrap: const resultAction = await dispatch(fetchUser(1)) // resultAction — это либо fulfilled-action, либо rejected-action, // но Promise, который вернул dispatch, — всегда RESOLVED. if (fetchUser.fulfilled.match(resultAction)) { const user = resultAction.payload // ✅ типизированный User } else { const err = resultAction.error // ❌ SerializedError } // С unwrap: try { const user = await dispatch(fetchUser(1)).unwrap() // сразу User } catch (e) { // e — payload из rejectWithValue ИЛИ SerializedError }

Promise.all / Promise.race / Promise.allSettled работают с поведением Promise: им важен reject/fulfill. Без .unwrap() thunk всегда fulfilled — Promise.all никогда не упадёт, даже если половина thunk'ов отклонилась. Это тихий баг. Всегда пиши .unwrap() для управляющей логики.

Паттерн 1: Sequential — цепочка

// Второй thunk использует payload первого — только sequential. async function loadFeed() { const user = await dispatch(fetchUser()).unwrap() const posts = await dispatch(fetchPosts(user.id)).unwrap() return { user, posts } } // Время ≈ t(user) + t(posts).

Паттерн 2: Parallel — Promise.all

// Независимые запросы — параллельно. async function loadDashboard() { const [a, b, c] = await Promise.all([ dispatch(fetchA()).unwrap(), dispatch(fetchB()).unwrap(), dispatch(fetchC()).unwrap(), ]) return { a, b, c } } // Время ≈ max(ta, tb, tc). // ❗ При первом reject — всё отваливается (fail-fast).

Паттерн 3: Race — Promise.race

// Кто первый ответил, того берём (fallback, retry, timeout…) async function withTimeout() { const p = dispatch(slowFetch()) const winner = await Promise.race([ p.unwrap(), new Promise((_, reject) => setTimeout(() => reject(new Error('timeout')), 500)), ]) // ❗ slowFetch продолжает идти в фоне! Прерывание не автоматическое. // Если нужно отменить — p.abort() (AsyncThunk поддерживает AbortSignal). return winner }

Паттерн 4: AllSettled — «собрать всё, что получилось»

// Несколько независимых запросов. Упавший не должен ломать остальные. async function loadEverything() { const results = await Promise.allSettled([ dispatch(fetchA()).unwrap(), dispatch(fetchB()).unwrap(), dispatch(failingFetch()).unwrap(), ]) for (const r of results) { if (r.status === 'fulfilled') console.log('ok', r.value) else console.warn('fail', r.reason) } }

Сравнение по временам и устойчивости

Паттерн Время На reject одного Когда выбрать
Sequential сумма цепочка прерывается зависимости по данным (user → posts)
Promise.all max fail-fast — всё ломается всё-или-ничего (критичные данные)
Promise.race min reject может выиграть timeout, fastest-first, fallback
Promise.allSettled max не влияет на остальные независимые фичи, «best effort»
Подводные камни:
  • Race не отменяет проигравшего. Победитель выиграл — второй Promise продолжает идти в фоне, делает сетевой запрос, диспатчит fulfilled/rejected action. Твой reducer получит «призрачный» апдейт от проигравшего. Решение: const p = dispatch(thunk()); Promise.race([p.unwrap(), timeout]); p.abort() — AsyncThunkAction поддерживает .abort() из коробки (см. урок 60).
  • Promise.all и fail-fast. Если один из трёх thunk'ов отвалится через 50мс, два остальных дойдут до конца — но Promise.all уже reject'нулся. Они дадут .fulfilled действие, которое попадёт в state после ошибки. Иногда это ok (опишет какое-то валидное состояние), иногда нет (перетрёт error-flag на success). Решение: Promise.allSettled или явная проверка requestId в reducer'е.
  • Sequential — легко сделать медленным случайно. Если у тебя for (const id of ids) await dispatch(fetchOne(id)).unwrap() — это 10 последовательных запросов вместо 10 параллельных. Решение: await Promise.all(ids.map(id => dispatch(fetchOne(id)).unwrap())). Sequential имеет смысл только если id's зависят друг от друга.
  • .unwrap() на Promise.all(...).unwrap() не работает. unwrap() — метод на AsyncThunkAction, не на Promise. Пиши Promise.all([p1.unwrap(), p2.unwrap()]), а не Promise.all([p1, p2]).unwrap().
  • AllSettled не даёт узнать какой request упал без доп. информации. Массив результатов идёт в том же порядке, что и входной — но если у тебя динамический массив thunk'ов, индекса может не хватить. Решение: собирай {name, promise} и пробегайся по ним после allSettled, либо используй оригинальные indices.
