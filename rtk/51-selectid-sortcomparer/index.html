Опции createEntityAdapter

const booksAdapter = createEntityAdapter<Book, string>({ selectId: (b) => b.isbn, // ключ — любое поле sortComparer: (a, b) => a.title.localeCompare(b.title), })

Цитата — RTK docs

«If the adapter is provided a sortComparer, it will keep the state.ids array in sorted order by passing two entities at a time to the comparer. This is useful when sorting is a view concern and is the same everywhere in the app.»
«Если adapter'у передан sortComparer, он поддерживает state.ids всегда отсортированным, передавая пары записей в comparer. Полезно, когда порядок — это бизнес-правило, единое по всему приложению.»

Исходник — sorted_state_adapter.ts

// Бинарный поиск позиции вставки в отсортированный массив export function findInsertIndex(sortedItems, item, comparisonFunction): number { let lowIndex = 0 let highIndex = sortedItems.length while (lowIndex < highIndex) { let middleIndex = (lowIndex + highIndex) >>> 1 const res = comparisonFunction(item, sortedItems[middleIndex]) if (res >= 0) lowIndex = middleIndex + 1 else highIndex = middleIndex } return lowIndex } const mergeFunction = (state, addedItems, appliedUpdates, replacedIds) => { const currentEntities = getCurrent(state.entities) const currentIds = getCurrent(state.ids) let sortedEntities = [] for (const id of currentIds) { const entity = currentEntities[id] if (entity) sortedEntities.push(entity) } const wasPreviouslyEmpty = sortedEntities.length === 0 for (const item of addedItems) { stateEntities[selectId(item)] = item if (!wasPreviouslyEmpty) { insert(sortedEntities, item, comparer) // ← findInsertIndex + splice } } if (wasPreviouslyEmpty) { sortedEntities = addedItems.slice().sort(comparer) } else if (appliedUpdates) { sortedEntities.sort(comparer) // mostly-sorted → Timsort быстрый } const newSortedIds = sortedEntities.map(selectId) if (!areArraysEqual(currentIds, newSortedIds)) { state.ids = newSortedIds } }

Как это работает на практике

  1. addOne: бинарный поиск (O(log n)) позиции + splice (O(n) из-за массива). Итого O(n) на вставку.
  2. updateOne: мутируем поле, потом sortedEntities.sort(comparer) для «почти отсортированного» массива. В V8 это Timsort — тоже почти O(n).
  3. ids пересобирается только если фактически изменился (через areArraysEqual) — чтобы не трогать ссылку зря.

Цитата о «sort всё в adapter»

«Because sorting is controlled by the adapter, you should perform sorting through its CRUD methods only — adding items through your own reducer logic bypasses the sortComparer.»
«Сортировка — ответственность adapter'а. Используй его CRUD-методы; прямые мутации state.ids.push или state.entities[…] = … обходят sortComparer.»

Подводные камни

1. Дорого на больших коллекциях. При 10k+ элементов sort после каждой мутации становится заметным (~O(n log n), на каждый add/update). Если сортировок не избежать — рассматривай sorted data structures (например, skip list) или поддерживай порядок «руками» через extra-reducers.

2. Нестабильный comparer ломает индексы. comparer(a, a) должен быть 0, не любое число. В бинпоиске используется res >= 0 → lowIndex = mid + 1, и нестабильный compare (например, Math.random() - 0.5) приведёт к непредсказуемой позиции.
Дополнительно: comparer должен быть транзитивным. Для строк — всегда localeCompare, для чисел — a - b.

3. Смена поля, по которому сортируем. Если sortComparer смотрит на price, а ты делаешь updateOne({ id, changes: { price: 999 } }) — RTK вызовет mergeFunction(state, [], true) → sort, и запись переедет. Если забудешь пройти через adapter (прямая мутация entities[id].price = 999 в обход), позиция останется старой. Ошибка синхронизации.

4. Порядок ≠ стабильный для равных. Если два title одинаковые — порядок между ними может меняться между addOne. Добавь вторичный ключ: (a, b) => a.title.localeCompare(b.title) || a.isbn.localeCompare(b.isbn).
