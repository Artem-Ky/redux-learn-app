Как builder решает, кого вызвать

Здесь два разных «порядка», их часто путают. Разделим:

  1. Правило написания кода (constraint API). Builder бросает ошибку, если нарушить порядок: сначала идут все addCase, затем все addMatcher, и в самом конце один addDefaultCase. После первого addMatcher уже нельзя писать addCase.
  2. Порядок выполнения при dispatch (runtime). Для каждого action reducer:
    (а) ищет ОДИН подходящий addCase по точному action.type →
    (б) затем перебирает все addMatcher в том порядке, в каком они были добавлены, и вызывает handler'ы у тех, что вернули true →
    (в) addDefaultCase вызывается, только если ни (а), ни (б) ничего не нашли.

Эти два порядка совпадают: правило (1) делает невозможным написать addMatcher раньше addCase, поэтому в (2) addCase-handler всегда вызывается первым, а matcher'ы — после, в порядке записи.

createReducer(initial, (builder) => { builder .addCase(userLoggedIn, handlerA) // 1 .addCase(userLoggedOut, handlerB) // 2 .addMatcher(isUserAction, handlerC) // 3 .addMatcher(isRejected, handlerD) // 4 .addDefaultCase(fallbackHandler) // 5 — последний })

Теория — addMatcher + addDefaultCase

Из доки

«The addMatcher method accepts a matcher function and a reducer. If multiple addCase or addMatcher reducers match the action, they will each be called in the order they were added to the builder.»
— addMatcher API
«Метод addMatcher принимает matcher-функцию и reducer. Если несколько addCase или addMatcher совпадают с action, они все вызываются в том порядке, в котором были добавлены в builder.»

addDefaultCase — из доки

«addDefaultCase registers a reducer that will be called if no other reducer was called for a given action.»
— addDefaultCase API
«addDefaultCase регистрирует reducer, который будет вызван, если ни один другой reducer не отработал для данного action.»

Два разных «порядка» — без путаницы

A. Правила НАПИСАНИЯ кода (enforced API)

  1. addCase можно вызывать только до первого addMatcher. Иначе builder бросит: builder.addCase should only be called before calling builder.addMatcher.
  2. addMatcher можно цепочкой, сколько угодно — но до addDefaultCase.
  3. addDefaultCase можно вызвать только один раз и только последним.
  4. Дублировать addCase на один и тот же action.type запрещено.
// ✗ Throws: addCase after addMatcher builder .addMatcher(m, h) .addCase(a, h2) // ошибка // ✓ Правильный порядок builder .addCase(a1, h1) .addCase(a2, h2) // несколько addCase подряд — ок .addMatcher(m1, h3) .addMatcher(m2, h4) // несколько addMatcher подряд — ок .addDefaultCase(fallback) // только один и только последним

B. Порядок ВЫПОЛНЕНИЯ при dispatch (runtime)

Внутри createReducer каждый dispatch проходит три шага. Вот фактический псевдокод из createReducer.ts:

// 1. Кладём в массив один (или ноль) handler из addCase // + ВСЕ handler'ы из addMatcher, чьи matcher'ы вернули true, // в порядке, в котором они были добавлены в builder. let caseReducers = [ actionsMap[action.type], // addCase ...finalActionMatchers .filter(({ matcher }) => matcher(action)) // addMatcher (все совпавшие) .map(({ reducer }) => reducer), ] // 2. Если не совпал НИКТО — берём defaultCase if (caseReducers.filter(Boolean).length === 0) { caseReducers = [finalDefaultCaseReducer] } // 3. Прогоняем state через каждый handler по очереди return caseReducers.reduce((prev, h) => h ? h(prev, action) : prev, state)
  • addCase — максимум ОДИН срабатывает на dispatch (ключ = exact action.type).
  • addMatcher — срабатывают все, чьи matcher'ы вернули true, в порядке добавления.
  • addDefaultCase — только если ни один addCase и ни один addMatcher не совпал.
  • Совпавшие handler'ы применяются накопительно: каждый получает state, который вернул предыдущий. Это важно — порядок может влиять на результат, если они трогают одно и то же поле state.

Поскольку правило (A) запрещает писать addMatcher до addCase, в runtime addCase-handler всегда идёт первым в массиве. Поэтому утверждения «сначала addCase, потом addMatcher» (про код) и «в порядке добавления» (про matcher'ы) — не противоречат друг другу: addCase ⇒ строго один и первый; matcher'ы ⇒ дальше по порядку записи.

Типичные matcher'ы в addMatcher

// 1) Type-predicate от actionCreator builder.addMatcher(myAction.match, handler) // 2) isAnyOf / isAllOf из RTK builder.addMatcher(isAnyOf(a, b, c), handler) // 3) Matcher на группу thunk-actions builder.addMatcher(isPending(fetchUser, fetchPosts), (s) => { s.loading = true }) builder.addMatcher(isRejected(fetchUser, fetchPosts), (s, a) => { s.error = a.error.message }) // 4) Произвольная функция builder.addMatcher( (action) => action.type.endsWith('/rejected'), (s, a) => { s.lastError = a.error } )

Пример: глобальная обработка loading/error

const uiSlice = createSlice({ name: 'ui', initialState: { loading: false, error: null }, reducers: {}, extraReducers: (builder) => { builder .addMatcher(isPending, (s) => { s.loading = true; s.error = null }) .addMatcher(isFulfilled, (s) => { s.loading = false }) .addMatcher(isRejected, (s, a) => { s.loading = false; s.error = a.error.message }) } }) // Теперь любой thunk в приложении обновляет ui.loading автоматически

addDefaultCase в редких случаях

addDefaultCase нужен в двух сценариях:

  1. Логирование неизвестных actions в dev: addDefaultCase((_, a) => { console.warn('unhandled', a) }).
  2. Fallback для динамически подгружаемых slice'ов, где action может прийти от неинжектированного slice.

В 99% случаев addDefaultCase не нужен — builder и так возвращает state unchanged.

Подводный камень: все matcher'ы вызываются — кумулятивно

Если у вас addCase(a, h1) и addMatcher(anyOf(a, b), h2) — при dispatch(a) выполнятся оба: сначала h1, потом h2. State, который вернул h1, передаётся в h2. Если оба пишут в одно поле — выиграет последний. Это фича, а не баг: на ней строится «сквозная логика» вроде toast'ов и analytics (см. урок 23a).

Что НЕ нужно делать с порядком

Не закладывайтесь на «обходной» порядок matcher'ов для бизнес-логики. Если matcher A должен обязательно отработать раньше matcher B — это сигнал, что они оба читают/пишут одно поле state и должны быть объединены в один handler с явной логикой. Иначе любой рефакторинг (перетасовка addMatcher) сломает поведение молча.
