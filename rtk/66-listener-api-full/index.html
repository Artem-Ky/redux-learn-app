Теория — 66. listenerApi = MiddlewareAPI + эффекты

Что такое listenerApi

«The listenerApi object is the second argument to each listener callback. It contains several utility functions that may be called anywhere inside the listener's logic.»
— RTK docs — listener api
«listenerApi — второй аргумент коллбека. Набор утилит: dispatch/getState как у MiddlewareAPI плюс getOriginalState, condition, take, delay, pause, fork, signal, cancelActiveListeners, cancel, subscribe/unsubscribe, throwIfCancelled, extra.»

Состав объекта (в порядке, как в notifyListener)

dispatch / getState
унаследованы из MiddlewareAPI
getOriginalState()
снапшот до next(action). Sync only
condition(p, ms?)
Promise<boolean> — ждёт, пока predicate станет true
take(p, ms?)
Promise<[action, cur, prev] | null> — возвращает тройку
delay(ms)
abort-aware setTimeout
pause(promise)
abort-aware await: кинет TaskAbortError при cancel
extra
аргумент из createListenerMiddleware({ extra })
signal
AbortSignal этой конкретной итерации effect'а
fork(exec, opts?)
параллельная задача (урок 67)
subscribe() / unsubscribe()
unsubscribe — снять entry, subscribe — добавить обратно
cancelActiveListeners()
отменить все прочие pending-инстансы этого entry
cancel()
отменить текущую итерацию
throwIfCancelled()
бросить TaskAbortError если signal.aborted

Как устроен внутренний AbortSignal (notifyListener)

// packages/toolkit/src/listenerMiddleware/index.ts, notifyListener (фрагмент) const internalTaskController = new AbortController() entry.pending.add(internalTaskController) await entry.effect( action, assign({}, api, { getOriginalState, condition: (p, t) => take(p, t).then(Boolean), // condition = take + Boolean take, delay: createDelay(internalTaskController.signal), pause: createPause(internalTaskController.signal), extra, signal: internalTaskController.signal, fork: createFork(internalTaskController.signal, autoJoinPromises), unsubscribe: entry.unsubscribe, subscribe: () => { listenerMap.set(entry.id, entry) }, cancelActiveListeners: () => { entry.pending.forEach((c, _, set) => { if (c !== internalTaskController) { c.abort(listenerCancelled); set.delete(c) } }) }, cancel: () => { internalTaskController.abort(listenerCancelled) entry.pending.delete(internalTaskController) }, throwIfCancelled: () => validateActive(internalTaskController.signal), }), )
Подводные камни listenerApi:
  • getOriginalState() работает только синхронно. Middleware обнуляет свой внутренний токен после прогона reducer'а (originalState = INTERNAL_NIL_TOKEN в finally). Любой вызов после await кинет Error: «listenerMiddleware: getOriginalState can only be called synchronously».
  • cancelActiveListeners() не отменяет fork'и текущего effect'а. Метод проходит по entry.pending и пропускает текущий internalTaskController — то есть отменяет «других меня», а не собственные побочные задачи. Чтобы уронить fork — держите ForkedTask в closure и зовите .cancel().

condition vs take

API Что возвращает Когда
take(pred) [action, current, previous] нужны данные action'а/state'а
take(pred, ms) [action, cur, prev] | null с таймаутом — null если не дождались
condition(pred) boolean нужен только факт «произошло/нет»
