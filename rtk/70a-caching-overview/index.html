Что RTK даёт вам до подключения RTK Query

RTK Query решает проблему server-state автоматически: cache key, deduplication, TTL, invalidation. Но RTK сам по себе уже содержит 5 слоёв кэша/оптимизации. Понимание этих слоёв — это (а) понимание того, что даёт RTK Query «бесплатно», (б) знание, какой слой заменяет RTKQ, а какой остаётся актуальным и после него.

LAYER 1 — state
Structural sharing (ссылки как кэш-ключи)
Immer + ===
Reducer возвращает ту же ссылку, если не было мутации. useSelector сравнивает === — нет изменения, нет ререндера. Это бесплатный кэш на уровне state.
→ 40a 26
LAYER 2 — view
Memoized selectors
createSelector / weakMapMemoize
Когда === недостаточно (фильтр, сумма, merge двух слайсов). Reselect кэширует computed value по ссылкам на inputs. Это кэш на уровне view.
→ 41 43 44
LAYER 3 — data shape
Нормализация — O(1) кэш-таблица
createEntityAdapter
{ ids, entities } — это хеш-таблица. selectById = O(1) lookup, как cache-hit. Кэш на уровне данных.
→ 46 47 52
LAYER 4 — requests
Request dedup + condition
condition / listener takeLatest
Не запускать thunk, если inflight. Отменять предыдущий, если пришёл новый. Кэш на уровне запросов.
→ 59 60 69
LAYER 5 — lifetime
TTL cache + invalidation руками
fetchedAt + condition
Храним fetchedAt в state, в condition проверяем свежесть. Invalidate — через reducer. Кэш на уровне времени жизни.
→ Урок 70f

Сводная таблица

Слой Что кэшируется Ключ кэша Когда инвалидируется Заменяется ли RTKQ
1. Structural sharing Ссылки на под-объекты state === на identity Любая реальная мутация в Immer Нет — остаётся актуальным
2. Memoized selectors Computed values (filter/sort/sum) Ссылки на inputs Изменение любого input по === Нет — остаётся актуальным
3. Нормализация Entity-таблица для O(1) lookup selectId(entity) CRUD-методы adapter'а Частично — RTKQ хранит по cacheKey
4. Request dedup Сам факт inflight-запроса loading === 'pending' Fulfilled/rejected thunk'а Да — RTKQ делает это автоматом по cacheKey
5. TTL cache Момент последней загрузки fetchedAt per id Таймер (TTL) + ручной invalidate Да — RTKQ делает через keepUnusedDataFor + tags

Ментальная модель: когда какой слой включается

// Типичный путь одного update'а: dispatch(cartItemAdded(item)) ↓ // [4] condition сказал ok → pending диспатчится // [3] adapter.addOne(state, item) — O(1) вставка в {ids, entities} // [1] structural sharing: state.cart изменился, state.user — нет (ссылка та же) ↓ // React обходит useSelector-subscribers: useSelector(s => s.user) // === совпало → skip render useSelector(s => s.cart.ids) // !== → render + [2] селектор recompute useSelector(selectCartTotal) // [2] createSelector: recompute useSelector(selectCartItems) // [2] weakMapMemoize hit если refs те же

Что НЕ будет покрыто в этой секции

1. autoBatchEnhancer — это оптимизация рендеров, не кэша. Он уже разбирался в уроке 15.
2. Оптимистичные апдейты — это UX-оптимизация, тоже не кэш. Ручной паттерн разбирался в листенерах (67–69). В RTKQ — onQueryStarted + updateQueryData.
3. Код-сплиттинг slices (lazy injection, урок 35/35a) — оптимизация бандла, не runtime-кэша.

Рассуждение: зачем вообще этим заниматься, если есть RTKQ

Три причины знать слои 4/5 руками:
• RTKQ покрывает только server-state. Клиентский state (фильтры, UI, draft-формы) живёт в обычных slices — и там кэш/дедуп полезны.
• Миграция: вы приходите в проект без RTKQ и должны поддерживать условный condition + TTL. Без понимания слоёв — правите вслепую.
• Внутри RTKQ те же идеи: cacheKey = нормализация, keepUnusedDataFor = TTL, tags = invalidation. Зная «руками» — быстрее читаете документацию RTKQ.
