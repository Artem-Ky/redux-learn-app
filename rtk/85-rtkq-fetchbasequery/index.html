Каждая опция fetchBaseQuery — отдельный endpoint + кнопка

fetchBaseQuery — это тонкая обёртка над нативным fetch. Его задача: унифицировать работу с baseUrl, JSON-сериализацию тела, query-параметры, заголовки, timeout и парсинг ответа. Ниже — по одному demo-блоку на каждую опцию.

Все error-формы (discriminated union)

export type FetchBaseQueryError = | { status: number; data: unknown } // HTTP !2xx | { status: 'FETCH_ERROR'; error: string } // network down | { status: 'PARSING_ERROR'; originalStatus: number; data: string; error: string } | { status: 'TIMEOUT_ERROR'; error: string } // timeoutSignal сработал | { status: 'CUSTOM_ERROR'; data?: unknown; error: string } // из своего queryFn

Сравни с SerializedError из createAsyncThunk — там другая форма. TS-narrowing: 'status' in error && typeof error.status === 'number' → HTTP.

Порядок сборки запроса

Подводные камни

1. prepareHeaders возвращает void → использовать переданный объект headers.
Если вернул undefined, RTKQ использует переданный headers (мутированный). Если вернул новый Headers — используется он. Ловушка: return; вместо return headers; — работает, но не очевидно читается.

2. body = FormData / Blob / ArrayBuffer → content-type УДАЛЯЕТСЯ.
RTKQ проверяет isJsonifiable(body): если это plain-object/array/has toJSON, ставит content-type: application/json. Иначе — удаляет content-type, чтобы браузер выставил правильный multipart/form-data; boundary=.... Не ставь content-type вручную для FormData.

3. validateStatus default = 200–299.
Если сервер возвращает 3xx (редирект) — fetch обычно следует по умолчанию, но если ты redirect: 'manual', то 3xx попадёт в validateStatus и вернёт {status: 3xx, data} как error. Так же для 4xx/5xx. Переопределяй, если твой API умеет status=201 с JSON body, и ты хочешь, чтобы он был ошибкой.

4. timeout: используется через anySignal.
RTKQ комбинирует api.signal (abort при unsubscribe) и timeoutSignal(timeout) в один AbortSignal. Если хоть один сработал — fetch abort → {status: 'TIMEOUT_ERROR'} (для timeout) или silent reject (для unsubscribe abort).

5. responseHandler: 'json' (default) бросает на не-JSON ответах.
Если endpoint иногда возвращает пустое тело (204 No Content) — response.json() бросает SyntaxError → {status: 'PARSING_ERROR'}. Для таких случаев — responseHandler: 'content-type' (смотрит заголовок) или custom (res) => res.text().then(t => t ? JSON.parse(t) : null).

6. paramsSerializer для массивов.
Default (URLSearchParams) сериализует {ids: [1,2,3]} как ids=1&ids=2&ids=3. Если твой сервер хочет ids=1,2,3 или ids[]=1&ids[]=2 — передай paramsSerializer: (p) => qs.stringify(p).

Когда НЕ хватит fetchBaseQuery — переходи на свой baseQuery
