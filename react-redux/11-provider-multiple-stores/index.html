Интерактив — Два store через кастомные контексты

Два независимых Redux store, каждый в своём <Provider> с кастомным React Context. Dispatch в один store не влияет на компоненты, подписанные на другой.

<Provider store={userStore} context={UserContext}> <Provider store={productStore} context={ProductContext}> <UserProfile /> ← читает из userStore через UserContext <ProductCatalog /> ← читает из productStore через ProductContext </Provider> </Provider>
Теория — Множественные store и кастомные контексты

Зачем нужны множественные store?

Redux был спроектирован для работы с одним store. Но иногда (микрофронтенды, интеграция legacy-систем, независимые модули) возникает необходимость использовать несколько store одновременно.

«Redux was designed to use a single store. However, if you are in an unavoidable position of needing to use multiple stores, as of v6 you may do so by providing (multiple) custom contexts.»
— react-redux.js.org — Accessing the Store
«Redux был спроектирован для использования одного store. Однако, если вы оказались в ситуации, когда необходимо использовать несколько store, начиная с v6 вы можете это сделать, предоставив (несколько) кастомных контекстов.»
— перевод
«Instead of using the default context instance from React Redux, you may supply your own custom context instance.»
— react-redux.js.org — Accessing the Store
«Вместо использования экземпляра контекста по умолчанию из React Redux, вы можете предоставить свой собственный кастомный экземпляр контекста.»
— перевод

Как это работает

Каждый <Provider> принимает проп context. Если передан кастомный контекст — store кладётся именно в него. Хуки useSelector и useDispatch тоже принимают context как аргумент, чтобы указать, из какого store читать.

import React from 'react' import { Provider, useSelector, useDispatch, createDispatchHook, createSelectorHook } from 'react-redux' // Создаём кастомные контексты const UserContext = React.createContext(null) const ProductContext = React.createContext(null) // Опционально: создаём типизированные хуки для каждого store const useUserSelector = createSelectorHook(UserContext) const useUserDispatch = createDispatchHook(UserContext) const useProductSelector = createSelectorHook(ProductContext) const useProductDispatch = createDispatchHook(ProductContext) // Вложенные Provider'ы function App() { return ( <Provider store={userStore} context={UserContext}> <Provider store={productStore} context={ProductContext}> <UserProfile /> <ProductCatalog /> </Provider> </Provider> ) }

Каждый store — своя Subscription

Каждый <Provider> создаёт свою собственную Subscription (linked list слушателей). При dispatch в один store — Subscription этого Provider проходит по своему списку, для каждого слушателя запускает его selector, сравнивает результат с предыдущим (===). Если отличается — вызывает forceRender() только для этого компонента. Компоненты, подписанные на другой store через другой контекст, не затрагиваются.

Когда это действительно нужно

Множественные store — это не рекомендуемый паттерн для обычных приложений. Используйте его только в специфических сценариях:

  • Микрофронтенды — каждый микрофронтенд имеет свой store, они не должны знать друг о друге
  • Интеграция legacy-модулей — старый модуль со своим store встраивается в новое приложение
  • Изоляция данных — например, админ-панель внутри пользовательского приложения
Не используйте множественные store вместо slice'ов!
Если вы хотите разделить state на части — используйте combineReducers или Redux Toolkit slice'ы. Множественные store усложняют отладку (DevTools работают только с одним store), делают невозможным cross-store dispatch и ломают middleware-цепочки.
Альтернатива createSelectorHook / createDispatchHook:
Вместо создания специальных хуков через createSelectorHook(context), можно передавать контекст напрямую: useSelector(selector, { context: UserContext }). Но кастомные хуки удобнее — не нужно каждый раз указывать контекст.
