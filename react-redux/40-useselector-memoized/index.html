Интерактив — три способа фильтровать todos

Три компонента делают одно и то же — показывают завершённые todos. Отличается только способ описания selector'а. Счётчик рендеров + счётчик вызовов .filter() показывает, насколько по-разному они работают на одних и тех же dispatch'ах.

Теория — useSelector + createSelector

Проблема: inline selector, возвращающий массив

В уроке 37 мы увидели: если selector возвращает новый объект, по умолчанию useSelector ре-рендерит на каждом dispatch. С массивами ситуация ровно такая же — .filter(), .map(), .slice() всегда возвращают новый массив, даже если содержимое совпадает с предыдущим.

// ❌ Плохо: и сам selector, и массив-результат — новые на каждом рендере/dispatch const completed = useSelector((state: RootState) => state.todos.list.filter((t) => t.completed) )

Что даёт вынос selector'а наружу компонента

// Selector стабилен как функция, но результат filter() каждый раз новый const selectCompletedTodos = (state: RootState) => state.todos.list.filter((t) => t.completed) function Component() { const completed = useSelector(selectCompletedTodos) // сама функция selectCompletedTodos — одна и та же, // но .filter() всё равно возвращает новый массив на каждом вызове. // По-прежнему ре-рендер на каждом dispatch. }

Это работает только для selector'ов, возвращающих примитив или уже стабильную ссылку из state (например, state.todos.list целиком, без .filter).

Решение: createSelector из reselect

createSelector принимает один или несколько input selector'ов и одну output-функцию. Он запоминает последний набор результатов input'ов и последний output. Если inputs совпали с предыдущим вызовом по === — output-функция не вызывается, возвращается тот же самый закешированный результат.

import { createSelector } from 'reselect' const selectCompletedTodos = createSelector( // input selector(s): (state: RootState) => state.todos.list, // output: (list) => list.filter((t) => t.completed) ) function Component() { const completed = useSelector(selectCompletedTodos) // если state.todos.list та же ссылка → возвращается // ТОТ ЖЕ массив, что и в прошлый раз → useSelector не рендерит }

Ключевое правило: createSelector вне компонента

"When using useSelector with an inline selector as shown above, a new instance of the selector is created whenever the component is rendered. This works as long as the selector does not maintain any state. However, memoizing selectors (e.g. created via createSelector from reselect) do have internal state, and therefore care must be taken when using them."
— react-redux.js.org/api/hooks
«Когда useSelector используется с inline-selector'ом, как показано выше, новый экземпляр selector'а создаётся при каждом рендере компонента. Это работает, пока selector не хранит внутреннее состояние. Однако memoizing-selector'ы (например, созданные через createSelector из reselect) имеют внутреннее состояние, поэтому при их использовании требуется осторожность.»
— перевод
"When the selector does only depend on the state, simply ensure that it is declared outside of the component so that the same selector instance is used for each render"
— react-redux.js.org/api/hooks
«Когда selector зависит только от state, просто убедитесь, что он объявлен вне компонента — тогда при каждом рендере используется один и тот же экземпляр selector'а.»
— перевод

Если создавать createSelector внутри компонента — внутренний кэш создаётся заново на каждом рендере и никогда не срабатывает. Результат: нулевая выгода от мемоизации. Поэтому — только на уровне модуля.

Когда inline selector допустим

Для примитивов, которые просто читаются из state без вычислений, inline-вариант абсолютно нормален:

// ✔ OK — возвращается примитив, === работает const count = useSelector((state: RootState) => state.counter.value) // ✔ OK — возвращается стабильная ссылка из state (если todos.list не пересоздаётся) const todos = useSelector((state: RootState) => state.todos.list)

Связь с системой Subscription

Ре-рендер после dispatch происходит по тому же пути: linked list callback'ов → Subscription.notify() → для каждого слушателя checkForUpdates → запустить selector → сравнить через === → forceRender если отличается. Мемоизация через createSelector не меняет механику подписки, она лишь гарантирует, что результат selector'а будет той же ссылкой, если inputs не изменились — и шаг сравнения === вернёт true, ре-рендер не произойдёт. Подробнее про Subscription — урок 03a.

Частая ошибка: передать в useSelector результат вызова createSelector(...)(). Правильно: передавайте саму функцию selectCompletedTodos, а не её вызов. useSelector сам подставит state.
Рабочий рецепт с createSelector:
  1. Объявите createSelector на уровне модуля (вне компонента).
  2. Input selector'ы возвращают сырые данные из state (как можно более «прямые»).
  3. Output-функция делает вычисление — она вызовется, только если хоть один input поменялся.
  4. Передайте итоговый selector в useSelector по ссылке, без вызова.
