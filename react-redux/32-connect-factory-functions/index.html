Интерактив — один selector на всех vs per-instance

Две колонки рендерят по 20 карточек ItemDetail — каждая со своим itemId. Слева — обычный mapStateToProps с одним общим memoized-selector'ом. Справа — factory-функция, которая при mount каждой карточки создаёт свой memoized-selector в замыкании. После любого dispatch смотрите на счётчики: слева — 20 cache miss'ов (кэш перезаписывается на каждом соседе), справа — 20 cache hit'ов (каждая карточка имеет свой стабильный кэш).

Теория — factory-функции mapStateToProps / mapDispatchToProps

Что такое factory-форма

Обычно mapStateToProps — это функция, которая на каждый dispatch возвращает объект props. Factory-форма — это «функция, возвращающая функцию»: внешняя вызывается один раз на инстанс компонента при mount, внутренняя становится настоящим mapStateToProps и вызывается при каждом dispatch. То же самое работает и для mapDispatchToProps.

«If your mapStateToProps or mapDispatchToProps functions return a function, they will be called once when the component instantiates, and their returns will be used as the actual mapStateToProps, mapDispatchToProps, functions respectively, in their subsequent calls.»
— react-redux.js.org/api/connect
«Если ваши функции mapStateToProps или mapDispatchToProps возвращают функцию, то они будут вызваны один раз при создании инстанса компонента, а их возвращаемые значения станут настоящими mapStateToProps и mapDispatchToProps в последующих вызовах.»
— перевод
«The factory functions are commonly used with memoized selectors. This gives you the ability to create component-instance-specific selectors inside the closure»
— react-redux.js.org/api/connect
«Factory-функции обычно используют с memoized-селекторами. Это даёт возможность создавать селекторы, специфичные для конкретного инстанса компонента, прямо в замыкании.»
— перевод

Проблема общего селектора

createSelector из reselect запоминает последний набор аргументов (cache size = 1). Если один и тот же селектор вызывают из разных инстансов с разными аргументами (например, разные ownProps.itemId), каждый следующий вызов «вытесняет» предыдущий кэш: после прохода по 20 компонентам кэш хранит результат только последнего, для всех 19 других на следующем dispatch будет cache miss и полный пересчёт.

// ✖ ПЛОХО — один selector на всех const selectItemById = createSelector( (state: RootState) => state.items, (_: RootState, itemId: number) => itemId, (items, itemId) => items.find(i => i.id === itemId) ) const mapStateToProps = (state: RootState, ownProps: { itemId: number }) => ({ item: selectItemById(state, ownProps.itemId) }) connect(mapStateToProps)(ItemDetail) // Проблема: у ВСЕХ 20 инстансов ОДИН кэш. // Каждый вызов с другим itemId перезаписывает кэш → мемоизация сломана.

Решение: factory создаёт свой selector в замыкании

// ✔ ХОРОШО — per-instance selector через factory const makeMapStateToProps = () => { const selectItemByIdForThisInstance = createSelector( (state: RootState) => state.items, (_: RootState, itemId: number) => itemId, (items, itemId) => items.find(i => i.id === itemId) ) return (state: RootState, ownProps: { itemId: number }) => ({ item: selectItemByIdForThisInstance(state, ownProps.itemId) }) } connect(makeMapStateToProps)(ItemDetail) // Каждый инстанс при маунте вызовет makeMapStateToProps() // и получит СВОЙ собственный selector со СВОИМ кэшем.

Когда это реально нужно

  • Списки, где каждый элемент получает свой срез state по ownProps (itemId, userId, координаты ячейки и т. п.).
  • Дорогие вычисления в селекторе, которые вы не хотите повторять на каждый dispatch (поиск в большой коллекции, сортировка, фильтрация, nested lookup).
  • Любой раз, когда в DevTools / React Profiler вы видите, что mapStateToProps у списка — «горячий» участок.

Factory для mapDispatchToProps

Работает симметрично. Это реже нужно: типичный случай — когда action creators связаны с ownProps через bindActionCreators и вы хотите сохранить ссылочную стабильность на инстанс. Выглядит так:

const makeMapDispatchToProps = (_dispatch, ownProps: { itemId: number }) => { const removeThis = () => ({ type: 'items/remove', payload: ownProps.itemId }) return (dispatch) => ({ remove: () => dispatch(removeThis()) }) } connect(null, makeMapDispatchToProps)(ItemDetail)

Связь с системой Subscription

Сам механизм подписки не меняется. При dispatch Subscription проходит по linked list callback'ов → Subscription.notify() → для каждого слушателя checkForUpdates → запустить selector (это и есть ваша внутренняя mapStateToProps) → сравнить с prevValue через === (shallow над объектом) → forceRender если отличается. Разница только в том, какая именно функция выступает селектором: без factory — одна и та же на все инстансы, с factory — у каждого инстанса своя замкнутая функция с собственным reselect-кэшем. Подробнее о Subscription — урок 03a.

Мини-правило: как только в mapStateToProps появляется createSelector + ownProps, которые «разнят» вызовы между инстансами — переходите на factory. Без factory reselect превращается в обычную функцию, которая пересчитывает всё при каждом dispatch.
Не злоупотребляйте: для одиночного компонента (не списка) factory не нужен — общий селектор справится. Factory решает конкретную проблему «много инстансов — один кэш». Для всего остального стандартный mapStateToProps и reselect работают отлично.
