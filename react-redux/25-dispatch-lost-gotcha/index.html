Интерактив — props.dispatch есть или нет?

Два счётчика бок о бок. Слева — connect(mapStateToProps) без второго аргумента, props.dispatch доступен. Справа — connect(mapStateToProps, mapDispatchToProps); кнопка «Ручной dispatch» вызывает props.dispatch(...), но там его нет. Нажмите красную кнопку справа — ErrorBoundary поймает TypeError.

Теория — куда девается dispatch

Правило поведения connect по умолчанию

Если mapDispatchToProps не указан — компонент получает props.dispatch.
Если mapDispatchToProps передан (функция ИЛИ объект) — компонент получает только то, что вернул mapDispatchToProps. dispatch в props пропадает.

Цитата из документации

«Once you've wrapped all our action creators with functions that dispatch the actions, the component is free of the need of dispatch. Therefore, if you define your own mapDispatchToProps, the connected component will no longer receive dispatch.»
— react-redux.js.org/using-react-redux/connect-mapdispatch
«Когда вы обернули все ваши action creators функциями, которые диспатчат экшены, компонент больше не нуждается в dispatch. Поэтому, если вы определяете собственный mapDispatchToProps, connected-компонент больше не получает dispatch.»
— перевод

Симптом

«This is a common error that happens when you try to call this.props.dispatch, but dispatch is not injected to your component.»
— react-redux.js.org/using-react-redux/connect-mapdispatch
«Это частая ошибка, которая случается, когда вы пытаетесь вызвать this.props.dispatch, но dispatch не инжектирован в ваш компонент.»
— перевод

В консоли вы увидите что-то вроде:

TypeError: this.props.dispatch is not a function

Почему так сделано

Логика команды react-redux: если вы уже подготовили обёрнутые функции (через свой mapDispatchToProps) — значит сырой dispatch компоненту не нужен, и API становится «чище». Подробнее — обсуждение в issue #255.

Способ 1 — явно вернуть dispatch

import { bindActionCreators } from 'redux' const mapDispatchToProps = (dispatch) => ({ ...bindActionCreators({ increment, decrement }, dispatch), dispatch, // ← вручную добавляем обратно }) connect(mapStateToProps, mapDispatchToProps)(Counter) // props = { count, increment, decrement, dispatch }

Способ 2 — просто не передавать mapDispatchToProps

Если в компоненте всего один-два редких экшена, проще вообще не писать mapDispatchToProps и использовать props.dispatch напрямую (поведение по умолчанию, урок 20). Это нормально для маленьких локальных случаев.

Способ 3 — хуки вместо connect

В современных приложениях часто используют useDispatch() (будет в уроке 44). Он возвращает dispatch напрямую, не связан с connect и не имеет этой ловушки.

Как ведут себя разные формы

форма connect props.dispatch?
connect()(C) есть
connect(mapStateToProps)(C) есть
connect(null, null)(C) есть
connect(mapStateToProps, mapDispatchToProps)(C) пропал
connect(mapStateToProps, {'{ increment }'})(C) пропал
connect(null, mapDispatchToProps)(C) пропал
Частая ошибка: вы перешли с props.dispatch на объектную форму и забыли, что в соседнем месте компонента ещё остался старый вызов props.dispatch({'{'} type: ... {'}'}). При клике всё падает с TypeError. Решение — либо переписать вызов через новое поле (props.increment()), либо вернуть dispatch в props явно.

Связь с Subscription-системой

Subscription здесь ни при чём — эта ошибка возникает до цикла обновления. this.props.dispatch(...) падает на этапе клика, ещё до того как какой-либо action дойдёт до store. Для справки: Subscription — это linked list callback'ов (урок 03a): при успешном dispatch store дёргает Subscription.notify(), для каждого подписчика checkForUpdates запускает mapStateToProps, сравнивает поля через === с предыдущими и при изменении вызывает forceRender(). Но если dispatch в props отсутствует, action даже не успеет попасть в store — исключение бросается в обработчике клика.

Итог урока: передача любого mapDispatchToProps «съедает» props.dispatch. Запомните правило и, если всё-таки нужен dispatch — возвращайте его явно или используйте useDispatch().
