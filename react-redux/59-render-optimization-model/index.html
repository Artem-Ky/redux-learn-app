Интерактив — три способа раздать одно состояние

Три колонки — одно и то же дерево из нескольких компонентов. В каждой колонке только C2 зависит от counter.value. Остальные читают данные, которые не меняются. Следите за тем, какие узлы загораются красным при каждой кнопке.

Теория — почему так выходит

Нативный React Context

Когда вы кладёте все данные в Context.Provider как value и вызываете useContext в потребителях — это «фича», а не «баг»: любое изменение value перерисовывает всех потребителей, даже если изменилось поле, которое им не нужно. Причина — React сравнивает value через Object.is и, если ссылка другая, уведомляет всех.

«Internally, React Redux uses React's "context" feature to make the Redux store accessible to deeply nested connected components.»
— react-redux.js.org/using-react-redux/accessing-store

Секрет React-Redux

Библиотека обходит проблему «всех потребителей» простым трюком. В Context кладётся стабильный объект { store, subscription } — он ссылочно не меняется всё время жизни приложения. Само значение state туда не идёт никогда. Уведомления об изменениях доставляются мимо Context — через собственную систему Subscription.

Что это даёт:

  • Context.value не меняется — React не рассылает принудительных перерисовок.
  • При dispatch Subscription проходит по linked list callback'ов → Subscription.notify() → для каждого слушателя checkForUpdates → запускает selector → сравнивает результат с prev через === → вызывает forceRender только если значение действительно изменилось.
  • Подробнее о Subscription — урок 03a.

connect — nested Subscription

Каждый connect-обёрнутый компонент создаёт собственную Subscription, встраиваясь в иерархию. У connect есть два мощных эффекта:

  • Подписка на изменения store (через Subscription). После mapStateToProps сравнивает результат с предыдущим через === (shallow по объекту верхнего уровня) — перерисовка только при реальной разнице.
  • Внешне connect работает как «чистый компонент»: если ownProps не изменились, он не перерисует обёрнутый компонент при обычном React-рендере родителя.

Hooks — плоская Subscription + React.memo

useSelector регистрирует слушателя у корневого Subscription. При dispatch каждому слушателю запускается checkForUpdates и, если его селектор вернул новое значение (по ===), — компонент перерисовывается. Но в отличие от connect, хуки не блокируют перерисовки, вызванные родителем. Чтобы получить то же поведение — оборачивают компонент в React.memo. Об этом подробно — урок 60.

Сводная таблица

Механизм dispatch → лишний рендер? parent rerender → лишний рендер?
Нативный Context да, все потребители да, обычный React
connect (nested Subscription) только подписанные на изменившуюся часть нет — ownProps сверяются по shallow
hooks (useSelector) без memo только подписанные да, parent тащит всех детей
hooks + React.memo только подписанные нет — memo блокирует при неизменных props
Мораль: в hooks-API оптимизация против «parent rerender» — ваша забота (через React.memo). Зато вы получаете плоскую Subscription, никаких nested-wrapperов и лишних Component-узлов в дереве. В большинстве приложений разницы в производительности между правильно настроенными connect и hooks+memo не видно.
Не путайте: даже связка hooks + React.memo не защищает от сценария «селектор каждый раз возвращает новый объект». Если useSelector возвращает новый массив/объект при каждом вызове, === всегда ложен и forceRender запускается каждый раз. Лечится мемоизированным селектором (createSelector) — урок 40 и 62.
