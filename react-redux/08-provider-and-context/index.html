Интерактив — Нативный Context vs React-Redux Provider

Нажимайте + и наблюдайте за счётчиками рендеров. Слева — нативный подход через React.createContext (все компоненты перерисовываются). Справа — настоящий React-Redux Provider (перерисовывается только подписанный компонент).

Теория — Provider и React Context: как это работает внутри

React-Redux использует Context, но не как вы думаете

"Internally, React Redux uses React's context feature to make the Redux store accessible to deeply nested connected components."

Внутри React Redux использует механизм context React для предоставления Redux store глубоко вложенным подключённым компонентам.

— react-redux.js.org, Accessing the Store

Ключевое отличие React-Redux от нативного использования Context: <Provider> помещает в Context стабильный объект {'{store, subscription}'}, который никогда не меняется. Context value остаётся === равным самому себе при каждом рендере.

Почему нативный Context — это проблема

При использовании React.createContext напрямую для хранения state, каждое изменение значения context вызывает re-render всех потребителей (useContext), даже если конкретному компоненту это значение не нужно.

// Нативный подход — ПЛОХО для производительности const StoreContext = React.createContext(null) function NaiveProvider({ children }) { const [state, setState] = useState(initialState) // Каждый setState создаёт НОВЫЙ объект value // → React пересчитывает ВСЕХ потребителей! return ( <StoreContext.Provider value={{ state, setState }}> {children} </StoreContext.Provider> ) }

Как работает React-Redux Provider

"React Redux's <Provider> component uses <ReactReduxContext.Provider> to put the Redux store and internal subscription wrappers into context. useSelector, useDispatch, and connect call useContext(ReactReduxContext) to read those values and handle updates."

<Provider> компонент React Redux использует <ReactReduxContext.Provider> для помещения Redux store и внутренних обёрток подписки в context. useSelector, useDispatch и connect вызывают useContext(ReactReduxContext) для чтения этих значений и обработки обновлений.

— react-redux.js.org, Accessing the Store
// Реальный React-Redux Provider (упрощённо) function Provider({ store, children }) { // contextValue создаётся ОДИН раз и НИКОГДА не меняется const contextValue = useMemo(() => ({ store, subscription: new Subscription(store) }), [store]) // Context.Provider получает стабильную ссылку // → React НЕ перерисовывает потребителей при dispatch! return ( <ReactReduxContext.Provider value={contextValue}> {children} </ReactReduxContext.Provider> ) }

Механизм уведомлений: Subscription вместо Context

Уведомления о dispatch проходят не через Context, а через Subscription — систему linked list коллбэков. При каждом dispatch:

  1. Store вызывает callback из store.subscribe()
  2. Это Root Subscription (созданная <Provider>), которая проходит по своему linked list
  3. Для каждого слушателя (каждый useSelector) вызывает checkForUpdates()
  4. checkForUpdates запускает selector, сравнивает результат через ===
  5. Если значение изменилось — вызывает forceRender() только для этого компонента

Подробности работы Subscription — в уроке 03a.

Итог: Context используется только для передачи ссылки на store. Перерисовка компонентов управляется через Subscription, а не через Context. Поэтому React-Redux не страдает от проблемы «все потребители перерисовываются».

Сравнение подходов

Нативный Context React-Redux Provider
Context value Меняется при каждом dispatch Стабильный {store, subscription}
Re-render ВСЕ потребители Только подписанные с изменением
Уведомления Через React Context Через Subscription (linked list)
Производительность O(N) re-renders при каждом dispatch Только изменённые компоненты
