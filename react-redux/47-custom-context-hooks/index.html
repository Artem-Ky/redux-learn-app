Интерактив — два изолированных store с собственными хуками

UserStore и ProductStore живут в разных React-контекстах. Для каждого своя тройка хуков: useUserSelector/useUserDispatch/useUserStore и useProductSelector/useProductDispatch/useProductStore. Dispatch в один store не вызывает ре-рендер компонентов другого — следите за счётчиками рендеров.

Теория — кастомный context и createHook-функции

Зачем кастомный контекст

По умолчанию все встроенные хуки (useSelector, useDispatch, useStore) смотрят во внутренний контекст React-Redux — ReactReduxContext. В 99% случаев этого достаточно. Но иногда нужен изолированный контекст:

  • Переиспользуемая библиотека со своим Redux-store, которая не должна конфликтовать со store приложения-потребителя.
  • Микрофронтенды: несколько независимых модулей, каждый со своим store, собранные в одной странице.
  • Сценарий «главный + вспомогательный store» — когда первый управляет основным состоянием приложения, а второй, например, отвечает за сложный виджет.

Documentation: context prop у <Provider>

"The <Provider> component allows you to specify an alternate context via the context prop. This is useful if you're building a complex reusable component, and you don't want your store to collide with any Redux store your consumers' applications might use."
— react-redux.js.org/api/hooks
«Компонент <Provider> позволяет указать альтернативный контекст через prop context. Это полезно, когда вы создаёте сложный переиспользуемый компонент и не хотите, чтобы ваш store конфликтовал с каким-либо Redux-store приложений-потребителей.»
— перевод

Documentation: create-hook функции

"To access an alternate context via the hooks API, use the hook creator functions"
— react-redux.js.org/api/hooks
«Чтобы получить доступ к альтернативному контексту через hooks API, используйте функции-создатели хуков.»
— перевод

Три функции-создателя

createSelectorHook(context?) => useSelector
createDispatchHook(context?) => useDispatch
createStoreHook(context?) => useStore

Каждая возвращает новый экземпляр соответствующего хука, привязанный к переданному контексту. Если контекст не передан — используется встроенный ReactReduxContext (то есть вы получите «обычный» useSelector).

Паттерн использования

import { createContext } from 'react' import { Provider, createStoreHook, createDispatchHook, createSelectorHook, } from 'react-redux' // 1. Собственный React-контекст const UserContext = createContext(null) // 2. Три хука, привязанных к нему export const useUserStore = createStoreHook(UserContext) export const useUserDispatch = createDispatchHook(UserContext) export const useUserSelector = createSelectorHook(UserContext) // 3. Провайдер с prop context <Provider store={userStore} context={UserContext}> ... </Provider>

Два независимых store с разными контекстами

В нашем демо userStore хранит данные профиля, productStore — каталог товаров. Каждый живёт в своём контексте, и у каждого свои хуки:

const UserContext = createContext(null) const ProductContext = createContext(null) export const useUserSelector = createSelectorHook(UserContext) export const useUserDispatch = createDispatchHook(UserContext) export const useProductSelector = createSelectorHook(ProductContext) export const useProductDispatch = createDispatchHook(ProductContext) function UserPanel() { const name = useUserSelector(s => s.name) return <div>{name}</div> } function ProductList() { const items = useProductSelector(s => s.items) return <ul>{items.map(p => <li>{p.name}</li>)}</ul> } // Вложенные Provider'ы с разным context <Provider store={userStore} context={UserContext}> <Provider store={productStore} context={ProductContext}> <UserPanel /> <ProductList /> </Provider> </Provider>

Use case 1: переиспользуемая библиотека со своим store

Ваш пакет экспортирует сложный компонент с собственной Redux-логикой (например, редактор форм или дашборд с фильтрами). Потребитель скорее всего использует свой Redux — если оба воспользуются встроенным ReactReduxContext, получится конфликт. Решение: библиотека создаёт MyLibContext, свой Provider и свои хуки — и никак не зависит от store приложения.

Use case 2: микрофронтенды

Несколько независимых модулей, каждый со своим store — собираются в общее приложение. Модуль A читает данные только через useAStore/useASelector/useADispatch, модуль B — через свои. Ни один модуль не может случайно прочитать или испортить состояние соседа.

Связь с системой Subscription

Каждый <Provider> поднимает свой корневой Subscription. Когда в userStore происходит dispatch, цепочка linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender проходит только по подпискам, зарегистрированным useUserSelector/connect через UserContext. Компоненты, подписанные на ProductContext, в эту цепочку не попадают и не ре-рендерятся. Подробнее о Subscription — урок 03a.

Важно: никогда не смешивайте контексты

Частая ошибка: <Provider store={storeA} context={CtxA}>, а внутри вызывается обычный useSelector (без кастомного контекста). Обычный useSelector ищет ReactReduxContext — его в этом поддереве нет, результат — ошибка «could not find react-redux context value». Нужно использовать именно useASelector, созданный через createSelectorHook(CtxA).

Можно ли обойтись одним Provider?

Если вам просто нужно разделить state на несколько slice'ов — используйте combineReducers и один store, это проще и типичнее. Кастомный контекст нужен именно для изоляции: когда одному куску UI вообще не должны быть видны чужие подписки и чужой store. Пример такого разделения с одним store — урок 27. Пример с двумя store (но без кастомных хуков через create-*) — урок 11.

Правило: создавайте кастомный контекст и createSelectorHook / createDispatchHook / createStoreHook в одном модуле — ./contexts/UserContext.ts, и экспортируйте оттуда все три хука. Компоненты импортируют их под именами useUserSelector и т.д. — это читается естественно и сразу понятно, к какому store относится.
