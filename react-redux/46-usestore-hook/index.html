Интерактив — два сценария useStore

Слева — компонент с useSelector, который подписан на изменения и ре-рендерится при каждом релевантном dispatch. Справа — компонент, читающий значение через useStore().getState() прямо в рендере. Второй не обновляется — useStore не подписывает компонент. Ниже — кнопки экспорта и замены reducer'а, где useStore уместен.

Теория — useStore

Что делает useStore

"This hook returns a reference to the same Redux store that was passed in to the <Provider> component."
— react-redux.js.org/api/hooks
«Этот хук возвращает ссылку на тот же самый Redux-store, который был передан в компонент <Provider>.»
— перевод

Сигнатура

useStore<State = unknown, Action extends Action = AnyAction>(): Store<State, Action>

Возвращается тот же объект store, который вы передавали в <Provider store={...}>. Методы у него стандартные: getState(), dispatch(action), subscribe(listener), replaceReducer(nextReducer).

Главное отличие от useSelector: нет подписки

useSelector подписывает компонент на изменения: при каждом dispatch вызывается selector, результат сравнивается со старым по === — если отличается, компонент ре-рендерится.

useStore не подписывает. Он просто достаёт объект store из контекста и возвращает вам. Если вы вызовете store.getState() прямо в рендере — получите актуальное значение в момент рендера, но следующий dispatch ре-рендер не вызовет. Компонент застрянет на устаревшем значении, пока что-то другое его не ре-рендерит.

Когда useStore уместен

"This hook should probably not be used frequently. Prefer useSelector() as your primary choice. However, this may be useful for less common scenarios that do require access to the store, such as replacing reducers."
— react-redux.js.org/api/hooks
«Этим хуком вряд ли стоит пользоваться часто. В качестве основного выбора предпочтительнее useSelector(). Однако он может быть полезен в менее частых сценариях, где действительно нужен доступ к самому store — например, для замены reducer'ов.»
— перевод

Сценарий 1: снимок state в обработчике

Вам нужно прочитать state в момент клика, но вы не хотите ре-рендерить компонент при каждом изменении этого куска state. Классический пример — кнопка «экспорт» или «сохранить»: в момент клика снимается полный state, сериализуется, отправляется куда-то.

function ExportButton() { const store = useStore<RootState>() const onExport = () => { const state = store.getState() const json = JSON.stringify(state, null, 2) console.log(json) // Снимок state в момент клика, без подписки на ре-рендеры } return <button onClick={onExport}>Export state as JSON</button> }

Если бы вы сделали это через useSelector(state => state), компонент ре-рендерился бы на каждом dispatch'е всего приложения.

Сценарий 2: замена reducer (code splitting)

Динамическая подгрузка модуля с новым reducer'ом — пример из документации. При lazy-loaded фиче вы подгружаете её reducer и встраиваете в корневой reducer через store.replaceReducer(newRootReducer).

function FeatureLoader() { const store = useStore<RootState>() const loadFeature = async () => { const { reducer } = await import('./features/chat') store.replaceReducer(combineReducers({ ...staticReducers, chat: reducer })) } return <button onClick={loadFeature}>Загрузить фичу "чат"</button> }

Сценарий 3: доступ к getState внутри thunk-подобной логики

Иногда нужно диспатчить action, значение которого зависит от актуального state — и при этом не подписывать компонент на этот state. Типично используется в комбинации с useDispatch:

function SaveButton() { const store = useStore<RootState>() const dispatch = useDispatch() const onSave = () => { const draft = store.getState().editor.draft dispatch({ type: 'SAVE', payload: draft }) } return <button onClick={onSave}>Сохранить</button> }

Антипаттерн: чтение state в рендере

Не делайте так! Компонент не обновится при изменении state:
// ❌ Компонент не подписан — при dispatch не ре-рендерится function BadTodoCount() { const store = useStore<RootState>() return <div>{store.getState().todos.length}</div> } // ✔ Правильный вариант — useSelector подписывает на изменения function GoodTodoCount() { const count = useSelector((state: RootState) => state.todos.length) return <div>{count}</div> }

В демо слева как раз такой «плохой» компонент: он прочитает значение один раз при первом рендере — и потом будет показывать его навсегда. Справа — правильный, на useSelector.

Связь с системой Subscription

useStore не регистрирует callback в подписке store. Поэтому при dispatch цепочка linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender не задействует компоненты, использующие только useStore. Они обновятся только когда их родитель ре-рендерится по другой причине (например, сработал соседний useSelector или изменился prop). Подробнее о Subscription — урок 03a.

Практический совет: если вы пишете useStore и затем сразу же store.getState() в рендере — это почти всегда ошибка. Замените на useSelector. useStore нужен только когда доступ к store отложен во времени (обработчики, эффекты) или вам нужны методы вроде replaceReducer/subscribe.
