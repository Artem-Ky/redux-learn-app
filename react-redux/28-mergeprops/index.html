Интерактив — профиль пользователя: без и с mergeProps

Компонент UserProfile получает userId как ownProps и должен показывать todos именно этого юзера и давать кнопку «добавить» с автоподстановкой userId. Слева — реализация без mergeProps: компонент сам фильтрует и сам оборачивает addTodo. Справа — та же логика, но через mergeProps: компонент получает todos и addTodo уже готовыми.

Теория — mergeProps

Что такое mergeProps

«If specified, defines how the final props for your own wrapped component are determined. If you do not provide mergeProps, your wrapped component receives { ...ownProps, ...stateProps, ...dispatchProps } by default.»
— react-redux.js.org/api/connect
«Если указан, определяет, как формируются финальные props для вашего обёрнутого компонента. Если mergeProps не указан, обёрнутый компонент получает { ...ownProps, ...stateProps, ...dispatchProps } по умолчанию.»
— перевод

Сигнатура

«mergeProps should be specified with maximum of three parameters. They are the result of mapStateToProps(), mapDispatchToProps(), and the wrapper component's props, respectively: stateProps, dispatchProps, ownProps»
— react-redux.js.org/api/connect
«mergeProps должен принимать максимум три параметра — результаты mapStateToProps(), mapDispatchToProps() и props самой обёртки: stateProps, dispatchProps, ownProps.»
— перевод
function mergeProps( stateProps, // результат mapStateToProps(state, ownProps) dispatchProps, // результат mapDispatchToProps(dispatch, ownProps) ownProps // props, переданные в connected-компонент снаружи ) { return /* финальные props */ } connect(mapStateToProps, mapDispatchToProps, mergeProps)(Comp)

Зачем нужен custom mergeProps

По умолчанию react-redux делает shallow merge: { ...ownProps, ...stateProps, ...dispatchProps }. Custom mergeProps нужен в трёх основных случаях:

  • Выборочная фильтрация: передать в компонент только подмножество полей из stateProps.
  • Привязка ownProps к dispatch: обернуть action creators так, чтобы они автоматически брали часть аргументов из ownProps.
  • Вычисление производных props: скомбинировать данные из state и ownProps во что-то новое.

Сравнение: без mergeProps vs с mergeProps

Без custom mergeProps — компонент сам разбирается:

const mapStateToProps = (state) => ({ allTodos: state.todos }) const mapDispatchToProps = { addTodo } function UserProfile({ userId, allTodos, addTodo }) { const todos = allTodos[userId] || [] const handleAdd = (text) => addTodo(userId, text) // компонент знает про userId И про структуру allTodos И про сигнатуру addTodo } connect(mapStateToProps, mapDispatchToProps)(UserProfile)

С custom mergeProps — компонент получает готовые props:

const mapStateToProps = (state) => ({ allTodos: state.todos }) const mapDispatchToProps = { addTodo } function mergeProps(stateProps, dispatchProps, ownProps) { return { ...ownProps, todos: stateProps.allTodos[ownProps.userId] || [], addTodo: (text) => dispatchProps.addTodo(ownProps.userId, text), } } function UserProfile({ todos, addTodo }) { // компонент работает с готовыми todos и addTodo(text) } connect(mapStateToProps, mapDispatchToProps, mergeProps)(UserProfile)

Опасность: новые ссылки при каждом вызове

«Note: Creating new values in mergeProps will cause re-renders. It is recommended that you memoize fields in order to avoid unnecessary re-renders.»
— react-redux.js.org/api/connect
«Замечание: создание новых значений в mergeProps вызовет ре-рендеры. Рекомендуется мемоизировать поля, чтобы избежать лишних ре-рендеров.»
— перевод

Обратите внимание: addTodo: (text) => dispatchProps.addTodo(ownProps.userId, text) — это новая функция при каждом вызове mergeProps. Даже если stateProps и ownProps не поменялись, areMergedPropsEqual (default shallowEqual) сравнит функции по ссылке и увидит разницу. Для тяжёлых случаев фактор функции нужно мемоизировать — например, через замыкание-кэш по ownProps.userId.

// Наивный вариант — новая функция каждый раз function mergeProps(s, d, o) { return { ...o, addTodo: (text) => d.addTodo(o.userId, text) } } // Мемоизированный вариант — кэш по userId function makeMergeProps() { let cachedUserId: any let cachedAdd: any return (s, d, o) => { if (o.userId !== cachedUserId) { cachedUserId = o.userId cachedAdd = (text: string) => d.addTodo(o.userId, text) } return { ...o, addTodo: cachedAdd } } }

Порядок вызова внутри connect

  1. ownProps приходят от родителя.
  2. Вызывается mapStateToProps(state, ownProps) → stateProps.
  3. Вызывается mapDispatchToProps(dispatch, ownProps) → dispatchProps.
  4. Вызывается mergeProps(stateProps, dispatchProps, ownProps) → финальные props.
  5. Финальные props сравниваются с предыдущими через areMergedPropsEqual.
  6. Если отличаются — компонент ре-рендерится.

Связь с системой Subscription

Когда происходит dispatch, Subscription проходит по linked list callback'ов → notify() → checkForUpdates для connected-компонента → заново запускает mapStateToProps с новым state → затем mergeProps(stateProps, dispatchProps, ownProps) → сравнение результата через areMergedPropsEqual (по умолчанию shallowEqual) → если поля отличаются по ===, вызывается forceRender. mergeProps выполняется на каждой проверке — вот почему создание новых ссылок внутри него опасно. Подробнее о Subscription — урок 03a.

Когда НЕ нужен mergeProps: в подавляющем большинстве случаев дефолтного shallow merge достаточно. mergeProps — инструмент для специфических задач, где нужно «запечь» ownProps в action creator или выбрать кусок state по ownProps. Если вы просто хотите фильтровать данные по ownProps, чаще удобнее сделать это в mapStateToProps(state, ownProps) — см. урок 15.
