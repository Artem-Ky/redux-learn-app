Интерактив — два store, два контекста, два connect

На странице есть два независимых store: userStore и productStore. Каждый проброшен через свой собственный <Provider context={UserContext}> и <Provider context={ProductContext}>. Компоненты подключены через connect(..., ..., ..., { context: UserContext }) и connect(..., ..., ..., { context: ProductContext }) соответственно. Dispatch в один store не ре-рендерит компоненты подключённые к другому store.

Теория — кастомный контекст

Зачем нужен custom context

По умолчанию react-redux использует один встроенный контекст (ReactReduxContext). Все <Provider store={...}> и все connect / хуки видят друг друга через него. Когда в одном приложении нужно несколько независимых store (например, изолированная библиотека с собственным state и главное приложение), вы можете создать свой React.Context и передать его и в Provider, и в connect через options.context. Тогда эти два слоя не будут конфликтовать.

API options.context

«React-Redux v6 allows you to supply a custom context instance to be used by React-Redux. You need to pass the instance of your context to both <Provider /> and your connected component.»
— react-redux.js.org/api/connect
«React-Redux v6 позволяет передавать свой собственный инстанс контекста. Нужно передать инстанс вашего контекста и в <Provider />, и в ваш connected-компонент.»
— перевод
«You may pass the context to your connected component either by passing it here as a field of option, or as a prop to your connected component in rendering.»
— react-redux.js.org/api/connect
«Вы можете передать контекст в connected-компонент либо здесь как поле опции, либо как prop при рендере connected-компонента.»
— перевод

Два способа передать context

Способ 1: через options.context при создании connected-компонента:

const MyContext = React.createContext(null) const Connected = connect( mapStateToProps, null, null, { context: MyContext } )(Comp) // В рендере: <Provider store={myStore} context={MyContext}> <Connected /> </Provider>

Способ 2: как prop при рендере:

const Connected = connect(mapStateToProps)(Comp) <Provider store={myStore} context={MyContext}> <Connected context={MyContext} /> </Provider>

Два независимых store на странице

const UserContext = React.createContext(null) const ProductContext = React.createContext(null) const userStore = createStore(userReducer) const productStore = createStore(productReducer) const UserInfo = connect( (state) => state, null, null, { context: UserContext } )(UserInfoComponent) const ProductList = connect( (state) => state, null, null, { context: ProductContext } )(ProductListComponent) <Provider store={userStore} context={UserContext}> <Provider store={productStore} context={ProductContext}> <UserInfo /> <ProductList /> </Provider> </Provider>

Обязательное правило: контексты должны совпадать

Invariant Violation — если Provider передаёт один context, а connect ожидает другой, react-redux не сможет найти store и бросит ошибку «Could not find store» / «Invariant Violation». Provider и connect должны ссылаться на один и тот же инстанс контекста.

Связь с системой Subscription

Каждый Provider создаёт свой Subscription внутри своего context'а и отдаёт его подключённым компонентам. Когда вы диспатчите action в userStore, Subscription именно этого store проходит по linked list callback'ов → notify() → checkForUpdates → селектор → сравнение через === → forceRender если отличается. Подписчики productStore живут в другом Subscription и о событиях userStore ничего не знают — поэтому их ре-рендеры не запускаются. Подробнее — урок 03a. Это же мы делали через custom hooks в уроке 11 — там использовались createSelectorHook / createDispatchHook, теперь — через options.context для connect.

Когда использовать custom context: библиотека/виджет со своим state, которую нужно дропнуть внутрь хост-приложения; микро-фронтенды с изолированными store; тесты, где нужно подменить store. Для большинства обычных SPA — достаточно одного store и дефолтного контекста.
