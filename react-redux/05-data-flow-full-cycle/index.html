Интерактив — Полный цикл данных от клика до обновления UI

Нажмите + или − и наблюдайте как каждый шаг цикла подсвечивается в реальном времени. В пошаговом режиме можно проходить шаги по одному.

1
Пользователь кликает кнопку
React вызывает onClick handler компонента.
onClick={() => dispatch(increment())}
↓
2
useDispatch() → dispatch(action)
Handler вызывает dispatch с объектом action. Action отправляется в Redux store.
dispatch({ type: 'counter/increment' })
↓
3
Store вызывает reducer
Redux store передаёт текущий state и action в reducer. Reducer возвращает новый state.
reducer({ value: N }, { type: 'counter/increment' }) → { value: N+1 }
↓
4
Store сохраняет новый state → Subscription оповещает слушателей
Store заменяет текущий state на новый. Затем вызывает callback из store.subscribe() — это Root Subscription (созданная Provider'ом). Subscription проходит по своему linked list слушателей и вызывает checkForUpdates() каждого (подробнее в 03a).
store → Root Subscription.notify() → listener.callback() для каждого
↓
5
checkForUpdates(): selector(newState) !== prevResult?
Для каждого компонента с useSelector: берёт новый state из store.getState(), запускает его selector-функцию, сравнивает результат с предыдущим через ===. Если значение отличается — вызывает forceRender() (через useReducer внутри). Если совпадает — ничего не делает, компонент спит.
selector({ value: N+1 }) → N+1. prev(N) !== N+1 → true → forceRender()
↓
6
Компонент ре-рендерится с новым значением
React перерисовывает компонент. UI обновлён. Цикл завершён.
Counter() → <span>{N+1}</span>
Теория — Полный цикл данных React-Redux

Цикл данных: 6 шагов

Каждое взаимодействие пользователя с React-Redux приложением проходит через 6 последовательных шагов. Понимание этого цикла — фундамент для всего дальнейшего обучения.

// Полный цикл одного клика: // Шаг 1: Пользователь кликает <button onClick={() => dispatch(increment())}> // Шаг 2: dispatch отправляет action в store store.dispatch({ type: 'counter/increment' }) // Шаг 3: Store вызывает reducer counterReducer({ value: 0 }, { type: 'counter/increment' }) // → возвращает { value: 1 } // Шаг 4: Store сохраняет новый state и уведомляет Subscription // (внутренний механизм React-Redux) // Шаг 5: useSelector запускает selector с новым state const newValue = selector(newState) // → 1 const changed = prevValue !== newValue // 0 !== 1 → true → RERENDER // Шаг 6: React перерисовывает компонент // Counter() вызывается заново, useSelector возвращает 1

Ключевой шаг: checkForUpdates (шаги 4–5)

Шаги 4–5 — самые важные для производительности. Вот что происходит внутри (подробнее — в уроке 03a):

  1. Store вызывает callback из store.subscribe()
  2. Этот callback — метод Root Subscription (созданной Provider'ом)
  3. Subscription проходит по linked list слушателей
  4. Для каждого слушателя (каждый useSelector в приложении) вызывает checkForUpdates()
  5. checkForUpdates берёт store.getState(), запускает selector-функцию, и сравнивает результат с предыдущим через ===
  6. Если prevResult !== newResult → вызывает forceRender() (внутри это useReducer(c => c + 1, 0) — трюк для принудительного рендера)
  7. Если prevResult === newResult → ничего не делает, компонент спит

Сравнение по умолчанию: === (строгое равенство ссылок). Это означает, что если selector возвращает новый объект каждый раз (например state => ({ a: state.a, b: state.b })) — компонент будет ре-рендериться при каждом dispatch, даже если данные не изменились. Это одна из главных «ловушек», которую мы подробно разберём в уроках про useSelector.

Что если данные НЕ изменились?

// Пример: dispatch изменяет state.other, но НЕ state.counter // Компонент подписан на state.counter: const count = useSelector(state => state.counter) // При dispatch({ type: 'other/update' }): // Шаг 5: selector(newState) → state.counter → 0 (не изменился) // prevValue === newValue → 0 === 0 → true → SKIP RERENDER // Компонент НЕ перерисовывается!
Вывод: React-Redux перерисовывает компонент только если selector(newState) !== selector(prevState). Это работает автоматически — вам нужно лишь правильно писать селекторы. Мы научим этому в разделе F.

Аналогия с Classic Redux

В Classic Redux мы вручную делали store.subscribe(() => { ... }) и сами решали, когда обновлять UI. React-Redux автоматизирует этот процесс:

Шаг Classic Redux React-Redux
Подписка store.subscribe() вручную Автоматически через useSelector
Чтение store.getState() вручную useSelector(selector) автоматически
Проверка Вручную (если вообще есть) === сравнение автоматически
Обновление UI DOM-манипуляции вручную React re-render автоматически
Отписка unsubscribe() вручную Автоматически при unmount
