Дерево компонентов — кто ре-рендерится?

Dispatch изменяет state.counter. Компоненты с синей полосой подключены к store. Только подключённые компоненты с изменёнными данными подсветятся.

Теория — Внутренняя архитектура React-Redux

Слои архитектуры

React-Redux — это многослойная система, в которой каждый слой выполняет свою роль. Понимание этих слоёв — ключ к эффективному использованию библиотеки.

1. Redux Store

Знакомый нам из Classic Redux store: createStore(reducer). Ничего нового. React-Redux не создаёт свой store — он использует тот, что вы передаёте в Provider.

2. Provider + React Context

«Internally, React Redux uses React's "context" feature to make the Redux store accessible to deeply nested connected components. As of React Redux version 6, this is normally handled by a single default context object instance generated by `React.createContext()`, called `ReactReduxContext`.»
— react-redux.js.org — Accessing the Store
«Внутри React Redux использует фичу "context" React, чтобы сделать Redux store доступным для глубоко вложенных подключённых компонентов. Начиная с версии 6, это обрабатывается одним объектом контекста по умолчанию, созданным через `React.createContext()`, называемым `ReactReduxContext`.»
— перевод

Важный нюанс: Provider НЕ просто кладёт store в Context. Он делает три вещи:

  1. Кладёт в Context стабильный объект { store, subscription } — этот объект НИКОГДА не меняется, поэтому useContext() НИКОГДА не вызывает ре-рендер.
  2. Создаёт root Subscription — linked list callback'ов. Один раз вызывает store.subscribe() и при каждом dispatch проходит по списку слушателей.
  3. Для каждого слушателя (компонент с useSelector): запускает его selector, сравнивает результат с предыдущим (===), и если значение изменилось — вызывает forceRender() только для этого компонента.

Подробный разбор Subscription с исходным кодом — в уроке 03a.

3. Subscription System — как React-Redux решает проблему лишних рендеров

Это самый важный и наименее документированный механизм React-Redux. Он не описан как отдельная страница в docs, но упоминается в нескольких местах. Разберём по порядку.

Проблема: React Context работает так: когда значение Context.Provider меняется, ВСЕ компоненты, вызывающие useContext(), перерисовываются. Если бы React-Redux просто клал state в Context — при каждом dispatch перерисовывалось бы ВСЁ дерево. Это катастрофа для производительности.

Решение: React-Redux НЕ кладёт state в Context. Он кладёт ссылку на store (которая никогда не меняется) и объект Subscription. Уведомления об изменениях идут через Subscription, минуя React Context.

Как это работает пошагово:

  1. <Provider> вызывает store.subscribe() один раз и создаёт корневой Subscription-объект. Этот объект хранит список «слушателей» (listeners).
  2. Когда компонент вызывает useSelector(selector), React-Redux регистрирует этот компонент как слушателя в Subscription. При этом React Context НЕ меняется.
  3. При dispatch(action) Redux store обновляется. Store вызывает callback из шага 1. Subscription получает уведомление.
  4. Subscription проходит по списку слушателей. Для каждого запускает его selector с новым state. Сравнивает результат с предыдущим (===). Если отличается — вызывает forceRender() только для этого компонента.
// Упрощённая схема Subscription: class Subscription { listeners = [] // список подписанных компонентов addListener(listener) { this.listeners.push(listener) } notifyAll() { // Вызывается при каждом store.dispatch() this.listeners.forEach(listener => { const newValue = listener.selector(store.getState()) if (newValue !== listener.prevValue) { listener.prevValue = newValue listener.forceRender() // Ре-рендер ТОЛЬКО этого компонента! } }) } }

Разница между connect и useSelector:

  • connect() создаёт вложенную (nested) Subscription: каждый connect-компонент становится «мини-Provider» для своих потомков. Это формирует иерархию подписок (parent → child), что решает проблему «zombie children» (когда дочерний компонент ещё подписан на удалённые данные).
  • useSelector создаёт плоскую (flat) подписку: регистрируется напрямую в Subscription объекте Provider. Проще, но может иметь edge cases со stale props (подробнее в уроке 53).
«Starting with version 5, React Redux has attempted to guarantee that consistency with `ownProps`. In version 7, that is implemented using a custom `Subscription` class internally in `connect()`, which forms a nested hierarchy. This ensures that connected components lower in the tree will only receive store update notifications once the nearest connected ancestor has been updated.»
— react-redux.js.org — Stale Props and Zombie Children
«Начиная с версии 5, React Redux пытается гарантировать согласованность с ownProps. В версии 7 это реализовано с помощью кастомного класса Subscription внутри connect(), который формирует вложенную иерархию. Это гарантирует, что подключённые компоненты ниже по дереву получат уведомления об обновлении store только после того, как ближайший подключённый предок обновится.»
— перевод
«With hooks, there is no way to render a context provider, which means there's also no nested hierarchy of subscriptions. Because of this, the "stale props" and "zombie child" issues may potentially re-occur in an app that relies on using hooks instead of `connect()`.»
— react-redux.js.org — Stale Props and Zombie Children
«С хуками нет возможности рендерить context provider, а значит нет и вложенной иерархии подписок. Из-за этого проблемы "stale props" и "zombie children" потенциально могут возникнуть в приложении, использующем хуки вместо connect().»
— перевод

4. Публичный API (useSelector / useDispatch / connect)

Это то, что вы используете в своих компонентах. API скрывает всю сложность подписки, сравнения значений и оптимизации рендеров. Вы просто говорите «мне нужно state.counter.value» — и React-Redux делает всё остальное.

Как данные текут при ЧТЕНИИ

Store → Provider (React Context) → Subscription → useSelector → Component ↑ ↑ ↑ ↑ кладёт store создаёт подписку извлекает рендерится в Context на store данные с данными

Как данные текут при ЗАПИСИ (dispatch)

Component → useDispatch → store.dispatch(action) → reducer → новый state ↓ store уведомляет Subscription ↓ Subscription проверяет: данные изменились? ↓ ↓ Да → ре-рендер Нет → ничего
Ключевой инсайт: React-Redux использует React Context только для передачи ссылки на store, а НЕ для уведомления об изменениях. Уведомления идут через собственную систему Subscription, что обеспечивает точечные ре-рендеры только тех компонентов, чьи данные действительно изменились.
