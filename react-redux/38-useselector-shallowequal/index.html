Интерактив — четыре варианта equality

Четыре карточки используют разные варианты equalityFn. Первые три читают один и тот же объект {'{ a, b }'}. Четвёртая читает дробное precision с кастомным сравнением. Кнопки меняют state — считайте рендеры каждой карточки.

Теория — shallowEqual и кастомные equality-функции

Зачем нужен второй аргумент useSelector

По умолчанию useSelector сравнивает результаты через === (см. урок 37). Это плохо работает, когда selector возвращает объект или массив, так как каждый вызов создаёт новую ссылку. Второй аргумент equalityFn позволяет задать своё правило сравнения — ре-рендер будет только когда equalityFn(prev, next) вернул false.

shallowEqual — shallow-сравнение из react-redux

"Use the shallowEqual function from React-Redux as the equalityFn argument to useSelector()"
— react-redux.js.org/api/hooks
«Используйте функцию shallowEqual из React-Redux в качестве аргумента equalityFn для useSelector().»
— перевод

shallowEqual сравнивает два объекта (или массива) на один уровень глубины: каждое поле сравнивается через ===. Если все поля равны — объекты считаются одинаковыми, и ре-рендера не будет, даже если ссылки разные.

Два способа передать equalityFn

// Способ 1: второй аргумент — сама функция import { shallowEqual } from 'react-redux' const data = useSelector( (state: RootState) => ({ a: state.pair.a, b: state.pair.b }), shallowEqual ) // Способ 2: объект options с полем equalityFn const data = useSelector( (state: RootState) => ({ a: state.pair.a, b: state.pair.b }), { equalityFn: shallowEqual } )

Способ 2 особенно удобен, когда кроме equalityFn нужно передать настройки dev-проверок (devModeChecks, урок 42).

Кастомная equality-функция

"Use a custom equality function as the equalityFn argument to useSelector()"
— react-redux.js.org/api/hooks
«Используйте свою функцию сравнения в качестве аргумента equalityFn для useSelector().»
— перевод
"The optional comparison function also enables using something like Lodash's _.isEqual() or Immutable.js's comparison capabilities."
— react-redux.js.org/api/hooks
«Необязательная функция сравнения также позволяет использовать, например, _.isEqual() из Lodash или возможности сравнения из Immutable.js.»
— перевод

Любая функция (prev, next) => boolean подходит. true означает «одинаковые — не ре-рендерить», false — «разные — ре-рендерить».

// Пример: считаем числа «равными», если отличаются меньше чем на 0.01 const customEqual = (a: number, b: number) => Math.abs(a - b) < 0.01 const precision = useSelector((state: RootState) => state.precision, customEqual)

shallowEqual vs createSelector

Оба подхода решают проблему возврата объекта из selector'а, но по-разному:

  • shallowEqual не мешает selector'у создавать новый объект при каждом вызове — мы просто не ре-рендерим, если все поля равны.
  • createSelector (из reselect) мемоизирует результат: если inputs не изменились — возвращается та же самая ссылка, и дефолтного === уже достаточно.

На практике createSelector предпочтительнее, когда вычисление тяжёлое или результат передаётся дальше в React.memo-компоненты. shallowEqual — лёгкое средство, когда selector просто «упаковывает» несколько полей.

Связь с системой Subscription

После dispatch: linked list callback'ов → Subscription.notify() → для каждого слушателя checkForUpdates → запустить selector → сравнить через equalityFn (по умолчанию ===, иначе — то, что вы передали) → forceRender если отличается. То есть equalityFn встраивается ровно в шаг сравнения, остальные этапы подписки те же. Подробнее про Subscription — урок 03a.

Осторожно с тяжёлыми сравнениями: equalityFn вызывается при каждом dispatch. Если вы положите туда _.isEqual на большой структуре — это станет узким местом. shallowEqual почти бесплатный, но глубокое сравнение больших массивов — нет.
Рабочий рецепт:
  1. Возвращайте примитив — дефолтного === достаточно.
  2. Возвращаете плоский объект из 2–5 полей — shallowEqual.
  3. Derived-значение с вычислением — createSelector (урок 40).
  4. Нестандартная семантика (округление, tolerant-сравнение) — кастомная equalityFn.
