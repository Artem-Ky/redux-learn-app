Интерактив — stabilityCheck и identityFunctionCheck

Два заведомо «сломанных» selector'а: один возвращает новый объект каждый раз, другой — весь state целиком. В dev-режиме useSelector вызывает selector второй раз с тем же state и пишет предупреждение в консоль браузера. Переключайте devModeChecks и смотрите, как меняется поведение.

Теория — dev-mode проверки в useSelector

Что это и зачем

"useSelector runs some extra checks in development mode to watch for unexpected behavior. These checks do not run in production builds."
— react-redux.js.org/api/hooks
«useSelector выполняет несколько дополнительных проверок в development-режиме, отслеживая неожиданное поведение. В production-сборке эти проверки не запускаются.»
— перевод
"These checks were first added in v8.1.0"
— react-redux.js.org/api/hooks
«Эти проверки впервые появились в v8.1.0.»
— перевод

Проверка 1 — stabilityCheck

"In development, the provided selector function is run an extra time with the same parameter during the first call to useSelector, and warns in the console if the selector returns a different result (based on the equalityFn provided)."
— react-redux.js.org/api/hooks
«В development provided selector вызывается ещё раз с тем же аргументом при первом вызове useSelector, и в консоль пишется предупреждение, если selector возвращает другой результат (с учётом переданной equalityFn).»
— перевод
"This is important, as a selector that returns a different result reference when called again with the same inputs will cause unnecessary rerenders."
— react-redux.js.org/api/hooks
«Это важно, потому что selector, возвращающий другую ссылку на результат при повторном вызове с теми же входными данными, будет вызывать ненужные ре-рендеры.»
— перевод

Типичный триггер — selector, собирающий inline-объект или массив:

// stabilityCheck ловит вот это: const data = useSelector((state: RootState) => ({ count: state.count, user: state.user })) // warning: "Selector unknown returned a different result when called // with the same parameters. This can lead to unnecessary rerenders."

Проверка 2 — identityFunctionCheck

"In development, a check is conducted on the result returned by the selector. It warns in the console if the result is the same as the parameter passed in, i.e. the root state."
— react-redux.js.org/api/hooks
«В development выполняется проверка результата selector'а. В консоль пишется предупреждение, если результат — это тот же самый объект, что был передан в selector, то есть весь root state.»
— перевод
"A useSelector call returning the entire root state is almost always a mistake, as it means the component will rerender whenever anything in state changes. Selectors should be as granular as possible, like state => state.some.nested.field."
— react-redux.js.org/api/hooks
«Вызов useSelector, возвращающий root state целиком, почти всегда ошибка: компонент ре-рендерится при любом изменении в state. Selector'ы должны быть максимально гранулированными, вроде state => state.some.nested.field.»
— перевод
// identityFunctionCheck ловит это: const everything = useSelector((state: RootState) => state) // warning: "Selector unknown returned the root state when called. // This can lead to unnecessary rerenders. Selectors that // return the entire state are almost certainly a mistake..."

До v8.1 эта проверка называлась noopCheck. С v8.1 переименована в identityFunctionCheck — более точное название (selector работает как функция тождества).

Значения frequencies

Значение Когда проверяется Когда брать
'never' Никогда Заведомо «сломанный» selector, который вы контролируете сами (например, memoized)
'once' (default) Только при первом вызове данного инстанса хука Большинство случаев — one-shot проверка на старте
'always' При каждом вызове selector'а Отладка регрессий — поймать селекторы, ломающиеся не сразу

Настройка: глобально через Provider

<Provider store={store} stabilityCheck="always" identityFunctionCheck="always" > {children} </Provider>

Настройка: per-hook через options

const count = useSelector(selectCount, { devModeChecks: { stabilityCheck: 'never' } }) const user = useSelector(selectUser, { devModeChecks: { identityFunctionCheck: 'never' } }) // Можно оба сразу: const data = useSelector(selectData, { devModeChecks: { stabilityCheck: 'always', identityFunctionCheck: 'never' } })

Важно: только dev

В production-сборке эти проверки полностью вырезаются (через process.env.NODE_ENV === 'production' guard). Накладных расходов в проде нет. Соответственно, полагаться на них в рантайме нельзя — они только для вас как разработчика.

Связь с системой Subscription

Проверки — это просто дополнительный вызов selector'а внутри функции checkForUpdates. На общий путь ре-рендера (linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender) они не влияют: просто запускают selector ещё раз и сравнивают результаты через ту же equalityFn. Warning летит в console.error. Подробнее про Subscription — урок 03a.

Рабочий совет: оставляйте 'once' по умолчанию. Переключайтесь на 'always', когда подозреваете, что selector ломается со временем — например, кэш reselect рассыпается после какого-то dispatch'а. На 'never' ставьте точечно там, где вы сознательно возвращаете новый объект и используете shallowEqual/кастомный equalityFn.
Мы подменяем console.error, чтобы показать предупреждения react-redux прямо в панели снизу. Реальные warning всё равно идут в консоль браузера (откройте DevTools — увидите их там же).
