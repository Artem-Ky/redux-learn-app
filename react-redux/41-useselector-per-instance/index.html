Интерактив — один selector на всех vs selector на инстанс

Девять инстансов компонента WordsByLetter. Каждый получает свою букву (A, B, C, ...) и выводит список слов из store, начинающихся на эту букву. Selector зависит от props. Слева — один общий createSelector на всех. Справа — отдельный selector на каждый инстанс через useMemo. Счётчики показывают, сколько раз вызвалась output-функция (.filter()) и сколько произошло ре-рендеров. Особенно показателен unrelated dispatch (tick) — words.list в store остаётся той же ссылкой.

Теория — per-instance мемоизация

Где ломается общий мемоизированный selector

createSelector из reselect по умолчанию хранит один последний результат — cache size = 1. Пока inputs приходят одни и те же, output-функция не вызывается. Но как только два инстанса вызывают тот же selector с разными props — они попеременно затирают друг другу кэш.

// ❌ ОБЩИЙ selector — cache ломается для разных props const selectCompletedTodosCount = createSelector( (state: RootState) => state.todos, (_: RootState, completed: boolean) => completed, (todos, completed) => todos.filter(t => t.completed === completed).length ) function CompletedTodosCount({ completed }: { completed: boolean }) { const count = useSelector((state: RootState) => selectCompletedTodosCount(state, completed) ) // Два инстанса с разными completed — кэш перезаписывается // ChildA (completed=true): cache = { inputs=[todos, true], output=3 } // ChildB (completed=false): cache = { inputs=[todos, false], output=5 } ← перезаписал A // ChildA снова: cache miss → пересчёт → cache = { ..., true, 3 } ← перезаписал B return <div>{count}</div> }

Решение 1: фабрика selector'ов + useMemo

Функция-фабрика makeSelect...() возвращает новый экземпляр createSelector(...) при каждом вызове. Компонент сохраняет результат фабрики в useMemo(..., []) — у каждого инстанса теперь свой selector со своим кэшем. Никто никому ничего не затирает.

// Фабрика — создаёт отдельный createSelector на каждый вызов const makeSelectCompletedTodosCount = () => createSelector( (state: RootState) => state.todos, (_: RootState, completed: boolean) => completed, (todos, completed) => todos.filter(t => t.completed === completed).length ) function CompletedTodosCount({ completed }: { completed: boolean }) { const selectCount = useMemo(makeSelectCompletedTodosCount, []) const count = useSelector((state: RootState) => selectCount(state, completed)) // Каждый инстанс имеет СВОЙ selector с СВОИМ кэшем return <div>{count}</div> }

Эквивалентная запись через inline-useMemo без отдельной фабрики: useMemo(() => createSelector(...), []). Фабричный вариант удобнее тем, что одну и ту же makeSelect... можно переиспользовать в разных компонентах.

Решение 2: + shallowEqual в useSelector

Фабрика спасает от лишних вычислений, когда изменилась «чужая» часть стора (words.list та же ссылка → все кэши хитнули, filter не запускался, 0 ре-рендеров). Но как только words.list получает новую ссылку, кэши миссуют у всех инстансов, filter отдаёт НОВЫЙ массив, а useSelector по умолчанию сравнивает результат через === — и ре-рендерятся все строки, хотя содержимое у них не менялось.

Дополним: вторым аргументом в useSelector передадим shallowEqual. Он сравнивает массивы поэлементно — если слова остались те же, ре-рендер пропускается. В итоге при ADD_WORD "Aew-15" ре-рендерится только строка A, остальные — нет.

import { useSelector, shallowEqual } from 'react-redux' function WordsByLetter({ letter }: { letter: string }) { const selectByLetter = useMemo(makeSelectByLetter, []) const words = useSelector( (state: RootState) => selectByLetter(state, letter), shallowEqual // content-equal → ре-рендер не триггерим ) return <div>{words.join(', ')}</div> }

Варианты решения

  1. useMemo + createSelector. Самый простой способ. Selector создаётся в замыкании инстанса, deps useMemo — нужные props.
  2. Фабрика selector'ов (selector factory). Функция вида makeSelectCount(), возвращающая новый createSelector(...) при каждом вызове. Компонент хранит результат фабрики в useMemo.
  3. reselect с cache size > 1. Через createSelectorCreator / weakMapMemoize (Reselect 5) — один и тот же selector держит много cache-записей, по одной на комбинацию inputs.
  4. Factory function в connect. Аналог useMemo для класс-компонентов: mapStateToProps возвращает ещё одну функцию, которая и есть per-instance mapStateToProps. Подробнее — урок 32.
// Вариант 2: selector factory const makeSelectCount = () => createSelector( (state: RootState) => state.todos, (_: RootState, completed: boolean) => completed, (todos, completed) => todos.filter(t => t.completed === completed).length ) function CompletedTodosCount({ completed }: { completed: boolean }) { const selectCount = useMemo(makeSelectCount, []) const count = useSelector((state: RootState) => selectCount(state, completed)) return <div>{count}</div> }

Как это выглядит в системе Subscription

Мемоизация selector'а — это локальная оптимизация внутри самой selector-функции. Она не меняет поведение подписки. Путь ре-рендера после dispatch всегда тот же: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender, если результат изменился. Но если у вас общий selector и N инстансов с разными props, то на каждом Subscription.notify() output-функция запускается N раз вместо одного — кэш стирается соседями. Per-instance selector ломает эту цепочку: каждый инстанс считает свой кусок один раз и попадает в кэш до следующего изменения state. Подробнее про Subscription — урок 03a.

Опасно: нельзя создавать createSelector(...) прямо в теле компонента без useMemo. Тогда selector пересоздаётся на каждом рендере — внутренний кэш reselect теряется ещё до первого хита.
Правило большого пальца:
  • Selector не зависит от props → объявляйте один экземпляр на уровне модуля.
  • Selector зависит от props и используется в одном инстансе → тоже на уровне модуля.
  • Selector зависит от props и используется в нескольких инстансах → useMemo + createSelector (per-instance).
