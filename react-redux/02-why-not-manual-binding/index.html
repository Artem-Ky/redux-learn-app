Интерактив — Ручное подключение vs React-Redux

Оба счётчика подключены к одному и тому же Redux store. Левый — через ручной store.subscribe(), правый — через React-Redux. Обратите внимание на счётчик рендеров — он показывает эффективность.

Теория — Почему ручное подключение Redux к React — плохая идея

Четыре проблемы ручного подключения

Технически можно подключить Redux к React вручную — через store.subscribe() в useEffect и store.getState() для чтения. Но у такого подхода есть серьёзные проблемы:

Проблема 1: Props Drilling

Без React-Redux store нужно передавать как prop от корневого компонента к каждому, кому он нужен. Если компонент на 5-м уровне вложенности нуждается в store — придётся «просверлить» prop через 4 промежуточных компонента, которым store вообще не нужен.

// Ручной подход — props drilling кошмар: <App store={store}> <Layout store={store}> // Layout'у store не нужен! <Sidebar store={store}> // Sidebar'у store не нужен! <UserPanel store={store}> // Вот ему нужен... <Avatar store={store} /> // И ему тоже </UserPanel> </Sidebar> </Layout> </App>

Проблема 2: Лишние ре-рендеры

При ручном store.subscribe() callback вызывается при каждом dispatch — даже если данные этого компонента не изменились. В демо выше видно: при dispatch несвязанного действия ручной счётчик ре-рендерится, а React-Redux — нет.

// Ручной подход — ре-рендер при КАЖДОМ dispatch: useEffect(() => { const unsubscribe = store.subscribe(() => { // Вызывается при КАЖДОМ dispatch, даже если наши данные не изменились! setLocalState(store.getState().counter) }) return unsubscribe }, [])

Проблема 3: Утечки памяти

Если забыть вернуть unsubscribe из useEffect, компонент останется подписан на store даже после размонтирования. Это утечка памяти. React-Redux управляет подпиской автоматически.

Проблема 4: Нет батчинга обновлений

При двух быстрых dispatch подряд ручной подход вызовет два отдельных setState, что может привести к двум ре-рендерам (в React < 18). React-Redux использует внутренний батчинг для объединения обновлений.

Как делает React-Redux

// React-Redux подход — просто, оптимизировано, безопасно: import { useSelector, useDispatch } from 'react-redux' function Counter() { // Автоподписка + ре-рендер ТОЛЬКО если state.counter.value изменился const count = useSelector(state => state.counter.value) const dispatch = useDispatch() return <button onClick={() => dispatch({ type: 'increment' })}>{count}</button> }
Вывод: React-Redux берёт на себя всю сложность интеграции: подписку, отписку, проверку изменений, оптимизацию рендеров, батчинг. Вы просто указываете какие данные нужны — всё остальное делается автоматически.

Итоговая таблица сравнения

Аспект Ручное подключение React-Redux
Передача store Props drilling Provider + Context
Подписка/отписка Вручную в useEffect Автоматически
Ре-рендеры При каждом dispatch Только при изменении данных
