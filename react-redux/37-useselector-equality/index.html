Интерактив — «плохой» vs «хороший» useSelector

Оба компонента читают одни и те же данные. «Плохой» возвращает из selector'а новый объект каждый раз. «Хороший» делает два отдельных useSelector с примитивами. Кнопка tick (unrelated) меняет state.other — «хороший» на это вообще не должен реагировать.

Теория — сравнение результатов selector'а

Как useSelector решает, рендериться ли

"When an action is dispatched, useSelector() will do a reference comparison of the previous selector result value and the current result value. If they are different, the component will be forced to re-render. If they are the same, the component will not re-render."
— react-redux.js.org/api/hooks
«Когда action диспатчится, useSelector() выполняет сравнение по ссылке между предыдущим и текущим результатом selector'а. Если они различаются — компонент принудительно ре-рендерится. Если одинаковы — компонент не ре-рендерится.»
— перевод
"useSelector() uses strict === reference equality checks by default, not shallow equality"
— react-redux.js.org/api/hooks
«useSelector() по умолчанию использует строгие сравнения по ссылке через ===, а не shallow-сравнение.»
— перевод

Ключевое отличие от connect

"This is different than connect(), which uses shallow equality checks on the results of mapState calls to determine if re-rendering is needed."
— react-redux.js.org/api/hooks
«Это отличается от connect(), который использует shallow-сравнение результатов вызовов mapStateToProps для принятия решения о ре-рендере.»
— перевод
"With mapStateToProps, all individual fields were returned in a combined object. It didn't matter if the return object was a new reference or not — connect() just compared the individual fields. With useSelector(), returning a new object every time will always force a re-render by default."
— react-redux.js.org/api/hooks
«В mapStateToProps все отдельные поля возвращались в составе одного общего объекта. Не имело значения, новая это ссылка или нет — connect() сравнивал поля по отдельности. С useSelector(), возвращение нового объекта каждый раз всегда принудительно вызывает ре-рендер по умолчанию.»
— перевод

Таблица: mapStateToProps vs useSelector

mapStateToProps (connect) useSelector

Решение Возвращайте плоский объект Примитивы · shallowEqual · createSelector

Три способа «починить» selector, возвращающий объект

// ❌ Проблема: новый объект на каждом вызове selector'а const data = useSelector((state: RootState) => ({ a: state.counter.a, b: state.counter.b })) // prev = { a: 1, b: 2 }; new = { a: 1, b: 2 } — но другая ссылка! // prev !== new → ре-рендер на каждом dispatch
// ✔ Вариант 1: отдельные useSelector const a = useSelector((state: RootState) => state.counter.a) const b = useSelector((state: RootState) => state.counter.b) // ✔ Вариант 2: shallowEqual (урок 38) const data = useSelector( (state: RootState) => ({ a: state.counter.a, b: state.counter.b }), shallowEqual ) // ✔ Вариант 3: createSelector из reselect (урок 40) const selectAB = createSelector( (state: RootState) => state.counter, (c) => ({ a: c.a, b: c.b }) ) const data = useSelector(selectAB)

Почему так сделано?

useSelector — хук для одного значения. Когда вы возвращаете объект, вы неявно «собираете» несколько значений — это тот же случай, что и mapStateToProps. Но API useSelector проще: он не пытается угадать, что вы комбинируете, а просто сравнивает по ссылке. Вы сами решаете, как собрать данные — через отдельные хуки, shallowEqual или memoized-selector.

Связь с системой Subscription

После dispatch: linked list callback'ов → Subscription.notify() → для каждого слушателя checkForUpdates → запустить selector → сравнить через === → forceRender если отличается. В случае selector'а, возвращающего inline-объект, каждый dispatch создаёт новый результат, а новая ссылка !== старой → forceRender. Отсюда и лишние ре-рендеры. Подробнее про Subscription — урок 03a.

Dev-предупреждение: начиная с React-Redux v8.1 useSelector в dev-режиме запускает selector ещё раз сразу после первого вызова и сравнивает результаты. Если selector возвращает разные ссылки при одинаковом state — в консоль летит warning «Selector unknown returned a different result when called with the same parameters». Это и есть сигнал «у тебя selector возвращает новый объект». Подробнее — урок 42.
Рабочее правило: пусть selector возвращает примитив (число, строку, boolean) или уже стабильную ссылку (например, state.items, которая не пересоздаётся, пока вы её не измените). Как только вам нужен объект или массив derived — переходите к createSelector.
