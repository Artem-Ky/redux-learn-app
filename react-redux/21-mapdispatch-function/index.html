Интерактив — mapDispatchToProps в функциональной форме

Слева — подход из урока 20 (props.dispatch({type:'...'}) прямо в обработчике). Справа — тот же счётчик, но через mapDispatchToProps-функцию: компонент получает в props готовые increment, decrement, reset и вызывает их напрямую. Нажмите кнопки и сравните записи в консоли.

Теория — mapDispatchToProps (функциональная форма)

Зачем вообще нужен mapDispatchToProps?

В уроке 20 мы видели: если не передать второй аргумент в connect, компонент получает props.dispatch «бесплатно». Это работает, но заставляет компонент знать о Redux: onClick={() => props.dispatch({ type: 'counter/increment' })}. mapDispatchToProps решает эту проблему — превращает «сырой» dispatch в удобные именованные функции, которые компонент воспринимает как обычные колбэки.

«Providing a mapDispatchToProps allows you to specify which actions your component might need to dispatch. It lets you provide action dispatching functions as props. Therefore, instead of calling props.dispatch(() => increment()), you may call props.increment() directly.»
— react-redux.js.org/using-react-redux/connect-mapdispatch
«Передача mapDispatchToProps позволяет указать, какие именно экшены будет диспатчить компонент. Она превращает функции диспатча в props. Поэтому вместо вызова props.dispatch(() => increment()) вы можете вызвать props.increment() напрямую.»
— перевод

Синтаксис функциональной формы

import { connect } from 'react-redux' // (1) Функция принимает dispatch и возвращает объект const mapDispatchToProps = (dispatch) => ({ increment: () => dispatch({ type: 'counter/increment' }), decrement: () => dispatch({ type: 'counter/decrement' }), reset: () => dispatch({ type: 'counter/reset' }), }) // (2) Передаём вторым аргументом в connect export default connect(mapStateToProps, mapDispatchToProps)(Counter) // (3) Внутри компонента — вместо props.dispatch(...) function Counter({ count, increment, decrement, reset }) { return <button onClick={increment}>+</button> }

Причина №1 — декларативность

«First, encapsulating the dispatch logic into function makes the implementation more declarative. Dispatching an action and letting the Redux store handle the data flow is how to implement the behavior, rather than what it does.»
— react-redux.js.org/using-react-redux/connect-mapdispatch
«Во-первых, упаковка логики диспатча внутрь функции делает реализацию более декларативной. Диспатч экшена и обработка потока данных внутри store — это как реализовать поведение, а не что именно оно делает.»
— перевод

Сравните два обработчика кнопки:

// Императивно — компонент знает о dispatch и типе экшена: <button onClick={() => dispatch({ type: 'SOMETHING' })}> // Декларативно — компонент знает только про "doSomething": <button onClick={doSomething}>

Причина №2 — передача колбэков дочерним компонентам

«In addition, you also gain the ability to pass down the action dispatching functions to child (likely unconnected) components. This allows more components to dispatch actions, while keeping them "unaware" of Redux.»
— react-redux.js.org/using-react-redux/connect-mapdispatch
«Кроме того, вы получаете возможность передавать функции-диспатчеры в дочерние (как правило не подключённые к store) компоненты. Это позволяет большему числу компонентов диспатчить экшены, при этом оставаясь "не осведомлёнными" о Redux.»
— перевод

Дочерний компонент (например, кнопка) получает onClick как обычный колбэк и даже не догадывается, что он связан с Redux. Это упрощает переиспользование UI и тесты: дочерку можно рендерить в Storybook без Provider.

// Родитель — connected function TodoListRaw({ todos, toggleTodo }) { return todos.map(t => <Todo todo={t} onClick={toggleTodo} /> ) } // Дочерний Todo — unconnected, ничего не знает про Redux function Todo({ todo, onClick }) { return <li onClick={() => onClick(todo.id)}>{todo.text}</li> }

Параметры функции mapDispatchToProps

Функция может быть арности 1 или 2:

  • (dispatch) — получает только dispatch, вызывается один раз при монтировании connected-компонента;
  • (dispatch, ownProps) — получает дополнительно ownProps (пропсы, переданные родителем). Переинициализируется при смене этих пропсов. Подробнее — в уроке 22.

Что возвращать

Функция должна возвращать plain object: ключи станут именами пропсов, значения — функциями, которые внутри себя вызывают dispatch.

Внимание: если вы указали mapDispatchToProps (любой формы), props.dispatch больше не передаётся компоненту автоматически. Это отдельная частая ошибка — ей посвящён урок 25.

Связь с Subscription-системой

Когда компонент вызывает props.increment(), внутри функция делает dispatch({ type: 'counter/increment' }). Store прогоняет reducer, получает новый state и вызывает Subscription.notify() (детали в уроке 03a). Subscription — это linked list callback'ов: для каждого подключённого компонента checkForUpdates запускает mapStateToProps с новым state, сравнивает каждое поле через === с предыдущим значением, и если хотя бы одно изменилось — вызывает forceRender(). Сами функции из mapDispatchToProps при этом не пересоздаются — connect кэширует их.

Итог урока: mapDispatchToProps в функциональной форме превращает сырой dispatch в набор именованных колбэков. Компонент становится декларативным и может передавать эти колбэки дочерним компонентам. В следующих уроках посмотрим на ownProps, bindActionCreators и короткую объектную форму.
