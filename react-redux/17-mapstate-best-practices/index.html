Интерактив — Анти-паттерны и исправления

Четыре распространённых ошибки при написании mapStateToProps. Каждая вкладка показывает плохой и хороший варианты с подсчётом рендеров.

Теория — Лучшие практики mapStateToProps

1. mapStateToProps должна «перекраивать» данные

"mapStateToProps functions can, and should, do a lot more than just return state.someSlice. They have the responsibility of 're-shaping' store data as needed for that component." — react-redux.js.org/using-react-redux/connect-mapstate
«Функции mapStateToProps могут и должны делать гораздо больше, чем просто возвращать state.someSlice. Они отвечают за "перекраивание" данных из store в ту форму, которая нужна конкретному компоненту.»

2. mapStateToProps должна быть быстрой

"mapStateToProps Functions Should Be Fast" — react-redux.js.org/using-react-redux/connect-mapstate (заголовок раздела)
«Функции mapStateToProps должны быть быстрыми.»

Эта функция вызывается при каждом изменении store. Если внутри выполняются тяжёлые вычисления — вся система Subscription замедляется. Subscription проходит по linked list callback'ов, вызывает checkForUpdates для каждого подключённого компонента, а checkForUpdates запускает mapStateToProps и сравнивает результат через ===. Если mapStateToProps медленная — всё тормозит (подробнее в уроке 03a).

3. mapStateToProps должна быть чистой и синхронной

"mapStateToProps Functions Should Be Pure and Synchronous" — react-redux.js.org/using-react-redux/connect-mapstate (заголовок раздела)
«Функции mapStateToProps должны быть чистыми и синхронными.»

Никаких fetch(), setTimeout(), побочных эффектов или мутаций. mapStateToProps — чистая функция: получает state, возвращает объект. Точка.

4. Используйте селекторы

"We highly encourage the use of 'selector' functions to help encapsulate the process of extracting values from specific locations in the state tree." — react-redux.js.org/using-react-redux/connect-mapstate
«Мы настоятельно рекомендуем использовать функции-"селекторы", чтобы инкапсулировать процесс извлечения значений из конкретных мест дерева state.»

Таблица анти-паттернов

Плохо Хорошо Почему
state => state state => ({ count: state.counter }) Вернуть весь state = ре-рендер при любом изменении
state.todos.map(...) Мемоизированный селектор map() создаёт новый массив → новая ссылка → всегда !==
async / fetch() Чистая синхронная функция mapStateToProps должна быть чистой
Тяжёлые вычисления createSelector (reselect) Мемоизация пересчитывает только при изменении входных данных
Внимание: state => state — самый частый анти-паттерн. Компонент будет перерисовываться при каждом dispatch, даже если нужные ему данные не менялись. Всегда извлекайте конкретные поля.
