Интерактив — один компонент в двух реализациях

Оба компонента читают одно и то же: count * multiplier. Слева — connect, справа — useSelector. Слайдер меняет multiplier (это ownProps родителя). Кнопка «render parent» форсирует ре-рендер родителя без изменения store. Смотрите, как по-разному ведут себя счётчики.

Теория — mapStateToProps vs useSelector

Главное отличие: что возвращает selector

"The selector may return any value as a result, not just an object."
— react-redux.js.org/api/hooks
«Selector может возвращать любое значение — не только объект.»
— перевод
"The selector normally should return just a single value, and not an object. If you do return an object or an array, be sure to use a memoized selector to avoid unnecessary re-renders."
— react-redux.js.org/api/hooks
«Selector обычно должен возвращать одно значение, а не объект. Если вы всё же возвращаете объект или массив — используйте мемоизированный selector, чтобы избежать ненужных ре-рендеров.»
— перевод

mapStateToProps обязан возвращать объект — его поля становятся props компонента. useSelector возвращает одно значение любого типа и может вызываться несколько раз в одном компоненте — по одному на интересующий slice.

Ре-рендер от родителя

"unlike connect(), useSelector() does not prevent the component from re-rendering due to its parent re-rendering, even if the component's props did not change."
— react-redux.js.org/api/hooks
«В отличие от connect(), useSelector() не препятствует ре-рендеру компонента при ре-рендере его родителя, даже если props компонента не изменились.»
— перевод

connect под капотом вызывает shouldComponentUpdate — если ни state, ни ownProps не изменились, React не ре-рендерит. useSelector — это просто хук, его результат проверяется только внутри этого рендера. Если родитель ре-рендерится — функциональный компонент вызывается заново, selector запускается заново, и только после сравнения === решается, нужно ли ещё один ре-рендер (forceRender). Но этот ре-рендер уже случился.

Лечится React.memo:

const Child = memo(function Child({ multiplier }: Props) { const count = useSelector((state: RootState) => state.counter.value) return <div>{count * multiplier}</div> })

ownProps

"The selector function does not receive an ownProps argument. However, props can be used through closure (see the examples above) or by using a curried selector."
— react-redux.js.org/api/hooks
«Selector не получает аргумент ownProps. Однако props можно использовать через замыкание или через curried selector.»
— перевод
// mapStateToProps — ownProps явно вторым аргументом const mapStateToProps = (state: RootState, ownProps: Props) => ({ scaled: state.counter.value * ownProps.multiplier }) // useSelector — ownProps через замыкание function Child({ multiplier }: Props) { const scaled = useSelector((state: RootState) => state.counter.value * multiplier // ← из closure ) }

Когда что брать

  • Новые проекты, функциональные компоненты — useSelector + useDispatch. Меньше бойлерплейта, проще типизация, лучшая интеграция с хуками.
  • Legacy-кодовая база на class-компонентах — connect. Или переписывайте на функциональные в рамках постепенной миграции.
  • Нужна nested subscription hierarchy (редкий случай — SSR, несколько store в ветвях) — connect. Хуки всегда подписываются на ближайший <Provider> корнем, без promote-parent-notify-child логики.

Про миграцию

Смешивать connect и хуки в одном приложении абсолютно нормально — это официально поддерживаемый паттерн, и именно так выглядит большинство реальных миграций. Пошаговое руководство и типовые трансформации — в уроке 54.

Связь с системой Subscription

Оба механизма используют один и тот же Subscription. После dispatch: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender (если результат изменился). Разница — в том, откуда берётся selector: для connect это комбинированный mapStateToProps с shallow-сравнением, для хуков — пользовательская функция с ===-сравнением. В обоих случаях уровень механизма подписки один и тот же.

Практика: не бойтесь использовать useSelector в уже существующем приложении с connect. Они работают поверх одного и того же <Provider>, один и тот же store, одна и та же подписка — только разный API на уровне компонента.
