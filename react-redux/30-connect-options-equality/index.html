Интерактив — areStatesEqual в действии

Оба компонента подключены к одному store с тяжёлым mapStateToProps. Левый — дефолтный (areStatesEqual: ===). Правый — с кастомным areStatesEqual, который пропускает вызов mapStateToProps, если срез entities.todos не изменился. Нажимайте «tick» (меняет не todos) — справа mapStateToProps не вызывается.

Теория — четыре equality функции connect

Четвёртый аргумент connect: options

connect принимает четыре аргумента: (mapStateToProps, mapDispatchToProps, mergeProps, options). Объект options содержит, в том числе, четыре функции сравнения, которые управляют кэшированием на разных уровнях конвейера connect.

Опция Что сравнивает По умолчанию
areStatesEqual Новый state и предыдущий state strictEqual (===)
areOwnPropsEqual Новые ownProps и предыдущие ownProps shallowEqual
areStatePropsEqual Результат mapStateToProps и предыдущий shallowEqual
areMergedPropsEqual Результат mergeProps и предыдущий shallowEqual

areStatesEqual

«areStatesEqual: Compares incoming store state to its previous value. default value: strictEqual: (next, prev) => prev === next»
— react-redux.js.org/api/connect
«areStatesEqual: сравнивает новый state store с предыдущим. Значение по умолчанию: strictEqual: (next, prev) => prev === next.»
— перевод
«You may wish to override areStatesEqual if your mapStateToProps function is computationally expensive and is also only concerned with a small slice of your state.»
— react-redux.js.org/api/connect
«Вы можете переопределить areStatesEqual, если ваш mapStateToProps вычислительно дорогой и при этом касается только маленького среза вашего state.»
— перевод
const areStatesEqual = (next, prev) => prev.entities.todos === next.entities.todos connect(mapStateToProps, null, null, { areStatesEqual })(Comp) // Если entities.todos не изменился — mapStateToProps даже не вызывается

areOwnPropsEqual

«areOwnPropsEqual: Compares incoming props to its previous value. default value: shallowEqual»
— react-redux.js.org/api/connect
«areOwnPropsEqual: сравнивает новые props с их предыдущим значением. Значение по умолчанию: shallowEqual.»
— перевод

Когда родитель ре-рендерится, он может передать обёртке те же ownProps. Дефолтный shallowEqual проверяет все поля через === и, если всё совпало, пропускает пересчёт. Переопределить стоит редко — например, если вы знаете что часть ownProps можно безопасно игнорировать.

areStatePropsEqual

«areStatePropsEqual: Compares the result of mapStateToProps to its previous value. default value: shallowEqual»
— react-redux.js.org/api/connect
«areStatePropsEqual: сравнивает результат mapStateToProps с его предыдущим значением. Значение по умолчанию: shallowEqual.»
— перевод

Это последний рубеж перед решением «нужен ли ре-рендер». mapStateToProps уже вызван, но если возвращённый объект shallow-равен предыдущему — рендера не будет. Переопределение — например, для глубокого сравнения (осторожно, это дорого).

areMergedPropsEqual

«areMergedPropsEqual: Compares the result of mergeProps to its previous value. default value: shallowEqual»
— react-redux.js.org/api/connect
«areMergedPropsEqual: сравнивает результат mergeProps с его предыдущим значением. Значение по умолчанию: shallowEqual.»
— перевод

Применяется только если указан свой mergeProps (урок 28). Финальная проверка после объединения stateProps / dispatchProps / ownProps.

Каскад проверок внутри connect

1
areStatesEqual
До вызова mapStateToProps. Если true → mapStateToProps даже не запускается, рендера не будет.
2
areOwnPropsEqual
Между входящими ownProps и предыдущими. Если true и state не менялся — пропускаем mapStateToProps / mapDispatchToProps.
3
areStatePropsEqual
После mapStateToProps. Если новый результат shallow-равен прежнему → ре-рендера не будет.
4
areMergedPropsEqual
После mergeProps. Последний барьер: если merged-объект shallow-равен предыдущему — рендера не будет.

Связь с системой Subscription

При dispatch Subscription проходит по linked list callback'ов → notify() → checkForUpdates для каждого connected-компонента. Именно внутри checkForUpdates разворачивается описанный каскад: сначала areStatesEqual (новый state vs старый), затем areOwnPropsEqual, затем (если пропустили) — вызов mapStateToProps, areStatePropsEqual, mergeProps, areMergedPropsEqual. Только если финальное сравнение показывает отличия, вызывается forceRender и React планирует ре-рендер. Подробнее о Subscription — урок 03a.

Практика: чаще всего вы НЕ будете трогать эти опции. Дефолты оптимальны для 99% случаев. Переопределяйте areStatesEqual, только если профилировщик показал, что mapStateToProps — «горячий» участок и при этом смотрит в маленький срез state. Для обычных селекторов лучше использовать reselect (урок 18) — он решает ту же задачу без изменения опций connect.
