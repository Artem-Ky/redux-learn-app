Интерактив — два подхода к типизации хуков

Слева — «старый» способ через TypedUseSelectorHook, справа — «новый» через .withTypes() из React-Redux v9.1.0+. Оба компонента читают один и тот же count из общего store и диспатчат один и тот же thunk через типизированный useAppDispatch. На уровне runtime они идентичны; разница только в том, как записывается тип.

Теория — типизированные хуки

Почему нужны типизированные хуки

"While it's possible to import the RootState and AppDispatch types into each component, it's better to create pre-typed versions of the useDispatch and useSelector hooks for usage in your application."
— react-redux.js.org/using-react-redux/usage-with-typescript
«Хотя можно импортировать типы RootState и AppDispatch в каждый компонент, лучше создать заранее типизированные версии хуков useDispatch и useSelector для использования во всём приложении.»
— перевод
"For useSelector, it saves you the need to type (state: RootState) every time"
— react-redux.js.org/using-react-redux/usage-with-typescript
«Для useSelector это избавляет вас от необходимости каждый раз писать (state: RootState).»
— перевод
"For useDispatch, the default Dispatch type does not know about thunks or other middleware. In order to correctly dispatch thunks, you need to use the specific customized AppDispatch type from the store that includes the thunk middleware types."
— react-redux.js.org/using-react-redux/usage-with-typescript
«Для useDispatch дефолтный тип Dispatch ничего не знает о thunk-ах или другом middleware. Чтобы корректно диспатчить thunks, нужно использовать специфический кастомизированный тип AppDispatch из store, включающий типы thunk-middleware.»
— перевод

Типы из store

"Using configureStore should not need any additional typings. You will, however, want to extract the RootState type and the Dispatch type so that they can be referenced as needed. Inferring these types from the store itself means that they correctly update as you add more state slices or modify middleware settings."
— react-redux.js.org/using-react-redux/usage-with-typescript
«Использование configureStore не должно требовать дополнительной типизации. Однако вы захотите извлечь типы RootState и Dispatch, чтобы на них можно было ссылаться по мере необходимости. Выведение этих типов из самого store означает, что они корректно обновляются по мере добавления новых state-slices или изменения настроек middleware.»
— перевод
// app/store.ts — типы выводятся из самого store export type RootState = ReturnType<typeof store.getState> export type AppDispatch = typeof store.dispatch export type AppStore = typeof store

Принцип: типы не объявляем, а выводим от самого store. Когда добавляем новый reducer или новое middleware — типы подтягиваются сами, без ручного правки.

Подход 1 — «старый» через TypedUseSelectorHook

// app/hooks.ts import type { TypedUseSelectorHook } from 'react-redux' import { useDispatch, useSelector, useStore } from 'react-redux' import type { AppDispatch, AppStore, RootState } from './store' export const useAppDispatch: () => AppDispatch = useDispatch export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector export const useAppStore: () => AppStore = useStore

Работает на всех версиях React-Redux и привычен любому, кто писал TypeScript с React-Redux до версии 9.1. Небольшой минус — три импортируемых имени (useDispatch, useSelector, useStore) и отдельный тип TypedUseSelectorHook.

Подход 2 — новый .withTypes() (v9.1.0+)

"React Redux v9.1.0 adds a new .withTypes method to each of these hooks, analogous to the .withTypes method found on Redux Toolkit's createAsyncThunk."
— react-redux.js.org/using-react-redux/usage-with-typescript
«React Redux v9.1.0 добавляет новый метод .withTypes к каждому из этих хуков, аналогичный методу .withTypes у createAsyncThunk из Redux Toolkit.»
— перевод
// app/hooks.ts import { useDispatch, useSelector, useStore } from 'react-redux' import type { AppDispatch, AppStore, RootState } from './store' export const useAppDispatch = useDispatch.withTypes<AppDispatch>() export const useAppSelector = useSelector.withTypes<RootState>() export const useAppStore = useStore.withTypes<AppStore>()

.withTypes<T>() возвращает ту же самую функцию на уровне runtime — это чистая TypeScript-обёртка, которая «прибивает» генерик к хуку. Выигрыш — меньше бойлерплейта и один единообразный синтаксис.

Использование в компоненте

import { useAppDispatch, useAppSelector } from './hooks' function Counter() { const count = useAppSelector(state => state.counter.value) // state: RootState const dispatch = useAppDispatch() // dispatch: AppDispatch (умеет про thunks) return ( <div> <p>{count}</p> <button onClick={() => dispatch(incrementAsync())}> async +1 </button> </div> ) }

Ни в селекторе, ни в dispatch больше нигде не нужно писать state: RootState или AppDispatch. Типы придут сами. dispatch(incrementAsync()), где incrementAsync возвращает thunk — проходит проверку типов благодаря AppDispatch, унаследованному от store с thunk-middleware.

Сравнение двух подходов

Аспект TypedUseSelectorHook .withTypes()
Версия React-Redux любая с TS-типами v9.1.0+
Импорты из react-redux useSelector, useDispatch, useStore + тип TypedUseSelectorHook useSelector, useDispatch, useStore — и всё
Синтаксис Три разных формы: generic-тип, явная сигнатура, явная сигнатура Один способ: hook.withTypes<T>()
Runtime сами хуки сами хуки (.withTypes() — это no-op)
Параллель с другим API — Redux Toolkit: createAsyncThunk.withTypes()

Как это положить в проект

  1. Создать app/store.ts, туда — store и экспорт типов RootState, AppDispatch, AppStore.
  2. Создать app/hooks.ts, туда — типизированные useAppSelector, useAppDispatch, useAppStore.
  3. В компонентах импортировать хуки из app/hooks, а не из react-redux. ESLint-правило no-restricted-imports помогает запретить импорт обычных хуков напрямую.
  4. При добавлении middleware / slice типы обновятся автоматически: RootState и AppDispatch выводятся из store.

Ошибки, которые ловят типы

  • Опечатка в имени slice (state.couter.value) — ошибка компиляции.
  • Читаем несуществующее поле — ошибка компиляции.
  • Dispatch'им не-action-объект или не-thunk — ошибка компиляции.
  • Меняем форму state в reducer'е — все места использования засветятся как broken.

Связь с системой Subscription

Типизация чисто TS-уровня и никак не влияет на поведение подписки. Runtime тот же: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender. Подробности — урок 03a.

Рекомендация: на новой версии React-Redux (v9.1.0+) берите .withTypes() — это короче, единообразно и явно проще читается. Старый TypedUseSelectorHook остаётся рабочим; переписывать существующий файл hooks.ts «ради красоты» необязательно.
