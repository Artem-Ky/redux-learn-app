Интерактив — один и тот же счётчик двумя способами

Слева — знакомый с уроков 12–27 счётчик через connect. Справа — такой же счётчик через хуки: useSelector для чтения state и useDispatch для отправки экшенов. Оба подписаны на один store, внешний dispatch обновляет обе карточки. Обратите внимание на количество строк исходного кода в каждой карточке.

Теория — Hooks API и почему он рекомендуется

Что добавили в react-redux

Hooks API — набор встроенных React-хуков, которые подключают функциональный компонент к Redux-store напрямую, без connect. Их три основных: useSelector (чтение state), useDispatch (получить функцию dispatch), useStore (редкий случай — получить сам store для императивных операций).

Минимальный пример

import { useSelector, useDispatch } from 'react-redux' function Counter() { const count = useSelector((state: RootState) => state.counter.value) const dispatch = useDispatch() return ( <div> <button onClick={() => dispatch({ type: 'counter/decrement' })}>−</button> <span>{count}</span> <button onClick={() => dispatch({ type: 'counter/increment' })}>+</button> </div> ) } <Provider store={store}><Counter /></Provider>

Ни mapStateToProps, ни mapDispatchToProps, ни HOC'а, ни export default connect(...) — весь «клей» с Redux помещается в три строки внутри компонента.

connect vs hooks — сравнение

Критерий connect hooks
Строк кода на простой компонент ~20 ~10
Обёртка HOC в дереве да (Connect(MyComp)) нет
TypeScript inference ConnectedProps (урок 34) прямее: типизирован useSelector
Тестирование требует <Provider> или mock'ов HOC требует только <Provider>
Работает с class-компонентами да нет (хуки только в функциях)
Объявления для DevTools много обёрток Connect(...) чистый стек компонентов
Рекомендуется командой Redux поддерживается, legacy-friendly по умолчанию

Три основных хука

  • useSelector(selector, equalityFn?) — читает срез state, подписывается на store, ре-рендерит компонент, когда результат селектора изменился. Разбираем в уроках 36–43.
  • useDispatch() — возвращает store.dispatch. Ссылка стабильна между рендерами. Уроки 44–45.
  • useStore() — возвращает сам store. Нужен редко (доступ к getState() в обработчиках без подписки, чтение после диспатча в эффектах). Урок 46.

Что остаётся одинаковым

Вся «внутрянка» react-redux — одна и та же. И connect, и хуки используют один и тот же контекст (<Provider>), одну и ту же систему подписки (Subscription). При dispatch Subscription проходит по linked list callback'ов → Subscription.notify() → для каждого слушателя checkForUpdates → запустить selector (ваша функция из useSelector или mapStateToProps) → сравнить с prevValue через === → forceRender если отличается. Разница — только в том, как вы подписываетесь: через HOC или через хук. Подробнее о Subscription — урок 03a.

Когда всё ещё использовать connect

  • Класс-компоненты (хуки недоступны).
  • Legacy-кодовая база — менять сотни компонентов разом нет смысла.
  • Явное «разделение ответственности» между контейнером и презентационным компонентом, когда презентационный должен оставаться «dumb» и принимать всё через props.
Стратегия миграции: команда React Redux не требует переписывать всё. Новые функциональные компоненты пишите на хуках, старые — оставляйте на connect, если они стабильны. Смешивать оба подхода в одном проекте совершенно нормально — они работают рядом и подписываются на один и тот же store.
Версии: хуки появились в v7.1.0 (лето 2019). Если вы на более старой версии — сначала обновитесь. В v9 (2024) API стабильный, типизированный и именно тот, что документирован на сайте.
