Рецепт 1: useActions(actions, deps?)

Был частью alpha-версии, но был удалён. Причина — не столько технические ограничения, сколько идеология:

"This hook was in our original alpha release, but removed in v7.1.0-alpha.4, based on Dan Abramov's suggestion. That suggestion was based on 'binding action creators' not being as useful in a hooks-based use case, and causing too much conceptual overhead and syntactic complexity."
— react-redux.js.org/api/hooks
«Этот хук был в исходном alpha-релизе, но удалён в v7.1.0-alpha.4 по предложению Дэна Абрамова. Его соображение основывалось на том, что «биндинг action creator'ов» в hooks-подходе не так полезен и приводит к слишком большим концептуальным накладным расходам и синтаксической сложности.»
— перевод
"You should probably prefer to call the useDispatch hook in your components to retrieve a reference to dispatch, and manually call dispatch(someActionCreator()) in callbacks and effects as needed."
— react-redux.js.org/api/hooks
«Скорее всего, вам стоит предпочесть вызов хука useDispatch в компонентах для получения ссылки на dispatch и ручной вызов dispatch(someActionCreator()) в обработчиках и эффектах по мере необходимости.»
— перевод

Реализация useActions

import { bindActionCreators } from 'redux' import { useDispatch } from 'react-redux' import { useMemo } from 'react' export function useActions(actions, deps?) { const dispatch = useDispatch() return useMemo(() => { if (Array.isArray(actions)) { return actions.map(a => bindActionCreators(a, dispatch)) } return bindActionCreators(actions, dispatch) }, deps ? [dispatch, ...deps] : [dispatch]) }

bindActionCreators оборачивает каждый action creator в функцию, которая автоматически диспатчит результат. useMemo гарантирует, что объект забинженных actions — стабильная ссылка (важно, если вы передаёте его в useEffect или в дочерний компонент с React.memo).

Использование useActions

const increment = () => ({ type: 'INCREMENT' }) const decrement = () => ({ type: 'DECREMENT' }) function Counter() { const { increment, decrement } = useActions({ increment, decrement }) return ( <> <button onClick={increment}>+</button> <button onClick={decrement}>−</button> </> ) }

Почему Дэн Абрамов предложил удалить useActions

  • Неявный dispatch. Чтение кода: видим вызов функции increment() — но где происходит dispatch? Скрыто внутри. Явный dispatch(increment()) читается сразу.
  • Два способа делать одно и то же. bindActionCreators делает из action creator'а функцию, которую можно звать как обычную функцию. Это ценно в старом connect, где такие функции передаются в дочерние компоненты как props и могут не знать про dispatch. В hooks-стиле dispatch доступен везде через useDispatch — обёртка не экономит ничего.
  • Синтаксическая сложность. Два API (useDispatch + useActions) вместо одного — лишняя когнитивная нагрузка.

Рецепт 2: useShallowEqualSelector

Частая потребность — чтобы useSelector сравнивал результат через shallowEqual, а не через ===. Каждый раз писать второй аргумент скучно, поэтому делают обёртку:

import { useSelector, shallowEqual } from 'react-redux' export function useShallowEqualSelector<Selected>( selector: (state: RootState) => Selected ): Selected { return useSelector(selector, shallowEqual) }

Используется там же, где обычно useSelector(..., shallowEqual): selector возвращает объект с несколькими полями, а вы хотите ре-рендер только при изменении значений, а не просто новой ссылки:

function UserPanel() { const { name, age } = useShallowEqualSelector(state => ({ name: state.user.name, age: state.user.age, })) return <div>{name}, {age}</div> }

Чем отличается от простого useSelector(..., shallowEqual)?

Ничем — это именно обёртка. Единственная выгода — не повторять shallowEqual на каждом call-site. Если в проекте это частый паттерн, обёртка делает код короче и однообразнее. Если редкий — обычный useSelector(selector, shallowEqual) полностью ок.

Связь с системой Subscription

Оба рецепта не меняют механику подписки. При dispatch: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === (или shallowEqual) → forceRender. В useActions шаг forceRender вообще неактуален — этот хук только создаёт обёртки над dispatch и сам по себе компонент не подписывает (хотя useDispatch внутри и не подписывает, а useMemo — это уже React-мемо, не Redux-подписка). В useShallowEqualSelector подписка — та же, что у обычного useSelector, просто последний шаг сравнения использует shallowEqual вместо ===. Подробнее о Subscription — урок 03a.

Вывод: рецепты — это конструктор. Копируйте их в свой проект, если видите реальную выгоду в конкретном сценарии. Современный идиоматичный react-redux — явный useDispatch + прямой вызов action creator'а и стандартный useSelector (опционально с shallowEqual). Но знать, как это сделать иначе — полезно.
