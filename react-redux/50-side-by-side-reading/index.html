Интерактив — один компонент в двух реализациях

Слева connect(mapStateToProps), справа useSelector. Оба читают state.counter.value и state.user.name. Ниже — три кнопки для разных dispatch'ей и одна для ре-рендера родителя без изменения state. Следите за счётчиками рендеров.

Теория — mapStateToProps vs useSelector

Концептуальная эквивалентность

"The selector is approximately equivalent to the mapStateToProps argument to connect conceptually."
— react-redux.js.org/api/hooks
«Selector концептуально примерно эквивалентен аргументу mapStateToProps функции connect.»
— перевод

Слева — connect + mapStateToProps

interface StateProps { count: number userName: string } interface OwnProps { title: string } type Props = StateProps & OwnProps const mapStateToProps = (state: RootState): StateProps => ({ count: state.counter.value, userName: state.user.name, }) const UserCounterConnect = ({ title, count, userName }: Props) => ( <div> <h3>{title}</h3> <p>User: {userName}</p> <p>Count: {count}</p> </div> ) export default connect(mapStateToProps)(UserCounterConnect)

Справа — useSelector

interface OwnProps { title: string } const UserCounterHook = ({ title }: OwnProps) => { const count = useSelector((state: RootState) => state.counter.value) const userName = useSelector((state: RootState) => state.user.name) return ( <div> <h3>{title}</h3> <p>User: {userName}</p> <p>Count: {count}</p> </div> ) } export default UserCounterHook

Дерево в DevTools <Connect(UserCounter)> → <UserCounter> Плоско: <UserCounter>
Типы StateProps, OwnProps, DispatchProps, ConnectedProps Только RootState + обычные props компонента
Parent re-render без изменения state Компонент не ре-рендерится: connect делает shouldComponentUpdate-подобную проверку Компонент ре-рендерится — как обычный React-потомок. Нужен React.memo, чтобы избежать

selector в useSelector — любое значение

"The selector may return any value as a result, not just an object."
— react-redux.js.org/api/hooks
«Selector может вернуть любое значение в качестве результата, а не только объект.»
— перевод

mapStateToProps обязан возвращать объект (поля становятся props). useSelector не ограничен — можно вернуть примитив state.counter.value, массив state.todos.list, строку, булево, функцию, что угодно.

ownProps: через второй параметр vs через closure

"The selector function does not receive an ownProps argument. However, props can be used through closure (see the examples above) or by using a curried selector."
— react-redux.js.org/api/hooks
«Selector-функция не получает аргумент ownProps. Однако props могут использоваться через замыкание (см. примеры выше) или с помощью curried-selector'а.»
— перевод

В connect:

const mapStateToProps = (state, ownProps) => ({ todo: state.todos.byIds[ownProps.id] })

В useSelector через closure:

function Todo({ id }: { id: number }) { const todo = useSelector((state: RootState) => state.todos.byIds[id]) return ... }

Ключевое отличие поведения: parent re-render

"unlike connect(), useSelector() does not prevent the component from re-rendering due to its parent re-rendering, even if the component's props did not change."
— react-redux.js.org/api/hooks
«В отличие от connect(), useSelector() не предотвращает ре-рендер компонента из-за ре-рендера его родителя, даже если props компонента не изменились.»
— перевод

connect — это HOC, который внутри сравнивает новые props с предыдущими (в т.ч. результат mapStateToProps) через shallowEqual. Если ничего не изменилось, дочерний рендер пропускается. useSelector ничего такого не делает: хук работает внутри функционального компонента, а родитель всё равно вызовет функцию-компонент дочернего при своём ре-рендере.

Чтобы получить поведение «не ре-рендерить, если props и результат selector'а не менялись», нужно обернуть компонент в React.memo:

const UserCounterHook = React.memo(({ title }: OwnProps) => { const count = useSelector((state: RootState) => state.counter.value) return <div>{title}: {count}</div> })

Теперь React.memo не даст компоненту перерисоваться при ре-рендере родителя, если props не поменялись. А сам useSelector по-прежнему подписан на store и ре-рендерит компонент при изменении count — независимо от родителя.

Связь с системой Subscription

Обе реализации подписываются на store через одну и ту же систему: при dispatch проходит цепочка linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === (или shallowEqual) → forceRender. Разница только в том, что connect внутри своей HOC-обёртки ещё и проверяет props родителя на shallow equality и таким образом блокирует «лишний» спуск рендера от родителя. useSelector реагирует только на подписку — рендер от родителя проходит сквозь него беспрепятственно. Подробнее о Subscription — урок 03a.

Практический вывод: в hooks-подходе по умолчанию компоненты перерендериваются чаще, потому что нет встроенной «защиты» от parent re-render. Это редко критично в реальных приложениях (React быстро рендерит чистые компоненты), но если заметите реальную проблему — React.memo возвращает поведение, близкое к connect.
