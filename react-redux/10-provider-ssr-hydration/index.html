Теория — Provider и SSR: serverState

Проблема: Hydration Mismatch

При Server-Side Rendering (SSR) сервер рендерит React-компоненты в HTML-строку и отправляет клиенту. Клиент затем «гидрирует» этот HTML — подключает обработчики событий и делает его интерактивным, используя hydrateRoot() вместо createRoot().

Проблема возникает, когда серверный state и клиентский state расходятся. Между моментом серверного рендера и клиентской гидрации может произойти dispatch — например, другой компонент в store инициализируется позже. В результате HTML с сервера не совпадает с тем, что React пытается отрендерить на клиенте.

Hydration mismatch — одна из самых частых ошибок при SSR. React 18 выдаёт предупреждение: "Text content does not match server-rendered HTML". Это может привести к визуальным артефактам и потере интерактивности.

Решение: prop serverState

"<Provider> now accepts a serverState prop for use in SSR hydration scenarios. This is necessary if you are calling hydrateRoot in order to avoid hydration mismatches."

<Provider> теперь принимает prop serverState для сценариев SSR-гидрации. Это необходимо при использовании hydrateRoot, чтобы избежать hydration mismatch.

— react-redux.js.org, Provider

Prop serverState говорит React-Redux: «Вот state, который сервер использовал для рендера этого HTML. Используй его для первого рендера на клиенте, чтобы HTML совпал». После первого рендера React-Redux переключается на актуальный store.getState().

Как это работает внутри

  1. Сервер создаёт store с preloadedState, рендерит HTML через renderToString()
  2. Сервер отправляет HTML + сериализованный state клиенту
  3. Клиент создаёт свой store с тем же preloadedState
  4. Клиент вызывает hydrateRoot() с serverState={window.__PRELOADED_STATE__}
  5. При гидрации useSelector использует serverState (не store.getState())
  6. React видит, что HTML совпадает → гидрация проходит без ошибок
  7. После гидрации переключается на store.getState() для последующих обновлений

Полный пример SSR-гидрации

// === Серверный код (Node.js) === import { renderToString } from 'react-dom/server' import { Provider } from 'react-redux' const store = createStore(rootReducer, preloadedState) const html = renderToString( <Provider store={store}> <App /> </Provider> ) // Вставляем state в HTML для передачи клиенту const fullHtml = ` <div id="root">${html}</div> <script> window.__PRELOADED_STATE__ = ${JSON.stringify(store.getState())} </script> `
// === Клиентский код === import { hydrateRoot } from 'react-dom/client' import { Provider } from 'react-redux' const preloadedState = window.__PRELOADED_STATE__ delete window.__PRELOADED_STATE__ const store = createStore(rootReducer, preloadedState) hydrateRoot( document.getElementById('root'), <Provider store={store} serverState={preloadedState} > <App /> </Provider> )

preloadedState vs serverState

preloadedState serverState
Где Аргумент createStore(reducer, preloadedState) Prop <Provider serverState={...}>
Зачем Инициализация store с данными Фиксация state для первого рендера (hydration)
Когда При создании store При гидрации (hydrateRoot)
Что если без него Store начнёт с reducer default state Возможен hydration mismatch
Совет: Если вы не используете SSR — вам serverState не нужен. Он решает конкретную проблему расхождения HTML при гидрации. Для обычных SPA достаточно preloadedState в createStore().
