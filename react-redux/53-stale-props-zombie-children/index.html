Теория — Stale Props & Zombie Children

Откуда берётся проблема

"With hooks, there is no way to render a context provider, which means there's also no nested hierarchy of subscriptions. Because of this, the 'stale props' and 'zombie child' issues may potentially re-occur in an app that relies on using hooks instead of connect()."
— react-redux.js.org/api/hooks
«С хуками нет способа отрендерить провайдер контекста, а значит, нет и вложенной иерархии подписок. Из-за этого проблемы "stale props" и "zombie child" могут потенциально возникать в приложении, которое полагается на хуки вместо connect().»
— перевод

В connect каждая HOC-обёртка создаёт свой Subscription, вкладывающийся в subscription родителя. Уведомления идут сверху вниз: родительский Subscription дёргает детей только после того, как сам отработал. Это гарантирует, что к моменту, когда дочерний селектор видит новый state, родитель уже обновил свои props.

В хуках такой иерархии нет: все useSelector подписываются на один и тот же корневой Subscription (тот, что в <Provider>). Порядок уведомлений — в порядке монтирования, и селектор ребёнка может сработать до того, как селектор родителя «починил» свой output. Отсюда две беды: stale props и zombie child.

Stale props

"Specifically, 'stale props' means any case where: a selector function relies on this component's props to extract data, a parent component would re-render and pass down new props as a result of an action, but this component's selector function executes before this component has had a chance to re-render with those new props"
— react-redux.js.org/api/hooks
«Конкретно, "stale props" означает любой случай, когда: selector-функция опирается на props этого компонента для извлечения данных; родительский компонент должен перерисоваться и передать новые props в результате action; но selector этого компонента выполняется до того, как он успел перерисоваться с этими новыми props.»
— перевод

Пример: родитель передаёт ребёнку id, а ребёнок делает useSelector(s => s.byId[props.id]). Action меняет оба: и state.byId, и props.id (потому что родительский селектор теперь возвращает другой id). Но если селектор ребёнка запустился по старому props.id — он получит «устаревшие» данные.

Zombie child

"'Zombie child' refers specifically to the case where: Multiple nested connected components are mounted in a first pass, causing a child component to subscribe to the store before its parent. An action is dispatched that deletes data from the store, such as a todo item. The parent component would stop rendering that child as a result. However, because the child subscribed first, its subscription runs before the parent stops rendering it."
— react-redux.js.org/api/hooks
«"Zombie child" — это конкретный случай: несколько вложенных connected-компонентов монтируются в первом проходе, из-за чего ребёнок подписывается на store раньше родителя. Диспатчится action, удаляющий данные из store — например, элемент todo. Родитель из-за этого перестаёт рендерить этого ребёнка. Но поскольку ребёнок подписался первым, его подписка срабатывает раньше, чем родитель успевает перестать его рендерить.»
— перевод

Что происходит по шагам:

  1. Todo id=5 есть в state.todos.byId.
  2. Родитель TodoList рендерит <TodoItem id={5} />.
  3. Внутри TodoItem: useSelector(s => s.todos.byId[5]).
  4. Dispatch удаляет id=5 из store.
  5. Subscription уведомляет всех подписчиков. Ребёнок получает уведомление раньше родителя.
  6. Селектор ребёнка возвращает undefined.
  7. Если ребёнок пишет todo.text без защиты — TypeError.
  8. Чуть позже родительский селектор сработает, увидит, что id=5 больше нет, и перестанет рендерить ребёнка — но ошибка уже произошла.

Встроенная защита в useSelector

"useSelector() tries to deal with this by catching all errors that are thrown when the selector is executed due to a store update"
— react-redux.js.org/api/hooks
«useSelector() пытается бороться с этим, перехватывая все ошибки, брошенные при выполнении селектора из-за обновления store.»
— перевод

Если селектор выбросил исключение в ответ на update store, useSelector проглатывает ошибку и отдаёт предыдущий результат; расчёт, что родитель вот-вот перерендерится и уберёт «мёртвого ребёнка». Но это работает только если ошибка случилась внутри самого селектора. Если вы падаете уже в теле компонента (todo.text когда todo === undefined) — React поднимает ошибку наверх, до ближайшего ErrorBoundary.

Как защититься

Два простых правила:

// 1. Селектор возвращает undefined? Защитный return null. function TodoItem({ id }: Props) { const todo = useSelector((state: RootState) => state.todos.byId[id]) if (!todo) return null // защита от zombie child return <div>{todo.text}</div> }
// 2. Или default-значение прямо в селекторе. const todo = useSelector((state: RootState) => state.todos.byId[id] ?? { id, text: '', done: false } )

Второй вариант опасен тем, что возвращает новую ссылку на каждый рендер — это ломает === и вызывает лишние ре-рендеры. Держите fallback-объект стабильным:

const EMPTY_TODO: Todo = { id: -1, text: '', done: false } const todo = useSelector((state: RootState) => state.todos.byId[id] ?? EMPTY_TODO )

На практике это редкость

"In practice, these are a rare concern - we've received far more comments about these being in the docs than actual reports of these being a real problem in an app."
— react-redux.js.org/api/hooks
«На практике это редкая проблема — мы получили намного больше комментариев о том, что она присутствует в документации, чем реальных сообщений о том, что она кому-то навредила в приложении.»
— перевод

Большинство приложений никогда не ловят zombie child — именно потому что защитная проверка «нет данных — ничего не рисую» и так считается хорошим тоном. Тем не менее знать, почему на пустой момент возвращается undefined, полезно для отладки странных ошибок вида «cannot read property text of undefined» сразу после delete.

Как это устроено по внутренностям

В обоих моделях (connect и hooks) работает одна и та же цепочка: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender. Отличие — в том, сколько у вас Subscription-уровней и в каком порядке они нотифицируют своих listener'ов: у connect — дерево, у хуков — плоский список на уровне <Provider>. Все подробности — в уроке 03a.

Практический чеклист:
  • Любой селектор, читающий «запись по id» (byId[id]), должен уметь пережить undefined.
  • В компоненте — if (!data) return null сразу после хука.
  • Не падайте в data.field без проверки.
  • Если нужен fallback-объект — сделайте его модульной константой, чтобы не менять ссылку между рендерами.
