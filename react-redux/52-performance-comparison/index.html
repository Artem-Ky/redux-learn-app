Интерактив — 100 элементов × 3 реализации

В каждой колонке 100 компонентов TodoItem. Клетка подсвечивается красным каждый раз, когда её React-компонент ре-рендерится. Сравнивайте, сколько клеток вспыхивает при каждом действии, и как растут счётчики «total renders».

Теория — performance: connect vs hooks

Как работает оптимизация в connect

"React Redux internally implements the shouldComponentUpdate method such that the wrapper component re-renders precisely when the data your component needs has changed."
— react-redux.js.org/using-react-redux/connect-mapstate
«React Redux внутри реализует метод shouldComponentUpdate так, что компонент-обёртка ре-рендерится ровно тогда, когда изменились данные, нужные вашему компоненту.»
— перевод

Когда вы пишете connect(mapStateToProps)(Component), React-Redux создаёт класс-обёртку вокруг вашего компонента. Эта обёртка на каждом уведомлении от store вычисляет новый результат mapStateToProps и сравнивает его с предыдущим через shallowEqual. Если shallow-equal — shouldComponentUpdate возвращает false, и дочерний рендер не запускается.

Та же логика срабатывает и при ре-рендере родителя: обёртка сравнивает ownProps через shallow, и если они не изменились — снова пропускает рендер. Именно поэтому в connect-списке из 100 элементов при изменении одного элемента ре-рендерится ровно один компонент.

Почему у хуков поведение другое

"unlike connect(), useSelector() does not prevent the component from re-rendering due to its parent re-rendering, even if the component's props did not change."
— react-redux.js.org/api/hooks
«В отличие от connect(), useSelector() не предотвращает ре-рендер компонента из-за ре-рендера его родителя, даже если props компонента не изменились.»
— перевод

useSelector живёт внутри функционального компонента и срабатывает только на подписку к store. Он никак не участвует в решении «рендерить ли детей при ре-рендере родителя». Если родитель ре-рендерится — дети ре-рендерятся просто потому, что это их родитель их вызвал. useSelector при этом снова считает свою функцию-селектор — но рендер уже произошёл.

Решение: React.memo

"If further performance optimizations are necessary, you may consider wrapping your function component in React.memo()"
— react-redux.js.org/api/hooks
«Если необходимы дополнительные оптимизации производительности, вы можете обернуть свой функциональный компонент в React.memo().»
— перевод
const TodoItem = memo(function TodoItem({ id }: Props) { const todo = useSelector((state: RootState) => state.todos.byId[id]) return <div>{todo.text}</div> })

React.memo — аналог shouldComponentUpdate для функциональных компонентов. По умолчанию он сравнивает props через Object.is для каждого ключа. Если props не изменились — компонент не ре-рендерится при ре-рендере родителя. Внутренние useSelector-подписки продолжают работать независимо: когда меняется их slice — компонент ре-рендерится, несмотря на memo.

Где проявляется цена хуков

В трёх ситуациях hooks без memo дают больше лишних рендеров, чем connect:

  • Родительский компонент часто ре-рендерится (например, печать в инпуте родителя, таймер, анимация). Каждый его рендер проходит через всех детей.
  • Длинные списки — 100, 1000 элементов, у каждого useSelector. При каждом рендере родителя вызывается useSelector × N, а ещё и сами функциональные компоненты отрабатывают.
  • Тяжёлое тело компонента — сложная логика, вычисления, JSX с множеством условий. Даже если React потом сравнит vDOM и ничего не изменит в DOM, сам прогон функции стоит CPU.

Практическая стратегия

  • Не оптимизируйте заранее. Для большинства компонентов React достаточно быстр, и memo добавит больше когнитивной сложности, чем выигрыша.
  • Меряйте React DevTools Profiler. Если «flamegraph» показывает, что длинный список перерисовывается целиком по пустяку — тогда memo.
  • Стабильные props для memo. memo работает только если props сохраняют ссылочное равенство. Поэтому inline-объекты и свежесозданные коллбэки (onClick={() => ...}) ломают memo. Используйте useCallback и useMemo.
  • Для очень больших списков — виртуализация (react-window, react-virtuoso). memo решает «не перерисовывать невидимое», виртуализация вообще не рендерит невидимое.

Связь с системой Subscription

Во всех трёх вариантах подписки идут через один и тот же механизм: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender. Разница — в том, блокирует ли HOC рендер от родителя: у connect да, у useSelector нет. React.memo воссоздаёт поведение connect на уровне React, а не на уровне Redux. Подробнее про подписку — урок 03a.

Главный вывод: connect — это «hooks + memo», приклеенный одной строкой connect()(Component). Hooks дают гибкость, но требуют осознанно ставить memo, когда это действительно нужно. В большинстве экранов разница не заметна.
