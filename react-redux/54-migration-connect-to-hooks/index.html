Интерактив — stepper: одна и та же фича, 6 шагов

Один и тот же компонент Counter в шести состояниях миграции — от класс-компонента с connect до функционального компонента в React.memo. На каждом шаге слева код, справа живой рабочий превью. Все шаги работают с общим store — нажимая кнопки на любом шаге, вы двигаете тот же count.

Теория — миграция connect → hooks

Почему мигрируем

"We recommend using the React-Redux hooks API as the default approach in your React components."
— react-redux.js.org/api/hooks
«Мы рекомендуем использовать hooks-API React-Redux как подход по умолчанию в ваших React-компонентах.»
— перевод

Hooks-подход идёт в документации как подход по умолчанию и даёт: (1) плоское дерево в React DevTools (нет обёрток <Connect(Counter)>); (2) простую TypeScript-инференцию — не нужны StateProps / DispatchProps / ConnectedProps; (3) гибкость — несколько useSelector-ов в одном компоненте, возврат любого значения; (4) натуральную интеграцию с другими хуками (useCallback, useMemo, useEffect).

Чеклист миграции одного компонента

  • Класс-компонент → функциональный компонент (если ещё не). Состояние класса (this.state) — через useState; lifecycle — через useEffect.
  • Убрать mapStateToProps. Для каждого поля — свой useSelector внутри компонента.
  • Убрать mapDispatchToProps. Взять useDispatch(), коллбэки собирать на месте, заворачивать в useCallback при передаче вниз.
  • Убрать connect(...)(Counter) и сам export default connect(...). Экспортировать сам компонент.
  • Удалить типы StateProps, DispatchProps, ConnectedProps. Остаются только собственные Props компонента.
  • Опционально — обернуть в React.memo, если parent часто перерисовывается и это заметно в профилировщике (урок 52).
  • Проверить stale props / zombie children для компонентов, читающих записи по id: добавить if (!data) return null (урок 53).
  • Если в приложении используются thunk / saga и типизация — создать типизированные useAppSelector / useAppDispatch (урок 55).

Что получается в финале

import { memo, useCallback } from 'react' import { useSelector, useDispatch } from 'react-redux' interface Props { title: string } export const Counter = memo(({ title }: Props) => { const count = useSelector((state: RootState) => state.counter.value) const user = useSelector((state: RootState) => state.user) const dispatch = useDispatch() const incrementCounter = useCallback( () => dispatch({ type: 'counter/increment' }), [dispatch] ) return ( <div> <h3>{title}</h3> <p>user: {user.name}</p> <p>count: {count}</p> <button onClick={incrementCounter}>+1</button> </div> ) })

Можно смешивать

Полная миграция приложения сразу — редкость и рисково. Нормальная стратегия: новые компоненты пишем на хуках, старые трогаем при необходимости (фича, баг, рефакторинг). Connect и хуки отлично уживаются в одном дереве — они работают поверх одного и того же <Provider>, одного store, одной Subscription-системы.

Что не нужно мигрировать

  • Класс-компоненты, которые по другим причинам остаются классами (lifecycle-критичный код, внешние библиотеки, HOC-обёртки). Хуки работают только в функциональных компонентах — нет смысла переписывать целый класс просто ради useSelector.
  • Уже работающие компоненты, которые никто не планирует трогать — принцип «не чини, что не сломано». Аккуратная миграция — это время и риск регресса.
  • Компоненты с очень хитрым mergeProps / options.areStatesEqual. Там ещё нужно оценить, получится ли повторить логику на хуках (обычно — да, через useSelector с equalityFn и useMemo, но осторожно).

Связь с системой Subscription

Миграция меняет только способ подключения на уровне компонента. Под капотом работает одна и та же цепочка: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender. Отличие — connect делает свою shallowEqual-проверку агрегата, а useSelector сверяет один результат через ===. Подписка и уведомления — общие. Подробнее — урок 03a.

Практика: начните с одного простого компонента и проделайте все шесть шагов. Когда прошагаете пару штук — уже поймаете ритм: «connect снять, useSelector поставить, useDispatch + useCallback, типы убрать». Остальные компоненты пойдут быстрее.
