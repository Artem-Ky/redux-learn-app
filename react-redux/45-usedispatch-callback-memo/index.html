Интерактив — inline callback vs useCallback

Два дочерних компонента обёрнуты в React.memo. В первый прокидывается inline () => dispatch(...) — новая ссылка на каждом рендере родителя. Во второй прокидывается useCallback(..., [dispatch]) — ссылка стабильна. Кнопка «render parent» меняет только локальное состояние родителя, не трогая store.

Теория — memoизация callback'ов поверх dispatch

Цитата из документации

"When passing a callback using dispatch to a child component, you may sometimes want to memoize it with useCallback. If the child component is trying to optimize render behavior using React.memo() or similar, this avoids unnecessary rendering of child components due to the changed callback reference."
— react-redux.js.org/api/hooks
«Когда вы передаёте callback, использующий dispatch, в дочерний компонент, иногда полезно мемоизировать его через useCallback. Если дочерний компонент оптимизирует ре-рендер через React.memo() или аналоги, это предотвращает ненужные ре-рендеры из-за новой ссылки на callback.»
— перевод

Проблема: inline-функция — новая ссылка на каждом рендере

const Parent = () => { const dispatch = useDispatch() const [unrelated, setUnrelated] = useState(0) // ❌ новая функция на каждом рендере Parent const onClick = () => dispatch({ type: 'counter/increment' }) return ( <> <button onClick={() => setUnrelated(unrelated + 1)}> Render parent </button> <IncrementButton onClick={onClick} /> </> ) } const IncrementButton = memo(({ onClick }: { onClick: () => void }) => { // ре-рендерится при каждом рендере Parent: // memo делает shallow-сравнение props, onClick — новая ссылка. return <button onClick={onClick}>+</button> })

Решение: useCallback

const onClick = useCallback(() => { dispatch({ type: 'counter/increment' }) }, [dispatch]) // dispatch стабилен → deps не меняются → useCallback возвращает ОДНУ функцию // → shallow-compare в memo → ребёнок НЕ ре-рендерится

Как это работает

React.memo делает shallow-сравнение props, возвращает тот же React element, если ничего не изменилось, и React просто не пересчитывает дерево ниже. Критично: сравнение идёт по ссылкам. Две identity-equal функции считаются разными.

useCallback(fn, deps) возвращает ту же функцию между рендерами, пока deps (по ссылке) совпадают. Если хотя бы один dep изменился — возвращается новая функция (и memo-ребёнок ре-рендерится).

dispatch из useDispatch — стабильная ссылка (см. урок 44), поэтому [dispatch] никогда не меняется, и callback живёт до размонтирования компонента.

Когда memoизация нужна

  • Callback передаётся в React.memo-компонент или PureComponent.
  • Callback используется как dep в других хуках (useEffect, useMemo).
  • Callback передаётся в тяжёлый ребёнок (большое дерево, heavy render).

Когда memoизация не нужна

  • Callback используется внутри того же компонента — в собственном JSX. Здесь ре-рендер всё равно происходит, и память на inline-функцию не стоит запуска useCallback.
  • Ребёнок — обычный компонент без React.memo. Он ре-рендерится вместе с родителем в любом случае.

Аккуратно с deps

// ❌ забыли добавить id в deps → callback замораживает старый id const onClick = useCallback(() => { dispatch({ type: 'todos/remove', payload: id }) }, [dispatch]) // id не в deps → stale closure // ✔ правильно const onClick = useCallback(() => { dispatch({ type: 'todos/remove', payload: id }) }, [dispatch, id])

Если callback зависит от props / state, все зависимости должны быть в deps. Иначе будет stale closure: callback будет использовать устаревшее значение переменной.

Альтернатива: bindActionCreators + useMemo

import { bindActionCreators } from 'redux' const actions = useMemo( () => bindActionCreators({ increment: () => ({ type: 'counter/increment' }), decrement: () => ({ type: 'counter/decrement' }), }, dispatch), [dispatch] ) // actions.increment — стабильная ссылка, тот же эффект, что useCallback

Связь с системой Subscription

useCallback не влияет ни на Redux store, ни на подписку. Он живёт только в React-слое: экономит React'у работу по сравнению props. Но важно: если из-за новой ссылки ребёнок ре-рендерится и внутри него тоже есть useSelector — тот запускает selector ещё раз и сравнивает результат. Лишняя работа на каждом пустом рендере. Путь подписки после dispatch остаётся тем же: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender. useCallback просто экономит render-работу родительских и дочерних компонентов между этими dispatch'ами. Подробнее про Subscription — урок 03a.

Паттерн:
const dispatch = useDispatch() const onIncrement = useCallback(() => dispatch({ type: 'counter/increment' }), [dispatch]) const onRemove = useCallback((id: number) => dispatch({ type: 'todos/remove', payload: id }), [dispatch])
Не оборачивайте всё подряд: useCallback не бесплатен — он держит функцию и массив deps в памяти и делает сравнение. Если получатель — не React.memo, оверхед только добавляется. Используйте точечно там, где есть реальный выигрыш.
