Интерактив — один useSelector-объект vs пять отдельных

Оба компонента показывают одни и те же 5 полей из разных срезов state. Слева — один useSelector с объектом и shallowEqual. Справа — пять отдельных useSelector-вызовов. Счётчики рендеров: оба ре-рендерятся одинаково, благодаря автоматическому батчингу React 18+.

Теория — несколько useSelector в одном компоненте

Каждый вызов useSelector — отдельная подписка

"You may call useSelector() multiple times within a single function component. Each call to useSelector() creates an individual subscription to the Redux store."
— react-redux.js.org/api/hooks
«Можно вызывать useSelector() много раз внутри одного функционального компонента. Каждый вызов useSelector() создаёт отдельную подписку на Redux-хранилище.»
— перевод

«Отдельная подписка» здесь означает, что хук регистрирует свой checkForUpdates-callback в Subscription (урок 03a). Если в компоненте 5 вызовов useSelector — в linked list Subscription добавятся 5 узлов-слушателей, и каждый из них после dispatch запустит свой selector и сравнит свой результат через ===.

Один dispatch → один ре-рендер (батчинг)

"Because of the React update batching behavior used in React Redux v7, a dispatched action that causes multiple useSelector()s in the same component to return new values should only result in a single re-render."
— react-redux.js.org/api/hooks
«Благодаря поведению батчинга обновлений React, которое используется в React Redux v7, один dispatched action, заставляющий несколько useSelector() в одном компоненте вернуть новые значения, должен привести только к одному ре-рендеру.»
— перевод

То есть даже если все пять ваших useSelector увидят новое значение после одного dispatch — React объединит их в один рендер. А в React 18+ это работает уже и для произвольных синхронных последовательностей dispatch внутри обработчика события — все они попадают в одну «пачку».

Вариант 1: один useSelector + shallowEqual

const data = useSelector((state: RootState) => ({ userName: state.user.name, userAge: state.user.age, todoCount: state.todos.list.length, theme: state.ui.theme, loading: state.ui.loading, }), shallowEqual)

Вариант 2: пять отдельных useSelector

const userName = useSelector((state: RootState) => state.user.name) const userAge = useSelector((state: RootState) => state.user.age) const todoCount = useSelector((state: RootState) => state.todos.list.length) const theme = useSelector((state: RootState) => state.ui.theme) const loading = useSelector((state: RootState) => state.loading)

Что рекомендуется

  • Один useSelector на одно значение — самый читаемый и безопасный паттерн. Каждое поле — отдельная подписка, дефолтного === достаточно.
  • Объект с shallowEqual — компактнее в коде, но легко ошибиться и забыть shallowEqual, а тогда компонент будет ре-рендериться на каждом dispatch.
  • Для derived данных (фильтрация, сортировка) — createSelector из reselect (урок 40), а не inline-объект.

Связь с системой Subscription

Смоделируем что происходит при dispatch в компоненте с 5 хуками: linked list callback'ов → Subscription.notify() → для каждого слушателя checkForUpdates → запустить selector → сравнить через === → forceRender если отличается. Если несколько хуков одного компонента вызовут forceRender — React видит это как один компонент и выполнит единый рендер (батчинг). Внутри этого рендера каждый useSelector просто вернёт актуальное значение. Подробнее про Subscription — урок 03a.

Совет: не бойтесь большого числа useSelector. Эти хуки очень дешёвые — подписка и сравнение крошечные, а читаемость кода выше, чем у одного «комбайна». Оптимизируйте только если профайлер показал реальную проблему.
Что не делать: не превращайте один useSelector в «глобальный аггрегатор»: useSelector(state => state) или useSelector(state => ({ …state })). Такой хук в dev-режиме триггерит identityFunctionCheck (урок 42) и заставляет компонент ре-рендериться от любого изменения store.
