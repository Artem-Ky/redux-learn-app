Теория — Subscription: внутренний механизм React-Redux

Зачем вообще нужна Subscription?

Чтобы понять Subscription, сначала нужно понять проблему, которую она решает.

React Context работает так: когда значение Context.Provider меняется, ВСЕ компоненты, которые вызывают useContext(), перерисовываются. Без исключений. Это дизайн React.

Если бы React-Redux просто клал state в Context:

// ❌ ПЛОХОЙ подход (React-Redux так НЕ делает): const StoreContext = React.createContext(null) function NaiveProvider({ store, children }) { const [state, setState] = useState(store.getState()) useEffect(() => { return store.subscribe(() => { setState(store.getState()) // state меняется → Context value меняется }) // → ВСЕ useContext() ре-рендерятся! }, [store]) return ( <StoreContext.Provider value={state}> // каждый dispatch = новый value {children} </StoreContext.Provider> ) }

При каждом dispatch все 100 компонентов, использующих useContext(StoreContext), перерисовались бы — даже если 99 из них не нуждаются в обновлении. Это катастрофа для производительности.

Что делает React-Redux вместо этого

React-Redux кладёт в Context НЕ state, а ссылку на store (которая НИКОГДА не меняется) и объект Subscription:

// ✔ Как React-Redux на самом деле работает (упрощённо): // Context value — СТАБИЛЬНЫЙ объект, НЕ меняется при dispatch const contextValue = useMemo(() => ({ store, // ссылка на store (стабильна) subscription: createSubscription(store) // объект подписок }), [store]) <ReactReduxContext.Provider value={contextValue}> {children} </ReactReduxContext.Provider> // contextValue НИКОГДА не меняется → useContext() НИКОГДА // не вызывает ре-рендер! Уведомления идут через Subscription.

Как Subscription уведомляет компоненты

Subscription — это связный список (linked list) слушателей. Каждый компонент, использующий useSelector или connect, добавляет себя в этот список. Когда dispatch меняет state, Subscription проходит по списку и для каждого слушателя:

  1. Запускает его selector с новым state
  2. Сравнивает результат с предыдущим значением (=== для useSelector, shallow equal для connect)
  3. Если значение изменилось — вызывает forceRender() только для этого компонента
  4. Если НЕ изменилось — ничего не делает, компонент спит

Исходный код Subscription (из react-redux)

Вот реальный код из src/utils/Subscription.ts (упрощённый для наглядности):

// Коллекция слушателей — двусвязный список (linked list) // Реальный код: github.com/reduxjs/react-redux/blob/master/src/utils/Subscription.ts function createListenerCollection() { let first: Listener | null = null let last: Listener | null = null return { // Оповестить всех слушателей notify() { batch(() => { // батчим обновления React let listener = first while (listener) { listener.callback() // вызываем callback каждого слушателя listener = listener.next // идём к следующему по linked list } }) }, // Подписать нового слушателя subscribe(callback) { const listener = { callback, next: null, prev: last } if (last) last.next = listener else first = listener last = listener return function unsubscribe() { // удаляем из linked list при отписке } } } }
// Сама Subscription — связывает store и слушателей function createSubscription(store, parentSub?) { let listeners = createListenerCollection() function trySubscribe() { if (parentSub) { // Если есть родительская Subscription (connect-иерархия) // → подписываемся на родителя, а НЕ напрямую на store parentSub.addNestedSub(handleChange) } else { // Если нет родителя (корневая, от Provider) // → подписываемся напрямую на store.subscribe() store.subscribe(handleChange) } } function handleChange() { // При изменении state — вызвать onStateChange // (это callback который проверяет selector и решает рендерить или нет) subscription.onStateChange?.() } function addNestedSub(listener) { trySubscribe() return listeners.subscribe(listener) // добавляем в linked list } function notifyNestedSubs() { listeners.notify() // оповещаем всех вложенных слушателей } }

Два типа подписки: nested (connect) vs flat (useSelector)

connect() — вложенная (nested) подписка:

// connect() создаёт СВОЮ Subscription, которая подписана на РОДИТЕЛЬСКУЮ Provider (Root Subscription) └─ ConnectedComponentA (Subscription A, подписана на Root) └─ ConnectedComponentB (Subscription B, подписана на A) └─ ConnectedComponentC (Subscription C, подписана на B) // При dispatch: Root → A → B → C // Каждый уровень ЖДЁТ пока родитель обновится, потом обновляет себя. // Это гарантирует: родитель рендерится РАНЬШЕ ребёнка. // Если родитель решил не рендерить ребёнка (unmount) → ребёнок не получит // уведомление → нет "zombie children".

useSelector — плоская (flat) подписка:

// useSelector регистрируется напрямую в Root Subscription Provider (Root Subscription) ├─ ComponentA (useSelector — listener в Root) ├─ ComponentB (useSelector — listener в Root) └─ ComponentC (useSelector — listener в Root) // При dispatch: Root уведомляет A, B, C одновременно (в порядке подписки) // Проще, но нет гарантии порядка parent → child. // Потенциальная проблема: "zombie children" (редко на практике).
«Starting with version 5, React Redux has attempted to guarantee that consistency with ownProps. In version 7, that is implemented using a custom Subscription class internally in connect(), which forms a nested hierarchy. This ensures that connected components lower in the tree will only receive store update notifications once the nearest connected ancestor has been updated.»
— react-redux.js.org — Stale Props and Zombie Children
«Начиная с версии 5, React Redux пытается гарантировать согласованность с ownProps. В версии 7 это реализовано с помощью кастомного класса Subscription внутри connect(), который формирует вложенную иерархию. Это гарантирует, что подключённые компоненты ниже по дереву получат уведомления только после обновления ближайшего подключённого предка.»
— перевод
«With hooks, there is no way to render a context provider, which means there's also no nested hierarchy of subscriptions. Because of this, the "stale props" and "zombie child" issues may potentially re-occur in an app that relies on using hooks instead of connect().»
— react-redux.js.org — Stale Props and Zombie Children
«С хуками нет возможности рендерить context provider, а значит нет и вложенной иерархии подписок. Из-за этого проблемы "stale props" и "zombie children" потенциально могут возникнуть при использовании хуков вместо connect().»
— перевод

Что именно делает useSelector при получении уведомления

Когда Subscription вызывает callback компонента с useSelector, происходит следующее (реальная логика из src/hooks/useSelector.ts ):

// Упрощённая логика useSelector при уведомлении от Subscription: function checkForUpdates() { const newStoreState = store.getState() const newSelectedValue = selector(newStoreState) // Сравниваем с предыдущим значением if (equalityFn(newSelectedValue, lastSelectedValue)) { // Значение НЕ изменилось → ничего не делаем return } // Значение изменилось → запоминаем новое и принудительно ре-рендерим lastSelectedValue = newSelectedValue forceRender() // useReducer(c => c + 1, 0) — трюк для принудительного рендера }

Полная цепочка: dispatch → ре-рендер

1. store.dispatch({ type: 'counter/increment' }) ↓ 2. Redux store вызывает reducer → новый state = { counter: 1, theme: 'dark' } ↓ 3. store вызывает callback'и из store.subscribe() ↓ 4. Root Subscription (от Provider) получает уведомление → вызывает subscription.onStateChange() → это вызывает notifyNestedSubs() → listeners.notify() ↓ 5. Для каждого слушателя (компонент с useSelector): → вызывается checkForUpdates() → запускает selector(newState) → сравнивает с prevValue ↓ 6a. Counter: selector(state) → state.counter → 1 prev = 0, new = 1 → 0 !== 1 → true → forceRender() → РЕ-РЕНДЕР! 6b. ThemeDisplay: selector(state) → state.theme → 'dark' prev = 'dark', new = 'dark' → 'dark' === 'dark' → false → ПРОПУСТИТЬ

Итого: Subscription — это НЕ магия. Это linked list callback'ов. При dispatch: store → Subscription.notify() → для каждого listener: запустить selector, сравнить результат, если отличается → forceRender(). Вот и всё. React Context используется только для передачи ссылки на store (она стабильна и никогда не меняется), а не для уведомления о изменениях.
