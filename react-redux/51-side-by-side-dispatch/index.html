Интерактив — три реализации одного счётчика

Слева — mapDispatchToProps как функция, посередине — mapDispatchToProps как объект (shorthand), справа — хук useDispatch. Визуально и функционально все три ведут себя одинаково. Смотрите на панель «Стабильность ссылок коллбэков»: пересоздаются ли функции increment / decrement от рендера к рендеру.

Теория — mapDispatchToProps vs useDispatch

Форма 1. mapDispatchToProps как функция

Первый и самый «ручной» вариант — передать функцию, которая принимает dispatch и возвращает объект с коллбэками. Каждый коллбэк явно вызывает dispatch с нужным action-объектом.

const mapDispatchToProps = (dispatch: Dispatch) => ({ increment: () => dispatch({ type: 'counter/increment' }), decrement: () => dispatch({ type: 'counter/decrement' }), reset: () => dispatch({ type: 'counter/reset' }), }) connect(mapStateToProps, mapDispatchToProps)(Counter)

connect вызывает эту функцию один раз при монтировании (или при каждой смене owner props, если mapDispatchToProps двухарный) и кэширует результат. В props компонента приходят уже обёрнутые функции, идентичные между рендерами.

Форма 2. mapDispatchToProps как объект (shorthand)

Более короткий и идиоматичный вариант — передать объект из action creators. React-Redux сам обернёт каждую функцию через bindActionCreators:

const increment = () => ({ type: 'counter/increment' }) const decrement = () => ({ type: 'counter/decrement' }) const reset = () => ({ type: 'counter/reset' }) connect(mapStateToProps, { increment, decrement, reset })(Counter)
"We recommend always using the 'object shorthand' form of mapDispatchToProps, unless you have a specific reason to customize the dispatching behavior."
— react-redux.js.org/using-react-redux/connect-mapdispatch
«Мы рекомендуем всегда использовать форму "object shorthand" для mapDispatchToProps, если только у вас нет особой причины настраивать поведение dispatch.»
— перевод

Что это даёт: (1) меньше бойлерплейта — не надо писать (dispatch) => ({ ... }) вручную; (2) все обёртки создаются один раз и не пересоздаются между рендерами; (3) action creators остаются чистыми и переиспользуемыми вне компонента (в тестах, thunk-ах, саге).

Форма 3. useDispatch

В hooks-подходе нет «маппера» как такового. Хук useDispatch возвращает ссылку на сам dispatch, а вы уже сами решаете, как его вызывать. Обычно это выглядит так:

function Counter() { const dispatch = useDispatch() const increment = useCallback( () => dispatch({ type: 'counter/increment' }), [dispatch] ) // ... }
"You should probably prefer to call the useDispatch hook in your components to retrieve a reference to dispatch, and manually call dispatch(someActionCreator()) in callbacks and effects as needed."
— react-redux.js.org/api/hooks
«Скорее всего, вам лучше вызывать хук useDispatch в компонентах, чтобы получить ссылку на dispatch, и вручную вызывать dispatch(someActionCreator()) в коллбэках и эффектах по мере необходимости.»
— перевод

Стабильность ссылки на dispatch

dispatch, возвращаемый useDispatch, — это та же самая функция между рендерами, пока не меняется <Provider> (а он практически никогда не меняется). Поэтому безопасно ставить его в зависимости useCallback, useEffect, useMemo:

const increment = useCallback( () => dispatch({ type: 'counter/increment' }), [dispatch] // стабильно — коллбэк создастся один раз )

Сам dispatch можно считать «живущим» на уровне store — именно поэтому он стабилен. В уроке 44 (useDispatch basics) это разобрано подробно.

Когда какой вариант брать

  • Новый код — useDispatch. Это официальный рекомендованный подход.
  • Остаёмся на connect — object shorthand. Минимум бойлерплейта, и это прямая рекомендация документации.
  • Нужен доступ к ownProps внутри коллбэка и при этом connect — функциональная форма mapDispatchToProps(dispatch, ownProps). Редкий случай.
  • Нестандартное поведение при dispatch (батчинг, логирование, условный dispatch, оборачивание thunk-ов) — функциональная форма, там у вас полный контроль над тем, что попадёт в props.

Связь с системой Subscription

В отличие от useSelector / mapStateToProps, dispatch сам по себе не подписывается на store — он только отправляет action. Подписки живут на стороне читателей: connect и useSelector. После dispatch запускается уже знакомая цепочка: linked list callback'ов → Subscription.notify() → checkForUpdates → selector → === → forceRender. Подробности — в уроке 03a.

Практический вывод: все три формы отправки actions функционально идентичны — они все в конце вызывают store.dispatch(action). Разница — в эргономике кода, количестве бойлерплейта, стабильности ссылок и в том, как легко протестировать action creator отдельно от UI.
