Теория — Что такое Middleware?

Определение

Middleware — это точка расширения между отправкой action и моментом, когда он достигает reducer'а. Middleware может перехватывать, логировать, модифицировать, задерживать или даже блокировать action'ы.

«Redux middleware provides a third-party extension point between dispatching an action, and the moment it reaches the reducer. People use Redux middleware for logging, crash reporting, talking to an asynchronous API, routing, and more.»
— Redux Fundamentals, Part 4

Аналогия с Express middleware

Если вы знакомы с Express.js, концепция похожа: запрос проходит через цепочку обработчиков, каждый может что-то сделать и передать дальше через next(). В Redux action проходит через цепочку middleware, каждый вызывает next(action) для передачи action следующему.

// Express middleware: app.use((req, res, next) => { console.log(req.method, req.url) next() // передать дальше }) // Redux middleware (похожая идея): const logger = storeAPI => next => action => { console.log('action:', action) return next(action) // передать дальше }

Что может middleware?

  • Логирование — записывать каждый action и state до/после
  • Crash reporting — отлавливать ошибки в dispatch
  • Async операции — общаться с API, работать с промисами
  • Модификация — изменять или заменять action'ы
  • Блокировка — не вызывать next(action) вовсе

Место в потоке данных

Без middleware поток данных простой: dispatch(action) → reducer → newState. С middleware между dispatch и reducer появляется конвейер:

// Без middleware: dispatch(action) → reducer(state, action) → newState // С middleware: dispatch(action) → middleware1(action) → middleware2(action) → middleware3(action) → reducer(state, action) → newState

Побочные эффекты (Side Effects)

Reducer'ы должны быть чистыми функциями — без побочных эффектов. Но реальным приложениям нужно делать HTTP-запросы, писать в localStorage, логировать. Middleware — именно то место, где живут побочные эффекты в Redux.

Middleware — это «расширяемый конвейер» между dispatch и reducer. Каждый middleware в цепочке получает action, может что-то с ним сделать, и передаёт дальше через next(action). Последний в цепочке передаёт action в reducer.
Если middleware не вызывает next(action), action никогда не дойдёт до reducer'а и state не обновится. Это может быть полезно для фильтрации, но чаще — это баг.

Ключевые моменты

  • Middleware — точка расширения между dispatch и reducer
  • Может логировать, модифицировать, задерживать, блокировать action'ы
  • Побочные эффекты живут в middleware, не в reducer'ах
  • Аналогия с Express: цепочка обработчиков с next()
  • Подключается через applyMiddleware() (следующий урок)
