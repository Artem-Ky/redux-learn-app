        Теория — combineReducers на масштабе
        
      
        
        
        Масштабирование reducer'ов
        
        
        
          В реальных приложениях state содержит множество независимых «доменов» данных:
          пользователи, посты, комментарии, уведомления и т.д. Каждый домен — это
          отдельный слайс со своим reducer'ом.
        
        

        
        
          «The key concept is that each slice reducer "owns" its slice of state.
          It is responsible for managing that slice, and not any other slices.»
          
        — Redux Fundamentals, Part 3
        
        

        
        Три слайса — три reducer'а
        
        
        
          Каждый слайс-reducer полностью независим: он не знает о существовании
          других слайсов и управляет только своей частью state.
        
        

        
        
const rootReducer = combineReducers({
  users: usersReducer,
  posts: postsReducer,
  comments: commentsReducer
})

// state = {
//   users: [...]     ← usersReducer
//   posts: [...]     ← postsReducer
//   comments: [...]  ← commentsReducer
// }
        
        

        
        Форма state зеркалит combineReducers
        
        
        
          Структура объекта, переданного в combineReducers, точно
          повторяется в итоговом state. Если вы передали три ключа — в state
          будет три ключа с теми же именами.
        
        

        
        
// Ключи combineReducers...
combineReducers({ users, posts, comments })

// ...становятся ключами state:
store.getState()
// → { users: [...], posts: [...], comments: [...] }
        
        

        
        Каждый слайс независим
        
        
        
          При dispatch любого action combineReducers вызывает
          все слайс-reducer'ы. Каждый из них решает, нужно ли
          ему реагировать на этот action. Если нет — возвращает state без изменений.
        
        

        
        
// dispatch({ type: 'users/userAdded', ... })
// → usersReducer: обрабатывает ✓
// → postsReducer: возвращает state (default) ✗
// → commentsReducer: возвращает state (default) ✗
        
        

        
        Когда создавать новый слайс?
        
        
        
          Новый слайс стоит создать, когда:
        
        
        
          
          
        • Данные представляют отдельный домен (пользователи ≠ посты)
          • 
          
        • Данные обновляются независимо от других частей state
          • 
          
        • У данных свой набор actions и своя логика обработки
          • 
          
        • Reducer текущего слайса становится слишком большим
          • 
        
        

        
        
          Добавьте данные в каждый из трёх слайсов в демо выше и наблюдайте,
          как state tree растёт. Каждый слайс полностью независим — изменения
          в одном не влияют на другие.
        
        

        
        
          Не создавайте слишком мелкие слайсы. Если два типа данных всегда
          обновляются вместе и тесно связаны — лучше держать их в одном слайсе.
          combineReducers — для логически независимых частей state.
        
        

        
        Ключевые моменты
        
        
          
          
        • combineReducers масштабируется на любое количество слайсов
          • 
          
        • Каждый слайс-reducer независим и изолирован
          • 
          
        • Форма state зеркалит ключи combineReducers
          • 
          
        • При dispatch вызываются все слайс-reducer'ы
          • 
          
        • Новый слайс = новый домен данных с независимой логикой
