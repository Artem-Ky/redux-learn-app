Теория — Почему нельзя мутировать state

      

        
Store не защищает state от мутаций

        

          store.getState() возвращает прямую ссылку на объект state.
          Redux не создаёт копию, не замораживает объект (Object.freeze), не использует Proxy.
          Технически, вы можете изменить state напрямую — и это молча сломает
          всё.
        


        

          «Redux does not actually prevent you from mutating your state. There is nothing
          in the store code that stops you from doing store.getState().someField = someValue.
          However, doing so will break time-travel debugging, and components will not
          re-render as expected.»
          
— Redux Fundamentals, Part 4
        


        
Что происходит при прямой мутации

        

// ❌ ПЛОХО: прямая мутация
store.getState().value = 999

// Что произошло:
// ✓ Значение в памяти изменилось
// ✗ Subscribers НЕ вызваны → UI НЕ обновился
// ✗ DevTools НЕ увидели изменение
// ✗ Time-travel сломан
        


        

// ✅ ПРАВИЛЬНО: через dispatch
store.dispatch({ type: 'counter/set', payload: 999 })

// Что произошло:
// ✓ Reducer создал новый объект state
// ✓ Subscribers вызваны → UI обновился
// ✓ DevTools записали action
// ✓ Time-travel работает
        


        
Почему getState() возвращает ссылку?

        

          Производительность. Глубокое копирование объекта при каждом вызове getState()
          было бы слишком дорого. Redux полагается на конвенцию: вы
          обещаете не мутировать state напрямую.
        


        
Что ломается при мутации

        
    
          
  • React-Redux — connect и useSelector сравнивают ссылки (===). Если state мутирован, ссылка та же → React не перерендерит
    • 
          
  • DevTools — не видят изменений, т.к. action не был dispatched
    • 
          
  • Time-travel — невозможно «откатить» мутацию, предыдущий state потерян
    • 
          
  • Middleware — не вызывается, т.к. dispatch не произошёл
    • 
        


        

          Нажмите «Прямая мутация» в демо, затем «Проверить реальный state». Вы увидите,
          что значение в памяти изменилось, но UI остался старым — subscribe не был вызван.
        


        

          Redux не защищает от мутаций. Это ваша ответственность.
          В продакшене используйте redux-immutable-state-invariant middleware
          для отлова мутаций в development-режиме.
        


        
Ключевые моменты

        
    
          
  • getState() возвращает прямую ссылку, не копию
    • 
          
  • Мутация state молча ломает UI, DevTools, time-travel
    • 
          
  • Только dispatch запускает reducer и уведомляет подписчиков
    • 
          
  • Redux полагается на конвенцию иммутабельности, а не на защиту
    • 
          
  • Используйте линтеры и middleware для отлова мутаций
