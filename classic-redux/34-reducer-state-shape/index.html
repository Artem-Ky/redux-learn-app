Теория — Проектирование формы state

      

        
Почему форма state важна?

        

          Структура state определяет, насколько просто будет писать reducer'ы,
          обновлять данные и читать их в компонентах. Неправильная форма приводит
          к сложным обновлениям и дублированию данных.
        


        

          «In a larger application, state shape is a key concern. Much of the
          pain of managing state is due to organizing it in a way that makes
          the rest of the code complex.»
          
— Redux Docs: Normalizing State Shape
        


        
Вложенная (Nested) структура

        

          Первый инстинкт — хранить данные так, как они выглядят в API или UI:
          посты содержат автора и комментарии внутри себя.
        


        

// ❌ Вложенная структура (проблемная)
{
  posts: [
    {
      id: 1,
      title: 'Мой пост',
      author: { id: 1, name: 'Алексей' },
      comments: [
        { id: 1, text: 'Отлично!' }
      ]
    }
  ]
}
        


        
Плоская (Normalized) структура

        

          Лучший подход для Redux — нормализация: каждый тип данных
          хранится отдельно, а связи — через ID.
        


        

// ✅ Плоская (нормализованная) структура
{
  posts: [
    { id: 1, title: 'Мой пост', authorId: 1 }
  ],
  users: [
    { id: 1, name: 'Алексей' }
  ],
  comments: [
    { id: 1, text: 'Отлично!', postId: 1 }
  ]
}
        


        
Почему плоская структура лучше для Redux?

        
    
          
  • Нет дублирования: один пользователь — одна запись, даже если он автор 100 постов
    • 
          
  • Простые обновления: чтобы изменить имя автора, обновляем одну запись в users
    • 
          
  • Простые reducer'ы: каждый reducer работает с плоским массивом
    • 
          
  • Предсказуемость: нет глубокой вложенности = нет сложных spread-операций
    • 
        


        

          «The recommended approach to managing relational or nested data in a Redux store
          is to treat a portion of your store as if it were a database, and keep that data
          in a normalized form.»
          
— Redux Docs: Normalizing State Shape
        


        
Три категории state

        

          State в Redux-приложении обычно делится на три категории:
        

        
    
          
  • Domain data — данные приложения (пользователи, посты, товары)
    • 
          
  • App state — состояние логики (загрузка, ошибки, пагинация)
    • 
          
  • UI state — состояние интерфейса (открытые модальные окна, активные вкладки)
    • 
        


        

{
  // Domain data (нормализованные)
  users: [...],
  posts: [...],

  // App state
  loading: { posts: false, users: true },
  errors: { posts: null },

  // UI state
  ui: { modalOpen: false, activeTab: 'posts' }
}
        


        

          Переключайте режимы в демо выше, чтобы увидеть разницу между вложенной
          и плоской структурами. Обратите внимание на console: там показаны
          плюсы и минусы каждого подхода.
        


        

          Вложенная структура создаёт проблему дублирования: если
          один пользователь автор 10 постов, его данные копируются 10 раз.
          Обновление имени потребует изменить все 10 копий — это ошибкоопасно.
        


        
Ключевые моменты

        
    
          
  • Плоская (нормализованная) структура — рекомендация Redux
    • 
          
  • Каждый тип данных — отдельный слайс, связи через ID
    • 
          
  • Нет дублирования, простые обновления, простые reducer'ы
    • 
          
  • State делится на domain data, app state и UI state
    • 
          
  • Нормализация = база данных в виде JS-объекта
    •
