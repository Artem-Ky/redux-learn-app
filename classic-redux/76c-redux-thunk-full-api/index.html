Теория — redux-thunk: полный справочник

      


        
Исходный код redux-thunk (~14 строк)

        

          Вся библиотека redux-thunk — это буквально несколько строк.
          Вот её полный исходный код:
        


        

function createThunkMiddleware(extraArgument?) {
  return ({ dispatch, getState }) => (next) => (action) => {
    if (typeof action === 'function') {
      return action(dispatch, getState, extraArgument)
    }
    return next(action)
  }
}

const thunk = createThunkMiddleware()
thunk.withExtraArgument = createThunkMiddleware

export default thunk
        


        

          14 строк. Это одна из самых маленьких и при этом самых используемых
          библиотек в экосистеме JavaScript. Миллионы загрузок в неделю — за 14 строк кода.
        


        
API: thunk function

        
Thunk-функция получает три аргумента:


        

const myThunk = (arg1, arg2) => (dispatch, getState, extraArgument) => {
  // dispatch — отправить action (проходит ВЕСЬ pipeline)
  // getState — прочитать текущий state
  // extraArgument — дополнительный аргумент (API клиент и т.д.)
}
        


        
1. dispatch — отправка экшенов

        

const fetchUsers = () => async (dispatch) => {
  // Dispatch обычного action
  dispatch({ type: 'users/loading' })

  const data = await fetch('/api/users')
  dispatch({ type: 'users/loaded', payload: data })

  // Dispatch другого thunk(!)
  dispatch(showNotification('Пользователи загружены'))
}
        


        
2. getState — чтение state

        

const conditionalFetch = () => (dispatch, getState) => {
  const state = getState()

  // Проверяем текущий state перед действием
  if (state.users.status === 'loaded') {
    console.log('Данные уже загружены, пропускаем')
    return
  }

  // getState() можно вызывать ПОСЛЕ dispatch — увидим обновлённый state
  dispatch({ type: 'users/loading' })
  console.log(getState().users.status) // 'loading'
}
        


        
3. withExtraArgument — инъекция зависимостей

        

          Это killer-фича для архитектуры. Позволяет передать
          любой объект (API клиент, роутер, аналитику) в каждый thunk:
        


        

// Настройка store
import { thunk } from 'redux-thunk'

const api = {
  getUsers: () => fetch('/api/users').then(r => r.json()),
  saveUser: (user) => fetch('/api/users', { method: 'POST', body: JSON.stringify(user) }),
}

const store = createStore(
  reducer,
  applyMiddleware(thunk.withExtraArgument(api))
)

// В thunk'е — третий аргумент = наш api
const fetchUsers = () => async (dispatch, getState, api) => {
  const users = await api.getUsers()  // ← не нужен import!
  dispatch({ type: 'users/loaded', payload: users })
}
        


        

          Зачем withExtraArgument?
          
— В тестах передаём mock API → thunk не знает, что API ненастоящий
          
— В SSR передаём серверный API клиент
          
— Можно передать несколько зависимостей: { api, analytics, router }
          
— Thunk не зависит от импортов → чистая архитектура
        


        
4. Thunk вызывает thunk (композиция)

        

// Маленькие переиспользуемые thunk'и
const showNotification = (text) => (dispatch) => {
  dispatch({ type: 'ui/notification', payload: text })
  setTimeout(() => dispatch({ type: 'ui/clearNotification' }), 3000)
}

const logAnalytics = (event) => (dispatch) => {
  dispatch({ type: 'analytics/track', payload: event })
}

// Большой thunk компонует маленькие
const saveAndNotify = (data) => async (dispatch) => {
  await dispatch(saveData(data))         // thunk!
  dispatch(showNotification('Сохранено'))  // thunk!
  dispatch(logAnalytics('data_saved'))    // thunk!
}
        


        
5. Обработка ошибок

        

const fetchSafely = () => async (dispatch) => {
  try {
    dispatch({ type: 'data/loading' })
    const data = await fetch('/api/data')
    dispatch({ type: 'data/loaded', payload: data })
  } catch (err) {
    dispatch({ type: 'data/error', payload: err.message })
    // Можно пробросить ошибку дальше для await в UI:
    throw err
  }
}

// В UI:
try {
  await store.dispatch(fetchSafely())
  showSuccess()
} catch {
  showError()
}
        


        
6. Тестирование thunks

        

// Thunk легко тестировать — передаём mock dispatch и getState
test('fetchUsers dispatches correct actions', async () => {
  const dispatch = jest.fn()
  const getState = jest.fn(() => ({ status: 'idle' }))
  const mockApi = { getUsers: () => Promise.resolve([{ id: 1 }]) }

  // Вызываем thunk напрямую, без store!
  await fetchUsers()(dispatch, getState, mockApi)

  expect(dispatch).toHaveBeenCalledWith({ type: 'users/loading' })
  expect(dispatch).toHaveBeenCalledWith({
    type: 'users/loaded',
    payload: [{ id: 1 }]
  })
})
        


        
redux-thunk vs альтернативы

        
Почему thunk, а не redux-saga / redux-observable?

        
    
          
  • Простота: 14 строк кода. Ноль нового API. Обычные функции и async/await.
    • 
          
  • Порог входа: если вы знаете Promise/async/await — вы уже знаете thunks.
    • 
          
  • Официальность: входит в Redux Toolkit по умолчанию.
    • 
          
  • Достаточность: покрывает 95% use cases для async в Redux.
    • 
        


        

          «The Redux "Thunk" middleware lets us pass functions to dispatch.
          "Thunk" functions let us write async logic ahead of time, without
          knowing what Redux store is being used. A Redux thunk function receives
          dispatch and getState as arguments, and can dispatch actions like
          "this data was received from an API response".»
          
— Redux Fundamentals, Part 6
