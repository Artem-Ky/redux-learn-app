Теория — Enum для состояния загрузки

      

        
Почему строковый enum?

        

          Загрузка данных — не бинарный процесс. Есть минимум 4 состояния:
          ещё не начата, в процессе, завершена успешно, завершена с ошибкой.
          Строковый enum точно описывает каждое из них.
        


        

interface TodosState {
  status: 'idle' | 'loading' | 'succeeded' | 'failed'
  entities: Todo[]
  error: string | null
}
        


        
Dispatching loading/loaded actions

        

          Thunk dispatch-ит экшены, которые переключают status в reducer.
          Каждый экшен — чёткий переход между состояниями.
        


        

function fetchTodos() {
  return async (dispatch: any) => {
    dispatch({ type: 'todos/loading' })
    try {
      const todos = await fetchFromAPI()
      dispatch({ type: 'todos/succeeded', payload: todos })
    } catch (err) {
      dispatch({ type: 'todos/failed', payload: err.message })
    }
  }
}
        


        
Организация state shape

        

          Хорошая практика — хранить status, entities (данные)
          и error рядом друг с другом. Это делает состояние
          самодокументированным.
        


        

// Начальное состояние
const initialState: TodosState = {
  status: 'idle',     // ещё не загружали
  entities: [],       // пустой массив данных
  error: null         // ошибки нет
}
        


        

          «We recommend using a 'status' enum field that will be one of
          'idle' | 'loading' | 'succeeded' | 'failed', as a standard approach
          for tracking loading state for async requests.»
          
— Redux Essentials, Part 5: Loading State for Requests
        


        

          Enum-подход масштабируется: легко добавить новые состояния, например
          'refreshing' или 'retrying', без изменения структуры.
        


        

          Не используйте isLoading: boolean для асинхронных операций —
          это покрывает только 2 из 4 возможных состояний. Подробнее — в уроке 74.
