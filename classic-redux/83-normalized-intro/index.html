Теория — Что такое нормализация?

      

        
Нормализация = нет дубликатов + доступ по ID

        

          Нормализация данных — это процесс структурирования данных так, чтобы
          каждая сущность хранилась в одном месте и на неё ссылались по ID.
          Точно как в реляционных базах данных — вместо дублирования данных автора
          в каждом посте, мы храним авторов отдельно и используем authorId.
        


        
Аналогия с базами данных

        

          В SQL нельзя хранить полный объект автора в каждой строке таблицы постов —
          это нарушает нормальные формы. Вместо этого используется FOREIGN KEY.
          Redux рекомендует тот же подход: отдельные «таблицы» для каждого типа сущностей.
        


        

// ❌ Денормализованные (вложенные, дубликаты)
const posts = [
  { id: 1, title: 'Redux',
    author: { id: 1, name: 'Иван' } },  // ← автор #1
  { id: 2, title: 'React',
    author: { id: 1, name: 'Иван' } }   // ← ДУБЛИКАТ автора #1!
]
        


        

// ✅ Нормализованные (раздельно, без дубликатов)
const state = {
  authors: {
    1: { id: 1, name: 'Иван' }     // ← один экземпляр
  },
  posts: {
    1: { id: 1, title: 'Redux', authorId: 1 },
    2: { id: 2, title: 'React', authorId: 1 }
  }
}
        


        
Проблемы вложенных / дублированных данных

        

          Если автор «Иван» меняет имя — нужно найти и обновить каждую копию
          во всех постах. Забыли одну — данные рассинхронизированы. С нормализацией
          обновляется одно место — authors[1].name.
        

        
    
          
  • Дубликаты → рассинхронизация при обновлении
    • 
          
  • Вложенность → сложные пути обновления (posts[0].author.name)
    • 
          
  • Лишние данные → больше памяти, медленнее перерисовка
    • 
        


        

          «The recommended approach to managing relational or nested data in a Redux store
          is to treat a portion of your store as if it were a database, and keep that data
          in a normalized form.»
          
— Redux Docs — Normalizing State Shape
        


        

          Нормализация в Redux — это тот же принцип, что и в реляционных базах данных:
          «каждый факт хранится в одном месте». Это упрощает обновления и предотвращает
          рассинхронизацию данных.
        


        

          Не нормализуйте всё подряд. Данные, которые приходят только для отображения
          и не обновляются, можно хранить вложенными. Нормализация нужна, когда
          одни и те же сущности встречаются в нескольких местах.
        


        
Ключевые моменты

        
    
          
  • Нормализация = каждая сущность хранится один раз, доступ по ID
    • 
          
  • Аналогия с SQL: отдельные таблицы + внешние ключи
    • 
          
  • Денормализация → дубликаты → рассинхронизация при обновлении
    • 
          
  • Redux рекомендует нормализованный state для связанных данных
    • 
          
  • Обновление одной сущности — в одном месте
