Теория — Множество подписчиков

      

        
Несколько subscribe на один store

        

          Redux поддерживает множество подписчиков. Каждый вызов
          store.subscribe(listener) добавляет новый callback в список.
          При каждом dispatch все зарегистрированные listener'ы
          вызываются последовательно.
        


        

          «The subscriptions are snapshotted just before every dispatch() call.
          If you subscribe or unsubscribe while the listeners are being invoked,
          this will not have any effect on the dispatch() that is currently in progress.»
          
— Redux API Reference: Store — subscribe(listener)
        


        
Порядок вызова — порядок регистрации

        

          Listener'ы вызываются в том порядке, в котором они были зарегистрированы.
          Первый subscribe — первый callback:
        


        

// Регистрация трёх подписчиков
store.subscribe(() => console.log('#1 Counter Display'))
store.subscribe(() => console.log('#2 Action Log'))
store.subscribe(() => console.log('#3 History'))

store.dispatch({ type: 'counter/incremented' })
// → #1 Counter Display
// → #2 Action Log
// → #3 History
        


        
Каждый подписчик — независимый

        

          Подписчики ничего не знают друг о друге. Каждый решает сам, что делать
          с новым state: обновить DOM, записать в лог, сохранить историю и т.д.
        


        

// Подписчик 1: обновляет отображение счётчика
store.subscribe(() => {
  counterEl.textContent = String(store.getState().value)
})

// Подписчик 2: ведёт лог действий
store.subscribe(() => {
  logEl.textContent += '\n' + lastAction.type
})

// Подписчик 3: сохраняет историю state
store.subscribe(() => {
  history.push(store.getState())
})
        


        
Сценарии использования

        
    
          
  • UI-обновление — перерисовка компонентов при изменении state
    • 
          
  • Логирование — запись каждого изменения для отладки
    • 
          
  • Персистенция — сохранение state в localStorage
    • 
          
  • Аналитика — отправка событий при определённых изменениях
    • 
          
  • Undo/Redo — сохранение истории state для отмены действий
    • 
        


        

          В демо выше три подписчика работают одновременно: один показывает число,
          второй ведёт лог action types, третий сохраняет историю state.
          Все три вызываются при каждом dispatch.
        


        
Snapshot-семантика

        

          Список подписчиков «замораживается» (snapshot) перед каждым dispatch.
          Если внутри listener добавить или удалить подписчика — это повлияет
          только на следующий dispatch, а не на текущий.
        


        

          Не полагайтесь на конкретный порядок вызова listener'ов в бизнес-логике.
          Порядок — это деталь реализации. Каждый listener должен быть
          независимым и не зависеть от результата другого.
        


        
Ключевые моменты

        
    
          
  • Можно регистрировать множество подписчиков через subscribe
    • 
          
  • Все вызываются после каждого dispatch в порядке регистрации
    • 
          
  • Каждый подписчик независим и решает, что делать с новым state
    • 
          
  • Snapshot-семантика: изменение списка подписчиков во время dispatch не влияет на текущий вызов
    • 
          
  • Каждый subscribe возвращает свою функцию unsubscribe
