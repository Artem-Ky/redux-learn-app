Теория — Кастомные проверки равенства в Reselect

      

        
Как работает мемоизация createSelector

        

          По умолчанию createSelector использует строгое равенство
          (===) для проверки входных значений. Если входные селекторы вернули
          те же ссылки — результирующая функция не перевычисляется.
        


        

import { createSelector } from 'reselect'

// Входной селектор каждый раз создаёт НОВЫЙ массив:
const selectItems = (state: State) =>
  state.items.filter(i => i.active)  // новая ссылка каждый раз!

// Результирующий селектор пересчитывается КАЖДЫЙ вызов,
// потому что filter() всегда возвращает новый массив (!== предыдущему)
const selectTotal = createSelector(
  [selectItems],
  (items) => items.reduce((sum, i) => sum + i.price, 0)
)
        


        
Решение: memoizeOptions

        

          createSelector принимает третий аргумент — объект настроек мемоизации.
          Ключевое свойство: memoizeOptions.equalityCheck — функция сравнения
          входных значений.
        


        

const deepEqual = (a: unknown, b: unknown): boolean =>
  JSON.stringify(a) === JSON.stringify(b)

const selectTotal = createSelector(
  [selectItems],
  (items) => items.reduce((sum, i) => sum + i.price, 0),
  {
    memoizeOptions: {
      equalityCheck: deepEqual
    }
  }
)
        


        
resultEqualityCheck

        

          Отдельная настройка resultEqualityCheck сравнивает
          результат вычисления. Если результат «глубоко» равен предыдущему,
          селектор вернёт старую ссылку:
        


        

const selectFiltered = createSelector(
  [selectItems, selectFilter],
  (items, filter) => items.filter(i => i.status === filter),
  {
    memoizeOptions: {
      resultEqualityCheck: deepEqual
    }
  }
)
        


        
Когда custom equality помогает?

        
    
          
  • Входной селектор реконструирует объект каждый вызов (filter, map, slice)
    • 
          
  • Данные структурно одинаковые, но === даёт false (новая ссылка)
    • 
          
  • Вы хотите избежать пересчёта тяжёлой результирующей функции
    • 
        


        
Компромиссы производительности

        

          Deep equality — это не бесплатно. JSON.stringify или рекурсивное
          сравнение имеют свою цену. Выбирайте стратегию в зависимости от ситуации:
        

        
    
          
  • === — O(1), но ложные пересчёты при новых ссылках
    • 
          
  • shallow equal — O(n) по ключам верхнего уровня, хороший баланс
    • 
          
  • deep equal — O(n·m) по всем вложенностям, максимальный кэш, но затраты
    • 
        


        

          «createSelector uses referential equality (===) to check if an input has changed.
          You can customize this behavior by passing an equalityCheck function in
          the memoizeOptions argument.»
          
— Reselect Documentation — createSelector
        


        

          Используйте кастомное сравнение точечно — только для селекторов,
          где входные значения гарантированно создают новые ссылки при каждом вызове.
          Для большинства случаев === достаточно.
        


        

          JSON.stringify — плохой выбор для production-кода. Он не обрабатывает
          undefined, функции, циклические ссылки и порядок ключей нестабилен.
          Используйте проверенные библиотеки (lodash.isEqual, fast-deep-equal).
