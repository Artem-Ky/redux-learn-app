Теория — Обработка ошибок в async thunks

      

        
try/catch в async thunks

        

          Асинхронные thunks должны оборачивать свою логику в try/catch,
          чтобы перехватить ошибки сети, парсинга и другие исключения.
        


        

          «Any time you write an async thunk, you should add error handling
          with try/catch blocks. The catch block should dispatch a "rejected"
          action containing the error message.»
          
— Redux Essentials, Part 5 — Async Logic and Data Fetching
        


        
Паттерн: loading / succeeded / failed


        

const fetchData = () => async (dispatch: any): Promise<void> => {
  dispatch({ type: 'fetch/pending' })

  try {
    const response = await fetch('/api/data')
    const data = await response.json()
    dispatch({ type: 'fetch/fulfilled', payload: data })
  } catch (err: any) {
    dispatch({ type: 'fetch/rejected', payload: err.message })
  }
}
        


        
Хранение ошибки в state

        

          Ошибка сохраняется в state, чтобы UI мог отобразить сообщение и предложить retry.
        


        

interface AppState {
  status: 'idle' | 'loading' | 'succeeded' | 'failed'
  data: Item[]
  error: string | null
}

// В reducer:
case 'fetch/rejected':
  return { ...state, status: 'failed', error: action.payload }
        


        
Recovery-паттерны

        
    
          
  • Retry — кнопка повторной попытки, которая заново вызывает thunk
    • 
          
  • Reset — сброс state в idle и очистка ошибки
    • 
          
  • Fallback — показ кэшированных данных при ошибке
    • 
        


        

// Retry: просто dispatch thunk ещё раз
btnRetry.addEventListener('click', () => {
  store.dispatch(fetchData() as any)
})
        


        

          Всегда сохраняйте error.message (строку), а не весь объект ошибки —
          объекты Error не сериализуются и нарушают принцип сериализуемости Redux-state.
        


        

          Без try/catch ошибка в async thunk приведёт к unhandled promise rejection.
          Это не обрушит приложение, но state останется в состоянии 'loading' навсегда.
        


        
Ключевые моменты

        
    
          
  • Оборачивайте async thunk в try/catch
    • 
          
  • В catch делайте dispatch({ type: 'rejected', payload: err.message })
    • 
          
  • Храните error: string | null в state
    • 
          
  • Предоставляйте пользователю возможность retry
    • 
          
  • Паттерн: { status: 'failed', error: 'Network error' }
