Теория — Immer: полный разбор

      

        
Что такое Immer?

        

          Immer (нем. «всегда») — библиотека Михеля Вестстрате (автора MobX),
          которая позволяет писать код, выглядящий как мутация, но под капотом
          создающий иммутабельные обновления. Вы «мутируете» черновик (draft), а Immer
          на основе ваших изменений создаёт новый объект.
        


        

import { produce } from 'immer'

const nextState = produce(currentState, draft => {
  // «мутируем» draft — на самом деле это Proxy
  draft.user.company.address.city = 'Новосибирск'
})
// nextState — новый иммутабельный объект
// currentState — не изменился
        


        
Как работает produce?

        

          Функция produce(currentState, recipe) принимает два аргумента:
        

        
    
          
  • currentState — текущее (замороженное) состояние
    • 
          
  • recipe — функция-рецепт, получающая draft
    • 
        

        

          draft — это Proxy-объект, который записывает
          все ваши «мутации». Когда recipe завершается, Immer строит новый объект,
          применяя записанные изменения к оригиналу.
        


        

// Как это работает внутри (упрощённо):
//
// 1. Immer создаёт Proxy вокруг currentState
// 2. draft = new Proxy(currentState, handler)
// 3. handler.set() перехватывает записи
// 4. Записи сохраняются в «патчи»
// 5. Из патчей строится новый объект
// 6. Неизменённые ветви === оригинал (structural sharing)
        


        
Structural Sharing

        

          Immer не копирует весь объект целиком. Он переиспользует
          все ветви дерева, которые не были изменены. Это означает, что ===
          сравнение работает для определения, изменилась ли часть state:
        


        

const state = { user: { name: 'Иван', tags: ['a', 'b'] }, config: { theme: 'dark' } }

const next = produce(state, draft => {
  draft.user.name = 'Пётр'
})

next === state              // false (корень изменился)
next.user === state.user      // false (user изменился)
next.user.tags === state.user.tags  // TRUE! (tags не трогали)
next.config === state.config  // TRUE! (config не трогали)
        


        

          Structural sharing — ключ к производительности. React/Redux используют ===
          для определения, нужен ли ре-рендер. Благодаря sharing, компоненты,
          зависящие от config, не перерисуются при изменении user.
        


        
Работа с массивами

        

          В отличие от ручного иммутабельного подхода, с Immer можно использовать
          все мутирующие методы массивов: push, pop,
          splice, sort, reverse, присвоение по индексу.
        


        

// Без Immer (ручная иммутабельность):
const added = { ...state, items: [...state.items, newItem] }
const removed = { ...state, items: state.items.filter(i => i.id !== id) }
const sorted = { ...state, items: [...state.items].sort(cmp) }

// С Immer — как обычный JS:
const next = produce(state, draft => {
  draft.items.push(newItem)
  draft.items.splice(index, 1)
  draft.items.sort(cmp)
  draft.items[0].name = 'Updated'
})
        


        
Curried Producers

        

          Если produce вызвать без первого аргумента
          (только с recipe), он возвращает функцию, которая принимает state.
          Это называется curried producer — переиспользуемый трансформер:
        


        

// Обычный produce:
const next = produce(state, draft => { draft.count++ })

// Curried produce (без state):
const increment = produce((draft: CounterState) => {
  draft.count++
})

// Теперь increment — это функция (state) => newState
const next = increment(currentState)

// С дополнительными аргументами:
const setCity = produce((draft: AppState, city: string) => {
  draft.user.address.city = city
})
const next = setCity(state, 'Казань')
        


        
Immer как Redux Reducer

        

          Curried producer — это готовый reducer! Сигнатура (state, action) => newState
          совпадает с тем, что ожидает Redux. Именно это делает createSlice под капотом:
        


        

// Классический reducer (spread-каскады):
function reducer(state, action) {
  switch (action.type) {
    case 'user/setCity':
      return { ...state, user: { ...state.user, address: { ...state.user.address, city: action.payload } } }
  }
}

// Immer reducer (curried produce):
const reducer = produce((draft, action) => {
  switch (action.type) {
    case 'user/setCity':
      draft.user.address.city = action.payload
      break
  }
})

// createSlice делает ИМЕННО ЭТО под капотом
        


        

          «Immer works by writing producers, which are functions that receive a proxy
          of the current state (the draft) and can freely mutate it. Once the producer
          finishes, Immer will produce the next immutable state based on the mutations
          to the draft.»
          
— Immer Documentation — Introduction
        


        
produce — мутация или return?

        

          Внутри recipe можно либо мутировать draft, либо
          вернуть новый объект. Нельзя делать и то, и другое:
        


        

// ✅ Мутация draft:
produce(state, draft => { draft.count++ })

// ✅ Полная замена через return:
produce(state, () => ({ count: 0 }))

// ❌ И то, и другое — ошибка!
produce(state, draft => {
  draft.count++
  return { count: 0 }  // Error!
})
        


        
Immer — основа Redux Toolkit

        

          Redux Toolkit использует Immer внутри createSlice.
          Когда вы пишете «мутирующий» код в reducers объекта createSlice, это не настоящая
          мутация — это Immer produce под капотом. Понимание Immer = понимание RTK.
        


        

// Redux Toolkit (createSlice) — Immer под капотом:
const todosSlice = createSlice({
  name: 'todos',
  initialState: [],
  reducers: {
    add(state, action) {
      state.push(action.payload)  // «мутация» — на самом деле produce!
    },
    toggle(state, action) {
      const todo = state.find(t => t.id === action.payload)
      if (todo) todo.completed = !todo.completed
    }
  }
})
        


        

          Draft — это Proxy, а не реальный объект. Нельзя выносить draft за пределы
          recipe-функции, передавать его в async-функции или сохранять в переменные
          вне produce. Draft «живёт» только внутри callback.
