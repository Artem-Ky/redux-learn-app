Теория — Middleware с асинхронным поведением

      

        
Middleware может содержать async-логику

        

          Reducer'ы должны быть чистыми функциями без побочных эффектов.
          Но middleware — это место, где побочные эффекты разрешены:
          setTimeout, fetch, WebSocket, localStorage и т.д.
        


        

          «Middleware have access to dispatch and getState, so they can dispatch
          new actions and read the current state. This makes them the ideal place
          for putting side effects like network requests, logging, and other
          asynchronous behavior.»
          
— Redux Fundamentals, Part 6
        


        
Паттерн: перехват и отложенный dispatch

        

const delayMiddleware =
  (storeAPI: any) => (next: any) => (action: any): any => {
    if (action.type === 'counter/delayedIncrement') {
      setTimeout(() => {
        storeAPI.dispatch({ type: 'counter/incremented' })
      }, 1000)
      return // не вызываем next() для этого action
    }
    return next(action)
  }
        


        
dispatch vs next

        

          Ключевое отличие: storeAPI.dispatch() отправляет action
          с самого начала pipeline (через все middleware).
          next() передаёт action следующему middleware.
          Для отложенных действий используем storeAPI.dispatch().
        


        

// storeAPI.dispatch() — через ВСЕ middleware
storeAPI.dispatch({ type: 'counter/incremented' })

// next() — только к следующему middleware в цепочке
next(action)
        


        
Прелюдия к thunk

        

          Этот паттерн — предвестник redux-thunk. Thunk middleware
          позволяет dispatch'ить функции вместо объектов. Функция получает
          dispatch и getState для асинхронной логики.
        


        

// Thunk — это функция вместо action-объекта
const delayedIncrement = (dispatch: any, getState: any): void => {
  setTimeout(() => {
    dispatch({ type: 'counter/incremented' })
  }, 1000)
}

// Thunk middleware позволяет dispatch'ить функции:
store.dispatch(delayedIncrement)
        


        
Побочные эффекты — в middleware, не в reducer

        
    
          
  • Reducer — чистая функция, только вычисление нового state
    • 
          
  • Middleware — сайд-эффекты: таймеры, API-вызовы, логирование
    • 
          
  • setTimeout — отложенное действие
    • 
          
  • fetch — сетевой запрос
    • 
          
  • localStorage — персистентность
    • 
        


        

          Нажмите «Delayed +1» — пульсирующая точка покажет ожидание. Через 1 секунду
          middleware сам dispatch'ит counter/incremented, и счётчик обновится.
          Посмотрите временные метки в консоли.
        


        

          Middleware с async-логикой усложняет дебаг: действие происходит «само по себе»
          через время. Всегда логируйте, когда и почему middleware инициирует dispatch.
          В продакшене используйте redux-thunk или redux-saga вместо ручных setTimeout.
        


        
Ключевые моменты

        
    
          
  • Middleware может содержать async-логику (setTimeout, promises)
    • 
          
  • Побочные эффекты принадлежат middleware, не reducer
    • 
          
  • storeAPI.dispatch() отправляет через весь pipeline
    • 
          
  • Этот паттерн — основа redux-thunk
    • 
          
  • Используйте storeAPI.dispatch(), не next(), для отложенных действий
