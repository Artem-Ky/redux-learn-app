Теория — «Луковая» модель middleware pipeline

      

        
Onion Model (луковая модель)

        

          Middleware в Redux образуют конвейер (pipeline). Каждый middleware
          «оборачивает» следующий, как слои луковицы. Порядок входа совпадает с порядком
          регистрации, а порядок выхода — обратный.
        


        

          «Redux middleware are written as a series of three nested functions...
          When an action is dispatched, it actually goes through all of the middleware
          in the pipeline first.»
          
— Redux Fundamentals, Part 4: Store — Middleware
        


        
Порядок выполнения

        

          Если middleware зарегистрированы как applyMiddleware(A, B, C):
        

        
    
          
  • Вход (до next): A → B → C (порядок регистрации)
    • 
          
  • Reducer: выполняется в центре
    • 
          
  • Выход (после next): C → B → A (обратный порядок)
    • 
        


        

const middlewareA = (storeAPI: any) => (next: any) => (action: any) => {
  console.log('A enter')   // ← до next()
  const result = next(action)
  console.log('A exit')    // ← после next()
  return result
}
        


        
Что происходит при вызове next(action)?

        

          next(action) передаёт action следующему middleware
          в цепочке. Если это последний middleware, action попадает в reducer.
          Код до next() — «вход» в middleware,
          код после — «выход».
        


        

// Визуализация call stack:
dispatch(action)
  └─ middlewareA: "A enter"
       └─ middlewareB: "B enter"
            └─ middlewareC: "C enter"
                 └─ reducer(state, action) → newState
            middlewareC: "C exit"
       middlewareB: "B exit"
  middlewareA: "A exit"
        


        
Аналогия с функциями

        

          По сути pipeline middleware — это вложенные вызовы функций. Код до next()
          выполняется при «погружении» в стек вызовов, а код после next() — при
          «всплытии» обратно.
        


        

          Этот паттерн аналогичен middleware в Express.js или Koa. В Koa его так и называют —
          «onion model». Каждый middleware видит action на входе и результат на выходе.
        


        

          Если middleware не вызывает next(action), action
          не дойдёт до reducer и до следующих middleware. Это может быть намеренным
          (фильтрация действий), но часто — это баг.
        


        
Ключевые моменты

        
    
          
  • Middleware образуют pipeline — цепочку обработчиков
    • 
          
  • Порядок входа = порядок регистрации в applyMiddleware
    • 
          
  • Порядок выхода = обратный (как стек вызовов)
    • 
          
  • Код до next() — «enter», после next() — «exit»
    • 
          
  • Каждый middleware оборачивает следующий (onion model)
