Интерактив — Три подхода к асинхронности в Redux

Задача: загрузить данные с «сервера» (симуляция 1с) и положить в store. Попробуем три подхода — и увидим, почему первые два плохие:

Подход 1 Async в reducer (НЕЛЬЗЯ!)

Попытка сделать fetch прямо внутри reducer — нарушает правило чистоты.

—
Подход 2 Async снаружи, dispatch вручную (работает, но...)

Делаем fetch снаружи, потом dispatch результата. Работает, но логика разбросана и не переиспользуется.

—
Подход 3 Thunk — dispatch функции вместо объекта

Dispatch'им функцию. Thunk middleware перехватывает её и вызывает с (dispatch, getState). Логика инкапсулирована и переиспользуема.

—
Теория — Зачем нужны Thunks? Почему без них нельзя?

Корень проблемы: Redux store — синхронный

Redux store сам по себе ничего не знает об асинхронности. Он умеет только:

  1. Синхронно вызвать reducer: newState = reducer(currentState, action)
  2. Сохранить новый state
  3. Оповестить подписчиков

Всё. Никаких промисов, таймаутов, HTTP-запросов. Store — это синхронная машина.

«By itself, a Redux store doesn't know anything about async logic. It only knows how to synchronously dispatch actions, update the state by calling the root reducer function, and notify the UI that something has changed. Any asynchronicity has to happen outside the store.»
— Redux Fundamentals, Part 6

Подход 1: Async в reducer — ЗАПРЕЩЕНО

Reducer — это чистая функция. Нельзя делать в нём:

  • HTTP-запросы (fetch, axios)
  • Таймеры (setTimeout, setInterval)
  • Обращения к localStorage / sessionStorage
  • Генерацию случайных чисел или Date.now()
  • Любые побочные эффекты
// ❌ НЕЛЬЗЯ! Reducer должен быть чистой функцией function todosReducer(state, action) { switch (action.type) { case 'todos/fetch': // ❌ fetch — побочный эффект! fetch('/api/todos') .then(r => r.json()) .then(data => { // ❌ Мутация state! state.todos = data }) return state // ← state не изменился синхронно! } }
Даже если этот код «работает» технически, он ломает:
— Time-travel debugging (нельзя воспроизвести)
— Тестируемость (нужно мокать fetch)
— Предсказуемость (результат зависит от сети)
— Иммутабельность (state мутируется асинхронно)

Подход 2: Async снаружи — работает, но проблемы

// Это работает, но... async function loadTodos() { store.dispatch({ type: 'todos/loading' }) const response = await fetch('/api/todos') const data = await response.json() store.dispatch({ type: 'todos/loaded', payload: data }) } // Проблемы: // 1. Функция ЖЁСТКО привязана к конкретному store // 2. Нельзя переиспользовать (другой store? SSR?) // 3. Логика размазана по UI-коду // 4. Нельзя тестировать без реального store

Подход 3: Thunk — элегантное решение

// Thunk — функция, которая получает dispatch и getState как аргументы const fetchTodos = () => async (dispatch, getState) => { dispatch({ type: 'todos/loading' }) const response = await fetch('/api/todos') const data = await response.json() dispatch({ type: 'todos/loaded', payload: data }) } // Используем как обычный dispatch: store.dispatch(fetchTodos()) // Преимущества: // ✅ Не привязан к конкретному store // ✅ Переиспользуем в любом месте // ✅ Логика инкапсулирована в action creator // ✅ Легко тестировать (передаём mock dispatch) // ✅ Может читать state через getState()

Что такое «thunk» вообще?

«The word "thunk" is a programming term that means "a piece of code that does some delayed work". Rather than execute some logic now, we can write a function body or code that can be used to perform the work later.»
— Redux Fundamentals, Part 6

Слово «thunk» пришло из функционального программирования. Thunk — это функция, которая оборачивает вычисление для отложенного выполнения.

// Это НЕ thunk — значение вычисляется СРАЗУ: const x = 1 + 2 // → 3 прямо сейчас // Это THUNK — вычисление отложено до вызова: const thunk = () => 1 + 2 // → функция, вычислит 3 когда вызовем thunk() // → 3

Итог: почему именно middleware?

Middleware — идеальное место для побочных эффектов в Redux:
— Middleware перехватывает dispatch до того, как action дойдёт до reducer
— Может решить: пропустить, задержать, изменить или заменить action
— Имеет доступ к dispatch (отправить новые экшены) и getState (прочитать state)
— Reducer остаётся чистым — только синхронные вычисления
