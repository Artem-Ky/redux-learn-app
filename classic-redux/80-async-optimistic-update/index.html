Теория — Оптимистичные обновления

      

        
Что такое optimistic update

        

          Optimistic update — это паттерн, при котором UI обновляется до получения
          ответа от сервера. Если сервер подтверждает операцию — ничего не меняется.
          Если сервер вернул ошибку — выполняется rollback к предыдущему состоянию.
        


        

          «Optimistic updates are a pattern where the UI is updated immediately,
          before the server confirms the change. If the server rejects the change,
          the UI rolls back to the previous state.»
          
— Redux — Structuring Reducers
        


        
Алгоритм

        
    
          
  1. Сохранить текущий state (snapshot)
    2. 
          
  2. Dispatch action с оптимистичным обновлением
    3. 
          
  3. Отправить запрос на сервер
    4. 
          
  4. Если ошибка → dispatch rollback action с сохранённым snapshot
    5. 
        


        

const toggleTodo = (id: number) => {
  return async (dispatch: any, getState: any): Promise<void> => {
    const previousTodos = getState().todos // snapshot

    dispatch({ type: 'todo/toggled', payload: id }) // optimistic

    try {
      await apiToggle(id) // server request
    } catch {
      dispatch({ type: 'todo/rollback', payload: previousTodos }) // rollback
    }
  }
}
        


        
UX-преимущества

        
    
          
  • Мгновенная обратная связь — нет задержки между кликом и изменением UI
    • 
          
  • Ощущение скорости — приложение кажется быстрым даже при медленном сервере
    • 
          
  • Offline-first — можно работать без сети и синхронизировать позже
    • 
        


        
Rollback-паттерн


        

// В reducer:
case 'todo/rollback':
  return { ...state, todos: action.payload } // восстановить snapshot
        


        

          Оптимистичные обновления особенно эффективны для простых toggle-операций
          (like, bookmark, complete) — где вероятность ошибки низкая, а задержка
          заметна для пользователя.
        


        
Когда использовать

        
    
          
  • Toggle-операции (чекбоксы, лайки, закладки)
    • 
          
  • Сортировка и перетаскивание элементов
    • 
          
  • Быстрое редактирование (inline edit)
    • 
        


        
Когда НЕ использовать

        
    
          
  • Создание ресурсов (нужен ID от сервера)
    • 
          
  • Финансовые операции (платежи, переводы)
    • 
          
  • Удаление с невосстановимыми данными
    • 
        


        

          При rollback UI может «моргнуть» — элемент переключится обратно. Это нормально,
          но стоит показать пользователю уведомление об ошибке, чтобы объяснить, почему
          состояние вернулось.
        


        
Ключевые моменты

        
    
          
  • Dispatch до запроса — мгновенное обновление UI
    • 
          
  • Сохраните getState() перед оптимистичным dispatch
    • 
          
  • При ошибке dispatch rollback с сохранённым snapshot
    • 
          
  • Мгновенная обратная связь — лучший UX
    • 
          
  • Не используйте для критичных операций (платежи, удаление)
