Теория — Custom API Middleware

      

        
Паттерн API middleware

        

          Вместо написания thunk для каждого API-запроса можно создать одну middleware,
          которая перехватывает любой action с полем meta.api
          и автоматически выполняет HTTP-запрос. Это централизует всю логику запросов
          в одном месте.
        


        

// Action с meta.api
const action = {
  type: 'users/fetch',
  meta: {
    api: {
      url: '/api/users',
      method: 'GET'
    }
  }
}
        


        
Middleware делает HTTP-запрос

        

          Когда middleware видит action.meta.api, она не пропускает action дальше.
          Вместо этого она сама делает запрос по указанному URL и диспатчит результат.
        


        

const apiMiddleware = (store: any) =>
  (next: any) => async (action: any) => {
    if (!action.meta?.api) return next(action)

    const { url, method } = action.meta.api

    try {
      const data = await fetch(url, { method })
      store.dispatch({
        type: `${action.type}/success`,
        payload: data
      })
    } catch (err) {
      store.dispatch({
        type: `${action.type}/failure`,
        error: err.message
      })
    }
  }
        


        
Диспатч success / failure

        

          Middleware автоматически формирует имена типов: к оригинальному type
          добавляется /success или /failure. Reducer просто
          обрабатывает эти типы — ему не нужно знать о HTTP.
        


        

          «Middleware provides a third-party extension point between dispatching an action,
          and the moment it reaches the reducer.»
          
— Redux Docs — Middleware
        


        
Централизация API-логики

        

          Без API middleware каждый thunk содержит одинаковый код: try/catch,
          fetch, dispatch loading/success/failure. С API middleware этот
          шаблонный код пишется один раз. Action creators становятся
          простыми объектами с meta.api.
        


        
Уменьшение boilerplate

        

          Сравните: для 10 API-эндпоинтов без middleware нужно 10 thunk с
          повторяющимся кодом fetch/dispatch. С API middleware — 10 простых action объектов
          и одна middleware. Логика запроса описана декларативно в action.
        


        

          API middleware — это декларативный подход: action описывает запрос
          (URL, метод), а middleware выполняет его. Это разделение делает
          код чище и тестируемее.
        


        

          В реальных проектах API middleware сложнее: авторизация, retry, кеширование,
          нормализация. Здесь показан базовый паттерн для понимания концепции.
