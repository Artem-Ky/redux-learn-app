Теория — Debounce Middleware

Паттерн debounce в middleware

Debounce — это техника, при которой действие выполняется только после того, как пользователь прекратил ввод на заданное время. Middleware — идеальное место для debounce, потому что оно перехватывает действия до попадания в reducer.

const debounceMiddleware = (store: any) => { let timer: number | null = null return (next: any) => (action: any) => { if (action.type === 'search/queryChanged') { if (timer) clearTimeout(timer) timer = setTimeout(() => { next(action) }, 400) return } return next(action) } }

clearTimeout / setTimeout

Каждый раз, когда приходит целевой action, предыдущий таймер сбрасывается через clearTimeout, и запускается новый setTimeout. Только когда пользователь перестаёт печатать на 400 мс, действие пропускается в reducer через next(action).

Проходит только последнее действие

Если пользователь вводит «redux» по одной букве — будет 5 dispatch, но в reducer попадёт только последний action с payload: "redux". Промежуточные «r», «re», «red», «redu» — все заблокированы middleware.

«Middleware provides a third-party extension point between dispatching an action, and the moment it reaches the reducer.»
— Redux Docs — Middleware

Случаи применения

  • Поиск — не отправлять запрос на каждый символ
  • Авто-сохранение — сохранять только после паузы в редактировании
  • Resize/scroll — не реагировать на каждый пиксель
  • Валидация — проверять форму после остановки ввода

Состояние middleware (замыкание)

Переменная timer живёт в замыкании (closure) middleware. Она создаётся один раз при инициализации middleware и сохраняется между вызовами. Это легальный способ хранить состояние вне Redux store — middleware имеет на это право, в отличие от reducer.

Переменная таймера в замыкании middleware — это приватное состояние middleware. Оно не попадает в store и не видно в DevTools, но позволяет middleware координировать действия между вызовами.
Debounce в middleware блокирует action полностью — reducer его не увидит. Если нужно показать промежуточное состояние (например, «печатает...»), диспатчьте отдельный action перед debounce.

Ключевые моменты

  • Debounce middleware перехватывает action и задерживает его на N мс
  • clearTimeout + setTimeout — ядро debounce
  • В reducer попадает только последнее действие после паузы
  • Таймер хранится в замыкании middleware
  • Типичные случаи: поиск, авто-сохранение, resize
