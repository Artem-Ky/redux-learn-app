Теория — Crash Reporter middleware

      

        
Зачем ловить ошибки в middleware?

        

          Если reducer выбросит исключение, обычное приложение упадёт целиком.
          Crash Reporter middleware оборачивает вызов next(action) в
          try/catch, перехватывает ошибку и позволяет приложению продолжить работу.
        


        

          «Middleware provides a third-party extension point between dispatching an action, and
          the moment it reaches the reducer. People use Redux middleware for logging, crash
          reporting, talking to an asynchronous API, routing, and more.»
          
— Redux Docs — Middleware
        


        
Паттерн: try/catch вокруг next(action)

        

const crashReporter =
  (storeAPI: any) => (next: any) => (action: any): any => {
    try {
      return next(action)
    } catch (err) {
      console.error('Caught an exception!', err)
      // Отправить ошибку на сервер
      reportError(err, storeAPI.getState(), action)
      throw err // опционально — можно и не пробрасывать
    }
  }
        


        
Что происходит при ошибке?

        
    
          
  • try — нормальный поток: next(action) вызывает следующий middleware или reducer
    • 
          
  • catch — если reducer выбросил, ошибка перехвачена
    • 
          
  • Можно залогировать, отправить на сервер, показать пользователю
    • 
          
  • Можно throw err — пробросить дальше, или «проглотить» — приложение продолжит
    • 
        


        
Отправка ошибки на сервер

        

          В продакшене Crash Reporter отправляет отчёт на сервер мониторинга
          (Sentry, Bugsnag и т.д.) с полным контекстом: состояние store, действие,
          стек ошибки.
        


        

function reportError(error: Error, state: any, action: any): void {
  fetch('/api/error-report', {
    method: 'POST',
    body: JSON.stringify({
      error: error.message,
      stack: error.stack,
      state,
      action,
      timestamp: Date.now()
    })
  })
}
        


        
Graceful degradation

        

          Middleware решает: пробросить ошибку (throw err) или «проглотить».
          Проглатывание позволяет приложению работать дальше — это
          graceful degradation. Пользователь видит, что кнопка не сработала,
          но приложение не упало.
        


        

          В нашем демо middleware не пробрасывает ошибку: приложение продолжает
          работу после нажатия «Crash!». Нажмите «Normal +1» — счётчик всё ещё работает.
        


        

          Не злоупотребляйте «проглатыванием» ошибок в продакшене. Если reducer выбрасывает —
          это баг, который нужно исправить. Crash Reporter — страховочная сетка, а не замена
          правильной обработки ошибок.
        


        
Ключевые моменты

        
    
          
  • Crash Reporter оборачивает next(action) в try/catch
    • 
          
  • Перехваченная ошибка логируется и/или отправляется на сервер
    • 
          
  • Можно «проглотить» ошибку для graceful degradation
    • 
          
  • Контекст ошибки: state, action, stack trace
    • 
          
  • В продакшене используйте Sentry, Bugsnag или аналоги
