Теория — Иммутабельное обновление вложенных структур

      

        
Проблема вложенных объектов

        

          Чем глубже вложенность, тем больше кода нужно для иммутабельного обновления.
          Для каждого уровня вложенности нужно создать копию:
        


        

const state = {
  user: {
    name: 'Иван',
    company: {
      name: 'Рога и Копыта',
      address: {
        city: 'Москва',        // ← хотим изменить ЭТО
        zip: '101000'
      }
    }
  }
}

// Для обновления city нужно копировать КАЖДЫЙ уровень:
const newState = {
  ...state,                             // копия корня
  user: {
    ...state.user,                      // копия user
    company: {
      ...state.user.company,            // копия company
      address: {
        ...state.user.company.address,  // копия address
        city: 'Новосибирск'              // ← новое значение
      }
    }
  }
}
        


        

          Это одна из главных болевых точек классического Redux.
          Именно эту проблему решает библиотека Immer,
          которую мы изучим позже. Immer позволяет писать «мутирующий» код,
          а за кулисами создаёт иммутабельные обновления.
        


        
Что происходит со ссылками?

        
При правильном иммутабельном обновлении:

        
    
          
  • state === newState → false (новый корневой объект)
    • 
          
  • state.user === newState.user → false (скопирован)
    • 
          
  • state.user.company === newState.user.company → false
    • 
          
  • state.user.company.address === newState.user.company.address → false
    • 
          
  • Но! state.user.name === newState.user.name → true (строки-примитивы не копируются)
