Теория — Middleware как трансформер action'ов

      

        
Middleware может изменить action

        

          Middleware имеет полный контроль над action'ом перед передачей в next().
          Можно добавить поля, изменить payload, нормализовать формат —
          передать в next() модифицированный объект.
        


        

          «A middleware can do anything it wants when it sees a dispatched action.
          It can modify the action, delay it, ignore it, or pass it along.»
          
— Redux Fundamentals, Part 4
        


        
Паттерн: обогащение action'а

        

const enrichMiddleware =
  (storeAPI: any) => (next: any) => (action: any): any => {
    const enrichedAction = {
      ...action,
      timestamp: Date.now(),
      meta: { source: 'user' }
    }
    return next(enrichedAction)
  }
        


        
Что можно добавить?

        
    
          
  • timestamp — время отправки действия
    • 
          
  • meta.source — кто инициировал (user, system, api)
    • 
          
  • meta.correlationId — связь с цепочкой действий
    • 
          
  • meta.userId — ID текущего пользователя
    • 
        


        

// Исходный action от компонента:
{ type: 'counter/incremented' }

// После middleware:
{
  type: 'counter/incremented',
  timestamp: 1713100000000,
  meta: { source: 'user' }
}
        


        
Нормализация формата action'ов

        

          Middleware может приводить action'ы к единому формату. Например, если разные
          части приложения используют разные соглашения:
        


        

const normalizeMiddleware =
  (storeAPI: any) => (next: any) => (action: any): any => {
    const normalized = {
      ...action,
      type: action.type.toLowerCase(),
      meta: action.meta || {}
    }
    return next(normalized)
  }
        


        

          В демо каждый dispatch обогащается полями timestamp и
          meta: { source: 'user' }. Посмотрите в консоль — оригинальный
          action сравнивается с enriched версией.
        


        

          Изменяя action, вы создаёте новый объект через spread
          ({ ...action, ... }). Никогда не мутируйте оригинальный action
          напрямую — это может вызвать побочные эффекты в других middleware.
        


        
Ключевые моменты

        
    
          
  • Middleware может модифицировать action перед next()
    • 
          
  • Используйте spread: next({ ...action, newField })
    • 
          
  • Добавление timestamp, meta, correlationId — частые паттерны
    • 
          
  • Не мутируйте оригинальный action — создавайте копию
    • 
          
  • Reducer получит уже обогащённый action
