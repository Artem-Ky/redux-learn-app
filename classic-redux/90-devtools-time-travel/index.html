Теория — Time-Travel отладка

      

        
Что такое Time-Travel?

        

          Time-Travel (путешествие во времени) — это возможность «перемотать»
          приложение к любому предыдущему состоянию. Каждый dispatch создаёт снимок (snapshot)
          состояния. Можно перейти к любому из них и увидеть, как выглядело приложение
          в тот момент.
        


        

// Каждый dispatch создаёт снимок:
// [@@INIT] → state₀
// [todos/add] → state₁
// [todos/add] → state₂
// [todos/toggle] → state₃
//
// Слайдер: ◀──────●── → переход к state₁
        


        
Replay (воспроизведение) экшенов

        

          Time-Travel работает через replay: чтобы получить состояние N,
          DevTools «проигрывает» все экшены от @@INIT до N через reducer.
          Так как reducer — чистая функция, результат всегда одинаковый.
        


        

// Replay = прогон экшенов через reducer
let state = initialState
for (const action of actions.slice(0, targetIndex)) {
  state = reducer(state, action)
}
// state = точное состояние на момент targetIndex
        


        

          «Redux makes it possible to implement time-travel debugging, because
          reducers are pure functions with no side effects. Given the same actions
          and initial state, you always get the same result.»
          
— Redux Fundamentals, Part 4: Store
        


        
Почему иммутабельность важна

        

          Time-Travel возможен только благодаря иммутабельности.
          Если reducer мутирует state, снимки будут указывать на один и тот же объект,
          и все они покажут последнее состояние. Иммутабельные обновления создают
          новый объект при каждом изменении — все снимки сохраняются.
        


        
Почему чистые reducers важны

        

          Replay работает потому, что reducer — чистая функция:
          одинаковые входные данные (state + action) всегда дают одинаковый результат.
          Если reducer содержит side effects (запись в localStorage, HTTP-запросы),
          replay будет вызывать эти эффекты повторно и давать непредсказуемый результат.
        


        

          Time-Travel — одна из «киллер-фич» Redux. Это не просто удобство —
          это мощнейший инструмент отладки. Вы можете найти баг, перемотать к моменту
          его возникновения, увидеть какой экшен привёл к ошибке и какие данные
          были в payload.
        


        

          Time-Travel «ломается», если: reducer мутирует state, reducer имеет side effects,
          в reducer используется Date.now() или Math.random().
          Всегда создавайте новый объект вместо мутации и генерируйте случайные значения
          в action creator, а не в reducer.
